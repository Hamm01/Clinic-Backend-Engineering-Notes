# 08.6 - JSON Web Token (JWT)

> After a user successfully logs in, the backend needs a way to remember that user on future requests without asking for the email and password every time. JSON Web Tokens (JWT) solve this problem by providing a secure, stateless authentication mechanism.

# The Problem

Imagine a doctor logs into our application.

```
POST /auth/login

↓

Email

↓

Password

↓

Login Successful
```

Now the doctor wants to view patients.

```
GET /patients
```

How does the backend know

this request belongs to the same doctor?

Without JWT,

the doctor would have to send

```
Email

Password
```

with every request.

This would be insecure and impractical.

---

# Traditional Session Authentication

Before JWT became popular,

applications stored sessions.

```
Login

↓

Server

↓

Create Session

↓

Store Session

↓

Return Session ID

↓

Browser Cookie
```

Every future request required

```
Session ID

↓

Server

↓

Database Lookup

↓

User Found
```

This works,

but introduces problems in distributed systems.

---

# Why Sessions Become Difficult

Imagine

```
Load Balancer

↓

Server A

↓

Server B

↓

Server C
```

If the user's session exists only on

```
Server A
```

and the next request reaches

```
Server C
```

authentication fails.

Now sessions must be shared.

Usually through

- Redis
- Database
- Sticky Sessions

This increases complexity.

---

# Stateless Authentication

JWT removes this dependency.

Instead of storing authentication information on the server,

the information travels with every request.

```
Login

↓

JWT Generated

↓

Client Stores JWT

↓

Future Requests

↓

Authorization Header

↓

Backend Verifies JWT
```

No session lookup.

No Redis.

No database query to identify the user.

---

# What is a JWT?

JWT stands for

```
JSON Web Token
```

It is a digitally signed string containing user information.

Example

```
eyJhbGciOiJIUzI1NiIsInR5...
```

The token itself proves the user's identity.

---

# JWT Structure

A JWT contains three parts.

```
Header

↓

Payload

↓

Signature
```

Example

```
xxxxx.yyyyy.zzzzz
```

---

## Header

Contains metadata.

```json
{
  "alg": "HS256",
  "typ": "JWT"
}
```

---

## Payload

Contains claims.

```json
{
  "userId": "uuid",
  "role": "DOCTOR"
}
```

Never store passwords inside JWTs.

---

## Signature

Created using

```
Secret Key

+

Header

+

Payload
```

If anyone modifies the payload,

the signature becomes invalid.

# Create JWT Utility

Create

```
src/utils/jwt.ts
```

```ts
import jwt from 'jsonwebtoken'
import { jwtConfig } from '../config/jwt'

interface JwtPayload {
  id: string

  email: string

  role: string
}

export function generateAccessToken(payload: JwtPayload) {
  return jwt.sign(
    payload,

    jwtConfig.secret,

    {
      expiresIn: jwtConfig.expiresIn
    }
  )
}

export function verifyAccessToken(token: string) {
  return jwt.verify(token, jwtConfig.secret)
}
```

---

# Why a Utility?

JWT is used by

- Login
- Middleware
- Refresh Token
- Logout
- Current User

Instead of duplicating logic,

we create a reusable utility.

---

# Modify Auth Service

Replace

```ts
return {
  user
}
```

with

```ts
const accessToken = generateAccessToken({
  id: user.id,

  email: user.email,

  role: user.role
})

return {
  accessToken,

  user
}
```

---

# Why Generate JWT in Service?

Authentication is business logic.

The Service decides

whether a user is authenticated.

Repositories should never generate JWTs.

Controllers should never generate JWTs.

# Where Will JWT Be Used?

Every protected request.

```
GET /patients

Authorization

Bearer eyJhbGc...
```

The backend verifies the token before executing business logic.

---

# Production Notes

Always

- Store JWT secret in environment variables.
- Use HTTPS in production.
- Keep Access Tokens short-lived.
- Never store passwords inside JWT payloads.
- Keep payloads small.
- Generate JWTs only after successful authentication.

# Summary

JWT enables stateless authentication by allowing the client to carry its own authentication information.

Instead of maintaining server-side sessions, our backend signs a token during login and verifies that token on every protected request.

The authentication flow now has everything required to identify a user securely.

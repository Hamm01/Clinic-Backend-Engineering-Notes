# 08.2 - User Account Model

> Before implementing authentication, we must design how user accounts are stored. Authentication belongs to users, not to business entities like Doctors or Patients.

---

# Objective

After completing this chapter, you should understand:

- Why we need a separate User model.
- How User relates to Doctors and other staff.
- Why Patients are excluded.
- What fields belong in User.
- How authentication integrates with the existing database.

---

# The Problem

Currently our database contains

```
Clinic

Doctor

Patient

Appointment

MedicalRecord

Diagnosis

Prescription
```

Everything required to run the clinic exists.

But one important question remains.

> **Who logs into the application?**

A Doctor?

A Patient?

A Clinic?

An Appointment?

None of them.

Authentication requires an account.

---

# Business Entity vs Authentication Entity

This distinction is extremely important.

A Doctor is a business entity.

```
Doctor

↓

Specialization

License Number

Experience

Clinic
```

A User Account is an authentication entity.

```
User

↓

Email

Password

Role

Account Status
```

These represent different responsibilities.

---

# Why Not Store Password in Doctor?

Many beginners design

```prisma
model Doctor {

    id String @id @default(uuid())

    firstName String

    email String

    password String
}
```

Although this works,

it mixes two responsibilities.

Doctor stores

- Medical information
- Employment information
- Authentication information

This violates separation of concerns.

---

# Better Design

Separate them.

```
Doctor

↓

Medical Information
```

```
User

↓

Authentication Information
```

Each table has a single responsibility.

---

# Relationship

Our User Account belongs to one staff member.

```
Doctor

▲

│

User
```

One Doctor

↓

One User Account

This is a One-to-One relationship.

---

# User Model

```prisma
model User {

    id String @id @default(uuid())

    email String @unique

    password String

    role UserRole

    isActive Boolean @default(true)

    doctorId String?

    doctor Doctor?
        @relation(fields: [doctorId], references: [id])

    createdAt DateTime @default(now())

    updatedAt DateTime @updatedAt
}
```

Notice that this model contains only authentication-related fields.

---

# Why Email is Unique

```prisma
email String @unique
```

Every account must have a unique login.

Without this,

two users could share the same email.

Authentication would become impossible.

---

# Why Password Exists Here

```
Doctor

↓

No Password
```

```
User

↓

Password
```

The authentication system owns passwords.

Business entities do not.

---

# Why Role Exists Here

Authentication identifies the user.

Authorization checks permissions.

The role is part of authentication because it identifies the type of account.

Example

```
ADMIN

DOCTOR

RECEPTIONIST
```

Later,

authorization will use this role to decide which APIs can be accessed.

---

# Account Status

```prisma
isActive Boolean
```

Suppose a doctor resigns.

Instead of deleting the account,

the administrator simply disables it.

```
Doctor

↓

Account Disabled

↓

Cannot Login
```

Historical data remains intact.

---

# User Role Enum

```prisma
enum UserRole {

    ADMIN

    DOCTOR

    RECEPTIONIST
}
```

Using an enum prevents invalid roles.

Instead of

```
Doctor

doctor

DOCTOR

Doc
```

the database only accepts predefined values.

---

```

```

↓

Find User

Compare Password

↓

```

Notice

The Doctor table is not used for authentication.

---

# Why Patient Has No User Account

Our first version focuses on internal staff.

```
Administrator

Doctor

Receptionist
```

Patients do not log into the system.

If patient login is added in the future,

another User record can be linked to a Patient.

The authentication architecture does not change.

---

# Repository Example

Authentication searches the User table.

const user = await prisma.user.findUnique({

  }

```

Notice

Authentication never searches

```
Doctor

Patient

Appointment
```

It searches User.

# Common Mistakes

## Storing Passwords in Business Tables

Wrong

```
Doctor

password
```

Correct

```
User

password
```

---

## Using Role as Free Text

Wrong

```
role = "Doctor"

role = "doctor"

role = "Doc"
```

Correct

```prisma
enum UserRole
```

---

## Deleting Accounts

Instead of deleting,

disable them.

```
isActive = false
```

Historical data remains consistent.

---

## Mixing Business Logic with Authentication

Doctor information belongs in the Doctor table.

Authentication information belongs in the User table.

Keep them independent.

---

# Senior Engineer Notes

When designing authentication,

always ask:

- Who logs into the system?
- Is authentication separate from business data?
- Can new account types be added easily?
- Can an account be disabled without deleting history?
- Does each table have one responsibility?

A well-designed User model makes every authentication feature easier to implement.

---

# Summary

The User model is the foundation of the authentication system.

It stores only authentication-related information such as email, password, role, and account status, while business information remains in the Doctor table.

Separating these responsibilities creates a cleaner architecture, improves maintainability, and makes the authentication system easier to extend.

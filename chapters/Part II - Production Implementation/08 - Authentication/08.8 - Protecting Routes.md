# 08.8 - Protecting Routes and Role-Based Access Control (RBAC)

> Authentication identifies the user. Authorization determines what that authenticated user is allowed to do. This chapter introduces Role-Based Access Control (RBAC) to protect routes based on user roles.

---

# Objective

After completing this chapter, you should understand:

- Authentication vs Authorization.
- Why RBAC is important.
- How to build reusable authorization middleware.
- How roles are checked.
- How protected routes are implemented.
- Why Controllers should never perform authorization.

# Authentication vs Authorization

Authentication asks

```
Who are you?
```

Authorization asks

```
What are you allowed to do?
```

Example

```
Doctor

↓

Authenticated

↓

Can Create Diagnosis

↓

Cannot Delete Clinic
```

Authentication happens first.

Authorization happens second.

---

# Why RBAC?

Imagine every authenticated user can call

```
DELETE /doctors/:id
```

Should a receptionist be allowed?

No.

Should another doctor?

Probably not.

Should an administrator?

Yes.

This is why authorization exists.

# Step 1 — Create Authorization Middleware

Create

```
src/middlewares/authorization.middleware.ts
```

```ts
import { NextFunction, Request, Response } from 'express'
import { ForbiddenException } from '../errors/ForbiddenException'

export function authorize(...roles: string[]) {
  return (req: Request, res: Response, next: NextFunction) => {
    if (!roles.includes(req.user.role)) {
      throw new ForbiddenException(
        'You do not have permission to perform this action.'
      )
    }

    next()
  }
}
```

---

# Why a Higher-Order Function?

Notice

```ts
authorize('ADMIN')
```

returns middleware.

This allows every route to define its own permissions.

Examples

```ts
authorize('ADMIN')
```

```ts
authorize('DOCTOR')
```

```ts
authorize('ADMIN', 'DOCTOR')
```

One middleware.

Many use cases.

---

# Step 2 — Protect Routes

Before

```ts
router.post('/', createPatient)
```

After

```ts
import { authenticate } from '../../middlewares/authentication.middleware'
import { authorize } from '../../middlewares/authorization.middleware'

router.post('/', authenticate, authorize('DOCTOR', 'ADMIN'), createPatient)
```

---

# Execution Order

Express executes middleware in the order they are registered.

```
authenticate

↓

authorize

↓

controller
```

If authentication fails,

authorization never executes.

If authorization fails,

the controller never executes.

---

# Another Example

Delete Doctor

```ts
router.delete('/:id', authenticate, authorize('ADMIN'), deleteDoctor)
```

Only administrators can delete doctors.

# Receptionist Example

Appointments

```ts
router.post(
  '/appointments',
  authenticate,
  authorize('ADMIN', 'RECEPTIONIST'),
  createAppointment
)
```

Receptionists can create appointments.

Doctors cannot.

# Why Not Check Roles in Controllers?

Wrong

```ts
if (req.user.role !== "ADMIN") {
    ...
}
```

Every controller would repeat this logic.

Correct

```
Authentication Middleware

↓

Authorization Middleware

↓

Controller
```

Controllers should only contain business logic.

# Production Notes

Always

- Authenticate first.
- Authorize second.
- Keep permission checks inside middleware.
- Return HTTP 403 for insufficient permissions.
- Keep Controllers free from authorization logic.

---

# Common Mistakes

## Authorization Before Authentication

Wrong

```
authorize()

↓

authenticate()
```

Correct

```
authenticate()

↓

authorize()
```

---

## Hardcoding Roles Everywhere

Wrong

```ts
if(role==="ADMIN")
```

inside every controller.

Correct

Reusable middleware.

---

## Returning 401 Instead of 403

401

Means

```
Not authenticated
```

403

Means

```
Authenticated

↓

Permission denied
```

Understanding this difference is important for REST API design.

# Senior Engineer Notes

As applications grow,

authorization becomes more complex.

Instead of only roles,

many systems eventually introduce

- Permissions
- Policies
- Resource ownership
- Attribute-Based Access Control (ABAC)

Starting with RBAC provides a clean, maintainable foundation that can evolve without changing the authentication flow.

# Summary

Authentication verifies a user's identity, while Authorization determines what that authenticated user is allowed to do.

By introducing reusable authorization middleware, we centralize permission checks, keep Controllers clean, and enforce consistent access rules across the application.

This approach scales naturally as the application grows and new roles or permissions are introduced.

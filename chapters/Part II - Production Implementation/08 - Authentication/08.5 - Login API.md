# 08.5 - Login API

> The Login API authenticates an existing user by verifying their credentials. If the email and password are valid, the backend generates a JWT that will be used to authenticate future requests.

---

# Objective

After completing this chapter, you should understand:

- How the Login API works.
- How user credentials are verified.
- Why passwords are never compared directly.
- How authentication integrates with the Service layer.
- How custom exceptions improve production applications.
- How the Login API prepares the application for JWT authentication.

---

# Business Scenario

A doctor wants to access the Clinic Management System.

The doctor enters:

```
Email

doctor@clinic.com

Password

Doctor@123
```

The backend must answer one question.

> Is this user really who they claim to be?

If yes,

the backend returns an Access Token.

# Why a Separate DTO?

Registration requires

- role
- doctorId

Login requires only

- email
- password

Each request should have its own DTO.

Never reuse unrelated DTOs.

# Step 5 — Service

```
src/modules/auth/auth.service.ts
```

```ts
import { comparePassword } from "../../utils/password";
import { UnauthorizedException } from "../../errors/UnauthorizedException";
import { LoginDto } from "./auth.types";

async login(dto: LoginDto) {

    const user =
        await this.repository.findByEmail(dto.email);

    if (!user) {
        throw new UnauthorizedException(
            "Invalid email or password."
        );
    }

    const isPasswordValid =
        await comparePassword(
            dto.password,
            user.password
        );

    if (!isPasswordValid) {
        throw new UnauthorizedException(
            "Invalid email or password."
        );
    }

    /**
     * JWT generation
     * (implemented in the next chapter)
     */

    return {
        user,
    };
}
```

---

# Why Return The Same Error?

Notice

```ts
Invalid email or password.
```

is returned for both

- incorrect email
- incorrect password

This is intentional.

Imagine returning

```
Email not found.
```

Now an attacker knows

the email does not exist.

Returning the same message prevents attackers from discovering registered accounts.

---

# Why Compare Passwords?

Wrong

```ts
dto.password === user.password
```

Correct

```ts
await comparePassword(...)
```

The database stores

```
$2b$10$...
```

not

```
Doctor@123
```

bcrypt performs the secure comparison.

# Why Use Custom Exceptions?

Instead of

```ts
throw new Error(...)
```

we now use

```ts
throw new UnauthorizedException(...)
```

Advantages

- Correct HTTP Status Code
- Global Error Handler
- Consistent API Responses
- Easier Logging
- Easier Debugging

This follows the Error Handling architecture implemented earlier.

# Common Mistakes

## Returning Different Errors

Wrong

```
Email not found.

Password incorrect.
```

Correct

```
Invalid email or password.
```

---

## Comparing Passwords Manually

Always use

```ts
comparePassword()
```

Never compare strings directly.

---

## Business Logic In Controllers

Controllers should never

- compare passwords
- generate JWTs
- access Prisma

Those responsibilities belong to the Service.

---

# Senior Engineer Notes

A production Login API should

- Validate requests.
- Use reusable DTOs.
- Search users by email.
- Compare passwords securely.
- Return generic authentication errors.
- Generate JWTs.
- Integrate with centralized error handling.

Every authentication request should follow the same layered architecture used throughout the backend.

---

# Summary

The Login API authenticates users by verifying their credentials against the stored password hash.

The Service coordinates the authentication process, while Controllers remain thin and Repositories focus only on database access.

The only remaining step is generating a secure JWT after successful authentication.

# 08.1 - Authentication Overview

> Authentication is the process of verifying **who is making a request** before allowing access to protected resources.

- What we will build throughout this section.

---

# What is Authentication?

Authentication is the process of verifying the identity of a user.

When someone claims,

> "I am Dr. Amit."

the backend must verify whether that statement is true.

Only after successful verification does the backend trust the request.

---

# Why Authentication Exists

Imagine the following APIs are publicly available.

```
GET /patients

POST /appointments

PUT /medical-records/:id

DELETE /patients/:id
```

If authentication did not exist,

any person could:

- Read patient information.
- Delete appointments.
- Modify prescriptions.
- Access confidential medical records.

The application would have no way of knowing who made the request.

Authentication prevents this.

---

# Real World Example

Imagine a hospital.

A doctor arrives at work.

Before entering the operation theatre,

the hospital verifies

- Doctor Identity Card
- Fingerprint
- Face Scan

Only after verification is the doctor allowed to enter.

Authentication in software works exactly the same way.

```
Email

↓

Password

↓

Identity Verified

↓

Access Granted
```

# Authentication vs Authorization

These two concepts solve different problems.

## Authentication

Answers

> Who are you?

Example

```
Email

↓

Password

↓

Verified
```

---

## Authorization

Answers

> What are you allowed to do?

Example

```
Doctor

↓

Can Create Diagnosis

↓

Can Update Prescription

↓

Cannot Delete Clinic
```

Authentication always happens before authorization.

---

# Example Flow

```
Doctor Logs In

↓

Authentication Successful

↓

Doctor Requests

DELETE /patients

↓

Authorization Checks Permission

↓

Allowed?

↓

Yes / No
```

Authentication identifies the user.

Authorization checks permissions.

# What Information Verifies a User?

A user logs in using

```
Email

+

Password
```

The backend verifies

- User exists.
- Account is active.
- Password matches.

If everything is correct,

authentication succeeds.

---

# What Happens After Login?

After successful authentication,

the backend generates a JSON Web Token (JWT).

```
User

↓

Login Successful

↓

JWT Generated

↓

JWT Returned

↓

Client Stores JWT
```

Future requests include this token.

---

# Future Request Flow

```
Client

↓

Authorization Header

Bearer <JWT>

↓

Backend

↓

Verify JWT

↓

Identify User

↓

Execute Business Logic
```

The user does not log in again for every request.

The JWT proves their identity.

---

# Where Authentication Fits

Authentication occurs before business logic.

```
HTTP Request

↓

Route

↓

Authentication Middleware

↓

Authorization Middleware

↓

Controller

↓

Service

↓

Repository

↓

Database

↓

HTTP Response
```

If authentication fails,

the request stops immediately.

Controllers and Services are never executed.

# Why Separate Authentication Into Components?

Every component has one responsibility.

| Component        | Responsibility                         |
| ---------------- | -------------------------------------- |
| User Model       | Stores account information             |
| Password Hashing | Securely stores passwords              |
| Register API     | Creates new accounts                   |
| Login API        | Verifies credentials                   |
| JWT              | Represents authenticated identity      |
| Middleware       | Verifies every incoming request        |
| Protected Routes | Restrict access to authenticated users |
| Refresh Tokens   | Extend user sessions                   |
| Logout           | End authenticated sessions             |

Keeping responsibilities separate makes the application easier to understand, maintain, and test.

# Business Rules

Our authentication system follows these rules.

- Email must be unique.
- Passwords are never stored in plain text.
- Passwords are hashed before saving.
- Login returns a JWT.
- Every protected request requires a valid JWT.
- Expired or invalid tokens are rejected.
- Authenticated users are identified before executing business logic.

These rules remain consistent across the entire application.

# Key Takeaways

- Authentication verifies the identity of the user.
- Authentication is different from authorization.
- Authentication always happens before business logic.
- Passwords are never stored directly.
- JWTs allow users to remain authenticated across requests.
- Every protected API depends on successful authentication.

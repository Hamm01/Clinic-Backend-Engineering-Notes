# 08.9 - Current User Endpoint (`GET /auth/me`)

> Once a user has successfully authenticated, the frontend often needs to know who the current user is. Instead of asking the client to send its user ID, the backend identifies the user from the verified JWT and returns the authenticated user's information.

---

# Objective

After completing this chapter, you should understand:

- Why `/auth/me` exists.
- Why clients should never send their own user ID.
- How authenticated user information flows through the application.
- How Authentication Middleware and JWT work together.
- How to build the `/auth/me` endpoint using our layered architecture.

# The Problem

Imagine the frontend has already logged in.

```
POST /auth/login

↓

Access Token
```

The frontend stores

```
JWT
```

Now the application loads.

The navigation bar needs

- Doctor Name
- Email
- Role
- Profile

How does it obtain this information?

---

# Wrong Approach

Many beginners implement

```
GET /users/:id
```

and allow the frontend to send

```
User ID
```

Example

```
GET /users/123
```

This is dangerous.

Anyone can modify

```
123

↓

456

↓

789
```

and attempt to access another user's data.

Never trust identifiers sent by the client when the authenticated user is already known.

---

# Correct Approach

The client sends

```
Authorization

Bearer eyJhbGc...
```

The backend already knows

who the authenticated user is.

```
JWT

↓

Authentication Middleware

↓

req.user.id

↓

Database

↓

Response
```

No user ID is sent by the client.

---

# Why Query the Database Again?

You might ask,

> The JWT already contains the user's information. Why query the database?

Because the JWT only contains minimal identity information.

Example

```
JWT

↓

id

email

role
```

The frontend often requires additional information such as

- Doctor Profile
- Specialization
- Phone Number
- Clinic

That information lives in PostgreSQL.

---

# Step 2 — Service

Modify

```
auth.service.ts
```

```ts
import { NotFoundException } from "../../errors/NotFoundException";

async getCurrentUser(userId: string) {

    const user = await this.repository.findById(userId);

    if (!user) {

        throw new NotFoundException(
            "Authenticated user not found."
        );
    }

    return user;
}
```

---

# Why Service?

The Service owns business logic.

Business rule

```
Authenticated user must exist.
```

If not,

the request fails.

The Repository should never throw business-related exceptions.

---

# Step 3 — Controller

Modify

```
auth.controller.ts
```

```ts
export async function me(
  req: Request,

  res: Response
) {
  const user = await service.getCurrentUser(req.user.id)

  return res.status(200).json({
    message: 'Authenticated user retrieved successfully.',

    data: user
  })
}
```

Notice

We never read

```
req.params.id

req.body.id

req.query.id
```

Only

```
req.user.id
```

which came from the verified JWT.

# Why Not Trust Client IDs?

Imagine

```
GET /users/5
```

A malicious user changes it to

```
GET /users/10
```

Without proper authorization,

private information is exposed.

Using

```
req.user.id
```

eliminates this risk.

---

# Production Notes

Always

- Read authenticated user IDs from the verified JWT.
- Keep JWT payloads small.
- Load additional profile information from the database.
- Never trust client-provided identifiers for authenticated operations.

---

# Common Mistakes

## Reading User ID From URL

Wrong

```ts
req.params.id
```

Correct

```ts
req.user.id
```

---

## Returning Password

Never expose

```ts
password
```

or

```ts
hashedPassword
```

in API responses.

Always sanitize user objects before returning them.

---

## Skipping Authentication Middleware

Never expose

```
GET /auth/me
```

without

```
authenticate
```

The endpoint must always require a valid JWT.

---

# Senior Engineer Notes

The `/auth/me` endpoint is one of the most frequently used APIs in modern applications.

Instead of repeatedly asking the frontend to send identifiers, the backend trusts only the authenticated identity established by the JWT.

This design improves security, simplifies frontend development, and keeps authentication concerns centralized.

# Summary

The `GET /auth/me` endpoint allows the frontend to retrieve the currently authenticated user's profile without sending a user ID.

By relying on `req.user`, which is populated by the Authentication Middleware after verifying the JWT, the backend ensures that only the authenticated user's information is returned. This pattern is used in almost every production application and significantly reduces the risk of exposing unauthorized user data.

# 08.4 - Register API

> The Register API creates a new user account in the system. Before storing the account, the backend validates the request, checks for duplicate users, hashes the password, and saves the account securely.

---

# Objective

After completing this chapter, you should understand:

- Why registration is the first authentication API.
- How the request flows through the backend.
- How Controllers, Services, and Repositories work together.
- Where password hashing is performed.
- How duplicate users are prevented.
- How a new User is created using Prisma.

---

# Step 1 — Define the Request Body

Before writing any code,

we must decide what information is required to create a user.

```
POST /auth/register
```

Request

```json
{
  "email": "doctor@clinic.com",
  "password": "Doctor@123",
  "role": "DOCTOR",
  "doctorId": "uuid"
}
```

Notice

The client never sends

- id
- createdAt
- updatedAt

These values are generated by the backend.

---

# Step 2 — Create the Request Type

Create

```
src/modules/auth/auth.types.ts
```

```ts
export interface RegisterDto {
  email: string
  password: string
  role: 'ADMIN' | 'DOCTOR' | 'RECEPTIONIST'
  doctorId?: string
}
```

---

## Why Create a DTO?

Instead of passing

```ts
req.body
```

through the application,

we create a strongly typed object.

Benefits

- Better IntelliSense.
- Compile-time safety.
- Easier validation.
- Easier testing.

---

# Step 3 — Validation

Create

```
src/modules/auth/auth.validation.ts
```

```ts
import { z } from 'zod'

export const registerSchema = z.object({
  email: z.email(),

  password: z.string().min(8, 'Password must contain at least 8 characters'),

  role: z.enum(['ADMIN', 'DOCTOR', 'RECEPTIONIST']),

  doctorId: z.string().uuid().optional()
})
```

---

## Why Validate Here?

Validation happens before business logic.

If the request is invalid,

the Service should never execute.

Example

Wrong

```
Password

123
```

The request should fail immediately.

# Step 4 — Repository

Create

```
src/modules/auth/auth.repository.ts
```

```ts
import { prisma } from '../../database/prisma'

export class AuthRepository {
  async findByEmail(email: string) {
    return prisma.user.findUnique({
      where: {
        email
      }
    })
  }

  async create(data: {
    email: string
    password: string
    role: 'ADMIN' | 'DOCTOR' | 'RECEPTIONIST'
    doctorId?: string
  }) {
    return prisma.user.create({
      data
    })
  }
}
```

---

## Why Repository?

Repositories are responsible only for communicating with the database.

They should never

- hash passwords
- generate JWTs
- validate business rules

Those responsibilities belong elsewhere.

---

# Step 5 — Service

Create

```
src/modules/auth/auth.service.ts
```

```ts
import { hashPassword } from '../../utils/password'
import { AuthRepository } from './auth.repository'
import { RegisterDto } from './auth.types'

export class AuthService {
  constructor(private readonly repository = new AuthRepository()) {}

  async register(dto: RegisterDto) {
    const existingUser = await this.repository.findByEmail(dto.email)

    if (existingUser) {
      throw new Error('Email already registered.')
    }

    const hashedPassword = await hashPassword(dto.password)

    return this.repository.create({
      email: dto.email,

      password: hashedPassword,

      role: dto.role,

      doctorId: dto.doctorId
    })
  }
}
```

---

## Why Does Password Hashing Happen Here?

The Service owns business logic.

Business rule

```
Never save plain text passwords.
```

Therefore,

the Service hashes the password before calling the Repository.

---

# Step 6 — Controller

Create

```
src/modules/auth/auth.controller.ts
```

```ts
import { Request, Response } from 'express'
import { AuthService } from './auth.service'

const service = new AuthService()

export async function register(req: Request, res: Response) {
  const user = await service.register(req.body)

  res.status(201).json({
    message: 'User registered successfully.',

    data: user
  })
}
```

---

## Why Is the Controller So Small?

Controllers should not contain business logic.

The Controller only

- receives the request
- calls the Service
- returns the response

Keeping Controllers thin makes them easier to maintain.

# Common Mistakes

## Saving Password Directly

Wrong

```ts
password: dto.password
```

Correct

```ts
password: hashedPassword
```

---

## Querying Database in Controller

Wrong

```
Controller

↓

Prisma
```

Correct

```
Controller

↓

Service

↓

Repository

↓

Database
```

---

## Business Logic Inside Repository

Repositories should only execute database operations.

Business rules belong in Services.

---

# Senior Engineer Notes

A production Register API should always:

- Validate incoming requests.
- Prevent duplicate accounts.
- Hash passwords.
- Keep Controllers thin.
- Keep Repositories database-focused.
- Return meaningful responses.
- Never expose password hashes.

---

# Summary

The Register API is the first real authentication endpoint.

It demonstrates how every backend layer works together to create a secure user account while keeping responsibilities separated across Controllers, Services, Repositories, and Utilities.

This architecture will be reused throughout the entire backend.

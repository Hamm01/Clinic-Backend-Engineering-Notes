# 08.10 - Refresh Tokens

> Access Tokens are intentionally short-lived for security reasons. Refresh Tokens allow users to remain logged in without repeatedly entering their credentials while maintaining a secure authentication system.

# Why Not Keep One JWT Forever?

Imagine this JWT never expires.

```
eyJhbGc...
```

If someone steals it,

they can use your account forever.

There is no way to stop them.

This is a huge security risk.

---

# Why Does the Access Token Expire?

Instead of

```
Forever
```

we use

```
15 Minutes
```

If the token leaks,

the attacker only has

15 minutes.

This dramatically reduces the attack window.

---

# New Problem

After

```
15 Minutes
```

the real user is also logged out.

Imagine Gmail asking you to login

every

15 minutes.

Nobody would use it.

---

# Refresh Tokens Solve This

Login

↓

Generate

```
Access Token

15 Minutes
```

AND

```
Refresh Token

30 Days
```

The client stores both.

When

```
Access Token

↓

Expires
```

the client silently requests

```
POST /auth/refresh
```

using the Refresh Token.

The backend returns

```
New Access Token
```

The user never notices.

---

# Access Token vs Refresh Token

| Access Token                         | Refresh Token                                                        |
| ------------------------------------ | -------------------------------------------------------------------- |
| Short-lived                          | Long-lived                                                           |
| Used on every API request            | Used only when Access Token expires                                  |
| Stored in memory (or secure storage) | Stored securely (HttpOnly cookie for web, secure storage for mobile) |
| Cannot be revoked easily             | Stored in database and can be revoked                                |
| Contains user identity               | Used to obtain a new Access Token                                    |

# Why Store Refresh Tokens In Database?

Access Tokens are stateless.

Refresh Tokens are stateful.

Why?

Because we need the ability to

- Logout
- Revoke tokens
- Detect stolen tokens
- Support multiple devices
- Force logout

Without database storage,

none of this is possible.

# Why Token Rotation?

Imagine

Someone steals your Refresh Token.

If you keep reusing it,

both users can continuously generate

new Access Tokens.

Instead

```
Old Refresh Token

↓

Delete

↓

Generate New Refresh Token
```

The stolen token immediately becomes useless.

```

```

↓

↓

Delete From Database

↓

Success

```

the Refresh Token no longer exists.

Even if someone still has it,

it cannot be used.
```

```
# Why HttpOnly Cookies?

Never store Refresh Tokens in

```

localStorage

```

because JavaScript can read them.

Instead

```

HttpOnly Cookie

```

JavaScript cannot access it.

Even if XSS occurs,

the Refresh Token remains protected.

---

# Production Notes

Always

- Rotate Refresh Tokens.
- Store Refresh Tokens in database.
- Use separate JWT secrets.
- Use HttpOnly Cookies.
- Support multiple devices.
- Allow logout from all devices.
- Expire old Refresh Tokens automatically.
```

# Summary

Refresh Tokens complement Access Tokens by providing a secure way to maintain user sessions without requiring frequent logins.

Unlike Access Tokens, Refresh Tokens are stored and managed by the server, allowing features such as logout, session revocation, token rotation, and multi-device authentication.

Together, Access Tokens and Refresh Tokens provide a balance between security and usability, making them the standard authentication approach for modern backend applications.

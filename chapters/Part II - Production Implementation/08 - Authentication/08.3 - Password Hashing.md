# 08.3 - Password Hashing

> Password hashing is the process of transforming a user's password into an irreversible cryptographic value before storing it in the database. Production applications never store passwords in plain text.

---

# Objective

After completing this chapter, you should understand:

- Why passwords should never be stored directly.
- What password hashing is.
- How hashing differs from encryption.
- Why bcrypt is used.
- How passwords are hashed during registration.
- How passwords are verified during login.
- How password hashing fits into our backend architecture.
- How to implement password hashing in Express with TypeScript.

---

# The Problem

Let's assume a user registers in our application.

```
Email

doctor@clinic.com

Password

Doctor@123
```

If we save the password directly into the database,

```
User

----------------------------

Email

doctor@clinic.com

Password

Doctor@123
```

everything appears to work.

The user can log in successfully.

However, this introduces one of the biggest security risks in backend development.

---

# Why Plain Text Passwords Are Dangerous

Imagine someone gains access to the database.

```
Database

↓

User Table

↓

Passwords
```

If passwords are stored directly,

the attacker immediately knows every user's password.

```
doctor@clinic.com

↓

Doctor@123
```

Now imagine hundreds of users.

```
doctor1@gmail.com

↓

Password123

doctor2@gmail.com

↓

Welcome@123

doctor3@gmail.com

↓

Clinic2025
```

Every account becomes compromised.

---

# Real World Example

Imagine a hospital stores patient locker keys.

Wrong approach

```
Locker Number

↓

Master Key Hanging Beside It
```

Anyone entering the room can open every locker.

Instead,

the hospital stores only the lock information.

Only the correct key can open it.

Passwords work the same way.

The backend should never know the original password after storing it.

---

# What is Password Hashing?

Password hashing converts a password into a fixed-length value.

```
Doctor@123

↓

Hash Function

↓

$2b$10$Cn3vJj...
```

The resulting value is called a **hash**.

The hash is stored in the database instead of the password.

---

# Important Property

Hashing is **one-way**.

```
Password

↓

Hash
```

Possible.

```
Hash

↓

Original Password
```

Not possible.

This is what makes hashing secure.

---

# Hashing vs Encryption

Many beginners confuse these two concepts.

| Hashing                          | Encryption                     |
| -------------------------------- | ------------------------------ |
| One-way                          | Two-way                        |
| Cannot recover original password | Original data can be decrypted |
| Used for passwords               | Used for confidential data     |
| Verification                     | Data protection                |

Passwords should always be hashed, not encrypted.

---

# Why Not Build Our Own Hash Function?

You might think,

```
Password

↓

SHA256

↓

Store Result
```

Although SHA256 is a secure cryptographic algorithm,

it is **too fast** for password storage.

Attackers can generate billions of SHA256 hashes every second.

Production applications use algorithms specifically designed for passwords.

---

# Why bcrypt?

bcrypt is one of the most widely used password hashing algorithms.

It provides:

- Salt generation.
- Configurable computational cost.
- Protection against brute-force attacks.
- Proven security.
- Wide industry adoption.

For our clinic backend, bcrypt is an excellent choice.

# Where Should Password Hashing Live?

Password hashing is not business logic.

It should not exist inside

```
Controller
```

or

```
Repository
```

Instead,

create a shared utility.

```
src

├── utils
│
└── password.ts
```

This utility can be reused by

- Register API
- Login API
- Password Reset
- Change Password

---

# Creating the Password Utility

```ts
// src/utils/password.ts

import bcrypt from 'bcrypt'

const SALT_ROUNDS = 10

export async function hashPassword(password: string): Promise<string> {
  return bcrypt.hash(password, SALT_ROUNDS)
}

export async function comparePassword(
  password: string,
  hashedPassword: string
): Promise<boolean> {
  return bcrypt.compare(password, hashedPassword)
}
```

---

# Why Use a Utility?

Notice that password hashing is independent of Express.

It does not know about

- Controllers
- Services
- Routes
- Prisma

It performs only one responsibility.

```
Password

↓

Hash

or

Compare
```

This follows the Single Responsibility Principle.

---

# Understanding Salt Rounds

```ts
const SALT_ROUNDS = 10
```

bcrypt automatically generates a random salt.

```
Password

↓

Random Salt

↓

Hash
```

The salt makes every hash unique.

Even if two users choose the same password,

their hashes will be different.

Example

User A

```
Doctor@123

↓

$2b$10$A7h...
```

User B

```
Doctor@123

↓

$2b$10$L92...
```

Same password.

Different hashes.

---

# Registration Flow

When a new doctor registers,

the password is hashed before saving.

```
Client

↓

POST /auth/register

↓

Controller

↓

Service

↓

hashPassword()

↓

Repository

↓

Database
```

The database never receives the original password.

---

# Registration Example

```ts
const hashedPassword = await hashPassword(dto.password)

await prisma.user.create({
  data: {
    email: dto.email,
    password: hashedPassword,
    role: 'DOCTOR'
  }
})
```

---

# What Gets Stored?

Instead of

```
Doctor@123
```

the database stores

```
$2b$10$LJ7smnP...
```

Even the backend developer cannot determine the original password.

---

# Login Flow

During login,

the user enters

```
Email

↓

Password
```

The backend retrieves the stored hash.

```
Database

↓

Stored Hash
```

Then compares

```
Entered Password

↓

comparePassword()

↓

true / false
```

---

# Login Example

```ts
const isPasswordValid = await comparePassword(loginDto.password, user.password)

if (!isPasswordValid) {
  throw new UnauthorizedError('Invalid email or password.')
}
```

Notice

We never compare plain text passwords.

bcrypt performs the comparison securely.

---

# Why Not Compare Strings?

Wrong

```ts
if (password === user.password)
```

Correct

```ts
await bcrypt.compare(...)
```

The stored value is already hashed.

String comparison would always fail.

---

# Where Does This Fit?

```
POST /auth/login

↓

Route

↓

Controller

↓

Service

↓

Repository

↓

Database

↓

comparePassword()

↓

Generate JWT

↓

Response
```

Password verification belongs inside the Service layer because it is part of the authentication business logic.

---

# Production Notes

Always

- Hash passwords before saving.
- Use bcrypt or another password-specific algorithm.
- Store only hashes.
- Keep hashing logic inside shared utilities.
- Never expose password hashes in API responses.
- Never log passwords.

---

# Common Mistakes

## Saving Plain Text Passwords

Wrong

```ts
password: dto.password
```

Correct

```ts
password: hashedPassword
```

---

## Hashing Twice

Hash the password only during registration or password updates.

Do not hash an already hashed password.

---

## Using SHA256 Directly

General-purpose hash algorithms are not designed for password storage.

Use bcrypt, Argon2, or another password hashing algorithm.

---

## Logging Passwords

Never write

```ts
logger.info(password)
```

or

```ts
console.log(password)
```

Passwords should never appear in logs.

---

# Senior Engineer Notes

Password hashing is one of the first security layers in every backend.

A well-designed authentication system ensures:

- Original passwords never reach the database.
- Password verification happens securely.
- Hashing logic is reusable.
- Security concerns remain isolated from business logic.

Treat password handling as a reusable infrastructure component rather than application-specific logic.

---

# Summary

Password hashing protects user credentials by converting passwords into irreversible hashes before storing them in the database.

Using bcrypt ensures that passwords remain secure even if the database is compromised. The backend verifies passwords by comparing hashes rather than storing or comparing plain text values.

The password utility we created will be reused throughout the authentication system whenever passwords need to be created, updated, or verified.

# 08.7 - Authentication Middleware

> Authentication Middleware verifies the JWT sent by the client, identifies the authenticated user, and allows only authorized requests to reach the application's business logic.

---

- How Bearer Tokens work.

- How authenticated users are attached to the request.
- How Authentication Middleware fits into the request lifecycle.

A doctor has already logged in.

```

↓

↓

```

```

```

# Authorization Header

Every protected request includes

```
Authorization

```

```
↓

JWT
```

Only the JWT is verified.

# Step 1 — Create the Middleware

Create

```
src/middlewares/authentication.middleware.ts
```

```ts
import { NextFunction, Request, Response } from 'express'
import { verifyAccessToken } from '../utils/jwt'
import { UnauthorizedException } from '../errors/UnauthorizedException'

export async function authenticate(
  req: Request,

  res: Response,

  next: NextFunction
) {
  const authorizationHeader = req.headers.authorization

  if (!authorizationHeader) {
    throw new UnauthorizedException('Authentication token is required.')
  }

  const [scheme, token] = authorizationHeader.split(' ')

  if (scheme !== 'Bearer' || !token) {
    throw new UnauthorizedException('Invalid authentication header.')
  }

  const payload = verifyAccessToken(token)

  req.user = payload

  next()
}
```

---

# Why Split the Header?

The header looks like

```
Bearer eyJhbGc...
```

We only need

```
eyJhbGc...
```

The word

```
Bearer
```

simply identifies the authentication scheme.

---

# Why Attach User to Request?

Without this,

every controller would need to decode the JWT again.

Instead

```
JWT

↓

Middleware

↓

req.user

↓

Controller
```

Every controller immediately knows

who is making the request.

---

# Step 2 — Extend Express Request

TypeScript does not know

```
req.user
```

exists.

Create

```
src/types/express.d.ts
```

```ts
import 'express'

declare global {
  namespace Express {
    interface Request {
      user: {
        id: string

        email: string

        role: string
      }
    }
  }
}
```

---

# Why Extend Request?

Without this,

TypeScript reports

```
Property 'user' does not exist on type Request
```

Declaration merging teaches Express about our custom property.

# Production Notes

Always

- Verify JWT inside middleware.
- Never trust client-provided user IDs.
- Attach authenticated user to `req.user`.
- Protect private routes with middleware.
- Keep middleware focused on authentication only.

---

# Common Mistakes

## Verifying JWT Inside Controllers

Wrong

```
Controller

↓

Verify JWT
```

Correct

```
Middleware

↓

Controller
```

---

## Trusting Request Body

Wrong

```json
{
  "userId": "123"
}
```

Correct

```
req.user.id
```

The authenticated user should always come from the verified JWT, never from the client.

---

## Forgetting Bearer Validation

Always verify

```
Bearer
```

before reading the token.

---

# Senior Engineer Notes

Authentication Middleware is one of the most reused components in a backend.

Every protected route depends on it.

A good middleware should

- verify JWTs
- reject invalid requests
- attach the authenticated user
- remain independent of business logic

It should never query unrelated business entities or execute application logic.

# Summary

Authentication Middleware acts as the security gatekeeper of the application.

Instead of verifying JWTs inside every controller, the middleware validates the token once, extracts the authenticated user's information, attaches it to the request, and forwards the request to the business layer.

This keeps Controllers clean, avoids duplicated logic, and ensures that every protected route follows the same authentication process.

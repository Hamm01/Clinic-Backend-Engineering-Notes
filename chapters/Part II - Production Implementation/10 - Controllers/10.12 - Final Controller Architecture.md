Imagine the receptionist creates a new patient.

```

```

```
Client

↓

Express Router

↓

↓

↓

Request Validation

↓

Patient Controller

↓

Patient Service

↓

Patient Repository

↓

the response travels back.

```

↓

The Controller coordinates both directions of this flow.

# Responsibility Matrix

| Layer          | Responsibility              |
| -------------- | --------------------------- |
| Router         | Map URL to Controller       |
| Authentication | Identify the logged-in user |
| Authorization  | Verify permissions          |
| Validation     | Validate incoming request   |
| Controller     | Coordinate the request      |
| Service        | Execute business logic      |
| Repository     | Access the database         |
| Prisma         | Generate database queries   |
| PostgreSQL     | Persist application data    |

Every layer performs one job.

---

# What Controllers Should Do

Controllers should

✅ Receive HTTP requests

✅ Read route parameters

✅ Read query parameters

✅ Read request body

✅ Read authenticated user

✅ Call the appropriate Service

✅ Return standardized HTTP responses

Controllers are responsible for coordinating communication between the HTTP layer and the application's business logic.

---

# What Controllers Should Never Do

Controllers should never

❌ Access Prisma directly

❌ Execute SQL queries

❌ Perform business validation

❌ Hash passwords

❌ Generate JWT tokens

❌ Check business permissions

❌ Send emails

❌ Upload files

❌ Schedule background jobs

❌ Process payments

❌ Perform calculations

These responsibilities belong to other layers of the application.

---

# Thin Controller Philosophy

A Controller should answer only one question.

```
Which Service
should handle
this request?
```

It should never answer

```
How
should
the business
work?
```

Business decisions belong inside Services.

# Controller Example

```ts
async create(
    req: Request,
    res: Response
) {

    const doctor =
        await this.doctorService.create({

            ...req.body,

            clinicId: req.user.clinicId,

            createdBy: req.user.id

        });

    return successResponse(

        res,

        201,

        "Doctor created successfully.",

        doctor

    );

}
```

This Controller

- Reads request data.
- Delegates business logic.
- Returns a standardized response.

Nothing more.

# Characteristics of a Production Controller

A production Controller is

- Small.
- Readable.
- Predictable.
- Stateless.
- Easy to test.
- Easy to maintain.
- Independent of database implementation.
- Focused on HTTP communication.

These characteristics make Controllers scalable as the application grows.

# Module Summary

In this module, we transformed Controllers from simple Express handlers into production-ready coordinators.

We learned that Controllers are responsible for interacting with the HTTP layer, while business rules belong inside Services and data access belongs inside Repositories. By keeping Controllers thin, using standardized responses, and centralizing error handling, we created an architecture that is easier to maintain, test, and scale.

The Controller is no longer just an Express file—it is the entry point into the application's business workflow.

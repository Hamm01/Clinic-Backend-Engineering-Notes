# 10.3 - Request Object

> Every HTTP request entering an Express application is represented by the Request object. It contains all information sent by the client, including URL parameters, query parameters, request body, headers, cookies, uploaded files, and authenticated user details. In this chapter, we will learn the Request object by implementing a real Doctor API instead of studying each property in isolation.

---

After completing this chapter, you will understand:

- What the Express Request object is.
- Where request data comes from.
- How to access different parts of a request.
- Which Request properties are commonly used in production.

# Controller

```ts
import { Request, Response } from 'express'

import { DoctorService } from './doctor.service'

export class DoctorController {
  private readonly doctorService = new DoctorService()

  async create(
    req: Request,

    res: Response
  ) {
    const doctor = await this.doctorService.create({
      clinicId: req.user.clinicId,

      createdBy: req.user.id,

      ...req.body
    })

    return res.status(201).json({
      success: true,

      message: 'Doctor created successfully.',

      data: doctor
    })
  }
}
```

# req.body

Example

```json
{
  "firstName": "John",

  "email": "john@example.com"
}
```

Express converts the JSON body into

```ts
req.body
```

Inside our controller

```ts
await doctorService.create(req.body)
```

The body contains the data used to create a new resource.

---

# Why Not Trust req.body Completely?

Imagine the client sends

```json
{
  "role": "ADMIN",

  "createdBy": "someone-else",

  "clinicId": "clinic-99"
}
```

If we simply do

```ts
repository.create(req.body)
```

the client controls sensitive fields.

Instead

```ts
clinicId: req.user.clinicId

createdBy: req.user.id
```

These values come from the authenticated user, not from the request body.

# req.headers

Every request contains headers.

Example

```http
Authorization:

Bearer eyJhbGc...
```

Controller

```ts
const token = req.headers.authorization
```

Normally

Authentication Middleware

reads this value.

Controllers rarely need it directly.

---

# req.user

This property

does not exist

by default.

Our Authentication Middleware

creates it.

Example

```ts
req.user = {
  id: 'doctor_1',

  clinicId: 'clinic_1',

  role: 'DOCTOR'
}
```

Later

Controllers can safely use

```ts
req.user.id
```

without decoding the JWT again.

---

# req.ip

Express also provides

```ts
req.ip
```

Useful for

- Audit Logs
- Security Monitoring
- Rate Limiting
- Login History

Example

```ts
console.log(req.ip)
```

A Receptionist creates a Doctor.

```json
}
```

```ts
req.user = {
```

```ts
await doctorService.create({

  clinicId: req.user.clinicId,

```

# Production Notes

Always

- Validate `req.body`.
- Never trust sensitive values from the client.
- Read resource IDs from `req.params`.
- Use `req.query` for pagination and filtering.
- Use `req.user` for authenticated user information.
- Keep Controllers responsible only for reading request data.

---

# Common Mistakes

## Trusting Client IDs

Wrong

```ts
clinicId: req.body.clinicId
```

Correct

```ts
clinicId: req.user.clinicId
```

---

## Using req.body for GET Requests

GET requests should primarily use

```
req.params

and

req.query
```

---

## Reading JWT Inside Every Controller

Authentication Middleware already decoded the JWT.

Use

```ts
req.user
```

instead.

---

## Forgetting Type Conversion

Query parameters are strings.

Convert them when numeric values are expected.

Example

```ts
Number(req.query.page)
```

# Summary

The Request object is the primary source of client data in an Express application.

It provides access to URL parameters, query parameters, request body, headers, authenticated user information, and other request metadata.

A production-ready Controller reads the necessary information from the Request object, enriches it with trusted server-side data such as `req.user`, and delegates the work to the Service layer without performing business logic.

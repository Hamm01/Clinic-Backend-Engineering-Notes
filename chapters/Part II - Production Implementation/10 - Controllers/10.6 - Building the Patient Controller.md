# 10.6 - Building the Patient Controller

> The Patient module is one of the most frequently used modules in a Clinic Management System. Receptionists register new patients, doctors search for patients, nurses update patient information, and administrators manage patient records. In this chapter, we will implement a production-ready Patient Controller that exposes RESTful endpoints while keeping the controller clean and focused on handling HTTP requests.

---

# Learning Objectives

After completing this chapter, you will understand:

- How to implement CRUD endpoints inside a Controller.
- How Controllers handle different HTTP methods.
- How to read Route Parameters.
- How to read Query Parameters.
- How Controllers interact with the Service layer.
- How to return standardized API responses.

---

# Business Requirement

Our clinic needs the following APIs.

```
Create Patient

↓

Get Patient By Id

↓

List Patients

↓

Update Patient

↓

Delete Patient
```

Instead of putting everything into one endpoint, we will expose REST APIs.

---

# API Design

| Method | Endpoint      | Purpose        |
| ------ | ------------- | -------------- |
| POST   | /patients     | Create Patient |
| GET    | /patients     | List Patients  |
| GET    | /patients/:id | Get Patient    |
| PUT    | /patients/:id | Update Patient |
| DELETE | /patients/:id | Delete Patient |

This follows REST conventions and makes the API predictable.

# Step 1 — Create Patient

```ts
async create(
    req: Request,
    res: Response
) {

    const patient =
        await this.patientService.create({

            ...req.body,

            clinicId: req.user.clinicId,

            createdBy: req.user.id

        });

    return res.status(201).json({

        success: true,

        message: "Patient created successfully.",

        data: patient

    });

}
```

---

# Why Add clinicId?

Every patient belongs to exactly one clinic.

Never allow

```ts
req.body.clinicId
```

Instead use

```ts
req.user.clinicId
```

The authenticated user determines which clinic the patient belongs to.

# Step 2 — Get Patient

Endpoint

```
GET

/api/v1/patients/:id
```

Controller

```ts
async findOne(
    req: Request,
    res: Response
) {

    const patient =
        await this.patientService.findOne(

            req.params.id

        );

    return res.status(200).json({

        success: true,

        message: "Patient fetched successfully.",

        data: patient

    });

}
```

---

# Why req.params?

Request

```
GET

/patients/patient_101
```

Express extracts

```
patient_101
```

into

```ts
req.params.id
```

Route parameters identify a single resource.

---

# Step 3 — List Patients

Endpoint

```
GET

/api/v1/patients
```

Controller

```ts
async findAll(
    req: Request,
    res: Response
) {

    const patients =
        await this.patientService.findAll({

            page: Number(req.query.page),

            limit: Number(req.query.limit),

            search: String(

                req.query.search ?? ""

            )

        });

    return res.status(200).json({

        success: true,

        message: "Patients fetched successfully.",

        data: patients

    });

}
```

---

# Why Query Parameters?

The client can request

```
/patients

?page=1

&limit=20

&search=Rahul
```

This avoids returning thousands of records in one request.

Query parameters are commonly used for

- Pagination
- Searching
- Filtering
- Sorting

# Step 4 — Update Patient

Endpoint


Controller

```ts
async update(
    req: Request,
    res: Response
) {

    const patient =
        await this.patientService.update(

        success: true,

        message: "Patient updated successfully.",

        data: patient

```

---

# Step 5 — Delete Patient

Endpoint

```
DELETE

/api/v1/patients/:id
```

Controller

```ts
async delete(
    req: Request,
    res: Response
) {

    await this.patientService.delete(

        req.params.id

    );

    return res.status(204).send();

}
```

---

# Why 204?

Deleting a resource

does not need to return data.

REST recommends

```
204 No Content
```

This indicates

the deletion was successful

without sending a response body.

---

# Complete Patient Controller

```text
POST     /patients
GET      /patients
GET      /patients/:id
PUT      /patients/:id
DELETE   /patients/:id
```

Every endpoint

calls the Service layer.

The Controller never accesses the database directly.

---

# Responsibilities

The Patient Controller

✅ Receives HTTP requests

✅ Reads request data

✅ Calls the Service

✅ Returns HTTP responses

It does **not**

❌ Validate business rules

❌ Execute database queries

❌ Check permissions

❌ Perform complex calculations

---

# Common Mistakes

## Returning Entire Database

Always paginate list endpoints.

---

## Using req.body for IDs

Wrong

```ts
req.body.id
```

Correct

```ts
req.params.id
```

---

## Forgetting Query Conversion

Remember

```ts
req.query.page
```

is a string.

Convert it to a number before using it.

---

## Business Logic in Controllers

Keep Controllers thin.

Business decisions belong inside Services.

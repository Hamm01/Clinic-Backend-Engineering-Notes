# 10.7 - Building the Appointment Controller

> Appointments are the core of any Clinic Management System. Every appointment connects a Patient, a Doctor, and a Clinic while also maintaining scheduling information. In this chapter, we will implement a production-ready Appointment Controller and learn how Controllers coordinate requests involving multiple entities without containing business logic.

---

# Learning Objectives

After completing this chapter, you will understand:

- How to build Controllers involving multiple entities.
- How Controllers handle relationships.
- Why Controllers should never validate appointment rules.
- How to keep Controllers clean.
- How to expose RESTful Appointment APIs.

---

# Step 1 — Create Appointment

```ts
async create(
    req: Request,
    res: Response
) {

    const appointment =
        await this.appointmentService.create({

            ...req.body,

            clinicId: req.user.clinicId,

            createdBy: req.user.id

        });

    return res.status(201).json({

        success: true,

        message: "Appointment created successfully.",

        data: appointment

    });

}
```

# Why Doesn't the Controller Check Availability?

Many beginners write

```ts
if(
doctor already booked
)
```

inside Controllers.

Don't.

Scheduling rules belong inside

```
AppointmentService
```

Controllers coordinate.

Services decide.

# Step 2 — Get Appointment

Endpoint

```
GET

/appointments/:id
```

```ts
async findOne(
    req: Request,
    res: Response
) {

    const appointment =
        await this.appointmentService.findOne(

            req.params.id

        );

    return res.status(200).json({

        success: true,

        message: "Appointment fetched successfully.",

        data: appointment

    });

}
```

---

# Step 3 — List Appointments

```ts
async findAll(
    req: Request,
    res: Response
) {

    const appointments =
        await this.appointmentService.findAll({

            page: Number(req.query.page),

            limit: Number(req.query.limit),

            doctorId: req.query.doctorId as string,

            patientId: req.query.patientId as string,

            date: req.query.date as string

        });

    return res.status(200).json({

        success: true,

        message: "Appointments fetched successfully.",

        data: appointments

    });

}
```

---

# Why Filters?

Receptionists rarely need

every appointment.

Instead

```
GET

/appointments

?doctorId=doctor_45

&date=2026-08-10
```

Only matching appointments

are returned.

Filtering improves performance

and user experience.

# Step 4 — Update Appointment

```ts
async update(
    req: Request,
    res: Response
) {

    const appointment =
        await this.appointmentService.update(

            req.params.id,

            req.body

        );

    return res.status(200).json({

        success: true,

        message: "Appointment updated successfully.",

        data: appointment

    });

}
```

---

# Step 5 — Cancel Appointment

```ts
async cancel(
    req: Request,
    res: Response
) {

    await this.appointmentService.cancel(

        req.params.id

    );

    return res.status(204).send();

}
```

Notice

We are not deleting

the appointment ourselves.

The Service decides

whether cancellation means

- Soft Delete
- Status Update
- Business Workflow

---

# What Doesn't Belong Here?

Never write

```ts
Doctor Availability

Patient Validation

Time Conflict

Clinic Working Hours

Double Booking Check
```

inside the Controller.

Those belong

inside the Service layer.

✅ Return Response

# Common Mistakes

## Checking Doctor Availability

Wrong

```ts
if(doctorBusy){

...
}
```

Move this to the Service.

---

## Accessing Prisma

Wrong

```ts
await prisma.appointment.create(...)
```

Controllers never access Prisma directly.

---

## Ignoring Query Parameters

Always support

```
Pagination

Filtering

Searching
```

for list endpoints.

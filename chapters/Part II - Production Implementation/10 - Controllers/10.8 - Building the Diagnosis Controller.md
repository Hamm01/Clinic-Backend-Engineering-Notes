# 10.8 - Building the Diagnosis Controller

> Diagnoses are one of the most sensitive resources in a healthcare application. A Diagnosis represents a doctor's medical assessment after examining a patient. Because medical information is highly sensitive, Controllers must remain simple and delegate all business rules, authorization checks, and validation to the Service layer. In this chapter, we will implement a production-ready Diagnosis Controller.

---

# Learning Objectives

After completing this chapter, you will understand:

- How to build a Controller for sensitive medical data.
- How Controllers work with authenticated users.
- Why Controllers should never contain medical business logic.

# Step 1 — Create Diagnosis

```ts
import { Request, Response } from 'express'

import { DiagnosisService } from './diagnosis.service'

export class DiagnosisController {
  private readonly diagnosisService = new DiagnosisService()

  async create(req: Request, res: Response) {
    const diagnosis = await this.diagnosisService.create({
      ...req.body,

      doctorId: req.user.id,

      clinicId: req.user.clinicId
    })

    return res.status(201).json({
      success: true,

      message: 'Diagnosis created successfully.',

      data: diagnosis
    })
  }
}
```

---

# Why Doesn't the Client Send doctorId?

Never trust

```json
{
  "doctorId": "doctor_999"
}
```

A malicious client could pretend to be another doctor.

Instead

```ts
doctorId: req.user.id
```

comes from the authenticated JWT.

This guarantees the diagnosis is linked to the logged-in doctor.

---

{

# Step 3 — List Diagnoses

```ts
async findAll(
    req: Request,
    res: Response
) {

    const diagnoses =
        await this.diagnosisService.findAll({

            patientId: req.query.patientId as string,

            doctorId: req.query.doctorId as string,

            page: Number(req.query.page),

            limit: Number(req.query.limit)

        });

    return res.status(200).json({

        success: true,

        message: "Diagnoses fetched successfully.",

        data: diagnoses

    });

}
```

# What Doesn't Belong in the Controller?

The Controller should **never** decide:

```
Can another doctor edit this diagnosis?

↓

Is the appointment completed?

↓

Is the patient active?

↓

Can the diagnosis still be modified?

↓

Should a prescription be generated?
```

These decisions belong in the **DiagnosisService**.

---

# Controller Responsibilities

The Diagnosis Controller should

✅ Receive HTTP requests

✅ Read route parameters

✅ Read query parameters

✅ Read authenticated user

✅ Call the Service

✅ Return HTTP responses

---

It should **never**

❌ Access Prisma

❌ Write SQL

❌ Validate business rules

❌ Perform authorization

❌ Generate prescriptions

❌ Update medical history directly

---

# Why Is This Separation Important?

Imagine a hospital changes its policy.

New Rule

```
A Diagnosis

cannot be edited

after 24 hours.
```

Where should this rule go?

Correct answer

```
DiagnosisService
```

The Controller should not change.

Only the Service changes.

This makes the application easier to maintain as business rules evolve.

# Common Mistakes

## Trusting doctorId from the Client

Wrong

```ts
doctorId: req.body.doctorId
```

Correct

```ts
doctorId: req.user.id
```

---

## Returning Different Response Formats

Every endpoint should follow the same API contract.

---

## Business Logic in Controllers

Controllers coordinate.

Services make decisions.

---

## Direct Database Access

Never write

```ts
await prisma.diagnosis.create(...)
```

inside Controllers.

# Summary

The Diagnosis Controller demonstrates how Controllers manage sensitive healthcare data while remaining simple and focused. It receives HTTP requests, enriches them with trusted information from the authenticated user, delegates all business decisions to the Service layer, and returns standardized responses.

This design ensures that medical rules, authorization policies, and data persistence remain isolated from the HTTP layer, making the application secure, maintainable, and easier to evolve.

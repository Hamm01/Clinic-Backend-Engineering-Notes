# 10.11 - Thin Controllers vs Fat Controllers

> One of the biggest differences between beginner and production backend applications is the distribution of responsibilities. Beginners often place business logic inside Controllers, creating large, difficult-to-maintain files. Production applications follow the Thin Controller pattern, where Controllers coordinate requests while Services contain business logic. In this chapter, we will compare both approaches and understand why thin Controllers are considered a best practice.

# What Is a Thin Controller?

A Thin Controller is responsible only for coordinating an HTTP request.

It should

- Receive the request.
- Read request data.
- Call the appropriate Service.
- Return the response.

Nothing more.

```
Request

↓

Controller

↓

Service

↓

Repository

↓

Database

↓

Controller

↓

Response
```

The Controller acts as a bridge between the client and the application's business logic.

---

# What Is a Fat Controller?

A Fat Controller contains business logic, validation, database access, and response handling all in one place.

Example

```ts
async create(
    req: Request,
    res: Response
) {

    if (!req.body.email) {

        return res.status(400).json({

            message: "Email is required."

        });

    }

    const existingDoctor =
        await prisma.doctor.findUnique({

            where: {

                email: req.body.email

            }

        });

    if (existingDoctor) {

        return res.status(409).json({

            message: "Doctor already exists."

        });

    }

    const hashedPassword =
        await bcrypt.hash(

            req.body.password,

            12

        );

    const doctor =
        await prisma.doctor.create({

            data: {

                ...req.body,

                password: hashedPassword

            }

        });

    return res.status(201).json({

        success: true,

        data: doctor

    });

}
```

This Controller is responsible for

- Validation
- Database Queries
- Password Hashing
- Business Rules
- Response Formatting

Everything is mixed together.

---

# Problems with Fat Controllers

As the application grows

Controllers become

```
800 Lines

↓

1200 Lines

↓

2000 Lines
```

Eventually

- Difficult to read.
- Difficult to debug.
- Difficult to test.
- Difficult to reuse.
- Difficult to maintain.

# Thin Controller Example

```ts
async create(
    req: Request,
    res: Response
) {

    const doctor =
        await this.doctorService.create({

            ...req.body,

            clinicId: req.user.clinicId,

            createdBy: req.user.id

        });

    return successResponse(

        res,

        201,

        "Doctor created successfully.",

        doctor

    );

}
```

The Controller now

- Reads request data.
- Delegates business logic.
- Returns a response.

Nothing else.

# Responsibility Distribution

## Controller

Responsible for

✅ Reading HTTP Requests

✅ Reading Parameters

✅ Reading Query Strings

✅ Reading Request Body

✅ Calling Services

✅ Returning HTTP Responses

---

## Service

Responsible for

✅ Business Rules

✅ Validation Rules

✅ Permission Checks

✅ Password Hashing

✅ JWT Generation

✅ Workflow Decisions

---

## Repository

Responsible for

✅ Database Queries

✅ CRUD Operations

✅ Transactions

✅ Prisma Operations

# Benefits of Thin Controllers

- Easier to understand.
- Easier to test.
- Easier to debug.
- Easier to maintain.
- Better separation of concerns.
- More reusable business logic.
- Smaller files.
- Cleaner architecture.

---

# Comparison

| Thin Controller            | Fat Controller                |
| -------------------------- | ----------------------------- |
| Small and focused          | Large and complex             |
| Delegates work             | Performs all work             |
| Easy to maintain           | Difficult to maintain         |
| Highly testable            | Hard to test                  |
| Business logic in Services | Business logic in Controllers |
| Production ready           | Beginner architecture         |

# Common Mistakes

## Database Queries Inside Controllers

Wrong

```ts
await prisma.doctor.findMany()
```

Move database access to the Repository.

---

## Password Hashing

Wrong

```ts
bcrypt.hash(...)
```

Move authentication logic to the Service.

---

## Permission Checks

Wrong

```ts
if(user.role !== "ADMIN")
```

Business authorization belongs in the Service or dedicated authorization layer.

---

## Validation

Wrong

```ts
if(!email)
```

Input validation belongs in Validation Middleware.

---

# Controller Checklist

Before finishing a Controller ask yourself

```
Does it access Prisma?

↓

Does it hash passwords?

↓

Does it generate JWTs?

↓

Does it contain business rules?

↓

Does it perform calculations?
```

If the answer is

```
Yes
```

the Controller is becoming too fat.

# Summary

Thin Controllers are a cornerstone of maintainable backend architecture. By limiting Controllers to handling HTTP requests and delegating business logic to Services, applications become easier to understand, test, and extend. This separation of responsibilities allows each layer of the application to evolve independently while keeping the overall architecture clean and scalable.

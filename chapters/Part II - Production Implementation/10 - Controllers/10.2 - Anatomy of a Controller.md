# 10.2 - Anatomy of a Controller

> A Controller is the entry point of every HTTP request after it passes through middleware. It is responsible for receiving the request, delegating work to the Service layer, and returning an appropriate HTTP response. In this chapter, we will build our first production-ready Controller and understand every line of code.

# What Is a Controller?

Think of a Controller as a receptionist.

A patient arrives.

The receptionist

- listens
- verifies
- forwards the request

The receptionist does **not**

- diagnose the patient
- prescribe medicine
- perform surgery

Exactly the same principle applies here.

```
Client

↓

Controller

↓

Service

↓

Repository

↓

Database
```

The Controller coordinates.

The Service performs business logic.

# Our First Controller

Create

```
doctor.controller.ts
```

```ts
import { Request, Response } from 'express'

import { DoctorService } from './doctor.service'

export class DoctorController {
  constructor(private readonly doctorService = new DoctorService()) {}

  async create(
    req: Request,

    res: Response
  ) {
    const doctor = await this.doctorService.create(req.body)

    return res.status(201).json({
      success: true,

      message: 'Doctor created successfully.',

      data: doctor
    })
  }
}
```

# What Should Never Be Inside Controllers?

Avoid

```ts
await prisma.doctor.create(...)
```

Controllers

should never

query databases.

---

Avoid

```ts
bcrypt.hash(...)
```

Authentication logic

belongs elsewhere.

---

Avoid

```ts
jwt.sign(...)
```

JWT generation

belongs

inside

Authentication Services.

---

Avoid

Large

business calculations.

Move them

to Services.

---

# Controller Responsibilities

A Controller should

✅ Receive Request

✅ Read Parameters

✅ Read Body

✅ Read Query

✅ Read Authenticated User

✅ Call Service

✅ Return Response

---

A Controller should NOT

❌ Access Database

❌ Perform Validation Logic

❌ Generate JWT

❌ Hash Passwords

❌ Write Business Rules

❌ Check Permissions

# Summary

Controllers are responsible for coordinating the flow of an HTTP request.

A well-designed Controller receives the request, delegates business logic to the Service layer, and returns a standardized HTTP response.

Keeping Controllers thin improves maintainability, testability, and scalability while allowing Services and Repositories to focus on their own responsibilities.

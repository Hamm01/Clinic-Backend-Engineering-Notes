# 10.10 - Error Handling Inside Controllers

> Error handling is a critical aspect of building reliable backend applications. A Controller should focus on coordinating requests, not managing exceptions. In production applications, errors are handled centrally through dedicated middleware, allowing Controllers to remain clean, consistent, and maintainable. In this chapter, we will understand why centralized error handling is preferred and how Controllers should participate in that architecture.

---

# Current Situation

Suppose our Controller looks like this.

```ts
async create(
    req: Request,
    res: Response
) {

    const doctor = await this.doctorService.create(req.body);

    return successResponse(
        res,
        201,
        "Doctor created successfully.",
        doctor
    );

}
```

Everything works.

But what happens if

```
Database Connection Lost

or

Validation Failed

or

Doctor Already Exists
```

Who handles these errors?

---

# Beginner Approach

Many developers write

```ts
async create(
    req: Request,
    res: Response
) {

    try {

        const doctor = await this.doctorService.create(req.body);

        return successResponse(
            res,
            201,
            "Doctor created successfully.",
            doctor
        );

    } catch (error) {

        return res.status(500).json({

            success: false,

            message: "Internal Server Error"

        });

    }

}
```

Initially

this looks correct.

---

# Why This Is a Problem

Imagine

```
120 Controllers
```

Every Controller contains

```
try

↓

catch

↓

500 Response
```

Now imagine

the company changes

the error response format.

Instead of changing

```
One File
```

you now modify

```
120 Controllers
```

This becomes difficult to maintain.

---

# Production Architecture

Instead of

```
Controller

↓

try/catch

↓

Response
```

Production applications use

```
Controller

↓

Service

↓

Repository

↓

Exception

↓

Global Error Middleware

↓

HTTP Response
```

Every exception

travels

to one place.

---

# Controller Without try/catch

```ts
async create(
    req: Request,
    res: Response
) {

    const doctor =
        await this.doctorService.create(req.body);

    return successResponse(

        res,

        201,

        "Doctor created successfully.",

        doctor

    );

}
```

Notice

There is

no

```
try

catch
```

inside the Controller.

---

# Where Does the Error Go?

Suppose

DoctorService throws

```ts
throw new Error('Doctor already exists.')
```

The flow becomes

```
Service

↓

Exception

↓

Express

↓

Global Error Middleware

↓

JSON Response
```

The Controller

never catches it.

---

# Request Lifecycle

```
Client

↓

Controller

↓

Service

↓

Repository

↓

Database

↓

Exception

↓

Global Error Middleware

↓

Client
```

Controllers remain focused

only on

request coordination.

---

# Why Centralized Error Handling?

One location

controls

- Status Codes
- Error Messages
- Logging
- Stack Traces
- Error Formatting

Every API

returns

the same error structure.

---

# Standard Error Response

Every API should return

```json
{
  "success": false,

  "message": "Doctor already exists.",

  "errors": []
}
```

Regardless of

which Controller

threw the exception.

---

# Controller Responsibilities

Controllers should

✅ Read Request

✅ Call Service

✅ Return Success Response

They should **not**

❌ Handle Database Errors

❌ Catch Business Exceptions

❌ Log Stack Traces

❌ Decide HTTP 500 Responses

---

# Responsibilities of Global Error Middleware

The middleware should

✅ Catch Exceptions

✅ Log Errors

✅ Determine Status Code

✅ Build Error Response

✅ Return Standardized JSON

---

# Common Mistakes

## try/catch in Every Controller

Avoid repeating the same error handling logic.

Centralize it.

---

## Returning Raw Errors

Wrong

```json
{
  "message": "PrismaClientKnownRequestError..."
}
```

Clients should not receive internal implementation details.

---

## Swallowing Exceptions

Never ignore an error.

Every exception should either be handled appropriately or allowed to reach the global error handler.

---

## Mixing Success and Error Responses

Success responses

and

error responses

should both follow standardized formats.

# Summary

Controllers should remain focused on handling HTTP requests and delegating work to the Service layer. Instead of repeating error handling logic in every Controller, production applications centralize exception handling using Global Error Middleware. This approach reduces duplication, ensures consistent error responses, simplifies maintenance, and provides a single location for logging and monitoring application failures.

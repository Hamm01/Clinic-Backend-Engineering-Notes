# 10.4 - Response Object

In this chapter, we'll build a standardized response structure that every endpoint in our Clinic Backend will use.

# What is Response?

The Response object represents

the server's reply

to the client.

```
Client

↓

Request

↓

Express

↓

Controller

↓

Response

↓

Client
```

Every request

must end

with exactly one response.

---

# Our Current Controller

```ts
async create(
    req: Request,
    res: Response
) {

    const doctor = await this.doctorService.create(
        req.body
    );

    return res.status(201).json({
        success: true,
        message: "Doctor created successfully.",
        data: doctor
    });

}
```

# Understanding Response

The Response object contains

methods for sending

HTTP responses.

Most commonly

```ts
res.status()

res.json()

res.send()

res.end()
```

---

# res.status()

Sets the HTTP Status Code.

Example

```ts
return res.status(201)
```

The status code tells the client

whether the request

was successful

or failed.

---

# Common Status Codes

| Code | Meaning               | When Used                |
| ---- | --------------------- | ------------------------ |
| 200  | OK                    | Successful GET or UPDATE |
| 201  | Created               | New Resource Created     |
| 204  | No Content            | Delete Success           |
| 400  | Bad Request           | Validation Failed        |
| 401  | Unauthorized          | Invalid Authentication   |
| 403  | Forbidden             | Permission Denied        |
| 404  | Not Found             | Resource Missing         |
| 409  | Conflict              | Duplicate Resource       |
| 500  | Internal Server Error | Unexpected Error         |

---

# Why 201?

Creating a Doctor

creates

a new resource.

Therefore

```http
HTTP 201 Created
```

is correct.

Returning

```
200
```

is not technically wrong,

but

```
201
```

is the REST standard.

# Why Standardize Responses?

Imagine

Doctor API

returns

```json
{
  "doctor": {}
}
```

Patient API

returns

```json
{
  "patient": {}
}
```

Appointment API

returns

```json
{
  "appointment": {}
}
```

Frontend

must handle

every endpoint differently.

---

Instead

every API returns

```json
{
  "success": true,
  "message": "...",
  "data": {}
}
```

Now

Frontend

handles

every response

the same way.

---

# Response Structure

Every successful response

should follow

```json
{
  "success": true,

  "message": "...",

  "data": {}
}
```

---

Every failed response

will follow

```json
{
  "success": false,

  "message": "...",

  "errors": []
}
```

We'll implement this

during

Error Handling.

# Updated Controller

```ts
async create(
    req: Request,
    res: Response
) {

    const doctor = await this.doctorService.create({

        ...req.body,

        createdBy: req.user.id

    });

    return res.status(201).json({

        success: true,

        message: "Doctor created successfully.",

        data: doctor

    });

}
```

# Why return?

Notice

```ts
return res.json(...)
```

instead of

```ts
res.json(...)
```

Using

```
return
```

immediately

stops

function execution.

It prevents

accidentally sending

multiple responses.

---

Wrong

```ts
res.json(data)

console.log('still running')
```

Even worse

```ts
res.json(data)

res.json(data)
```

Express throws

```
Cannot set headers

after they are sent.
```

---

Correct

```ts
return res.json(data)
```

Function ends

immediately.

# Production Notes

Always

- Return the correct HTTP status.
- Use a consistent response structure.
- Return meaningful messages.
- Return JSON for APIs.
- End the function after sending a response.

---

# Common Mistakes

## Returning Different Formats

Wrong

```json
{
  "id": 1
}
```

and

```json
{
  "doctor": {}
}
```

Use one standard format.

---

## Wrong Status Codes

Creating

↓

201

Updating

↓

200

Deleting

↓

204

---

## Multiple Responses

Never call

```ts
res.json()
```

twice.

---

## Exposing Internal Errors

Don't return

```json
{
  "message": "Prisma Error..."
}
```

Return

```json
{
  "message": "Internal server error."
}
```

Detailed errors belong

in logs.

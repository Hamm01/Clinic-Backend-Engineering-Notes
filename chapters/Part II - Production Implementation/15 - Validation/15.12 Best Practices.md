# 15.12 - Best Practices

## Overview

Writing validation code is only part of building a reliable backend.

Equally important is organizing validation so that it remains consistent, reusable, and easy to maintain as the application grows.

The following best practices are used throughout our Clinic Backend and are commonly found in production applications.

---

## Validate Every Client Input

Treat every value received from the client as untrusted.

Validate:

- Request Body
- Path Parameters
- Query Parameters
- Uploaded Files

Never assume client data is correct.

---

## Validate Before Business Logic

Validation should occur before the request reaches the Controller or Service layer.

```text
Client

↓

Validation

↓

Controller

↓

Service

↓

Repository
```

This keeps the rest of the application working with trusted input.

---

## One Schema Per Endpoint

Different endpoints usually require different validation rules.

Example

```text
POST /patients

↓

CreatePatientSchema
```

```text
PATCH /patients/:id

↓

UpdatePatientSchema
```

Keeping schemas specific to each endpoint improves readability and maintenance.

---

## Reuse Common Validation

Avoid duplicating validation rules.

Instead, centralize commonly used validators.

Example

```text
Shared Validators

↓

Patient

Doctor

Appointment

Prescription
```

This keeps validation consistent across the application.

---

## Keep Validation Separate from Business Logic

Validation checks whether a request is correctly formed.

Business logic determines whether an operation is allowed.

Example

Validation

```text
Email format is valid.
```

Business Logic

```text
Email is not already registered.
```

Mixing these responsibilities makes the application harder to maintain.

---

## Return Consistent Error Responses

Every validation failure should return the same response structure.

Clients should never receive different error formats for different endpoints.

Consistency improves API usability and simplifies frontend development.

---

## Keep Controllers Lightweight

Controllers should coordinate requests, not validate them.

Good

```text
Request

↓

Validation Middleware

↓

Controller

↓

Service
```

Bad

```text
Controller

↓

Validation

↓

Business Logic
```

Validation belongs in middleware, not inside Controllers.

---

## Infer Types from Validation Schemas

Avoid creating duplicate TypeScript interfaces.

Instead, infer types directly from Zod schemas.

```ts
type CreatePatientRequest = z.infer<typeof createPatientSchema>
```

This keeps validation and TypeScript definitions synchronized.

---

## Write Clear Validation Messages

Validation messages should clearly explain the problem.

Good

```text
Age must be greater than or equal to 18.
```

Bad

```text
Invalid input.
```

Clear messages improve the developer experience for API consumers.

---

## Summary

Following consistent validation practices results in APIs that are easier to maintain, simpler to extend, and more predictable for clients.

By validating every request, separating responsibilities, and reusing validation logic, we create a strong foundation for scalable backend applications.

---

## Interview Discussion

**Question**

> What are the most important validation practices in a production backend?

**Expected Answer**

A production backend should validate every client input, perform validation before business logic, reuse common validation schemas, keep validation separate from business rules, use consistent error responses, and centralize validation through middleware.

# Senior Backend Notes

## Validation Should Be Predictable

Every endpoint should follow the same validation pipeline.

```text
Client

↓

Validation Middleware

↓

Controller

↓

Service

↓

Repository
```

Avoid creating special validation logic for individual endpoints unless absolutely necessary.

---

## Organize Validation by Feature

As the project grows, group validation files by feature instead of placing every schema in a single directory.

Example

```text
validation/

├── patient/

├── doctor/

├── appointment/

└── shared/
```

This keeps related schemas together and makes the project easier to navigate.

---

## Keep Validation Focused

Validation should answer one question:

> "Is this request valid?"

It should not:

- Query the database.
- Call external services.
- Perform business operations.

Keeping validation focused improves testability, readability, and long-term maintainability.

---

# Senior Backend Suggestion

## Create Validation Standards for the Entire Team

As backend teams grow, developers should follow the same validation conventions.

For example:

- Every endpoint has its own validation schema.
- Every request is validated through middleware.
- Every validation error follows the same response format.
- Shared validators are reused whenever possible.
- Business logic is never implemented inside validation schemas.

Documenting these conventions helps maintain consistency across the codebase, simplifies code reviews, and makes onboarding new team members easier.

Well-defined validation standards are a hallmark of mature backend engineering teams.

# 15.7 - Query Parameter Validation

## Overview

Many API endpoints accept additional information through query parameters.

Query parameters are commonly used for:

- Pagination
- Searching
- Sorting
- Filtering

Since query parameters are provided by the client, they should be validated before reaching the application's business logic.

---

## Why Validate Query Parameters?

Consider the following endpoint.

```http
GET /patients?page=1&limit=10
```

A client could also send:

```http
GET /patients?page=-10&limit=10000
```

or

```http
GET /patients?page=abc
```

Without validation:

- Invalid values reach the Controller.
- Database queries become inefficient.
- Unexpected runtime errors may occur.

Validation ensures that only acceptable query values are processed.

---

## Folder Structure

Reusable query validation belongs in the shared validation folder.

```text
src/

validation/

└── shared/

    └── pagination.validation.ts
```

Feature-specific query validation can extend these shared schemas.

---

## Creating a Query Schema

```ts
import { z } from 'zod'

export const paginationSchema = z.object({
  page: z.coerce.number().min(1).default(1),

  limit: z.coerce.number().min(1).max(100).default(10)
})
```

This schema validates:

- `page` must be at least **1**
- `limit` must be between **1** and **100**

---

## Why `z.coerce.number()`?

Query parameters always arrive as strings.

Example

```http
GET /patients?page=1
```

The server receives:

```ts
req.query.page === '1'
```

Using `z.coerce.number()` automatically converts the string into a number before validation.

---

## Using the Schema

```ts
const query = paginationSchema.parse(req.query)
```

If validation succeeds,

```ts
query.page
query.limit
```

are available as validated numbers.

---

## Request Lifecycle

```text

↓

GET /patients?page=1&limit=20

↓

Query Validation

↓

↓

Service

↓

Repository
```

Only validated query parameters continue through the application.

---

## Filtering Example

Additional query parameters can also be validated.

```http
GET /patients?gender=MALE
```

```ts
const filterSchema = z.object({
  gender: z.enum(['MALE', 'FEMALE']).optional()
})
```

---

## Sorting Example

```http
GET /patients?sort=createdAt
```

```ts
const sortSchema = z.object({
  sort: z.enum(['createdAt', 'firstName']).optional()
})
```

Restricting allowed values prevents invalid or unsupported sorting options.

---

## Best Practices

- Validate every query parameter.
- Convert query strings to their expected types.
- Set sensible default values.
- Limit pagination size.
- Restrict filtering and sorting to supported values.

---

## Common Mistakes

❌ Trusting raw query parameters.

❌ Allowing unlimited page sizes.

❌ Forgetting that query values are strings.

❌ Using unsupported sort fields.

❌ Performing manual conversions throughout the codebase.

---

## Summary

Query Parameter Validation ensures that pagination, filtering, and sorting values are correctly formatted before entering the application's business logic.

Validating query parameters results in more reliable APIs and protects the database from inefficient or invalid queries.

---

## Interview Discussion

**Question**

> Why should pagination parameters be validated?

**Expected Answer**

Pagination parameters come from the client and can contain invalid or excessive values. Validation ensures that page numbers and limits remain within acceptable ranges, protecting the application from inefficient database queries and improving API reliability.

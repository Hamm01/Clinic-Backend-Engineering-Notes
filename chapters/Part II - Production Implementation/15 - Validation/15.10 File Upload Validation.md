# 15.10 - File Upload Validation

## Overview

Many backend applications allow users to upload files such as images, documents, and reports.

Before storing an uploaded file, the application should validate that it meets the expected requirements.

File validation protects the application from invalid, oversized, or unsupported files.

---

## Why Validate Uploaded Files?

Consider an endpoint that uploads a patient's medical report.

```http
POST /patients/:id/report
```

Without validation, a client could upload:

- Unsupported file types
- Extremely large files
- Empty files
- Executable files

Accepting these files may waste storage, reduce performance, or introduce security risks.

---

## What Should Be Validated?

Every uploaded file should be checked for:

- File type
- MIME type
- File size
- Required file
- Maximum upload limit

These checks should occur before the file is processed or stored.

---

## Folder Structure

```text
src/

validation/

└── shared/

    └── file.validation.ts
```

Shared file validation rules can be reused across multiple upload endpoints.

---

## Example

Suppose the application accepts only PDF reports.

```text
Allowed File Types

✓ PDF
```

Requests containing other file types should be rejected immediately.

---

## File Size Validation

Applications should also define a maximum upload size.

Example

```text
Maximum File Size

10 MB
```

Files larger than the configured limit should not be processed.

---

## MIME Type Validation

File extensions should never be trusted.

Instead of checking:

```text
report.pdf
```

the application should verify the file's MIME type.

Example

```text
application/pdf
```

This provides more reliable validation than relying only on the filename.

---

## Request Lifecycle

```text
Client

↓

File Upload

↓

File Validation

↓

Controller

↓

Service

↓

Storage
```

Only validated files continue through the upload process.

---

## Best Practices

- Allow only supported file types.
- Limit maximum file size.
- Validate MIME types.
- Reject empty uploads.
- Validate files before storing them.

---

## Common Mistakes

❌ Trusting the file extension.

❌ Allowing unlimited upload sizes.

❌ Accepting every file type.

❌ Storing files before validation.

❌ Forgetting to handle empty uploads.

---

## Summary

File Upload Validation ensures that uploaded files meet the application's requirements before they are processed or stored.

Validating file type, size, and MIME type helps improve security, reliability, and resource management.

---

## Interview Discussion

**Question**

> Why is checking only the file extension not sufficient?

**Expected Answer**

File extensions can be renamed easily. A file named `report.pdf` may not actually be a PDF. Validating the MIME type provides a more reliable way to verify the uploaded file's format before processing or storing it.

# Senior Backend Suggestion

## Keep Upload Validation Independent of Storage

Avoid coupling validation logic to a specific storage provider.

The validation layer should not know whether files are stored in:

- Local Storage
- Amazon S3
- Cloudinary
- Google Cloud Storage

Instead, validate the file first and let the Service layer decide where it should be stored.

```text
Client

↓

File Validation

↓

Controller

↓

Service

↓

Storage Provider
```

This separation makes it easier to change storage providers in the future without modifying validation logic.

It also keeps validation focused on a single responsibility: verifying that uploaded files meet the application's requirements before further processing.

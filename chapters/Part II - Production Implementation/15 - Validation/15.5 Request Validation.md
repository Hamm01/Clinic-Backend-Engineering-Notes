# 15.5 - Request Validation

## Overview

Most API endpoints receive data through the request body.

Before the application processes that data, it should verify that the request matches the expected structure.

Request validation ensures that only correctly formatted data reaches the Controller and Service layers.

In our Clinic Backend, every POST, PUT, and PATCH endpoint validates its request body using Zod.

---

## Why Validate Request Bodies?

Consider the following request.

```json
{
  "firstName": "",
  "email": "invalid-email",
  "age": -5
}
```

Without validation:

- Invalid data reaches the Controller.
- Services execute business logic on bad input.
- Repositories receive malformed objects.
- Database errors become more common.

Instead of handling these problems later, we reject the request immediately.

---

## Folder Structure

Validation schemas are organized by feature.

```text
src/

validation/

└── patient/

    ├── create-patient.validation.ts

    └── update-patient.validation.ts
```

Each API operation owns its own validation schema.

---

## Creating a Validation Schema

Example

```ts
import { z } from 'zod'

export const createPatientSchema = z.object({
  firstName: z.string().min(2),

  lastName: z.string().min(2),

  email: z.email(),

  age: z.number().min(0),

  phone: z.string().optional()
})
```

This schema defines the expected structure for creating a patient.

---

## Valid Request

```json
{
  "firstName": "John",
  "lastName": "Doe",
  "email": "john@example.com",
  "age": 25
}
```

The request passes validation and continues to the Controller.

---

## Invalid Request

```json
{
  "firstName": "",
  "email": "abc",
  "age": -5
}
```

Validation fails because:

- `firstName` is too short.
- `email` is invalid.
- `age` cannot be negative.

The request is rejected before reaching the Controller.

---

## Using the Schema

The incoming request is validated before any business logic executes.

```ts
const request = createPatientSchema.parse(req.body)
```

If validation succeeds,

```ts
request
```

contains trusted data.

If validation fails,

Zod throws a validation error.

---

## Type Inference

The same schema can generate a TypeScript type.

```ts
export type CreatePatientRequest = z.infer<typeof createPatientSchema>
```

Now the validation schema becomes the single source of truth for both runtime validation and compile-time type checking.

---

## Why One Schema Per Endpoint?

Different operations require different validation rules.

Example

Creating a patient

```text
firstName

lastName

email

password
```

Updating a patient

```text
phone

address
```

Instead of one large schema, create a dedicated schema for each API operation.

---

## Best Practices

- Create one schema per endpoint.
- Keep schemas close to their feature.
- Infer TypeScript types from schemas.
- Validate before entering Controllers.
- Reuse common validation rules where appropriate.

---

## Common Mistakes

❌ Using one schema for every endpoint.

❌ Validating inside the Service layer.

❌ Writing manual validation instead of schemas.

❌ Ignoring validation errors.

❌ Duplicating validation logic.

---

## Summary

Request validation ensures that every incoming HTTP request matches the expected structure before entering the application.

By validating request bodies early, Controllers and Services receive trusted data, reducing complexity and improving application reliability.

---

## Interview Discussion

**Question**

> Why should each endpoint have its own validation schema?

**Expected Answer**

Each endpoint has different requirements. A create operation usually requires mandatory fields, while an update operation often allows partial updates. Separate schemas keep validation focused, reusable, and easier to maintain.

# 15.14 - Validation in Our Clinic Backend

## Overview

Throughout this module, we have learned how to validate request bodies, path parameters, query parameters, and uploaded files.

In this chapter, we will see how all of these pieces work together in our Clinic Backend.

The goal is to build a validation layer that is reusable, consistent, and easy to maintain.

## Validation Flow

Every request follows the same pipeline.

```text
Client

↓

Route

↓

Validation Middleware

↓

Controller

↓

Service

↓

Repository

↓

Database

↓

Response
```

Only validated requests continue through the application.

---

## Route Example

```ts
router.post(
  '/patients',

  validate(createPatientSchema, 'body'),

  patientController.create
)

router.get(
  '/patients/:id',

  validate(patientIdSchema, 'params'),

  patientController.findById
)

router.get(
  '/patients',

  validate(paginationSchema, 'query'),

  patientController.findAll
)
```

Every route applies validation before reaching the Controller.

---

## Controller Example

```ts
export const create = async (
  req: Request,

  res: Response
) => {
  const patient = await patientService.create(req.body)

  res.status(201).json(patient)
}
```

The Controller assumes the request has already been validated.

No additional validation logic is required.

---

## Service Example

```ts
export const create = async (data: CreatePatientRequest) => {
  const existingPatient = await patientRepository.findByEmail(data.email)

  if (existingPatient) {
    throw new ConflictException('Email already exists')
  }

  return patientRepository.create(data)
}
```

The Service focuses only on business rules.

---

## Repository Example

```ts
export const create = (data: CreatePatientRequest) => {
  return prisma.patient.create({
    data
  })
}
```

The Repository performs database operations using already validated data.


Every endpoint returns the same validation response format.

## Benefits

Following this architecture provides several advantages.

- Centralized validation.
- Lightweight Controllers.
- Focused Service layer.
- Reusable validation schemas.
- Consistent error responses.
- Reduced duplicate code.
- Easier testing.

---

## Summary

Our Clinic Backend validates every incoming request before it reaches the application's business logic.

By combining Zod schemas, reusable middleware, centralized error handling, and feature-based organization, we create a validation system that is reliable, maintainable, and suitable for production applications.

---

## Interview Discussion

**Question**

> Describe the complete validation flow in your backend application.

**Expected Answer**

Every request first passes through validation middleware, where the request body, path parameters, or query parameters are validated using Zod. If validation succeeds, the Controller coordinates the request, the Service executes business logic, the Repository interacts with the database, and a response is returned. If validation fails, the Global Error Handler returns a standardized `400 Bad Request` response.

# Senior Backend Notes

## Validation Is an Architectural Layer

Validation should be treated as a dedicated layer of the application rather than scattered throughout Controllers and Services.

```text
Client

↓

Validation Layer

↓

Application Layer

↓

Persistence Layer
```

Each layer has a single responsibility, resulting in cleaner and more maintainable code.

---

## Feature-Based Validation Scales Better

Organizing validation by feature keeps related schemas together and reduces the cognitive load when navigating the project.

Example

```text
validation/

├── patient/

├── doctor/

├── appointment/

└── shared/
```

As new modules are added, their validation schemas naturally fit into the existing structure.

---

## Validation Should Be Invisible to Business Logic

After a request passes validation, the Service layer should never need to ask whether the request is structurally correct.

Services should assume they always receive trusted, validated input and focus entirely on implementing business rules.

---

# Senior Backend Suggestion

## Build Validation as a Reusable Foundation

Validation is not a feature that belongs only to the Patient module or the Clinic Backend.

It is infrastructure that every feature can build upon.

Whenever you create a new module, follow the same implementation pattern.

```text
Feature

↓

Validation Schema

↓

Validation Middleware

↓

Controller

↓

Service

↓

Repository
```

Keeping this structure consistent across every module makes the codebase predictable and easier to maintain.

As the application grows, developers can introduce new endpoints by following the same validation pipeline instead of creating custom solutions for each feature.

# 15.13 - Common Mistakes

## Overview

Validation is one of the first layers of a backend application.

When implemented incorrectly, it can lead to duplicate code, inconsistent APIs, security issues, and difficult-to-maintain projects.

This chapter highlights common validation mistakes and how to avoid them.

---

## Mistake 1 - Trusting Client Input

Client input should never be trusted.

Bad

```ts
const patient = await patientService.create(req.body)
```

Good

```ts
const patient = createPatientSchema.parse(req.body)

await patientService.create(patient)
```

Always validate incoming data before using it.

---

## Mistake 2 - Validating Inside Controllers

Controllers should coordinate requests, not validate them.

Bad

```text
Controller

↓

Validation

↓

Service
```

Good

```text
Validation Middleware

↓

Controller

↓

Service
```

Move validation into reusable middleware.

---

## Mistake 3 - Mixing Validation with Business Logic

Validation checks request structure.

Business logic checks whether an operation is allowed.

Bad

```ts
const schema = z
  .object({
    email: z.email()
  })
  .refine(async () => {
    // Database Query ❌
  })
```

Good

```text
Validation

↓

Controller

↓

Service

↓

Database
```

Database checks belong in the Service layer.

---

## Mistake 4 - Repeating Validation Rules

Avoid writing the same validation logic multiple times.

Bad

```ts
z.string().uuid()
```

Repeated in multiple files.

Good

```ts
import { uuidSchema } from '@/validation/shared'
```

Reuse shared validators whenever possible.

---

## Mistake 5 - Using One Schema for Every Endpoint

Each endpoint has different requirements.

Bad

```text
PatientSchema
```

Used for:

- Create Patient
- Update Patient
- Patch Patient

Good

```text
CreatePatientSchema

UpdatePatientSchema

PatchPatientSchema
```

Create dedicated schemas for each operation.

---

## Mistake 6 - Returning Inconsistent Error Responses

Every validation error should follow the same structure.

Bad

```json
{
  "error": "Invalid Email"
}
```

Good

```json
{
  "success": false,
  "message": "Validation failed",
  "errors": [
    {
      "field": "email",
      "message": "Invalid email address"
    }
  ]
}
```

Consistency improves API usability.

---

## Mistake 7 - Ignoring Query and Path Validation

Many developers validate only the request body.

Remember to validate:

- Request Body
- Path Parameters
- Query Parameters
- File Uploads

Every client input should be validated.

---

## Summary

Most validation problems come from placing validation in the wrong layer, duplicating logic, or trusting client input.

By validating every request through middleware and keeping validation separate from business logic, the application becomes easier to maintain and more reliable.

---

## Interview Discussion

**Question**

> What are some common mistakes developers make when implementing validation?

**Expected Answer**

Common mistakes include trusting client input, validating inside Controllers, mixing validation with business logic, duplicating validation rules, using one schema for multiple endpoints, and returning inconsistent error responses.

# Senior Backend Notes

## Think of Validation as a Gateway

A useful mental model is to treat validation as the entry gate to your application.

```text
Client

↓

Validation

↓

Controller

↓

Service

↓

Repository
```

Nothing should pass through this gateway unless it has been validated.

This ensures that every layer after validation works with trusted data.

---

## Small Mistakes Become Large Problems

Skipping validation may seem harmless during development.

As the application grows, these small shortcuts often lead to:

- Duplicate code
- Inconsistent APIs
- Difficult debugging
- Unexpected runtime errors

Consistent validation practices early in the project prevent these issues from accumulating over time.

---

# Senior Backend Suggestion

## Add Validation to Every New Endpoint

A useful team convention is:

> **No endpoint is complete until validation has been implemented.**

Whenever a new API endpoint is created, ensure it includes:

- Validation schema
- Validation middleware
- Consistent error response
- Appropriate request types

Example development workflow:

```text
Create Route

↓

Create Validation Schema

↓

Attach Validation Middleware

↓

Implement Controller

↓

Implement Service

↓

Test Endpoint
```

Following this order helps keep validation as a standard part of API development rather than something added later.

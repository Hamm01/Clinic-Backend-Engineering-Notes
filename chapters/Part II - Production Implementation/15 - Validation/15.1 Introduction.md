# 15.1 - Introduction

## Overview

Every backend application receives data from external sources.

Clients can send valid requests, incomplete data, incorrect formats, or even malicious input. Before our application performs any business logic or stores data in the database, we must verify that the incoming data is correct.

This process is called **validation**.

Validation acts as the first line of defense for the application, ensuring that only valid and expected data enters the system.

In our Clinic Backend, every incoming request is validated before reaching the Controller or Service layer.

---

## Why Validation Exists

Applications should never trust client input.

Consider a patient registration request.

```json
{
  "firstName": "",
  "email": "invalid-email",
  "age": -5
}
```

Without validation, invalid data could:

- Reach the business layer.
- Cause unexpected runtime errors.
- Store incorrect data in the database.
- Produce inconsistent API behavior.

Validation prevents these problems by rejecting invalid requests early.

---

## Where Validation Happens

Validation is the first step after an HTTP request reaches the server.

```text
Client

↓

HTTP Request

↓

Validation

↓

Controller

↓

Service

↓

Repository

↓

Database
```

If validation fails, the request never reaches the business logic.

This keeps Controllers and Services focused on their actual responsibilities instead of checking whether the incoming data is valid.

## Interview Discussion

A common backend interview question is:

> **Why shouldn't validation be performed inside the Service layer?**

A good answer is:

- Validation checks whether incoming data has the correct structure and format.
- Services should focus on business logic.
- Keeping validation separate improves maintainability, reusability, and follows the Single Responsibility Principle.

# 15.8 - Validation Middleware

## Overview

As the number of API endpoints grows, validating requests inside every Controller leads to repetitive code.

Instead of writing validation logic for each endpoint, we can move it into a reusable middleware.

A Validation Middleware validates incoming requests before they reach the Controller, allowing Controllers to focus only on coordinating business logic.

---

## Why Use Validation Middleware?

Imagine every Controller contains validation.

```ts
const request = createPatientSchema.parse(req.body)

return patientService.create(request)
```

Now imagine having:

- 80 API endpoints
- 80 validation calls

The same pattern is repeated throughout the application.

Instead, we validate requests once using middleware.

---

## Creating Validation Middleware

```ts
import { ZodSchema } from 'zod'
import { Request, Response, NextFunction } from 'express'

export const validate = (schema: ZodSchema) => {
  return (req: Request, res: Response, next: NextFunction) => {
    req.body = schema.parse(req.body)

    next()
  }
}
```

The middleware:

- Receives a Zod schema.
- Validates the request body.
- Passes validated data to the next middleware or Controller.

If validation fails, Zod throws an error automatically.

---

## Using Middleware

```ts
router.post(
  '/patients',

  validate(createPatientSchema),

  patientController.create
)
```

The Controller no longer performs validation.

---

## Controller

```ts
export const create = async (
  req: Request,

  res: Response
) => {
  const patient = await patientService.create(req.body)

  res.json(patient)
}
```

The Controller assumes the incoming request is already valid.

## Why This Is Better

Validation Middleware provides several benefits.

- Eliminates duplicate validation code.
- Keeps Controllers lightweight.
- Makes validation reusable.
- Keeps request processing consistent.
- Follows the Single Responsibility Principle.

---

## Best Practices

- Validate requests before Controllers.
- Create one reusable middleware.
- Pass schemas as middleware arguments.
- Keep Controllers free from validation logic.
- Reuse middleware across every route.

---

## Common Mistakes

❌ Calling `schema.parse()` inside every Controller.

❌ Creating different validation middleware for every feature.

❌ Mixing validation with business logic.

❌ Skipping middleware on some endpoints.

❌ Ignoring validation errors.

---

## Summary

Validation Middleware centralizes request validation into a reusable component.

Instead of repeating validation logic in every Controller, each route applies a validation schema before business logic executes, resulting in cleaner Controllers and more maintainable code.

---

## Interview Discussion

**Question**

> Why is validation commonly implemented as middleware in Express applications?

**Expected Answer**

Middleware executes before the Controller, making it the ideal place to validate incoming requests. It removes duplicate validation logic, keeps Controllers focused on request handling, and ensures validation is applied consistently across all endpoints.

# Senior Backend Suggestion

## Build One Generic Validation Middleware

The middleware shown in this chapter validates only the request body.

As your application grows, you'll also need to validate:

- Request Body
- Path Parameters
- Query Parameters
- Headers (optional)

Instead of creating separate middleware for each request type, build a single reusable middleware that accepts both the validation schema and the request section to validate.

Example

```ts
validate(createPatientSchema, 'body')

validate(patientIdSchema, 'params')

validate(paginationSchema, 'query')
```

This allows every route to reuse the same middleware while validating different parts of the incoming request.

Example

```ts
router.post(
  '/patients',

  validate(createPatientSchema, 'body'),

  patientController.create
)

router.get(
  '/patients/:id',

  validate(patientIdSchema, 'params'),

  patientController.findById
)

router.get(
  '/patients',

  validate(paginationSchema, 'query'),

  patientController.findAll
)
```

Using a single middleware for all request validation keeps the codebase consistent, reduces duplication, and scales well as new endpoints are added.

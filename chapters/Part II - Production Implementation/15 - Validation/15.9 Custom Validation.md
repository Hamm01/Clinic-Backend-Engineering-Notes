# 15.9 - Custom Validation

## Overview

Built-in validators handle common rules such as required fields, email formats, and string lengths.

However, real-world applications often require additional validation based on business requirements.

Custom validation allows us to enforce these rules while keeping validation logic organized and reusable.

---

## Why Custom Validation?

Some validations cannot be expressed using basic validators.

Examples include:

- Password confirmation
- Appointment date cannot be in the past
- Age must be at least 18 years
- Start date must be before end date

These rules require comparing values or applying custom logic.

---

## Folder Structure

```text
src/

validation/

├── patient/

├── doctor/

├── appointment/

└── shared/
```

Each feature owns its own custom validation rules.

---

## Example 1 - Password Confirmation

```ts
import { z } from 'zod'

export const registerSchema = z
  .object({
    password: z.string().min(8),

    confirmPassword: z.string()
  })
  .refine(
    data => data.password === data.confirmPassword,

    {
      message: 'Passwords do not match',

      path: ['confirmPassword']
    }
  )
```

The validation succeeds only when both passwords match.

---

## Example 2 - Age Validation

```ts
const patientSchema = z
  .object({
    age: z.number()
  })
  .refine(
    data => data.age >= 18,

    {
      message: 'Patient must be at least 18 years old',

      path: ['age']
    }
  )
```

This adds a business-specific validation rule.

---

## Example 3 - Appointment Date

```ts
const appointmentSchema = z
  .object({
    appointmentDate: z.date()
  })
  .refine(
    data => data.appointmentDate > new Date(),

    {
      message: 'Appointment date must be in the future',

      path: ['appointmentDate']
    }
  )
```

Past appointment dates are rejected.

---

## What Should NOT Be Custom Validation?

Some checks require database access.

Examples

- Email already exists
- Username already exists
- Doctor belongs to the clinic
- Appointment slot already booked

These are **business rules**, not validation rules.

They belong inside the Service layer.

---

## Validation vs Business Logic

```text
Validation

↓

Request Structure

↓

Controller

↓

Service

↓

Business Rules

↓

Repository
```

Validation ensures the request is correctly formed.

The Service determines whether the requested operation is allowed.

---

## Best Practices

- Use custom validation for request data only.
- Keep validation independent of the database.
- Return meaningful validation messages.
- Keep business rules inside Services.
- Reuse custom validation where possible.

---

## Common Mistakes

❌ Querying the database inside validation.

❌ Mixing business rules with request validation.

❌ Writing complex validation inside Controllers.

❌ Creating duplicate custom validation logic.

❌ Ignoring validation error messages.

---

## Summary

Custom validation extends basic validation rules to support real-world application requirements.

It helps validate relationships between request fields while keeping validation separate from business logic and database operations.

---

## Interview Discussion

**Question**

> Should checking whether an email already exists be done using Zod?

**Expected Answer**

No.

Zod validates the structure and format of incoming data.

Checking whether an email already exists requires a database query and is part of the application's business logic. That responsibility belongs to the Service layer.

# Senior Backend Notes

## Keep Validation Pure

A validation schema should never communicate with external systems.

Avoid performing:

- Database queries
- API calls
- Redis lookups
- File system operations

inside validation schemas.

Validation should only inspect the incoming request.

---

## Separate Validation from Business Rules

Think of validation as answering one question:

> "Is this request correctly formed?"

Business logic answers a different question:

> "Can this operation be performed?"

Example

Validation

```text
Email format is valid.
```

Business Logic

```text
Email is not already registered.
```

Keeping these responsibilities separate results in a cleaner and more maintainable architecture.

---

## Real Project Examples

Good candidates for custom validation:

- Password confirmation
- Date comparisons
- Age restrictions
- Numeric ranges
- Conditional fields

Poor candidates for custom validation:

- Unique email
- Available appointment slot
- Existing patient record
- Doctor permissions

These require business logic and should remain inside the Service layer.

# Senior Backend Suggestion

## Create a Shared Validation Utilities Folder

As your application grows, you'll encounter validation rules that are reused across multiple features.

Instead of rewriting them, extract reusable helpers.

```text
src/

validation/

shared/

├── validators/

│   ├── password.validator.ts

│   ├── date.validator.ts

│   ├── phone.validator.ts

│   └── age.validator.ts

├── common.validation.ts

└── index.ts
```

Feature-specific schemas can then compose these reusable validators.

Example

```ts
const registerSchema = z.object({
  password: passwordSchema,

  phone: phoneSchema
})
```

This keeps validation rules centralized, avoids duplication, and makes maintenance easier as the project grows.

For large backend applications with many modules, shared validation utilities become an important part of maintaining consistency across the codebase.

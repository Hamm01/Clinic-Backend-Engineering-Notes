# 15.4 - Zod Basics

## Overview

Zod is a TypeScript-first validation library used to validate incoming data at runtime.

It allows us to define a schema once and use it for both:

- Runtime validation
- TypeScript type inference

In our Clinic Backend, every request is validated using Zod before it reaches the Controller.

---

## Why Zod?

Without Zod, we would manually validate every request.

Example

```ts
if (!req.body.firstName) {
  throw new Error('First name is required')
}

if (typeof req.body.age !== 'number') {
  throw new Error('Age must be a number')
}

if (!req.body.email.includes('@')) {
  throw new Error('Invalid email')
}
```

As applications grow, this approach becomes repetitive and difficult to maintain.

Zod allows us to define validation rules in a single schema.

```bash
npm install zod
```

## Creating Your First Schema

A schema describes the expected shape of incoming data.

```ts
import { z } from 'zod'

const patientSchema = z.object({
  firstName: z.string(),

  lastName: z.string(),

  age: z.number()
})
```

This schema expects:

- `firstName` → string
- `lastName` → string
- `age` → number

---

## Common Data Types

These are the Zod types we will use most frequently.

```ts
z.string()

z.number()

z.boolean()

z.date()

z.array()

z.object()

z.enum()
```

Each type validates that the incoming value matches the expected data type.

---

## Adding Validation Rules

Schemas can define additional constraints.

```ts
const patientSchema = z.object({
  firstName: z.string().min(2),

  lastName: z.string().min(2),

  age: z.number().min(0)
})
```

These rules ensure:

- Names contain at least two characters.
- Age cannot be negative.

---

## Optional Fields

Not every field is required.

```ts
const patientSchema = z.object({
  firstName: z.string(),

  phone: z.string().optional()
})
```

If `phone` is missing, validation still succeeds.

---

## Email Validation

Zod provides built-in validation for common formats.

```ts
const patientSchema = z.object({
  email: z.email()
})
```

Only valid email addresses are accepted.

---

## Parsing Data

Validation happens using `parse()`.

```ts
const patient = patientSchema.parse(req.body)
```

If the request is valid:

```ts
patient
```

contains the validated data.

If validation fails, Zod throws an error.

---

## Type Inference

One of Zod's biggest advantages is automatic TypeScript type generation.

```ts
type CreatePatientRequest = z.infer<typeof patientSchema>
```

Now both validation and TypeScript use the same schema.

There is no need to create duplicate interfaces.

---

## Where We Use Zod

Throughout our Clinic Backend, Zod will validate:

- Request Body
- Route Parameters
- Query Parameters
- File Metadata
- Shared Validation Schemas

Every incoming request follows the same pattern.

```text
HTTP Request

↓

Zod Schema

↓

Validated Data

↓

Controller
```

---

## Best Practices

- Create one schema per API operation.
- Keep schemas close to the feature they validate.
- Reuse shared validation schemas.
- Infer TypeScript types from Zod.
- Keep validation separate from business logic.

---

## Common Mistakes

❌ Writing manual validation instead of schemas.

❌ Creating duplicate TypeScript interfaces.

❌ Using one schema for every endpoint.

❌ Mixing validation with business logic.

❌ Ignoring validation errors.

---

## Summary

Zod provides a clean and maintainable way to validate incoming data.

By defining a schema once, we gain both runtime validation and TypeScript type safety, reducing duplication and improving consistency across the application.

## Interview Discussion

**Question**

> Why is Zod a good choice for TypeScript backends?

**Expected Answer**

Zod combines runtime validation with TypeScript type inference. A single schema validates incoming data and generates TypeScript types, eliminating duplicate interfaces and ensuring consistency between compile-time and runtime.

# 15.2 - Why Validation Matters

## Overview

Validation is more than checking whether a field is empty.

It protects the application from invalid data, unexpected behavior, security issues, and database inconsistencies.

Without validation, every layer of the application must assume that incoming data may be incorrect, making the system harder to maintain and more prone to errors.

---

## The Problem Without Validation

Imagine a client sends the following request.

```json
{
  "firstName": "",
  "email": "abc",
  "age": -10
}
```

Without validation:

- The Controller receives invalid data.
- The Service executes business logic on incorrect values.
- The Repository attempts to save bad data.
- The Database may reject the request or store inconsistent information.

The error is discovered much later than necessary.

---

## Validation Stops Problems Early

Instead of allowing invalid data to move through the application, validation rejects it immediately.

```text
Client

↓

HTTP Request

↓

Validation ❌

↓

400 Bad Request
```

The request never reaches the Controller, Service, or Database.

This keeps the rest of the application clean and predictable.

---

## Problems Validation Prevents

Validation helps prevent several common issues.

### Invalid Data

Example

```json
{
  "email": "not-an-email"
}
```

Validation ensures the data matches the expected format before it is processed.

---

### Missing Required Fields

Example

```json
{
  "lastName": "Singh"
}
```

If `firstName` is required, validation rejects the request instead of allowing incomplete records.

---

### Incorrect Data Types

Example

```json
{
  "age": "twenty"
}
```

The API expects a number, not a string.

Validation detects this before business logic executes.

---

### Database Errors

Suppose the database requires a unique email.

Instead of relying on the database to detect invalid requests, validation ensures the request structure is correct before reaching the persistence layer.

Business rules such as checking whether an email already exists are handled later in the Service layer.

---

### Security

Validation also reduces the application's attack surface.

By accepting only expected input, the backend becomes more resistant to malformed requests and unexpected payloads.

Validation is one part of building secure APIs, alongside authentication, authorization, and database protections.

---

## Validation vs Business Rules

Validation checks whether the request has the correct structure and format.

Business rules determine whether the requested operation is allowed.

Example

Validation

```text
Is the email valid?
```

Business Rule

```text
Does this email already belong to another patient?
```

These responsibilities should remain separate.

---

## Benefits of Validation

Validation provides several advantages.

- Prevents invalid data from entering the system.
- Keeps Controllers simple.
- Allows Services to focus on business logic.
- Reduces runtime errors.
- Improves API reliability.
- Produces consistent error responses.

---

## Best Practices

- Validate every incoming request.
- Reject invalid data as early as possible.
- Keep validation separate from business logic.
- Return meaningful validation errors.
- Apply validation consistently across every endpoint.

---

## Common Mistakes

❌ Trusting client input.

❌ Performing validation inside the Service layer.

❌ Waiting for database errors instead of validating requests.

❌ Mixing validation rules with business logic.

❌ Returning inconsistent validation errors.

---

## Summary

Validation protects the application by ensuring that only correctly structured data enters the system.

By validating requests before business logic executes, we create APIs that are more reliable, maintainable, and easier to debug.

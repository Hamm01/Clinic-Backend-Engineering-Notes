# 15.11 - Error Responses

## Overview

When validation fails, the API should return a clear and consistent error response.

A standardized error format helps:

- Frontend applications display errors.
- Developers debug requests.
- APIs remain predictable.
- Clients integrate more easily.

Every validation failure in our Clinic Backend follows the same response structure.

---

## Why Standardize Error Responses?

Imagine two endpoints returning different validation responses.

Endpoint A

```json
{
  "message": "Invalid email"
}
```

Endpoint B

```json
{
  "error": {
    "field": "email"
  }
}
```

The frontend must handle multiple response formats.

Instead, every validation error should follow one consistent structure.

---

## Standard Response Format

```json
{
  "success": false,
  "message": "Validation failed",
  "errors": [
    {
      "field": "email",
      "message": "Invalid email address"
    }
  ]
}
```

This format is easy to understand and can be reused across every endpoint.

---

## Multiple Validation Errors

A request may contain more than one validation error.

Example

```json
{
  "firstName": "",
  "email": "abc",
  "age": -5
}
```

Response

```json
{
  "success": false,
  "message": "Validation failed",
  "errors": [
    {
      "field": "firstName",
      "message": "First name is required"
    },
    {
      "field": "email",
      "message": "Invalid email address"
    },
    {
      "field": "age",
      "message": "Age must be greater than or equal to 0"
    }
  ]
}
```

Returning every validation error allows the client to correct all issues in a single request.

---

## Error Response Flow

```text
Client

↓

HTTP Request

↓

Validation

↓

Validation Error

↓

Global Error Handler

↓

400 Bad Request
```

The validation layer reports the error.

The Global Error Handler formats the response before sending it back to the client.

---

## HTTP Status Code

Validation failures should return:

```http
400 Bad Request
```

The request reached the server successfully, but the provided data was invalid.

---

## Keep Error Messages Helpful

Good

```text
Email must be a valid email address.
```

Bad

```text
Invalid input.
```

Clear error messages help API consumers quickly identify and fix problems.

---

## Best Practices

- Use one error response format.
- Return all validation errors.
- Include the field name.
- Write clear error messages.
- Return the correct HTTP status code.

---

## Common Mistakes

❌ Returning different error formats across endpoints.

❌ Returning only the first validation error.

❌ Exposing internal implementation details.

❌ Using generic error messages.

❌ Returning HTTP 500 for validation failures.

---

## Summary

Consistent validation error responses make APIs easier to consume, easier to debug, and simpler to maintain.

Every endpoint should return the same error structure whenever validation fails.

---

## Interview Discussion

**Question**

> Why should APIs use a standardized validation error format?

**Expected Answer**

A consistent error format makes APIs predictable for clients. Frontend applications can handle validation errors uniformly, reducing duplicate error-handling logic and improving the overall developer experience.

# Senior Backend Notes

## Use a Global Error Handler

Validation middleware should not manually build HTTP responses.

Instead, allow validation errors to propagate to a centralized Global Error Handler.

```text
Validation Middleware

↓

Throw Validation Error

↓

Global Error Handler

↓

HTTP Response
```

This keeps validation focused only on detecting invalid input, while the error handler is responsible for formatting API responses.

---

## Keep Error Responses Consistent

Every API should return validation errors using the same structure.

A predictable response format allows frontend applications to implement a single error-handling strategy instead of supporting multiple response formats.

---

## Never Expose Internal Errors

Validation responses should only include information that helps the client fix the request.

Avoid exposing:

- Stack traces
- Internal file paths
- Database errors
- Framework-specific implementation details

Clients should receive only the information necessary to correct their input.

---

# Senior Backend Suggestion

## Create a Shared Error Response Builder

Instead of formatting validation errors in multiple places, centralize the response structure.

Example

```text
src/

utils/

└── error-response.ts
```

This utility can generate consistent validation responses for every endpoint.

Benefits include:

- Consistent API responses
- Reduced duplicate code
- Easier maintenance
- Improved frontend integration

As the application grows, the same response builder can also be used for authentication, authorization, and business logic errors, ensuring that every API response follows a unified format.

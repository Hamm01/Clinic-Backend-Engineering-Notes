# 15.6 - Path Parameter Validation

## Overview

Not all client data is sent in the request body.

Many API endpoints receive data through the URL path, such as resource identifiers.

These values are called **Path Parameters** and should be validated before the application attempts to process them.

In our Clinic Backend, every route parameter is validated before reaching the Controller.

---

## Why Validate Path Parameters?

Consider the following endpoint.

```http
GET /patients/:id
```

A valid request might be:

```http
GET /patients/clz7f9n5m0000abc123xyz
```

However, a client could also send:

```http
GET /patients/abc
```

or

```http
GET /patients/!!!
```

Without validation:

- Invalid IDs reach the Controller.
- Database queries are executed unnecessarily.
- Errors become harder to debug.

Validation rejects invalid identifiers before any business logic executes.

---

## Folder Structure

Shared parameter validation can be reused across multiple features.

```text
src/

validation/

└── shared/

    └── id.validation.ts
```

A single validation schema can be imported by Patients, Doctors, Appointments, and other modules.

---

## Creating a Parameter Schema

Example

```ts
import { z } from 'zod'

export const idParamSchema = z.object({
  id: z.string().min(1)
})
```

This schema validates that the route contains an `id` parameter.

> **Note:** In a real project, you should validate the format of the identifier (such as CUID, UUID, or MongoDB ObjectId) instead of checking only that it is a non-empty string.

---

## Using the Schema

Validate the route parameters.

```ts
const params = idParamSchema.parse(req.params)
```

If validation succeeds:

```ts
params.id
```

contains a trusted identifier.

If validation fails, Zod throws a validation error.

## Reusing Parameter Validation

Most modules require an identifier.

Instead of creating duplicate schemas:

```text
Patient

Doctor

Appointment

Diagnosis
```

all modules can reuse the same shared parameter validation.

This reduces duplication and keeps validation consistent across the project.

---

## Best Practices

- Validate every route parameter.
- Reuse common parameter schemas.
- Validate identifiers before database queries.
- Keep parameter validation separate from business logic.
- Return consistent validation errors.

---

## Common Mistakes

❌ Assuming route parameters are always valid.

❌ Querying the database before validating the identifier.

❌ Creating duplicate parameter schemas.

❌ Mixing parameter validation with business logic.

❌ Ignoring malformed identifiers.

---

## Summary

Path Parameter Validation ensures that identifiers received through the URL are correctly structured before reaching the application's business logic.

Validating route parameters early reduces unnecessary database operations and improves the reliability of the API.

---

## Interview Discussion

**Question**

> Why should route parameters be validated before querying the database?

**Expected Answer**

Route parameters are user input. Validating them before database access prevents unnecessary queries, reduces invalid requests reaching the data layer, and allows the application to return clear and consistent error responses.

# Senior Backend Notes

## Reusable Validation Schemas

As applications grow, many API endpoints validate the same types of identifiers.

Instead of defining validation rules repeatedly, create reusable validation schemas inside a shared folder.

```text
src/

validation/

shared/

├── id.validation.ts

├── uuid.validation.ts

├── cuid.validation.ts

└── pagination.validation.ts
```

This keeps validation logic centralized and avoids duplication across different features.

---

## Example

Instead of writing the validation rule multiple times,

```ts
z.object({
  id: z.string().min(1)
})
```

create reusable validators.

```ts
import { z } from 'zod'

export const cuidSchema = z.string().cuid2()

export const uuidSchema = z.string().uuid()
```

Now feature-specific validation schemas become much cleaner.

```ts
import { z } from 'zod'
import { cuidSchema } from '@/validation/shared/cuid.validation'

export const patientIdParamSchema = z.object({
  id: cuidSchema
})
```

The same validator can now be reused for:

- Patients
- Doctors
- Appointments
- Diagnoses
- Prescriptions

---

## Why This Approach?

Imagine your project has 50 endpoints that validate an identifier.

If the validation rule changes, updating every schema individually becomes time-consuming and error-prone.

By centralizing identifier validation, you only update one file, and every endpoint immediately benefits from the change.

---

## Real Project Structure

```text
src/

validation/

├── patient/

│   ├── create-patient.validation.ts

│   ├── update-patient.validation.ts

│   └── index.ts

│

├── doctor/

├── appointment/

├── shared/

│   ├── cuid.validation.ts

│   ├── uuid.validation.ts

│   ├── pagination.validation.ts

│   └── common.validation.ts

└── index.ts
```

This feature-first structure keeps validation organized while allowing commonly used schemas to be shared across the entire application.

---

## Production Recommendation

As a project grows, many validation rules become reusable.

Examples include:

- Resource identifiers
- Pagination parameters
- Sorting options
- Search queries
- Date ranges

Keeping these validators inside a shared directory improves consistency, reduces duplication, and makes future maintenance significantly easier.

# 15.3 - Validation Flow

## Overview

Validation is not just a library or a function call.

It is a step in the request lifecycle that ensures only valid data enters the application.

In a production backend, validation happens before business logic executes. This allows every layer after validation to assume that the incoming data is already well-formed.

Understanding this flow is essential for designing clean and maintainable backend systems.

---

## Complete Validation Flow

Every request in our Clinic Backend follows the same lifecycle.

```text
Client

↓

HTTP Request

↓

Validation

↓

Controller

↓

Service

↓

Repository

↓

Database

↓

HTTP Response
```

Validation acts as a gatekeeper between the client and the application.

---

## Step 1 - Client Sends Request

Every request begins with the client.

Example

```http
POST /patients
```

```json
{
  "firstName": "John",
  "lastName": "Doe",
  "email": "john@example.com"
}
```

At this point, the server has no idea whether the data is correct.

---

## Step 2 - Validation

The incoming request is validated.

Validation checks:

- Required fields
- Data types
- String lengths
- Email format
- Number ranges
- Enum values

If validation fails:

```text
Client

↓

Validation ❌

↓

400 Bad Request
```

The request stops immediately.

Nothing else executes.

---

## Step 3 - Controller

If validation succeeds, the Controller receives trusted input.

```text
Client

↓

Validation ✓

↓

Controller
```

The Controller no longer needs to check:

- Is the email valid?
- Is firstName empty?
- Is age a number?

Those checks have already been completed.

The Controller simply coordinates the request.

---

## Step 4 - Service

The Service focuses only on business logic.

Examples

- Is the email already registered?
- Does the doctor belong to this clinic?
- Is the appointment slot available?

These are business rules, not validation rules.

---

## Step 5 - Repository

The Repository receives clean business data.

It performs database operations without worrying about malformed requests.

```text
Service

↓

Repository

↓

PostgreSQL
```

---

## Why Validate Early?

Imagine validation happened inside the Service.

```text
Controller

↓

Service

↓

Validation
```

Every Service method would need to repeat validation logic.

Controllers would pass invalid data deeper into the application.

Repositories might receive malformed objects.

This increases complexity and duplication.

Instead, validate once at the application's entry point.

---

## Structural Validation vs Business Validation

These are different responsibilities.

### Structural Validation

Checks whether the request is correctly formed.

Examples

- Email format
- Required fields
- String length
- Data types

↓

Validation Layer

---

### Business Validation

Checks whether the requested operation is allowed.

Examples

- Email already exists.
- Appointment overlaps another booking.
- Doctor belongs to the selected clinic.
- Patient account is active.

↓

Service Layer

Keeping these responsibilities separate makes the code easier to understand and maintain.

---

## Benefits of This Flow

Following this validation flow provides several advantages.

- Invalid requests stop immediately.
- Controllers remain lightweight.
- Services focus on business rules.
- Repositories receive trusted data.
- Database errors are reduced.
- Validation logic is reusable.

---

## Best Practices

- Validate every incoming request.
- Validate before reaching Controllers.
- Keep Controllers free from validation logic.
- Keep business rules inside Services.
- Reject invalid requests early.

---

## Common Mistakes

❌ Validating inside Controllers.

❌ Validating inside Services.

❌ Allowing invalid requests to reach Repositories.

❌ Mixing business rules with structural validation.

❌ Trusting client input without validation.

---

## Summary

Validation is the application's first line of defense.

By validating requests before Controllers and Services execute, every layer receives clean and predictable data, resulting in a simpler, safer, and more maintainable backend architecture.

---

## Interview Discussion

**Question**

> Why should validation happen before the Controller?

**Expected Answer**

Validation belongs at the application's entry point because Controllers should coordinate requests, not verify data. Early validation prevents invalid requests from reaching the Service and Repository layers, reduces duplicated checks, and allows the rest of the application to assume the incoming data is already valid.

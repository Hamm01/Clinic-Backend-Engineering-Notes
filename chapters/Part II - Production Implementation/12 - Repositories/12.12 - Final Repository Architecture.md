---
# Ignoring Pagination

Another common Repository mistake is returning every matching record from the database.

During development,

the database usually contains only a small amount of data.

As a result,

queries appear fast,

and developers often overlook pagination.

In production,

however,

tables may contain millions of records.

Returning every row becomes inefficient and can significantly impact application performance.
---

# Incorrect Implementation

Consider a Repository that retrieves all patients.

## Code

```ts
async findAll(): Promise<Patient[]> {

    return prisma.patient.findMany();

}
```

---

# What's Wrong?

This query has no limit.

```
Database

↓

Patient Table

↓

Return Every Record
```

As the number of patients grows,

the Repository returns increasingly larger result sets.

---

# Why Is This a Problem?

Returning every record affects multiple parts of the application.

The database performs more work.

↓

More data is transferred across the network.

↓

Node.js allocates more memory.

↓

The API response becomes larger.

↓

The client takes longer to process the response.

A simple query can become a major performance bottleneck.

---

# Business Example

The clinic has been operating for eight years.

The Patient table now contains:

```
Patient

↓

2,500,000 Records
```

The receptionist opens the patient search page.

The Repository returns every patient.

```
2,500,000 Patients

↓

Network

↓

Application

↓

Browser
```

The user only needs the first page,

yet the application retrieves the entire dataset.

---

# Correct Implementation

Repositories returning collections should support pagination.

## Code

```ts
async findAll(
    page: number,
    limit: number
): Promise<Patient[]> {

    return prisma.patient.findMany({

        skip: (page - 1) * limit,

        take: limit,

        orderBy: {

            createdAt: "desc"

        }

    });

}
```

---

# Explanation

Instead of retrieving every patient,

the Repository retrieves only the requested page.

```
Database

↓

Patient Table

↓

20 Records

↓

Application
```

The amount of work performed by both the database and the application is significantly reduced.

---

# Repository Responsibility

Whenever a Repository returns a collection,

consider whether the collection can continue growing over time.

Examples include:

- Patients
- Doctors
- Appointments
- Diagnoses
- Prescriptions

If the answer is yes,

the Repository should support pagination.

---

# Common Mistakes

## Returning Entire Tables

Avoid Repository methods that retrieve every record without a clear reason.

---

## Leaving Pagination to the Client

The database should return only the required records.

Clients should not receive thousands of unnecessary objects.

---

## Forgetting Ordering

Pagination without a consistent ordering may produce inconsistent results between requests.

Always combine pagination with an appropriate `orderBy`.

---

# Production Notes

Production applications almost never expose unbounded collection queries.

Instead,

Repositories provide paginated methods that:

- Limit returned rows.
- Reduce database load.
- Lower memory consumption.
- Improve response times.
- Scale as datasets grow.

Pagination should be considered the default behavior for Repository methods returning collections.

---

# Summary

Ignoring pagination may not cause problems during development,

but it becomes a serious scalability issue as data grows.

Repositories should return only the records required for the current request, ensuring efficient database access and predictable application performance.

# Repository Folder Structure

A production Repository layer should be organized in a way that makes it easy to locate, understand, and extend database access logic.

Instead of placing every Repository inside a single file,

each business entity should own its own Repository.

The Repository layer should also separate reusable infrastructure from business-specific implementations.

---

# Complete Folder Structure

```text
src/

├── controllers/
│
├── services/
│
├── repositories/
│   │
│   ├── doctor.repository.ts
│   ├── patient.repository.ts
│   ├── appointment.repository.ts
│   ├── diagnosis.repository.ts
│   │
│   ├── interfaces/
│   │   ├── doctor-repository.interface.ts
│   │   ├── patient-repository.interface.ts
│   │   ├── appointment-repository.interface.ts
│   │   └── diagnosis-repository.interface.ts
│   │
│   └── index.ts
│
├── prisma/
│
├── database/
│
└── app.ts
```

This structure keeps every Repository close to its business entity while providing a central location for database access.

---

# Why Organize Repositories This Way?

As applications grow,

the number of Repository methods also increases.

Keeping everything inside one large file quickly becomes difficult to maintain.

Instead,

organize Repositories by responsibility.

```
Doctor

↓

DoctorRepository

--------------------

Patient

↓

PatientRepository

--------------------

Appointment

↓

AppointmentRepository

--------------------

Diagnosis

↓

DiagnosisRepository
```

Each Repository owns the persistence logic for one business entity.

---

# Repository Responsibilities

Each Repository should answer questions related only to its entity.

Example

```
PatientRepository

↓

findById()

↓

findByEmail()

↓

findByPhoneNumber()

↓

create()

↓

update()
```

The Patient Repository should not retrieve doctor information or manage appointments.

Its responsibility is limited to patient persistence.

---

# Repository Interfaces

Large applications often define interfaces for each Repository.

Example

```ts
export interface PatientRepository {
  findById(id: string): Promise<Patient | null>

  create(data: CreatePatientDto): Promise<Patient>

  update(id: string, data: UpdatePatientDto): Promise<Patient>
}
```

Interfaces define the contract.

Concrete Repository implementations provide the implementation.

This separation improves testing and flexibility.

---

# Repository Index File

Instead of importing every Repository individually,

an index file can centralize exports.

Example

```ts
export * from './doctor.repository'
export * from './patient.repository'
export * from './appointment.repository'
export * from './diagnosis.repository'
```

This simplifies imports throughout the application.

---

# Dependency Flow

Repositories should never depend on Controllers or Services.

The dependency direction always flows downward.

```
Controller

↓

Service

↓

Repository

↓

Prisma

↓

Database
```

Keeping dependencies one-directional prevents circular references and maintains a clean architecture.

---

# Business Example

When a new feature is introduced,

such as managing prescriptions,

the architecture naturally expands.

```
repositories/

↓

prescription.repository.ts
```

No existing Repository requires modification.

The new Repository follows the same conventions,

making the architecture predictable and easy to extend.

---

# Benefits

Organizing Repositories in this manner provides:

- Clear ownership.
- Easy navigation.
- Better scalability.
- Reduced coupling.
- Improved maintainability.
- Consistent project structure.

Every engineer joining the project knows exactly where database logic belongs.

---

# Production Notes

Production backend teams prioritize consistency over creativity.

Every Repository follows the same structure,

naming convention,

and responsibility.

A predictable folder structure reduces onboarding time,

simplifies code reviews,

and makes long-term maintenance significantly easier.

---

# Summary

A well-organized Repository folder structure keeps database access clean, modular, and scalable.

Each Repository owns one business entity,

interfaces define contracts,

and dependencies always flow from Controllers to Services to Repositories before reaching the database.

# Complete Request Lifecycle

Understanding individual layers is important.

Understanding how those layers collaborate is even more important.

Every HTTP request follows the same journey through the application.

Once this flow is understood,

the architecture becomes predictable and easy to reason about.

---

# The Complete Flow

Consider the following request.

```
GET /patients/123
```

The request travels through every layer before reaching the database.

```
Client

↓

HTTP Request

↓

Controller

↓

Service

↓

Repository

↓

Prisma Client

↓

PostgreSQL

↓

Prisma Client

↓

Repository

↓

Service

↓

Controller

↓

HTTP Response

↓

Client
```

Every layer performs exactly one responsibility.

---

# Step 1 — Client Sends Request

A client application sends an HTTP request.

```
GET /patients/123
```

The client does not know:

- Business rules
- Database structure
- SQL
- Prisma

It only knows how to communicate using HTTP.

---

# Step 2 — Controller Receives Request

The Controller receives the request.

Example

```ts
@Get(":id")
async getPatient(

    @Param("id") id: string

) {

    return this.patientService.getPatient(id);

}
```

The Controller is responsible for:

- Receiving requests.
- Extracting parameters.
- Calling the appropriate Service.
- Returning the response.

The Controller does not access the database.

---

# Step 3 — Service Executes Business Logic

The Service receives the request from the Controller.

Example

```ts
async getPatient(id: string) {

    const patient = await this.patientRepository.findById(id);

    if (!patient) {

        throw new NotFoundException();

    }

    return patient;

}
```

The Service decides:

- What should happen.
- Which Repositories are required.
- Whether validation is necessary.
- Whether a transaction should begin.

The Service never communicates directly with Prisma.

---

# Step 4 — Repository Accesses the Database

The Repository translates the business request into a database query.

Example

```ts
async findById(id: string) {

    return prisma.patient.findUnique({

        where: {

            id

        }

    });

}
```

The Repository is responsible only for persistence.

It knows nothing about HTTP,

Controllers,

or business workflows.

---

# Step 5 — Prisma Generates SQL

Prisma converts the Repository query into SQL.

```
Repository

↓

Prisma

↓

Generated SQL
```

Example

```sql
SELECT *

FROM "Patient"

WHERE id = $1

LIMIT 1;
```

The database executes SQL,

not TypeScript.

---

# Step 6 — Database Executes Query

PostgreSQL executes the SQL.

```
SQL

↓

Query Planner

↓

Indexes

↓

Storage Engine

↓

Result
```

The database retrieves the requested data.

---

# Step 7 — Result Travels Back

The database returns the result.

```
Database

↓

Prisma

↓

Repository

↓

Service

↓

Controller

↓

HTTP Response

↓

Client
```

Each layer performs its own work before forwarding the result.

---

# Responsibility of Each Layer

```
Controller

↓

HTTP Communication

-------------------------

Service

↓

Business Logic

-------------------------

Repository

↓

Database Access

-------------------------

Prisma

↓

SQL Generation

-------------------------

Database

↓

Data Storage
```

Each layer has one responsibility.

No layer performs another layer's work.

---

# Why This Architecture Works

Consider a new business requirement.

The clinic now blocks inactive patients.

Only the Service changes.

```
Controller

✓ No Changes

↓

Service

✓ Business Rule Added

↓

Repository

✓ No Changes

↓

Database

✓ No Changes
```

Because responsibilities are separated,

changes remain isolated.

---

# Benefits

Following this request lifecycle provides:

- Predictable architecture.
- Easier debugging.
- Better testing.
- Lower coupling.
- Higher maintainability.
- Easier onboarding.

Every request follows exactly the same path.

---

# Production Notes

Production applications process thousands of requests every minute.

Keeping every request on the same architectural path makes the application easier to monitor,

debug,

optimize,

and extend.

Whenever an engineer investigates an issue,

they already know where to look.

The architecture itself becomes documentation.

---

# Summary

Every HTTP request should flow through the application in a consistent manner.

Controllers handle HTTP,

Services implement business rules,

Repositories access the database,

Prisma generates SQL,

and the database persists the data.

Maintaining this flow keeps responsibilities clear and allows each layer to evolve independently.

# Dependency Flow

A well-designed backend architecture is built on controlled dependencies.

Each layer is allowed to depend only on the layer directly below it.

This creates a predictable architecture that is easy to understand, test, and maintain.

Dependencies should always flow in one direction.

---

# Dependency Direction

The application should follow this dependency chain.

```
Controller

↓

Service

↓

Repository

↓

Prisma

↓

Database
```

Every layer depends only on the next layer.

No layer should skip responsibilities or access lower layers directly.

---

# Why Dependency Direction Matters

Imagine the following architecture.

```
Controller

↓

Service

↓

Repository

↓

Database
```

Each layer has a clear responsibility.

Now imagine this.

```
Controller

↘

Repository

↓

Database

↑

Service
```

The Controller bypasses the Service.

Business logic is skipped.

Responsibilities become unclear.

The architecture becomes difficult to maintain.

---

# Allowed Dependencies

The following dependencies are acceptable.

```
Controller

↓

Service
```

```
Service

↓

Repository
```

```
Repository

↓

Prisma
```

```
Prisma

↓

Database
```

Every dependency moves downward.

---

# Dependencies That Should Never Exist

Controllers should never access:

- Prisma
- Database
- SQL
- Repository internals

---

Services should never access:

- SQL
- Database
- HTTP Request
- HTTP Response

---

Repositories should never access:

- Controllers
- Services
- HTTP Objects
- Authentication Logic
- Authorization Logic

Each layer remains independent.

---

# Business Example

Suppose the clinic introduces a new authentication system.

Only the Controller and authentication middleware require modification.

```
Authentication

↓

Controller

↓

Service

↓

Repository
```

The Repository remains unchanged.

Similarly,

suppose PostgreSQL is replaced with another database.

Only the Repository implementation changes.

```
Controller

✓ No Changes

↓

Service

✓ No Changes

↓

Repository

✓ Updated

↓

New Database
```

Because dependencies are controlled,

the impact of change remains localized.

---

# Benefits of One-Way Dependencies

Following one-way dependencies provides several advantages.

- Easier testing.
- Lower coupling.
- Better maintainability.
- Easier refactoring.
- Clear architecture.
- Predictable request flow.

Every engineer understands where responsibilities belong.

---

# Circular Dependencies

One of the most dangerous architectural problems is a circular dependency.

Example

```
PatientService

↓

AppointmentService

↓

PatientService
```

Neither Service can operate independently.

Similar problems occur when Repositories depend on each other.

Circular dependencies make applications harder to understand,

harder to test,

and often introduce runtime issues.

---

# Dependency Inversion

Higher layers should depend on abstractions rather than concrete implementations.

Example

```
PatientService

↓

PatientRepository Interface

↓

PrismaPatientRepository
```

The Service knows only the contract.

It does not know how the Repository communicates with the database.

This improves flexibility and testability.

---

# Production Notes

Large backend applications rely heavily on dependency management.

Keeping dependencies one-directional prevents architectural drift,

reduces coupling,

and allows individual layers to evolve without affecting unrelated parts of the application.

As projects grow,

this consistency becomes increasingly valuable.

---

# Summary

A clean backend architecture depends on controlled dependency flow.

Controllers communicate with Services,

Services communicate with Repositories,

Repositories communicate with Prisma,

and Prisma communicates with the database.

Maintaining one-way dependencies keeps the application modular, predictable, and easier to maintain.

# Repository Architecture Blueprint

Throughout this module, we have implemented multiple Repositories, optimized queries, handled transactions, introduced pagination, and established clear architectural boundaries.

Taken together, these concepts form a production-ready Repository architecture.

This blueprint summarizes the principles that should guide every Repository implementation.

---

# Principle 1 — One Repository Per Business Entity

Every Repository should represent a single business entity.

```
Doctor

↓

DoctorRepository

--------------------

Patient

↓

PatientRepository

--------------------

Appointment

↓

AppointmentRepository

--------------------

Diagnosis

↓

DiagnosisRepository
```

A Repository should never be responsible for multiple unrelated entities.

---

# Principle 2 — One Responsibility Per Layer

Every application layer should perform one responsibility.

```
Controller

↓

HTTP Communication

------------------------

Service

↓

Business Logic

------------------------

Repository

↓

Database Access

------------------------

Database

↓

Persistence
```

Keeping responsibilities separate improves readability and maintainability.

---

# Principle 3 — Business Logic Lives in Services

Repositories retrieve and persist data.

They do not decide:

- Whether an operation is allowed.
- Which workflow should execute.
- How business rules are applied.

Those responsibilities belong to the Service layer.

---

# Principle 4 — Database Access Is Centralized

Every database operation should pass through the Repository.

```
Controller

↓

Service

↓

Repository

↓

Prisma

↓

Database
```

No Controller or Service should communicate directly with Prisma.

---

# Principle 5 — Repository Methods Should Express Business Intent

Repository methods should describe business operations rather than generic CRUD actions.

Prefer:

```ts
findUpcomingAppointments()

findPatientByPhoneNumber()

findDoctorsByClinic()
```

Instead of:

```ts
findAll()

update()

delete()
```

Business-oriented method names make the code easier to understand.

---

# Principle 6 — Optimize Before Scaling

Repositories should retrieve only the required data.

Use:

- `select`
- `include`
- Pagination
- Transactions
- Index-aware queries

Optimization should be based on measurement rather than assumptions.

---

# Principle 7 — Keep Repositories Independent

Repositories should never depend on one another.

```
Service

├── PatientRepository

├── AppointmentRepository

└── DoctorRepository
```

The Service coordinates business workflows.

Repositories remain independent.

---

# Principle 8 — Design for Growth

Applications evolve continuously.

Repositories should be designed so new business requirements can be added without restructuring the existing architecture.

Adding:

```
PrescriptionRepository
```

should not require modifying:

- PatientRepository
- DoctorRepository
- AppointmentRepository

A modular architecture scales naturally.

# Senior Backend Engineer Repository Checklist

Throughout this module, we explored the principles, patterns, and practices used to build production-ready Repositories.

Before considering a Repository complete, experienced backend engineers perform a final architectural review.

The following checklist summarizes the questions that should be asked during implementation and code reviews.

---

# Responsibility Checklist

Does the Repository have a single responsibility?

✓ Manages one business entity.

✓ Does not contain business logic.

✓ Does not perform validation.

✓ Does not return HTTP responses.

✓ Does not coordinate workflows.

A Repository should focus exclusively on data persistence.

---

# Architecture Checklist

Does the Repository follow the application's architecture?

✓ Controller calls Service.

✓ Service calls Repository.

✓ Repository communicates with Prisma.

✓ Prisma communicates with the database.

No architectural layer should be bypassed.

---

# Query Checklist

Review every query before merging the code.

✓ Retrieves only required columns.

✓ Uses `select` where appropriate.

✓ Uses `include` only when relationships are needed.

✓ Avoids unnecessary joins.

✓ Returns the smallest amount of data required.

Efficient queries improve both application and database performance.

---

# Performance Checklist

Consider how the Repository behaves with large datasets.

✓ Pagination implemented where required.

✓ No N+1 queries.

✓ No over-fetching.

✓ Appropriate ordering applied.

✓ Query performance measured.

Code that performs well with one hundred records should also perform well with one million records.

---

# Transaction Checklist

When multiple database operations belong to the same business workflow,

verify that transactional consistency has been considered.

✓ Transaction used when appropriate.

✓ Partial updates prevented.

✓ Database consistency preserved.

---

# Repository Design Checklist

Review the Repository structure.

✓ Repository methods have meaningful names.

✓ Methods describe business operations.

✓ Repository remains focused.

✓ No unnecessary abstractions.

✓ No Generic Repository abuse.

A Repository should communicate the business intent of the application.

---

# Dependency Checklist

Verify dependency direction.

✓ Repository does not access Controllers.

✓ Repository does not access Services.

✓ Repository does not depend on other Repositories.

✓ Dependencies always flow downward.

Keeping dependencies simple improves long-term maintainability.

---

# Testing Checklist

Repositories should be easy to test.

✓ Small methods.

✓ Predictable behavior.

✓ Clear inputs.

✓ Clear outputs.

✓ Easy to mock.

Good Repository design naturally leads to better tests.

---

# Code Review Questions

During every pull request,

ask the following questions.

- Is this Repository solving one problem?
- Can another Repository method already solve this?
- Is this query efficient?
- Is business logic placed in the Service?
- Is pagination required?
- Could this introduce an N+1 query?
- Is unnecessary data being returned?
- Will this design still make sense one year from now?

These questions help identify architectural problems before they reach production.

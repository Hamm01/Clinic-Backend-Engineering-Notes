# 12.1 - Repository Overview

> In the previous module, we learned that Services are responsible for implementing business logic. However, business logic is only one part of a backend application. Every Service eventually needs to store or retrieve data from a database. This chapter introduces the Repository layer—the component responsible for communicating with the database while keeping business logic independent from persistence.

# Introducing the Repository Layer

Production applications answer that question by introducing another layer.

```
Client

↓

Routes

↓

Middleware

↓

Controller

↓

Service

↓

Repository

↓

Prisma ORM

↓

PostgreSQL
```

The Repository becomes the application's gateway to the database.

---

# What Is a Repository?

A Repository is a class responsible for reading and writing data.

It hides the details of the database implementation from the rest of the application.

Instead of allowing every Service to write database queries,

Services communicate only with Repositories.

```
PatientService

↓

PatientRepository

↓

Prisma

↓

PostgreSQL
```

The Service knows **what** data it needs.

The Repository knows **how** to retrieve it.

---

# Mental Model

Think of each layer as having one responsibility.

```
Controller

↓

Communication

Service

↓

Business Decisions

Repository

↓

Database Communication

Database

↓

Data Storage
```

Each layer focuses on one job.

This separation makes the application easier to understand and maintain.

---

# Why Does This Layer Exist?

Imagine a Service that directly communicates with Prisma.

```ts
const patient = await prisma.patient.create({
  data: dto
})
```

This works.

However, the Service now performs two different jobs.

- Business logic.
- Database access.

As the project grows,

mixing these responsibilities makes the code harder to maintain.

---

# Separating Responsibilities

Instead,

the Service delegates persistence.

```
PatientService

↓

PatientRepository.create()

↓

Prisma

↓

PostgreSQL
```

Now each layer has a single responsibility.

The Service focuses on business.

The Repository focuses on persistence.

---

# Business Example

Suppose a receptionist registers a patient.

The complete request flows like this.

```
Receptionist

↓

POST /patients

↓

PatientController

↓

PatientService

↓

PatientRepository

↓

Prisma

↓

PostgreSQL
```

Notice that the Controller never communicates with the database.

The Service never writes SQL queries.

Only the Repository performs persistence.

---

# Repository Responsibilities

A Repository is responsible for:

- Creating records.
- Reading records.
- Updating records.
- Deleting records.
- Executing database queries.
- Mapping ORM operations.

Repositories should not contain:

- Business rules.
- HTTP logic.
- Authentication.
- Authorization.
- Validation logic.
- Response formatting.

Those responsibilities belong to other layers.

---

# Benefits of Using Repositories

Introducing a Repository layer provides several advantages.

## Clear Separation of Concerns

Business logic and persistence remain independent.

---

## Better Readability

Services become easier to understand because they describe business workflows instead of database queries.

---

## Easier Maintenance

Database changes remain isolated inside Repositories.

---

## Better Reusability

The same database operations can be reused by multiple Services.

---

## Improved Testability

Services can be tested without directly depending on database implementations.

# Repository Philosophy

Repositories answer one question.

> **"How should this data be stored or retrieved?"**

They do not answer questions like:

- Is the doctor available?
- Can this appointment be scheduled?
- Does the patient belong to this clinic?

Those are business questions.

Business questions belong to Services.

# Summary

The Repository layer separates database access from business logic.

Instead of allowing Services to communicate directly with Prisma, production applications introduce Repositories that act as a dedicated persistence layer.

This separation keeps the application organized, easier to test, and easier to maintain as the project grows.

By understanding the role of Repositories, we prepare ourselves to build a clean persistence layer for our Clinic Backend.

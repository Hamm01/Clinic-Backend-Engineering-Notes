# Implementing create()

# Explanation

- Compile-time validation.

# Implementing findById()

One of the most frequently used Repository methods is retrieving a single record by its primary key.

This method is used whenever the application needs to load a specific doctor.

Examples:

- View Doctor Profile
- Book Appointment
- Update Doctor
- Delete Doctor
- Verify Doctor Exists

Instead of repeating the same Prisma query throughout the application, we centralize it inside the Repository.

---

## Code

```ts
async findById(
    id: string
): Promise<Doctor | null> {

    return prisma.doctor.findUnique({

        where: {
            id
        }

    });

}
```

---

# Explanation

Let's understand what this method is doing.

The Repository receives a Doctor ID.

```
Doctor ID

↓

findUnique()

↓

Doctor

or

null
```

If the doctor exists,

Prisma returns the Doctor.

Otherwise,

it returns `null`.

The Repository does not decide what happens next.

It simply retrieves the data.

---

# Why Use findUnique()?

Because our `id` column is marked as a unique field inside Prisma Schema.

Example

```prisma
model Doctor {

    id String @id @default(uuid())

}
```

Since only one Doctor can have a specific ID,

`findUnique()` is the correct method.

---

# Business Example

Suppose a patient opens the doctor's profile.

```
GET /doctors/:id

↓

DoctorController

↓

DoctorService

↓

DoctorRepository.findById()

↓

Database
```

The Repository retrieves the doctor.

The Service decides whether returning that doctor is allowed.

---

# Production Notes

A Repository should never throw an error simply because no record exists.

Returning `null` allows the Service to decide how the business should respond.

For example,

```ts
const doctor = await doctorRepository.findById(id)

if (!doctor) {
  throw new ResourceNotFoundException('Doctor not found')
}
```

The Repository retrieves data.

The Service interprets the result.

---

# Implementing findByEmail()

Many systems allow doctors to log in using their email address.

Searching by email becomes another common persistence operation.

---

## Code

```ts
async findByEmail(
    email: string
): Promise<Doctor | null> {

    return prisma.doctor.findUnique({

        where: {

            email

        }

    });

}
```

---

# Explanation

Unlike `findById()`,

this method searches using the unique email field.

```
Email

↓

findUnique()

↓

Doctor

↓

null
```

Because email is unique,

Prisma guarantees that at most one record is returned.

---

# Why Not Repeat This Query Everywhere?

Without a Repository,

multiple Services would eventually write this query.

```ts
await prisma.doctor.findUnique({
  where: {
    email
  }
})
```

Eventually,

one Service adds `select`.

Another adds `include`.

Another changes filtering.

The application becomes inconsistent.

Keeping the query inside one Repository method ensures every Service retrieves doctors in the same way.

---

# Implementing exists()

Sometimes,

we don't need the complete Doctor object.

We only need to know whether it exists.

For example,

before scheduling an appointment.

---

## Code

```ts
async exists(
    id: string
): Promise<boolean> {

    const doctor =
        await prisma.doctor.findUnique({

            where: {
                id
            },

            select: {
                id: true
            }

        });

    return doctor !== null;

}
```

---

# Explanation

Notice that we only select

```ts
id
```

instead of loading the complete Doctor record.

This reduces the amount of data returned from the database.

```
Database

↓

Doctor ID

↓

true

or

false
```

This is a small optimization,

but production applications perform thousands of these checks every day.

Small improvements become significant over time.

---

# Business Example

Before creating an appointment,

the Service may perform the following workflow.

```
AppointmentService

↓

doctorRepository.exists()

↓

true

↓

Continue Appointment Creation
```

If the Doctor does not exist,

the Service throws a business exception.

# Implementing findMany()

Retrieving multiple records is one of the most common database operations.

Examples include:

- Doctor Listing
- Admin Dashboard
- Search Results
- Available Doctors
- Reporting

Instead of returning a single Doctor, this method returns a collection.

---

## Code

```ts
async findMany(): Promise<Doctor[]> {

    return prisma.doctor.findMany({

        where: {

            deletedAt: null

        },

        orderBy: {

            createdAt: "desc"

        }

    });

}
```

---

# Explanation

Unlike `findUnique()`,

`findMany()` returns zero or more records.

```
Database

↓

findMany()

↓

[]

↓

Doctor[]

```

Notice two important things.

First,

we ignore deleted records.

```ts
deletedAt: null
```

Second,

the newest doctors appear first.

```ts
orderBy: {
  createdAt: 'desc'
}
```

These are examples of application-wide persistence rules.

Keeping them inside the Repository ensures every Service behaves consistently.

---

# Why Not Return Every Record?

Imagine displaying doctors on the Admin Dashboard.

Showing deleted doctors would confuse users.

Instead of forcing every Service to remember

```ts
deletedAt: null
```

the Repository applies the filter automatically.

This prevents bugs caused by forgotten conditions.

---

# Business Example

```
GET /doctors

↓

DoctorController

↓

DoctorService

↓

DoctorRepository.findMany()

↓

Database
```

The Service simply requests all active doctors.

The Repository decides how to retrieve them.

# Implementing softDelete()

Most healthcare systems never permanently remove medical information.

Instead,

records are marked as deleted.

This is known as a **Soft Delete**.

---

## Code

```ts
async softDelete(
    id: string
): Promise<Doctor> {

    return prisma.doctor.update({

        where: {

            id

        },

        data: {

            deletedAt: new Date()

        }

    });

}
```

---

# Explanation

Instead of removing the row,

we record the deletion timestamp.

```
Doctor

↓

deletedAt

↓

Current Time
```

The Doctor still exists in the database.

Future queries simply ignore deleted records.

---

# Why Is Soft Delete Better?

Imagine a doctor has:

- Appointments
- Prescriptions
- Diagnoses
- Audit Logs

Deleting the Doctor permanently could break historical records.

Soft deletes preserve data integrity.

---

# Business Example

```

↓

DoctorService

↓

DoctorRepository.softDelete()

↓

Database
```

The Doctor disappears from the application,

but remains safely stored.

# Repository Design Principles

A good Repository should:

✔ Group related database operations.

✔ Return typed results.

✔ Hide Prisma implementation details.

✔ Keep persistence logic centralized.

✔ Avoid business decisions.

When reading a Repository,

every method should clearly express a database operation.

---

# Common Mistakes

## Loading More Data Than Necessary

Avoid

```ts
findMany()
```

when only a count is required.

Use

```ts
count()
```

instead.

---

## Hardcoding Business Rules

Avoid

```ts
if (doctor.isAvailable)
```

Availability is a business decision.

Repositories should not make business decisions.

---

## Deleting Records Permanently

Healthcare systems usually require historical records.

Prefer soft deletes whenever business requirements demand data retention.

---

## Returning HTTP Errors

Repositories should never know about

- Status Codes
- Express
- Responses

Those belong to Controllers.

---

Avoid generic methods that force Services to build database queries themselves.

Repositories should become the application's single source of truth for persistence.

# Loading Related Data

A Repository often needs to return more than a single database record.

For example,

when viewing a doctor's profile, we may also need:

- Clinic Information
- Upcoming Appointments
- Specializations

Prisma allows us to load related data using the `include` option.

---

## Code

```ts
async findByIdWithClinic(
    id: string
): Promise<Doctor | null> {

    return prisma.doctor.findUnique({

        where: {
            id
        },

        include: {
            clinic: true
        }

    });

}
```

---

# Explanation

Normally,

```ts
findUnique()
```

returns only the `Doctor` record.

By using `include`,

Prisma also loads the related `Clinic`.

```
Doctor

↓

Clinic

↓

Single Response
```

This reduces additional database queries.

---

# When Should We Use include?

Use `include` when the application needs related entities together.

Examples:

- Doctor Profile
- Appointment Details
- Patient History
- Diagnosis Details

Avoid loading relationships that are not required.

Loading unnecessary data increases query execution time and memory usage.

---

# Selecting Specific Fields

Sometimes we do not need every column.

For example,

when displaying a doctor list,

we may only require:

- Name
- Email
- Specialization

Instead of retrieving the complete record, we can use `select`.

---

## Code

```ts
async findSummaryById(
    id: string
) {

    return prisma.doctor.findUnique({

        where: {
            id
        },

        select: {

            id: true,

            name: true,

            email: true,

            specialization: true

        }

    });

}
```

---

# Explanation

Using `select` limits the returned fields.

Instead of retrieving every column,

only the required data is returned.

Benefits include:

- Less network traffic.
- Faster queries.
- Smaller API responses.
- Better performance.

---

# include vs select

| include                | select                     |
| ---------------------- | -------------------------- |
| Loads related entities | Loads specific columns     |
| Used for relationships | Used for projection        |
| Returns more data      | Returns only required data |

Choose the option that best matches the business requirement.

---

# Building Reusable Query Objects

As projects grow,

different methods often share the same query configuration.

Instead of repeating it,

extract reusable query objects.

---

## Code

```ts
const activeDoctorWhere = {
  deletedAt: null
}
```

Now reuse it.

```ts
async findMany() {

    return prisma.doctor.findMany({

        where: activeDoctorWhere

    });

}
```

---

# Why Is This Better?

Without reusable query objects,

developers duplicate filters.

```ts
deletedAt: null
```

appears throughout the application.

Eventually,

one query forgets the condition.

Soft-deleted records begin appearing unexpectedly.

Centralizing common filters prevents this problem.

# Code Review Checklist

Before merging a Repository into production, review the following.

- Are all database queries located inside the Repository?
- Does every method have a clear purpose?
- Are business rules kept out of the Repository?
- Is the shared Prisma Client being reused?
- Are `select` and `include` used appropriately?
- Are repeated query conditions extracted?
- Are method names descriptive?
- Is unnecessary data avoided?

A Repository that satisfies these questions is easier to maintain as the application evolves.

# Common Mistakes

## Returning Entire Objects

Avoid retrieving every column when only a few fields are needed.

Use `select` whenever possible.

---

## Overusing include

Do not eagerly load every relationship.

Only retrieve related data required by the current business use case.

---

## Duplicating Query Conditions

Repeated filters lead to inconsistent behaviour.

Extract common query objects where appropriate.

---

## Repository Becoming a Service

Avoid methods such as:

```ts
approveDoctorRegistration()

scheduleAppointment()

sendNotification()
```

These are business workflows.

Repositories should only perform persistence operations.

---

# Production Notes

Repositories should evolve alongside the application.

As new features are introduced,

new persistence methods can be added without affecting Controllers or Services.

This isolation makes database changes significantly easier to manage and keeps the persistence layer consistent across the application.

# 12.10 - Query Optimization

> Writing database queries that return the correct data is only the first step. Production backend applications must also execute those queries efficiently. As applications grow, inefficient queries increase response time, consume more database resources, and reduce the application's ability to scale. Query optimization focuses on retrieving the required data with the least possible cost.

# What Makes a Query Slow?

Developers often assume that slow APIs are caused by Node.js.

In reality,

many performance problems originate in the database.

Common causes include:

- Retrieving unnecessary columns.
- Loading unnecessary relationships.
- Missing indexes.
- Executing too many queries.
- Filtering in application memory.
- Large table scans.
- Inefficient pagination.

Most backend performance issues can be traced back to one of these patterns.

Query optimization is therefore part of the Repository's responsibility.

# Selecting Only Required Data

One of the simplest ways to improve database performance is to retrieve only the data required by the current business operation.

Every column returned by the database consumes resources.

Even if the application never uses that data,

the database still reads it,

the network still transfers it,

and the application still stores it in memory.

For this reason,

Repositories should avoid retrieving complete entities unless every field is actually needed.

---

# The Problem

Consider the Patient model.

```
Patient

↓

id

firstName

lastName

dateOfBirth

gender

phoneNumber

email

address

bloodGroup

insuranceNumber

medicalHistory

allergies

createdAt

updatedAt
```

Now imagine the receptionist opens the patient list.

The screen displays only:

- Patient ID
- Name
- Phone Number

Yet the Repository executes:

```ts
await prisma.patient.findMany()
```

Every column is returned,

including information that never appears on the screen.

---

# Why Is This Wasteful?

The database performs additional work.

```
Database

↓

Read Every Column

↓

Serialize Data

↓

Transfer Over Network

↓

Application Memory

↓

Discard Unused Fields
```

Most of the retrieved data is never used.

As the number of records grows,

this unnecessary work becomes increasingly expensive.

---

# Better Repository Implementation

Instead of retrieving the complete entity,

retrieve only the required fields.

## Code

```ts
async findPatientList() {

    return prisma.patient.findMany({

        select: {

            id: true,

            firstName: true,

            lastName: true,

            phoneNumber: true

        }

    });

}
```

---

# Explanation

The Repository explicitly tells Prisma which fields are required.

Instead of returning fourteen columns,

the database returns only four.

```
Patient

↓

id

firstName

lastName

phoneNumber
```

Everything else remains in the database.

---

# Business Example

Receptionist Dashboard

```
Patient List

↓

Name

↓

Phone Number

↓

Registration Date
```

The receptionist cannot view:

- Medical history
- Diagnoses
- Prescriptions
- Insurance details

The Repository should not retrieve them.

Every query should match the needs of the business feature.

---

# Visualizing Data Flow

Without `select`

```
Database

↓

14 Columns

↓

Network

↓

Application

↓

Uses 4 Columns
```

With `select`

```
Database

↓

4 Columns

↓

Network

↓

Application

↓

Uses 4 Columns
```

The application receives exactly what it needs.

---

# Benefits of Using select

Using `select` provides several advantages.

## Reduced Database I/O

The database reads less data from storage.

---

## Lower Network Usage

Smaller responses travel faster between the database and the application.

---

## Reduced Memory Consumption

Node.js stores fewer objects in memory.

This becomes important when thousands of records are returned.

---

## Faster Serialization

Prisma converts fewer fields into JavaScript objects.

Less work means faster responses.

---

# Repository Responsibility

Repositories should expose methods designed for business use cases.

For example,

instead of one generic method:

```ts
findAll()
```

consider purpose-specific methods.

```ts
findPatientList()
```

```ts
findPatientDetails()
```

```ts
findPatientSummary()
```

Each method retrieves only the fields required for its specific use case.

---

# Common Mistakes

## Returning Entire Entities

Avoid this pattern.

```ts
findMany()
```

when only a few fields are required.

---

## Selecting Every Field "Just in Case"

Do not retrieve data because it _might_ be needed later.

Repositories should return only what the current business operation requires.

---

## Ignoring Sensitive Information

Some fields should rarely be returned.

Examples include:

- Insurance numbers
- Medical history
- Clinical notes

Using `select` helps reduce accidental exposure of sensitive data.

---

# Production Notes

Large applications rarely expose complete database entities.

Instead,

Repositories define purpose-specific queries that retrieve only the fields required by each business feature.

This improves performance,

reduces memory usage,

and minimizes the risk of exposing unnecessary or sensitive information.

---

# Summary

Retrieving fewer columns is one of the simplest and most effective query optimization techniques.

Using Prisma's `select` option allows Repositories to return only the data required for the current business operation, reducing database work, network traffic, memory usage, and response time.

# Loading Related Data with `include`

Earlier, we learned how `select` retrieves only the required columns from a table.

However,

real business operations often require information from multiple related entities.

For example,

an appointment dashboard may display:

- Appointment Time
- Patient Name
- Doctor Name
- Clinic Name

This information exists in multiple database tables.

The Repository therefore needs a way to load related entities.

Prisma provides this using the `include` option.

---

# Business Example

Consider the Appointment model.

```
Appointment

↓

Patient

Doctor

Clinic
```

The receptionist opens today's appointments.

The interface displays:

```
09:00 AM

↓

John Doe

↓

Dr. Sarah

↓

Downtown Clinic
```

The Repository must retrieve data from multiple tables.

---

# Repository Implementation

## Code

```ts
async findTodayAppointments() {

    return prisma.appointment.findMany({

        include: {

            patient: true,

            doctor: true,

            clinic: true

        }

    });

}
```

---

# Explanation

The Repository retrieves:

```
Appointment

+

Patient

+

Doctor

+

Clinic
```

Prisma automatically loads the related entities.

The application receives one complete object containing all required information.

---

# Visualizing include

Without relationships

```
Appointment
```

With `include`

```
Appointment

├── Patient

├── Doctor

└── Clinic
```

The Repository can now return richer business objects without executing multiple manual queries.

---

# When Should We Use `include`?

Use `include` when the business feature genuinely requires related data.

Examples:

Patient Profile

↓

Medical Records

↓

Diagnoses

↓

Prescriptions

The user expects to see all of this information together.

Loading related entities is therefore appropriate.

---

# When Should We Avoid `include`?

Not every query needs relationships.

Suppose the receptionist only wants today's appointment count.

Avoid

```ts
await prisma.appointment.findMany({
  include: {
    patient: true,

    doctor: true,

    clinic: true
  }
})
```

These relationships are never used.

Instead,

retrieve only the appointment records—or even better, use `count()` if only the total is required.

---

# Combining `select` and `include`

Prisma also allows selecting specific fields from related entities.

## Code

```ts
await prisma.appointment.findMany({
  include: {
    patient: {
      select: {
        firstName: true,

        lastName: true
      }
    },

    doctor: {
      select: {
        firstName: true,

        lastName: true
      }
    }
  }
})
```

---

# Explanation

Instead of loading the complete Patient and Doctor objects,

the Repository retrieves only the fields required by the business feature.

```
Appointment

├── Patient

│     ├── firstName

│     └── lastName

│

└── Doctor

      ├── firstName

      └── lastName
```

This reduces:

- Database I/O
- Network traffic
- Memory usage

while still providing all required business information.

---

# Choosing Between `select` and `include`

| Situation                                    | Recommendation                     |
| -------------------------------------------- | ---------------------------------- |
| Need only columns from one table             | Use `select`                       |
| Need related entities                        | Use `include`                      |
| Need only a few fields from related entities | Use `include` with nested `select` |
| Need complete related entities               | Use `include`                      |

---

# Common Mistakes

## Loading Every Relationship

Avoid this pattern.

```ts
include: {

    patient: true,

    doctor: true,

    clinic: true,

    diagnosis: true,

    prescriptions: true,

    medicalRecord: true

}
```

Large relationship trees increase query complexity and response size.

---

## Using `include` by Default

Relationships should only be loaded when the business feature requires them.

Do not use `include` simply because the relationship exists.

---

## Returning Sensitive Data

Including an entity returns all of its fields unless restricted.

Always consider whether sensitive information should be excluded by using nested `select`.

---

# Production Notes

Experienced backend engineers treat every relationship as a cost.

Before adding an `include`, ask:

- Does this screen actually need this data?
- Can fewer fields be selected?
- Would loading this relationship significantly increase query complexity?

Repositories should retrieve only the relationships required for the current business operation.

---

# Summary

The `include` option allows Repositories to retrieve related entities in a single query, making it easier to build business-focused responses.

However,

relationships increase the amount of data retrieved.

Repositories should therefore load only the required relationships and combine `include` with nested `select` whenever possible to minimize unnecessary data transfer.

# Avoiding Over-Fetching

Earlier, we learned that the N+1 Query Problem increases the number of database queries.

Another common performance issue is **Over-Fetching**.

Over-fetching occurs when a Repository retrieves significantly more data than the business operation actually requires.

Although the application works correctly,

the database performs unnecessary work,

the network transfers unnecessary data,

and the application stores objects that are never used.

---

# What Is Over-Fetching?

Suppose the receptionist opens the patient list.

The screen displays only:

- Patient ID
- Full Name
- Phone Number

However,

the Repository executes:

```ts
await prisma.patient.findMany({
  include: {
    appointments: true,

    diagnoses: true,

    prescriptions: true,

    medicalRecords: true,

    clinic: true
  }
})
```

The database retrieves a large object graph,

even though most of the information never appears on the screen.

---

# Visualizing Over-Fetching

Business Requirement

```
Patient List

↓

Name

↓

Phone
```

Repository

```
Patient

├── Appointments

├── Diagnoses

├── Prescriptions

├── Medical Records

├── Clinic

└── Every Column
```

The Repository retrieves much more than the business requires.

---

# Why Is This Expensive?

Every additional field and relationship has a cost.

```
Database

↓

Read More Data

↓

Serialize Objects

↓

Network Transfer

↓

Application Memory

↓

Garbage Collection
```

As the number of records grows,

this unnecessary work becomes increasingly expensive.

---

Repositories should be designed around business use cases.

# Explanation

```

↓

id

↓

firstName

↓

lastName

↓

phoneNumber
```

Everything else remains inside the database.

# Index-Aware Queries

As applications grow, the number of records stored in the database increases dramatically.

A query that executes in a few milliseconds with one thousand records may become significantly slower when the table contains millions of rows.

Indexes exist to help the database locate records efficiently.

Repositories should therefore be designed with indexes in mind.

An optimized Repository not only retrieves the correct data,

it also retrieves it in a way that allows the database to use its indexes effectively.

---

# Why Indexes Exist

Imagine the Patient table contains:

```
Patient

↓

2,000,000 Records
```

The receptionist searches for a patient using their phone number.

Without an index,

the database must inspect every row until it finds the correct patient.

```
Patient Table

↓

Row 1

↓

Row 2

↓

Row 3

↓

...

↓

Row 2,000,000
```

This process is called a **Full Table Scan**.

As the table grows,

query performance decreases.

---

# Searching with an Index

Suppose the `phoneNumber` column has an index.

Instead of checking every row,

the database uses the index to jump directly to the matching records.

```
Query

↓

Phone Number Index

↓

Matching Record
```

The number of rows examined is drastically reduced.

---

# Repository Example

The Repository searches for a patient by phone number.

## Code

```ts
async findByPhoneNumber(
    phoneNumber: string
): Promise<Patient | null> {

    return prisma.patient.findUnique({

        where: {

            phoneNumber

        }

    });

}
```

---

# Explanation

This Repository method performs well because:

- `phoneNumber` is unique.
- The database can use its index.
- Only one record is expected.

The Repository does not need to scan the entire Patient table.

---

# Searching by Non-Indexed Columns

Now consider another Repository method.

## Code

```ts
async findByFirstName(
    firstName: string
) {

    return prisma.patient.findMany({

        where: {

            firstName

        }

    });

}
```

If `firstName` is not indexed,

the database may need to examine every row.

```
Patient Table

↓

Read Every Row

↓

Compare firstName

↓

Return Matches
```

Although the Repository implementation looks simple,

the database performs significantly more work.

---

# Sorting and Indexes

Indexes also improve sorting.

Consider this Repository method.

```ts
await prisma.patient.findMany({
  orderBy: {
    createdAt: 'desc'
  }
})
```

If `createdAt` is indexed,

the database can retrieve the records in the requested order efficiently.

Without an index,

the database may need to:

- Read every record.
- Sort the data.
- Return the results.

Sorting large datasets without indexes becomes increasingly expensive.

---

# Filtering and Indexes

Repositories often filter records.

Example

```ts
await prisma.appointment.findMany({
  where: {
    clinicId,

    status: 'SCHEDULED'
  }
})
```

If both columns are indexed,

the database can quickly locate matching appointments.

If not,

it may scan the entire Appointment table.

---

# Business Example

The receptionist opens today's appointments.

The Repository filters using:

- Clinic
- Appointment Date
- Status

These fields are used in almost every request.

They are good candidates for database indexes.

Repositories frequently reveal which columns deserve indexing.

---

# Repository Design and Indexes

When designing Repository methods, ask:

- Which columns are searched most often?
- Which columns are used in filters?
- Which columns are used for sorting?
- Which columns uniquely identify records?

These questions help identify where indexes provide the greatest benefit.

---

# Common Mistakes

## Filtering on Unindexed Columns

A Repository may work correctly,

but repeated filtering on unindexed columns can become a major performance bottleneck.

---

## Sorting Large Tables Without Indexes

Sorting millions of rows without an index forces the database to perform unnecessary work.

---

## Assuming Prisma Optimizes Everything

Prisma generates SQL,

but it does not automatically create every index your application needs.

Repository design and database indexing must work together.

---

## Creating Too Many Indexes

Indexes improve reads,

but they also increase the cost of:

- Inserts
- Updates
- Deletes

Every index must be justified by a real business requirement.

---

# Production Notes

Indexes should be created for columns that are:

- Frequently searched.
- Frequently filtered.
- Frequently sorted.
- Used in joins.
- Declared as unique.

Repositories often reveal these access patterns long before performance problems appear.

Database indexing should evolve alongside Repository usage.

---

# Summary

Indexes allow the database to locate records efficiently without scanning entire tables.

Repositories that filter, sort, or search using indexed columns scale far better as data grows.

Understanding how Repository methods interact with database indexes is essential for building production-ready backend systems.

# Understanding Prisma Generated SQL

Prisma provides a clean TypeScript API for interacting with the database.

For example,

a Repository method may look like this.

## Code

```ts
async findByEmail(
    email: string
): Promise<Patient | null> {

    return prisma.patient.findUnique({

        where: {

            email

        }

    });

}
```

This appears to be a TypeScript function.

However,

the database never understands TypeScript.

The database understands only SQL.

---

# Measuring Query Performance

Writing an optimized Repository method is only the first step.

The next responsibility of a backend engineer is verifying that the optimization actually improves performance.

Performance should never be based on assumptions.

It should always be measured.

A query that appears simple may execute slowly,

while a more complex query may perform efficiently because of proper indexing and query planning.

---

# Why Measuring Performance Matters

Consider two Repository methods.

Repository A

```ts
await prisma.patient.findMany({
  where: {
    status: 'ACTIVE'
  }
})
```

Repository B

```ts
await prisma.patient.findMany({
  where: {
    clinicId,

    status: 'ACTIVE'
  },

  orderBy: {
    createdAt: 'desc'
  }
})
```

Without measurement,

it is impossible to know which query performs better.

Never assume.

Always verify.

---

# What Should We Measure?

When evaluating Repository performance,

consider the following metrics.

| Metric                     | Why It Matters                                             |
| -------------------------- | ---------------------------------------------------------- |
| Query execution time       | Indicates how long the database spent executing the query. |
| Number of executed queries | Helps identify N+1 query problems.                         |
| Rows scanned               | Shows how much work the database performed.                |
| Rows returned              | Helps detect over-fetching.                                |
| Memory usage               | Large result sets consume more application memory.         |
| Response time              | Reflects the overall impact on the API.                    |

Looking at only one metric rarely tells the full story.

# Production Best Practices

Writing efficient Repository methods is not about memorizing Prisma APIs.

It is about consistently applying a set of engineering principles whenever data is retrieved from the database.

The following practices are commonly followed in production backend applications.

---

# Design Repositories Around Business Use Cases

Repositories should answer business questions,

not database questions.

Instead of writing generic methods such as:

```ts
findAll()

findOne()

findById()
```

prefer methods that reflect the business operation.

```ts
findPatientList()

findPatientProfile()

findUpcomingAppointments()
```

Each Repository method should have a single responsibility.

---

# Retrieve Only the Required Data

Always retrieve the smallest amount of data necessary.

Use:

- `select` for choosing specific columns.
- `include` only when related entities are required.
- Nested `select` when only part of a relationship is needed.

Avoid retrieving complete entities unless every field is required.

---

# Keep Database Queries Inside the Repository

The Repository should be the only layer responsible for communicating with the database.

```
Controller

↓

Service

↓

Repository

↓

Database
```

Controllers and Services should never execute Prisma queries directly.

This keeps database access centralized and easier to maintain.

---

# Avoid Duplicate Queries

Before writing a new Repository method,

ask:

- Does another Repository method already retrieve this data?
- Can the existing method be extended without affecting other use cases?

Repositories should encourage reuse without becoming overly generic.

---

# Prevent N+1 Queries

Never execute database queries inside loops.

Avoid patterns such as:

```ts
for (...) {

    await prisma...
}
```

Instead,

retrieve related data using optimized Repository queries.

Reducing database round trips often has a greater impact than optimizing application code.

---

# Keep Repository Methods Focused

Each Repository method should perform one business operation.

Avoid methods that:

- Search.
- Update.
- Delete.
- Aggregate.

all within the same function.

Small,

well-defined methods are easier to test,

maintain,

and optimize.

---

# Use Pagination for Large Collections

Never return an unbounded collection.

Repository methods returning lists should support pagination whenever large datasets are expected.

This protects both the application and the database from unnecessary load.

---

# Think About Indexes

Whenever a Repository introduces:

- filtering,
- sorting,
- searching,

consider whether the queried columns should be indexed.

Repository usage often reveals where indexes are required.

---

# Monitor Repository Performance

Performance should be monitored continuously.

Questions every backend engineer should ask include:

- How many queries does this endpoint execute?
- Is the database using indexes?
- Are unnecessary relationships being loaded?
- Is the response larger than necessary?

Optimization should be guided by measurement rather than assumptions.

---

# Keep Business Logic Out of Repositories

Repositories should focus on data persistence.

Avoid placing business decisions inside Repository methods.

Repository

```ts
findPatientById()
```

Service

```ts
if (!patient.isActive) {

    throw new Error(...);

}
```

The Repository retrieves data.

The Service decides what to do with that data.

Keeping these responsibilities separate improves maintainability.

---

# Repository Checklist

Before completing a Repository method,

review the following checklist.

✓ Does the query retrieve only the required fields?

✓ Are only the necessary relationships loaded?

✓ Is pagination required?

✓ Can the database use indexes efficiently?

✓ Does the method avoid N+1 queries?

✓ Does the method represent one business use case?

✓ Is business logic kept inside the Service layer?

✓ Has the query been measured or reviewed for performance?

If the answer is "Yes" to each question,

the Repository is likely suitable for production.

---

# Common Mistakes

## Building Generic Repositories

Repositories should model business operations,

not provide every possible CRUD operation by default.

---

## Returning Complete Database Models

Retrieve only the information required by the current feature.

---

## Mixing Responsibilities

Repositories should retrieve data.

Services should implement business rules.

Controllers should handle HTTP requests.

Each layer has its own responsibility.

---

## Optimizing Too Early

Write clear Repository methods first.

Measure performance.

Optimize only where necessary.

Premature optimization often increases complexity without delivering meaningful improvements.

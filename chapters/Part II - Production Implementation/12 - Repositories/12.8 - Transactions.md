# 12.8 - Repository Transactions

> Modern backend applications rarely modify a single database table. Most business operations involve multiple related entities that must either succeed together or fail together. This chapter introduces database transactions and explains how multiple repositories work together safely while keeping business logic inside the Service layer.

# Why Do Transactions Exist?

Imagine the following business operation.

A doctor completes a patient's consultation.

The application needs to perform several database operations.

- Create Diagnosis
- Create Prescription
- Create Laboratory Tests
- Update Medical Record
- Update Appointment Status

Although these are separate database operations,

from the business perspective,

they represent **one consultation**.

```
Consultation

↓

Diagnosis

↓

Prescription

↓

Lab Tests

↓

Medical Record

↓

Appointment
```

Either the consultation should be saved completely,

or nothing should be saved.

---

# The Problem Without Transactions

Suppose the application performs the following operations.

```
Save Diagnosis

✓

Save Prescription

✓

Save Lab Tests

✗

Update Appointment

✗
```

The database now contains:

```
Diagnosis

✓

Prescription

✓

Laboratory Tests

✗

Appointment Status

✗
```

The patient's medical record is now inconsistent.

The doctor believes the consultation completed,

but important information is missing.

---

# What Is a Transaction?

A transaction groups multiple database operations into one unit of work.

```
Operation 1

+

Operation 2

+

Operation 3

+

Operation 4

↓

Transaction
```

The database guarantees only two outcomes.

```
Everything succeeds

OR

Everything fails
```

There is no partial success.

---

# Atomicity

One of the fundamental properties of a transaction is **Atomicity**.

Think of it as:

```
All

or

Nothing
```

If any operation fails,

the database automatically rolls back every previous operation inside the transaction.

---

# Business Example

Doctor completes consultation.

```
Doctor

↓

Diagnosis

↓

Prescription

↓

Medical Record

↓

Appointment Completed
```

If saving the prescription fails,

the diagnosis should also disappear.

Otherwise,

the patient's consultation is incomplete.

---

# Where Do Transactions Belong?

One of the most common mistakes in backend applications is placing transaction logic in the wrong layer.

Transactions do **not** belong inside:

- Controllers
- Routes
- Middleware
- Repositories

Repositories should remain responsible for individual persistence operations.

Controllers should remain responsible for HTTP.

Middleware should remain responsible for request processing.

The **Service layer** coordinates the business workflow and decides when a transaction is required.

---

# Layer Responsibilities

```
Controller

↓

Receives HTTP Request

↓

Service

↓

Starts Transaction

↓

Repository

↓

Executes Database Queries

↓

PostgreSQL
```

Every layer continues to have a single responsibility.

---

# Why Should Services Control Transactions?

The Service understands the complete business operation.

For example,

creating a diagnosis may require:

- DiagnosisRepository
- PrescriptionRepository
- MedicalRecordRepository
- AppointmentRepository

Only the Service knows that these operations represent one business workflow.

The Repository does not.

---

# Common Mistakes

## Starting Transactions Inside Controllers

Controllers should never coordinate database operations.

Their responsibility ends after delegating the request to the Service.

---

## Business Logic Inside Repositories

Repositories should not decide whether multiple queries belong together.

They simply expose persistence methods.

---

## Long Transactions

Avoid keeping transactions open while:

- Calling external APIs.
- Sending emails.
- Uploading files.
- Waiting for user input.

Transactions should remain as short as possible.

---

# Production Notes

Transactions protect data consistency,

but they also consume database resources.

Large or slow transactions increase lock duration and reduce concurrency.

Production systems keep transactions:

- Small
- Fast
- Focused

---

# Summary

A transaction allows multiple database operations to behave as one business operation.

The Service layer decides when a transaction begins.

Repositories execute individual persistence operations.

The database guarantees that every operation succeeds together or fails together, ensuring the application's data remains consistent.

# Implementing a Transaction

Now that we understand why transactions exist, let's implement one using Prisma.

Consider the following business operation.

A doctor completes a consultation.

The application must:

1. Create the diagnosis.
2. Create one or more prescriptions.
3. Update the patient's medical record.
4. Mark the appointment as completed.

These operations represent a single business workflow.

If any operation fails, none of the changes should be stored.

---

## Without a Transaction

Consider the following Service implementation.

```ts
await diagnosisRepository.create(diagnosisData)

await prescriptionRepository.createMany(prescriptions)

await medicalRecordRepository.update(recordId, medicalRecord)

await appointmentRepository.complete(appointmentId)
```

At first glance, this code appears correct.

However, imagine the third operation fails because the database becomes unavailable.

```
Diagnosis

✓ Saved

↓

Prescription

✓ Saved

↓

Medical Record

✗ Failed

↓

Appointment

✗ Not Updated
```

The database now contains incomplete business data.

The patient has a diagnosis and prescription, but the consultation was never completed.

This leaves the system in an inconsistent state.

---

# Using Prisma Transactions

Prisma provides the `$transaction()` API to execute multiple database operations as a single unit of work.

---

## Code

```ts
await prisma.$transaction(async tx => {
  await tx.diagnosis.create({
    data: diagnosisData
  })

  await tx.prescription.createMany({
    data: prescriptions
  })

  await tx.medicalRecord.update({
    where: {
      id: recordId
    },

    data: medicalRecord
  })

  await tx.appointment.update({
    where: {
      id: appointmentId
    },

    data: {
      status: 'COMPLETED'
    }
  })
})
```

---

# Explanation

The transaction begins when Prisma executes:

```ts
prisma.$transaction()
```

Every database operation inside the callback becomes part of the same transaction.

```
Transaction

├── Diagnosis

├── Prescriptions

├── Medical Record

└── Appointment
```

If every query succeeds,

Prisma commits the transaction.

If any query fails,

Prisma automatically rolls back every previous operation.

No partial data remains in the database.

---

# Business Workflow

```
Doctor

↓

Finish Consultation

↓

Diagnosis Service

↓

Start Transaction

↓

Save Diagnosis

↓

Save Prescriptions

↓

Update Medical Record

↓

Complete Appointment

↓

Commit Transaction
```

This entire workflow behaves as one business operation.

---

# What Happens During Rollback?

Suppose creating the prescriptions fails.

The execution flow becomes:

```
Diagnosis

✓

↓

Prescription

✗

↓

Rollback

↓

Diagnosis Removed
```

After rollback,

the database returns to its original state.

It is as though the transaction never happened.

---

# Why Rollback Is Important

Imagine a patient visits the clinic.

The doctor records a diagnosis.

The prescription fails to save.

Without rollback,

the patient record now contains an incomplete consultation.

Later,

another doctor reviews the patient's history and assumes treatment has already been prescribed.

Incorrect medical decisions may follow.

Transactions prevent these situations by preserving consistency.

---

# Production Example

Consider an online consultation.

The doctor submits the consultation form.

```
POST

/consultations/complete
```

Internally,

the Service performs:

```
Diagnosis

↓

Prescription

↓

Medical Record

↓

Appointment

↓

Commit
```

The client receives a successful response only after the transaction completes successfully.

---

# Repository Participation

Notice that individual repositories still perform one responsibility.

```ts
DiagnosisRepository

↓

create()
```

```ts
PrescriptionRepository

↓

createMany()
```

```ts
MedicalRecordRepository

↓

update()
```

Repositories remain independent.

The Service coordinates them inside one transaction.

---

# Common Mistakes

## Opening Multiple Transactions

Avoid creating a separate transaction for every Repository method.

Instead,

wrap the complete business workflow inside one transaction.

---

## Mixing External APIs

Avoid code such as:

```ts
await prisma.$transaction(async () => {
  await paymentGateway.charge()
})
```

External services cannot participate in database transactions.

Complete the database transaction first.

Then call external systems.

---

## Long Running Transactions

Avoid keeping a transaction open while:

- Uploading files.
- Sending emails.
- Waiting for HTTP responses.
- Performing slow calculations.

The shorter the transaction,

the better the database performs.

---

# Production Notes

Every transaction acquires database resources.

Large transactions increase:

- Lock duration.
- Deadlock risk.
- Database contention.
- Response time.

Design transactions to contain only the database operations that must succeed together.

---

# Summary

Prisma transactions allow multiple persistence operations to execute as a single business operation.

Instead of treating every query independently, the Service coordinates related Repository methods inside one transaction.

This guarantees that either every database change is committed successfully or every change is rolled back, preserving consistency across the entire application.

---

# Making Repositories Transaction-Aware

In the previous section, we executed multiple database operations inside a Prisma transaction.

```ts
await prisma.$transaction(async (tx) => {

    ...

});
```

However, a new problem appears.

Our repositories currently use the global Prisma client.

Example

```ts
export class DiagnosisRepository {
  async create(data: Prisma.DiagnosisCreateInput) {
    return prisma.diagnosis.create({
      data
    })
  }
}
```

Notice that the Repository always uses:

```ts
prisma
```

instead of

```ts
tx
```

This creates an important limitation.

---

# Why Is This a Problem?

Suppose the Service starts a transaction.

```
Diagnosis Service

↓

Transaction Started

↓

Diagnosis Repository

↓

Prescription Repository

↓

Appointment Repository
```

If every Repository continues using the global Prisma client,

their queries execute **outside** the transaction.

```
Transaction

↓

Diagnosis Repository

❌ Outside

↓

Prescription Repository

❌ Outside
```

If one Repository fails,

the previous queries have already been committed.

Rollback is no longer possible.

The transaction becomes ineffective.

---

# Understanding Prisma Transaction Client

When Prisma starts a transaction,

it creates a special database client.

Example

```ts
await prisma.$transaction(async tx => {})
```

Inside the callback,

```
tx
```

behaves exactly like the normal Prisma client.

```
prisma.user.create()

↓

tx.user.create()
```

```
prisma.appointment.update()

↓

tx.appointment.update()
```

The API is identical.

The difference is that every query executed through `tx` belongs to the same transaction.

---

# Refactoring the Repository

Instead of hardcoding the global Prisma client,

the Repository should receive the database client as a dependency.

---

## Before

```ts
import { prisma } from '../config/prisma'

export class DiagnosisRepository {
  async create(data: Prisma.DiagnosisCreateInput) {
    return prisma.diagnosis.create({
      data
    })
  }
}
```

---

## After

```ts
import { PrismaClient, Prisma } from '@prisma/client'

export class DiagnosisRepository {
  constructor(private readonly db: PrismaClient) {}

  async create(data: Prisma.DiagnosisCreateInput) {
    return this.db.diagnosis.create({
      data
    })
  }
}
```

---

# Explanation

Previously,

the Repository depended directly on

```ts
prisma
```

Now,

it depends on

```ts
db
```

The Repository no longer cares whether that object is

```
Global Prisma Client

or

Transaction Client
```

It simply executes database queries.

This makes the Repository reusable in both situations.

---

# Business Example

Normal request

```
Controller

↓

Service

↓

Repository

↓

Prisma Client

↓

Database
```

Transaction request

```
Controller

↓

Service

↓

Transaction Client

↓

Repository

↓

Database
```

The Repository behaves exactly the same.

Only the database client changes.

---

# Dependency Injection Benefits

Passing the database client through the constructor provides several advantages.

## Reusability

The Repository works with:

- Global Prisma client
- Transaction client

without modification.

---

## Testability

During unit testing,

a mocked database client can be injected.

Example

```ts
const repository = new DiagnosisRepository(mockPrisma)
```

No real database is required.

---

## Flexibility

Future implementations can replace the underlying client without changing Repository logic.

The Repository depends only on the interface provided by Prisma.

---

# Repository Responsibility

Notice that the Repository still performs only one task.

```
Database Client

↓

Repository

↓

Database Query
```

It does not know:

- whether a transaction exists,
- who started the transaction,
- how many repositories are participating.

That responsibility remains inside the Service layer.

---

# Common Mistakes

## Importing Prisma Everywhere

Avoid this pattern.

```ts
import { prisma } from '../config/prisma'
```

inside every Repository.

Hardcoded dependencies reduce flexibility.

---

## Starting Transactions Inside Repositories

Avoid code like:

```ts
prisma.$transaction(...)
```

inside Repository methods.

Repositories should execute queries,

not coordinate business workflows.

---

## Creating Multiple Prisma Clients

A Repository should never create its own Prisma client.

Example

```ts
const prisma = new PrismaClient()
```

inside a Repository.

Creating multiple clients wastes connections and increases resource usage.

---

# Production Notes

Large applications often inject the database client into every Repository.

This design allows:

- Standard queries
- Transactional queries
- Mocked clients for testing

using the same Repository implementation.

The Repository remains completely unaware of the surrounding workflow.

---

# Summary

Repositories should depend on an abstract database client rather than the global Prisma instance.

By accepting the database client through dependency injection, the same Repository can execute queries using either the standard Prisma client or Prisma's transaction client.

This design keeps Repositories reusable, testable, and fully compatible with transactional business workflows.

# Using Multiple Repositories Inside One Transaction

Now that our repositories can accept a database client, we can execute multiple repository operations inside a single transaction.

This is where the Service layer becomes the coordinator of the business workflow.

The Service understands:

- what needs to happen,
- in what order,
- and whether all operations should succeed together.

The repositories simply execute their individual persistence operations.

---

# Business Workflow

Consider a doctor completing a patient's consultation.

Several persistence operations are required.

```
Doctor

↓

Complete Consultation

↓

Diagnosis

↓

Prescription

↓

Medical Record

↓

Appointment Status
```

From the doctor's perspective,

this is one action.

From the database perspective,

it involves multiple tables.

The Service combines these operations into one transaction.

---

# Service Before Refactoring

Without transaction support,

the Service may look like this.

## Code

```ts
await diagnosisRepository.create(diagnosis)

await prescriptionRepository.createMany(prescriptions)

await medicalRecordRepository.update(recordId, record)

await appointmentRepository.complete(appointmentId)
```

---

# Problem

Every repository call executes independently.

```
Diagnosis

✓ Saved

↓

Prescription

✓ Saved

↓

Medical Record

✗ Failed
```

The database now contains incomplete information.

The patient's consultation is no longer consistent.

---

# Service After Refactoring

Instead,

the Service starts a transaction.

## Code

```ts
await prisma.$transaction(async tx => {
  const diagnosisRepository = new DiagnosisRepository(tx)

  const prescriptionRepository = new PrescriptionRepository(tx)

  const medicalRecordRepository = new MedicalRecordRepository(tx)

  const appointmentRepository = new AppointmentRepository(tx)

  await diagnosisRepository.create(diagnosis)

  await prescriptionRepository.createMany(prescriptions)

  await medicalRecordRepository.update(recordId, medicalRecord)

  await appointmentRepository.complete(appointmentId)
})
```

---

# Explanation

Notice what changed.

Previously,

repositories used the global Prisma client.

Now,

every repository receives the same transaction client.

```
Transaction

↓

tx

↓

Diagnosis Repository

↓

Prescription Repository

↓

Medical Record Repository

↓

Appointment Repository
```

Every query belongs to the same database transaction.

---

# Visualizing the Transaction

```
Diagnosis Service

│

├── Start Transaction

│

├── Diagnosis Repository

│

├── Prescription Repository

│

├── Medical Record Repository

│

├── Appointment Repository

│

└── Commit
```

If one Repository throws an error,

Prisma automatically rolls back every operation.

---

# Rollback Scenario

Suppose the Prescription Repository fails.

Execution becomes:

```
Diagnosis

✓

↓

Prescription

✗

↓

Rollback

↓

Diagnosis Removed

↓

Medical Record Not Updated

↓

Appointment Not Updated
```

The patient record remains exactly as it was before the transaction started.

---

# Why This Design Matters

Notice something important.

Repositories never communicate with each other.

```
Diagnosis Repository

×

Prescription Repository
```

There are no Repository-to-Repository calls.

Instead,

the Service coordinates everything.

```
Service

↓

Repository A

↓

Repository B

↓

Repository C
```

This keeps each Repository independent and reusable.

---

# Business Example

A doctor finishes a consultation.

The Service performs the following steps.

1. Save diagnosis.
2. Save prescriptions.
3. Update medical record.
4. Mark appointment as completed.

The doctor sees only one operation.

The database performs four.

The transaction guarantees that all four succeed together.

---

# Why Services Own Workflows

Repositories understand persistence.

Services understand business processes.

For example,

the Repository knows how to save a diagnosis.

The Service knows that creating a diagnosis also requires:

- prescriptions,
- medical record updates,
- appointment completion.

That orchestration belongs in the Service layer.

---

# Common Mistakes

## Calling One Repository From Another

Avoid designs like:

```ts
DiagnosisRepository

↓

PrescriptionRepository
```

Repositories should never depend on other repositories.

Keep them independent.

---

## Starting Multiple Transactions

Avoid wrapping each repository method inside its own transaction.

Doing so prevents the entire business workflow from rolling back together.

Use one transaction for one business operation.

---

## Creating New Repository Instances Everywhere

Inside a transaction,

all repositories must receive the same transaction client.

If one repository accidentally uses the global Prisma client,

its queries execute outside the transaction.

---

# Production Notes

In large applications,

Services often coordinate five or more repositories inside a single transaction.

Examples include:

- Completing a consultation.
- Processing an insurance claim.
- Creating an invoice.
- Processing an e-commerce order.
- Recording a financial transfer.

Repositories remain simple.

Services orchestrate the complete workflow.

This separation makes the application easier to understand, test, and maintain.

---

# Summary

Transactions become truly powerful when multiple repositories participate in the same business workflow.

The Service starts the transaction, injects the shared transaction client into every Repository, and coordinates the order of operations.

Each Repository continues to focus on one responsibility—executing database queries—while the Service ensures the overall workflow either succeeds completely or fails safely.

---

# Handling Errors Inside Transactions

Every database transaction has two possible outcomes.

```
Success

↓

Commit
```

or

```
Failure

↓

Rollback
```

There is no third option.

If any operation inside the transaction throws an error,

Prisma immediately stops executing the remaining queries and rolls back every completed operation.

This behavior guarantees that the database remains consistent.

---

# Example Scenario

Suppose a doctor completes a consultation.

The Service performs the following operations.

```
Diagnosis

↓

Prescription

↓

Medical Record

↓

Appointment
```

Now imagine that saving the prescription fails.

Without transactions,

the diagnosis would already exist in the database.

With transactions,

Prisma automatically restores the database to its previous state.

```
Diagnosis

✓ Created

↓

Prescription

✗ Failed

↓

Rollback

↓

Diagnosis Removed
```

After the rollback,

it is as though the transaction never started.

---

# Catching Transaction Errors

Although Prisma automatically performs the rollback,

the application should still handle the exception.

---

## Code

```ts
try {
  await prisma.$transaction(async tx => {
    const diagnosisRepository = new DiagnosisRepository(tx)

    const prescriptionRepository = new PrescriptionRepository(tx)

    await diagnosisRepository.create(diagnosis)

    await prescriptionRepository.createMany(prescriptions)
  })
} catch (error) {
  logger.error(error)

  throw new Error('Unable to complete consultation.')
}
```

---

# Explanation

Notice what the `catch` block does **not** contain.

It does **not** attempt to manually undo database changes.

For example,

avoid code such as:

```ts
await diagnosisRepository.delete(id)
```

Prisma has already rolled back every operation.

The application's responsibility is to:

- Log the error.
- Return an appropriate response.
- Allow monitoring systems to capture the failure.

---

# Business Example

A doctor submits a consultation.

```
Doctor

↓

Diagnosis Service

↓

Transaction

↓

Database Error

↓

Rollback

↓

Return Error Response
```

The doctor receives an error message,

and the patient's records remain unchanged.

# What Should Be Logged?

When a transaction fails,

production systems usually log information such as:

- Request ID
- User ID
- Doctor ID
- Patient ID
- Appointment ID
- Error message
- Stack trace
- Timestamp

Example

```text
Consultation Failed

Appointment:
APT-2041

Doctor:
DOC-102

Patient:
PAT-890

Reason:
Unique constraint violation
```

These logs help engineers investigate production issues without inspecting the database manually.

---

# What Should Not Be Logged?

Avoid logging sensitive medical information.

For example,

never log:

- Diagnosis description
- Clinical notes
- Prescription contents
- Patient medical history

Instead,

log identifiers.

```
Doctor ID

Patient ID

Appointment ID
```

Sensitive healthcare information should remain protected.

---

# Should We Retry the Transaction?

Not every failure should be retried.

Examples of retryable failures include:

- Temporary database connection loss.
- Deadlocks.
- Network interruptions.

Examples of non-retryable failures include:

- Validation errors.
- Unique constraint violations.
- Foreign key violations.
- Invalid business rules.

The Service should understand the difference.

Blindly retrying every transaction may create duplicate operations or unnecessary load.

---

# Repository Responsibilities During Failure

Notice that repositories remain unchanged.

```ts
DiagnosisRepository

↓

create()
```

If an error occurs,

the Repository simply throws it.

Repositories should never:

- Catch transaction errors.
- Retry queries.
- Roll back data.
- Convert database errors into HTTP responses.

Those responsibilities belong to higher layers.

---

# Error Flow

```
Database

↓

Repository

↓

Service

↓

Controller

↓

HTTP Response
```

Each layer handles only the responsibilities assigned to it.

---

# Common Mistakes

## Swallowing Exceptions

Avoid code such as:

```ts
catch (error) {

    console.log(error);

}
```

Ignoring exceptions makes production debugging extremely difficult.

Always log meaningful information and propagate the error appropriately.

---

## Returning HTTP Responses from Services

Services should not return:

```ts
res.status(500)
```

HTTP belongs to Controllers.

Services communicate by throwing exceptions or returning domain results.

---

## Logging Sensitive Data

Healthcare systems must protect patient privacy.

Log identifiers instead of confidential medical information.

---

## Manual Rollback Logic

Avoid writing cleanup code after a transaction failure.

Prisma automatically rolls back the transaction.

Adding manual rollback logic often introduces new bugs.

---

# Production Notes

In mature applications,

transaction failures are monitored through centralized logging systems.

Typical production stack:

```text
Application

↓

Logger

↓

Log Aggregation

↓

Monitoring

↓

Alerting
```

This allows engineers to detect recurring transaction failures before they affect users.

---

# Summary

Prisma automatically handles transaction rollback whenever an error occurs.

Repositories simply execute database queries and propagate exceptions.

The Service logs failures, translates them into business errors, and allows the Controller to return an appropriate HTTP response.

This separation of responsibilities keeps transaction handling predictable, maintainable, and consistent across the application.

# Production Transaction Patterns

Writing a transaction is only part of the job.

Designing transactions correctly is what separates production systems from small projects.

Experienced backend engineers spend more time deciding **where a transaction should begin and end** than writing the actual database queries.

---

# Keep Transactions Small

Transactions should contain only the operations that must succeed together.

Good Example

```
Start Transaction

↓

Create Diagnosis

↓

Create Prescriptions

↓

Update Medical Record

↓

Complete Appointment

↓

Commit
```

Every operation modifies the database and belongs to the same business workflow.

---

## Bad Example

```
Start Transaction

↓

Save Diagnosis

↓

Upload File

↓

Send Email

↓

Call Payment API

↓

Generate PDF

↓

Commit
```

This transaction stays open far longer than necessary.

While waiting for external systems,

database resources remain locked.

This reduces concurrency and increases response time.

---

# One Transaction Per Business Operation

A useful guideline is:

> One business operation should usually use one transaction.

Examples

```
Complete Consultation

↓

One Transaction
```

```
Create Patient

↓

One Transaction
```

```
Register Doctor

↓

One Transaction
```

Avoid combining unrelated workflows into the same transaction.

For example,

```
Create Patient

+

Generate Monthly Report

+

Send Welcome Email
```

These operations should execute independently.

---

# Keep External Services Outside Transactions

Database transactions protect only database operations.

External systems cannot participate in the same transaction.

Avoid this pattern.

```ts
await prisma.$transaction(async tx => {
  await diagnosisRepository.create(data)

  await emailService.sendPrescription()
})
```

If the email fails,

the diagnosis has already been committed or rolled back independently.

Instead,

complete the transaction first.

```text
Transaction

↓

Commit

↓

Send Email

↓

Generate PDF

↓

Push Notification
```

This keeps transactions short and predictable.

---

# Idempotency

Some requests may be retried.

For example,

the client loses its network connection immediately after submitting a consultation.

The user clicks **Submit** again.

Without protection,

the application may create duplicate diagnoses.

Production systems often use:

- Unique constraints
- Request identifiers
- Idempotency keys

to ensure that retrying the same request does not create duplicate records.

Although idempotency is a broader architectural topic,

it is closely related to transactions because both aim to preserve data consistency.

---

# Deadlocks

When multiple transactions attempt to update the same records simultaneously,

the database may detect a deadlock.

Example

```
Transaction A

↓

Appointment

↓

Medical Record
```

At the same time,

```
Transaction B

↓

Medical Record

↓

Appointment
```

Each transaction waits for the other.

The database resolves this by cancelling one transaction.

Production applications should:

- Keep transactions short.
- Access tables in a consistent order.
- Retry only when appropriate.

---

# Transaction Boundaries

The Service should clearly define where a transaction begins and ends.

```
Controller

↓

Service

↓

Start Transaction

↓

Repository Operations

↓

Commit

↓

Return Response
```

Everything outside these boundaries should execute independently.

---

# Repository Review Checklist

Before introducing a transaction into a project, verify the following.

✓ Repositories contain only persistence logic.

✓ Services coordinate business workflows.

✓ Every repository receives the same transaction client.

✓ No repository starts its own transaction.

✓ External API calls remain outside the transaction.

✓ Transactions are short and focused.

✓ Errors are logged appropriately.

✓ Database constraints support business rules.

Following this checklist reduces bugs and improves maintainability.

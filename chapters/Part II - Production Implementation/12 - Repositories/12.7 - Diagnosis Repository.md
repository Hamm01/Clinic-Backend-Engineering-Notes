# 12.7 - Diagnosis Repository

> The previous repositories focused on storing and retrieving individual entities such as Doctors, Patients, and Appointments. The Diagnosis Repository introduces a different challenge. A diagnosis rarely exists by itself. It is usually created together with prescriptions, laboratory tests, and updates to a patient's medical history. In this chapter, we will learn how repositories support complex persistence operations while keeping business logic inside the Service layer.

# Why Is Diagnosis Repository Different?

Doctors can exist independently.

Patients can exist independently.

Appointments can exist independently.

A diagnosis usually cannot.

A diagnosis is created during a consultation and becomes part of a patient's medical history.

```
Patient

↓

Appointment

↓

Diagnosis

↓

Prescription

↓

Laboratory Tests
```

This makes diagnosis persistence significantly more complex than previous repositories.

# Repository Responsibilities

The Diagnosis Repository is responsible for persistence operations such as:

- Creating diagnoses.
- Retrieving diagnoses.
- Updating diagnosis information.
- Loading diagnosis history.
- Loading related prescriptions.
- Loading laboratory tests.

It is **not** responsible for deciding:

- Whether the doctor may create a diagnosis.
- Whether the appointment is completed.
- Whether the patient is currently admitted.
- Whether medication should be prescribed.

Those are business rules handled by the Service layer.

# Implementing create()

Creating a diagnosis is one of the most important persistence operations in the Clinic Backend.

Unlike creating a Doctor or Patient, a diagnosis is usually associated with an existing appointment.

The Repository is responsible only for storing the diagnosis.

It does not verify:

- Whether the appointment exists.
- Whether the doctor is assigned to the appointment.
- Whether the appointment has already been completed.
- Whether the diagnosis is medically valid.

Those validations belong to the Service layer.

---

## Code

```ts
import { Diagnosis, Prisma } from '@prisma/client'

import { prisma } from '../config/prisma'

export class DiagnosisRepository {
  async create(data: Prisma.DiagnosisCreateInput): Promise<Diagnosis> {
    return prisma.diagnosis.create({
      data
    })
  }
}
```

---

# Explanation

The Repository receives a validated diagnosis object.

```
DiagnosisService

↓

DiagnosisRepository.create()

↓

Prisma

↓

PostgreSQL
```

Its responsibility is simple.

Persist the diagnosis.

# Implementing findByAppointment()

In most healthcare systems, every completed appointment has one diagnosis.

Retrieving a diagnosis using the Appointment ID is therefore a common persistence operation.

---

## Code

```ts
async findByAppointment(
    appointmentId: string
): Promise<Diagnosis | null> {

    return prisma.diagnosis.findFirst({

        where: {

            appointmentId

        }

    });

}
```

---

# Explanation

Instead of searching by the Diagnosis ID,

the Repository searches using the Appointment ID.

```
Appointment

↓

Diagnosis

↓

Diagnosis Record
```

This matches how the business actually operates.

Doctors usually know the appointment,

not the diagnosis identifier.

---

# Business Example

```
Appointment Details

↓

Open Consultation

↓

DiagnosisRepository.findByAppointment()

↓

Diagnosis
```

This allows the Service to retrieve the diagnosis directly from the appointment workflow.

---

# Production Notes

If the business guarantees one diagnosis per appointment,

consider enforcing a unique constraint at the database level.

Doing so improves data integrity and allows `findUnique()` instead of `findFirst()`.

# Implementing findByPatient()

Patients accumulate diagnoses throughout their lifetime.

Medical professionals frequently review a patient's diagnosis history before making new clinical decisions.

The Repository provides a dedicated method for retrieving diagnoses by patient.

---

## Code

```ts
async findByPatient(
    patientId: string
): Promise<Diagnosis[]> {

    return prisma.diagnosis.findMany({

        where: {

            patientId

        },

        orderBy: {

            createdAt: "desc"

        }

    });

}
```

---

# Explanation

The Repository retrieves every diagnosis associated with a patient.

Results are sorted from newest to oldest.

```
Patient

↓

Diagnoses

↓

Newest First
```

Displaying recent diagnoses first matches the workflow followed by healthcare professionals during consultations.

---

# Business Example

DiagnosisRepository.findByPatient()

The Service receives the patient's complete diagnosis history and decides how to present or process it.

# Common Mistakes

## Mixing Medical Decisions into the Repository

Avoid logic such as:

```ts
if (diagnosis.requiresAdmission)
```

Repositories should never make clinical decisions.

---

## Searching by Incorrect Business Identifier

Doctors usually navigate through appointments and patients.

Provide Repository methods that match these workflows instead of forcing Services to translate between identifiers.

---

## Returning Unsorted Medical History

Diagnosis history should generally be returned in descending chronological order so that the most recent consultation appears first.

---

## Duplicating Queries

If multiple Services require patient diagnosis history,

avoid repeating Prisma queries.

Expose a dedicated Repository method instead.

# Loading Related Data

A diagnosis is rarely viewed on its own.

When a doctor opens a patient's consultation, they usually need additional information such as:

- Patient Details
- Doctor Details
- Appointment Information
- Prescriptions
- Laboratory Tests

Instead of executing multiple database queries,

the Repository can retrieve related data in a single operation.

---

## Code

```ts
async findWithRelations(
    id: string
): Promise<Diagnosis | null> {

    return prisma.diagnosis.findUnique({

        where: {

            id

        },

        include: {

            patient: true,

            doctor: true,

            appointment: true,

            prescriptions: true

        }

    });

}
```

---

# Explanation

Using Prisma's `include` option,

related entities are retrieved together.

```
Diagnosis

├── Patient

├── Doctor

├── Appointment

└── Prescriptions
```

The Repository returns a complete diagnosis instead of requiring multiple Repository calls.

---

# Business Example

Suppose a doctor opens a consultation.

The application displays:

- Diagnosis
- Patient Information
- Prescriptions

Workflow

```
Doctor

↓

Diagnosis Details

↓

DiagnosisRepository.findWithRelations()

↓

Database
```

Everything required for the screen is loaded in one Repository method.

---

# Why Use include?

Without `include`,

the Service would perform multiple queries.

Example:

```
Diagnosis

↓

Patient

↓

Doctor

↓

Prescription

↓

Appointment
```

This increases code complexity.

The Repository should encapsulate these persistence operations.

# Selecting Specific Fields

Sometimes,

the application needs only a small subset of information.

For example,

the patient dashboard may only display:

- Diagnosis Title
- Diagnosis Date
- Doctor Name

Instead of loading every column,

we use `select`.

---

## Code

```ts
async findSummary(
    id: string
) {

    return prisma.diagnosis.findUnique({

        where: {

            id

        },

        select: {

            id: true,

            diagnosis: true,

            createdAt: true,

            doctor: {

                select: {

                    firstName: true,

                    lastName: true

                }

            }

        }

    });

}
```

---

# Explanation

Only the required fields are returned.

```
Diagnosis

↓

Selected Fields

↓

Smaller Response
```

Benefits include:

- Faster queries
- Reduced memory usage
- Smaller API responses

# Preparing for Transactions

Saving a diagnosis often involves more than one table.

Example:

```
Diagnosis

+

Prescription

+

Medical Record

+

Lab Tests
```

If one database operation succeeds but another fails,

the application's data becomes inconsistent.

For example,

```
Diagnosis Saved

✓

Prescription Failed

✗
```

The patient now has a diagnosis without medication.

This is an inconsistent state.

---

# Why Transactions Matter

Imagine a doctor completes a consultation.

The application needs to:

1. Save the diagnosis.
2. Save prescriptions.
3. Save laboratory tests.
4. Update the patient's medical history.

Either all operations should succeed,

or none of them should.

This principle is known as **atomicity**.

The Repository itself should not decide when to start a transaction,

but it should provide methods that can safely participate in one.

In the next implementation chapter, we will combine multiple Repository operations using Prisma transactions.

---

# Common Mistakes

## Loading Every Relationship

Avoid retrieving unnecessary related data.

Only use `include` for information required by the current business operation.

---

## Returning Entire Objects

When only a few fields are needed,

prefer `select`.

Returning smaller responses improves application performance.

---

## Transaction Logic Inside Controllers

Controllers should never manage database transactions.

Transaction orchestration belongs inside the Service layer,

while Repositories provide the persistence methods used within those transactions.

---

## Mixing Business Decisions

Avoid logic such as

```ts
if (diagnosis.requiresAdmission)
```

Repositories should never make medical or business decisions.

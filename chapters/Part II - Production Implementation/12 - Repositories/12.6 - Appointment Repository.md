# Repository Responsibilities

The Appointment Repository is responsible for persistence operations such as:

- Creating appointments.
- Updating appointments.
- Finding appointments.
- Cancelling appointments.
- Loading schedules.
- Dashboard queries.
- Date filtering.

It is **not** responsible for deciding:

- Whether the doctor is available.
- Whether the clinic is open.
- Whether the patient already has another appointment.

Those decisions belong to the Service layer.

# Implementing create()

The first responsibility of the `AppointmentRepository` is creating a new appointment.

Unlike previous repositories, creating an appointment connects multiple business entities.

An appointment typically references:

- Patient
- Doctor
- Clinic
- Appointment Date
- Appointment Status

The Repository is responsible only for storing this information.

It does **not** verify:

- Whether the doctor is available.
- Whether the patient already has an appointment.
- Whether the clinic is open.
- Whether the appointment time overlaps another appointment.

Those are business rules implemented inside the Service layer.

---

## Code

```ts
import { Appointment, Prisma } from '@prisma/client'

import { prisma } from '../config/prisma'

export class AppointmentRepository {
  async create(data: Prisma.AppointmentCreateInput): Promise<Appointment> {
    return prisma.appointment.create({
      data
    })
  }
}
```

---

# Explanation

The Repository receives validated appointment data from the Service.

```

Prisma

PostgreSQL
```

The Repository performs one responsibility.

Persist the appointment.

Nothing more.

# Implementing findByDoctor()

One of the most common scheduling operations is retrieving all appointments assigned to a doctor.

Examples include:

- Doctor Dashboard
- Daily Schedule
- Upcoming Consultations

---

## Code

```ts
async findByDoctor(
    doctorId: string
): Promise<Appointment[]> {

    return prisma.appointment.findMany({

        where: {

            doctorId,

            deletedAt: null

        },

        orderBy: {

            appointmentDate: "asc"

```

---

# Explanation


Notice that the sorting logic is centralized inside the Repository rather than repeated in every Service.

# Common Mistakes

## Mixing Business Rules

Avoid code like:

```ts

```

The Repository should not decide whether another appointment may be scheduled.

That belongs inside the Service.

---

## Returning Unsorted Results

Appointment lists should generally have a predictable order.

Centralize sorting inside the Repository so every caller receives consistent results.

---

## Duplicating Scheduling Queries

If multiple Services require a doctor's appointments,

avoid repeating Prisma queries.

Create one Repository method and reuse it throughout the application.

# Implementing findToday()

One of the most common queries in a scheduling system is retrieving today's appointments.

Examples include:

- Reception Dashboard
- Doctor Dashboard
- Daily Schedule
- Check-In Queue

Instead of loading every appointment,

we only retrieve appointments scheduled for today.

---

## Code

```ts
async findToday(): Promise<Appointment[]> {

    const startOfDay = new Date();

    startOfDay.setHours(0, 0, 0, 0);

    const endOfDay = new Date();

    endOfDay.setHours(23, 59, 59, 999);

    return prisma.appointment.findMany({

        where: {

            appointmentDate: {

                gte: startOfDay,

                lte: endOfDay

            },

            deletedAt: null

        },

        orderBy: {

            appointmentDate: "asc"

        }

    });

}
```

---

# Explanation

The Repository calculates today's boundaries.

```ts
00:00:00

↓

23:59:59
```

Prisma then retrieves appointments that fall within that range.

```
Today

↓

Appointments

↓

Sorted Chronologically
```

This method is frequently used throughout healthcare systems.

---

# Business Example

# Implementing cancel()

In real-world healthcare systems, appointments are rarely deleted.

Instead, they are cancelled.

This preserves historical information while accurately reflecting the appointment's lifecycle.

Typical reasons include:

- Patient cancelled.
- Doctor unavailable.
- Clinic closed.
- Emergency situations.

Instead of deleting the appointment, we update its status.

---

## Code

```ts
async cancel(
    id: string
): Promise<Appointment> {

    return prisma.appointment.update({

        where: {

            id

        },

        data: {

            status: "CANCELLED"

        }

    });

}
```

---

# Explanation

The Repository updates only the appointment status.

```
Appointment

↓

Status

↓

CANCELLED
```

The appointment continues to exist in the database.

Only its state changes.

---

# Why Not Delete the Appointment?

Suppose a patient cancels an appointment.

The clinic may still need to know:

- Who cancelled?
- When was it cancelled?
- Which doctor was assigned?
- How many cancellations occurred this month?

Deleting the record permanently would remove valuable business information.

Status updates preserve historical data.

# Loading Related Data

Appointments are relationship-heavy entities.

A single appointment usually requires information from several tables.

Examples:

- Patient
- Doctor
- Clinic

Prisma's `include` option makes this straightforward.

---

## Code

```ts
async findByIdWithRelations(
    id: string
): Promise<Appointment | null> {

    return prisma.appointment.findUnique({

        where: {

            id

        },

        include: {

            patient: true,

            doctor: true,

            clinic: true

        }

    });

}
```

---

# Explanation

Instead of returning only the appointment,

the Repository loads related entities in one query.

```
Appointment

├── Patient

├── Doctor

└── Clinic
```

The Service receives a complete view of the appointment without performing additional database queries.

# Preparing for Pagination

As the number of appointments grows,

loading every record becomes inefficient.

Production systems introduce pagination.

Typical API requests look like:

```
GET /appointments?page=2&limit=20
```

The Repository supports this using `skip` and `take`.

---

## Code

```ts
async findPaginated(

    page: number,

    limit: number

): Promise<Appointment[]> {

    return prisma.appointment.findMany({

        skip: (page - 1) * limit,

        take: limit,

        where: {

            deletedAt: null

        },

        orderBy: {

            appointmentDate: "asc"

        }

    });

}
```

---

# Explanation

Pagination retrieves only the requested records.

```
Database

↓

Appointments

↓

20 Records

↓

API Response
```

Instead of loading thousands of appointments,

only the required page is returned.

# Repository Review Checklist

Before merging this Repository into production, verify:

- All Prisma queries are inside the Repository.
- Business validation remains inside the Service.
- Related entities are loaded only when required.
- Pagination is used for large result sets.
- Status changes update records instead of deleting them.
- Methods have descriptive names.
- Duplicate query logic is minimized.

Following this checklist keeps the persistence layer maintainable and scalable.

---

# Common Mistakes

## Deleting Appointments

Avoid permanently deleting appointments.

Use status changes or soft deletes unless business requirements explicitly demand removal.

---

## Loading Every Relationship

Do not use `include` indiscriminately.

Retrieve only the related entities required by the current use case.

---

## Missing Pagination

Large datasets should never be returned without pagination.

Loading thousands of appointments into memory can significantly impact performance.

---

## Business Logic in Repository

Avoid methods such as:

```ts
assignDoctor()

approveAppointment()

sendReminder()
```

These are business workflows.

Repositories should only retrieve and persist data.

---

Design repository methods around these access patterns to maintain predictable performance.

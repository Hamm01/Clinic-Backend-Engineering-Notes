# Why Another Repository?

At first glance,

DoctorRepository and PatientRepository seem similar.

Both perform CRUD operations.

So why build another Repository?

Because every business entity has different persistence requirements.

Doctors are primarily searched by:

- Email
- License Number
- Specialization

Patients are searched by:

- Clinic
- Mobile Number
- Medical Record Number
- Appointment History

The Repository should reflect the needs of the business entity it manages.

# Explanation

The structure is identical to the DoctorRepository.

- communicates only with the database.
- contains persistence methods.
- never contains business rules.

Consistency across repositories makes the project easier to navigate.

# Repository Responsibilities

The PatientRepository answers questions such as:

- How do we create a patient?
- How do we retrieve a patient?
- How do we update patient information?
- How do we search patients?

It does **not** answer:

- Can this patient register?
- Is this clinic active?
- Is this appointment allowed?

Those remain Service responsibilities.

# Implementing create()

The first responsibility of the `PatientRepository` is creating a new patient record.

Unlike the `DoctorRepository`, a patient usually belongs to a clinic and may later be associated with appointments, diagnoses, prescriptions, and medical records.

The Repository is only responsible for persisting the patient.

It does not verify whether:

- the clinic exists,
- the doctor exists,
- the patient already has an appointment.

Those validations belong to the Service layer.

---

## Code

```ts
import { Prisma, Patient } from '@prisma/client'
import { prisma } from '../config/prisma'

export class PatientRepository {
  async create(data: Prisma.PatientCreateInput): Promise<Patient> {
    return prisma.patient.create({
      data
    })
  }
}
```

---

# Explanation

The Repository receives validated data from the Service.

```
PatientService

↓

PatientRepository.create()

↓

Prisma

↓

PostgreSQL
```

The Repository does one thing:

Persist the Patient.

---

# Business Example

Suppose a receptionist registers a patient.

```
Receptionist

↓

PatientController

↓

PatientService

↓

PatientRepository.create()

↓

Database
```

The Repository stores the patient record.

It does not decide whether registration is allowed.

# Implementing findById()

Almost every patient workflow begins by loading a patient.

Examples include:

- View Profile
- Book Appointment
- Update Information
- Medical History
- Diagnosis

---

## Code

```ts
async findById(
    id: string
): Promise<Patient | null> {

    return prisma.patient.findUnique({

        where: {
            id
        }

    });

}
```

---

# Explanation

Because the `id` field is unique,

Prisma guarantees that only one Patient is returned.

```
Patient ID

↓

findUnique()

↓

Patient

or

null
```

If no record exists,

the Repository simply returns `null`.

Handling missing patients remains a business decision handled by the Service.

---

# Production Notes

Repositories should avoid throwing business exceptions such as:

```ts
throw new Error('Patient not found')
```

Instead,

allow the Service to decide how missing records affect the business workflow.

# Implementing exists()

Sometimes we only need to verify that a patient exists.

Loading the complete record would be unnecessary.

---

## Code

```ts
async exists(
    id: string
): Promise<boolean> {

    const patient =
        await prisma.patient.findUnique({

            where: {

                id

            },

            select: {

                id: true

            }

        });

    return patient !== null;

}
```

---

# Explanation

Only the primary key is selected.

```
Database

↓

Patient ID

↓

Exists?

↓

true / false
```

This avoids transferring unnecessary data from the database.

---

# Business Example

Before creating an appointment,

the Service performs:

```
AppointmentService

↓

patientRepository.exists()

↓

Continue

↓

AppointmentRepository.create()
```

The Repository confirms existence.

The Service decides what to do next.

# Common Mistakes

## Using Phone Numbers Inside Services

Avoid repeating Prisma queries throughout Services.

Instead of

```ts
await prisma.patient.findUnique(...)
```

call

```ts
patientRepository.findByPhone(...)
```

---

## Returning HTTP Responses

Repositories should never know about Express.

Avoid:

```ts
res.status(404)
```

---

## Business Validation

Avoid:

```ts
if (patient.age < 18)
```

Age validation belongs inside the Service.

---

# Production Notes

Repositories should expose persistence methods that match the language of the business.

Instead of forcing Services to build database queries,

Repositories should provide expressive methods such as:

```ts
findByPhone()

findByClinic()

findByMedicalRecordNumber()

findRecentPatients()
```

This keeps Services focused on workflows while Repositories remain the single source of truth for persistence.

# Loading Related Data

In healthcare applications, a patient rarely exists in isolation.

A patient is connected to multiple business entities.

Examples include:

- Clinic
- Doctor
- Appointments
- Diagnoses
- Prescriptions
- Medical Records

Very often, the application needs more than just the Patient record.

---

# Using include

Prisma allows us to load related entities together.

## Code

```ts
async findByIdWithAppointments(
    id: string
): Promise<Patient | null> {

    return prisma.patient.findUnique({

        where: {
            id
        },

        include: {

            appointments: true

        }

    });

}
```

---

# Explanation

Normally,

```ts
findUnique()
```

returns only the Patient.

Using

```ts
include
```

Prisma automatically loads the patient's appointments.

```
Patient

↓

Appointments

↓

Single Database Query
```

Instead of executing multiple queries,

the Repository returns everything required for the business operation.

---

# Business Example

Suppose the doctor opens a patient's profile.

The application should display:

- Patient Information
- Previous Appointments

The workflow becomes

```
Doctor

↓

Patient Profile

↓

PatientRepository.findByIdWithAppointments()

↓

Database
```

The Service receives everything it needs.

---

# Loading Multiple Relationships

Prisma allows multiple relationships to be loaded together.

## Code

```ts
async findCompleteProfile(
    id: string
): Promise<Patient | null> {

    return prisma.patient.findUnique({

        where: {

            id

        },

        include: {

            clinic: true,

            appointments: true,

            diagnoses: true,

            prescriptions: true

        }

    });

}
```

---

# Explanation

This query retrieves a complete patient profile.

```
Patient

├── Clinic

├── Appointments

├── Diagnoses

└── Prescriptions
```

Repositories expose business-oriented persistence methods.

The Service simply requests the complete profile.

# Selecting Specific Fields

Sometimes,

loading complete objects is unnecessary.

Suppose we only need:

- Name
- Phone
- Date of Birth

Instead of returning every column,

we use `select`.

---

## Code

```ts
async findPatientSummary(
    id: string
) {

    return prisma.patient.findUnique({

        where: {

            id

        },

        select: {

            id: true,

            firstName: true,

            lastName: true,

            phone: true,

            dateOfBirth: true

        }

    });

}
```

---

# Why Use select?

Retrieving fewer columns improves:

- Query performance.
- Memory usage.
- Network traffic.
- API response size.

Large healthcare systems process thousands of requests every minute.

Small optimizations become important over time.

---

# Building Reusable Query Objects

Notice that many methods contain the same filter.

```ts
deletedAt: null
```

Instead of repeating it,

extract reusable query objects.

---

## Code

```ts
const activePatientWhere = {
  deletedAt: null
}
```

Reuse it.

```ts
async findMany() {

    return prisma.patient.findMany({

        where: activePatientWhere

    });

}
```

---

# Why Is This Better?

Without reusable query objects,

the same condition appears throughout the Repository.

Eventually,

one method forgets the filter.

Soft deleted patients begin appearing unexpectedly.

Centralizing common filters reduces maintenance effort and improves consistency.

# Repository Review Checklist

Before merging a Repository into production, verify the following.

- Database queries exist only inside the Repository.
- No business validation exists.
- The shared Prisma Client is reused.
- Query methods have meaningful names.
- `select` is used where complete objects are unnecessary.
- `include` is used only when relationships are required.
- Repeated query conditions are extracted.
- Soft deleted records are handled consistently.

Following this checklist helps maintain a clean persistence layer.

---

# Common Mistakes

## Returning Every Relationship

Avoid loading all related entities by default.

Only include relationships required for the current business operation.

---

## Returning Sensitive Information

Repositories should avoid exposing unnecessary columns.

Use `select` whenever only part of an entity is needed.

---

## Mixing Repository and Service Logic

Avoid methods like

```ts
scheduleAppointment()

approveRegistration()

calculateOutstandingBills()
```

These are business workflows.

Repositories only retrieve and persist data.

---

## Duplicate Query Conditions

Repeated filters should be extracted into reusable query objects.

This reduces bugs and improves readability.

# 12.11 - Common Mistakes

> Repositories are one of the most important layers in a backend application. A poorly designed Repository can introduce tight coupling, poor performance, duplicated logic, and maintenance challenges. This chapter reviews the most common mistakes developers make when implementing Repositories and explains how to avoid them in production systems.

When reviewing a Repository, experienced backend engineers rarely ask:

Instead, they ask:

- Is the responsibility clear?
- Is business logic placed in the correct layer?
- Is the query efficient?
- Is the Repository reusable?

# Business Logic Inside the Repository

One of the most common Repository mistakes is placing business rules inside Repository methods.

Repositories exist to communicate with the database.

They should retrieve, create, update, and delete data.

They should **not** make business decisions.

When business logic is mixed with data access,

the Repository becomes difficult to reuse, test, and maintain.

---

# Incorrect Implementation

Consider a Repository method that retrieves a patient.

## Code

```ts
async findActivePatient(
    id: string
): Promise<Patient> {

    const patient = await prisma.patient.findUnique({

        where: {

            id

        }

    });

    if (!patient) {

        throw new Error("Patient not found");

    }

    if (!patient.isActive) {

        throw new Error("Patient account is inactive");

    }

    return patient;

}
```

---

# What's Wrong?

The Repository performs two different responsibilities.

```
Repository

↓

Database Access

+

Business Decisions
```

Checking whether a patient exists is acceptable.

Deciding whether an inactive patient can access the system is a business rule.

Business rules belong in the Service layer.

---

# Why Is This a Problem?

Suppose another feature needs inactive patients.

For example:

- Admin Dashboard
- Audit Reports
- Historical Records

The existing Repository method can no longer be reused because it always throws an exception.

The Repository has become tightly coupled to one business requirement.

---

# Correct Implementation

The Repository should only retrieve the data.

## Code

```ts
async findById(
    id: string
): Promise<Patient | null> {

    return prisma.patient.findUnique({

        where: {

            id

        }

    });

}
```

---

# Business Logic Belongs in the Service

## Code

```ts
const patient = await patientRepository.findById(id)

if (!patient) {
  throw new NotFoundException('Patient not found')
}

if (!patient.isActive) {
  throw new BadRequestException('Patient account is inactive')
}

return patient
```

---

# Explanation

The responsibilities are now clearly separated.

```
Repository

↓

Retrieve Patient
```

```
Service

↓

Validate Business Rules

↓

Return Result
```

Each layer has a single responsibility.

---

# Business Example

The clinic application has two features.

Doctor Portal

↓

Only active patients may be viewed.

---

Admin Portal

↓

Both active and inactive patients must be visible.

Using the corrected design,

both features reuse the same Repository method.

Each Service applies its own business rules.

The Repository remains reusable.

---

# Benefits

Keeping business logic out of Repositories provides several advantages.

- Better separation of concerns.
- Easier unit testing.
- Improved reusability.
- Simpler Repository methods.
- Easier maintenance.

Repositories become focused entirely on data persistence.

---

# Common Mistakes

## Throwing Business Exceptions

Avoid placing business validation inside Repository methods.

---

## Returning Different Results Based on Business Rules

Repositories should return database results.

Services should decide how those results are handled.

---

## Mixing Responsibilities

A Repository should not decide whether an operation is allowed.

It should only perform the requested database operation.

---

# Production Notes

In production applications,

Repositories are intentionally simple.

Business workflows evolve frequently.

Keeping business logic inside Services allows those workflows to change without modifying the data access layer.

This separation makes the architecture more flexible and easier to maintain.

---

# Summary

Repositories should focus exclusively on data persistence.

Business decisions, validations, and workflow rules belong in the Service layer.

Separating these responsibilities results in cleaner, more reusable, and more maintainable backend architectures.

# Accessing Prisma Outside the Repository

Another common mistake is using the Prisma Client directly inside Controllers or Services.

Although this approach works,

it breaks the application's architecture by bypassing the Repository layer.

Repositories exist to centralize all database access.

Every database operation should pass through the Repository before reaching Prisma.

---

# Incorrect Implementation

Suppose a Service needs to retrieve a patient.

Instead of calling the Repository,

it accesses Prisma directly.

## Code

```ts
export class PatientService {
  async getPatient(id: string) {
    return prisma.patient.findUnique({
      where: {
        id
      }
    })
  }
}
```

---

# What's Wrong?

The Service is performing two responsibilities.

```
Service

↓

Business Logic

+

Database Access
```

The Repository layer is completely skipped.

This breaks the separation of responsibilities established throughout the application.

---

# Why Is This a Problem?

Imagine another Service also needs to retrieve a patient.

It also writes:

```ts
prisma.patient.findUnique(...)
```

Later,

a third Service does the same.

Eventually,

the same query exists in multiple places.

```
Patient Service

↓

Prisma

-------------------

Appointment Service

↓

Prisma

-------------------

Doctor Service

↓

Prisma
```

The application now contains duplicated database logic.

---

# Correct Implementation

The Service should communicate only with the Repository.

## Repository

```ts
export class PatientRepository {
  async findById(id: string): Promise<Patient | null> {
    return prisma.patient.findUnique({
      where: {
        id
      }
    })
  }
}
```

---

## Service

```ts
export class PatientService {
  constructor(private readonly patientRepository: PatientRepository) {}

  async getPatient(id: string) {
    return this.patientRepository.findById(id)
  }
}
```

---

# Explanation

Responsibilities are now clearly separated.

```
Controller

↓

Service

↓

Repository

↓

Prisma

↓

Database
```

Every layer performs only one job.

---

# Business Example

Suppose the clinic introduces soft deletes.

Previously,

every Prisma query looked like this.

```ts
prisma.patient.findMany()
```

Now every query must exclude deleted patients.

```ts
where: {
  deletedAt: null
}
```

If Prisma is used directly across multiple Services,

every Service must be updated individually.

If all database access passes through the Repository,

the change is made in one place.

---

# Benefits

Using the Repository as the single database access layer provides:

- Centralized database logic.
- Easier maintenance.
- Better code reuse.
- Consistent query implementation.
- Improved testing.
- Cleaner architecture.

---

# Common Mistakes

## Injecting Prisma into Services

Services should depend on Repositories,

not the ORM.

---

## Mixing Repository and Prisma

Avoid implementations like this.

```ts
const patient = await patientRepository.findById(id);

const appointments = await prisma.appointment.findMany(...);
```

A Service should either use Repository methods or introduce another Repository.

Database access should remain centralized.

---

## Duplicating Queries

The same Prisma query should not appear in multiple Services.

Repositories exist to prevent duplication.

---

# Production Notes

In production applications,

changing database behavior should require modifications in as few places as possible.

Centralizing database access inside Repositories makes it easier to:

- Introduce caching.
- Add logging.
- Optimize queries.
- Change the ORM.
- Modify database schemas.

Without a Repository layer,

these changes become significantly more expensive.

---

# Summary

Services should never communicate directly with Prisma.

All database operations should pass through the Repository layer.

This keeps database access centralized, reduces duplication, and preserves the layered architecture of the application.

# Repository Calling Another Repository

Another common architectural mistake is allowing one Repository to depend on another Repository.

At first,

this may seem like a good way to reuse existing code.

However,

Repositories are responsible only for accessing their own data source.

They should remain independent.

Coordinating multiple Repositories is the responsibility of the Service layer.

---

# Incorrect Implementation

Suppose the Appointment Repository needs patient information.

A developer writes:

## Code

```ts
export class AppointmentRepository {
  constructor(private readonly patientRepository: PatientRepository) {}

  async findAppointmentWithPatient(appointmentId: string) {
    const appointment = await prisma.appointment.findUnique({
      where: {
        id: appointmentId
      }
    })

    if (!appointment) {
      return null
    }

    const patient = await this.patientRepository.findById(appointment.patientId)

    return {
      appointment,

      patient
    }
  }
}
```

---

# What's Wrong?

The Appointment Repository now depends on another Repository.

```
Appointment Repository

↓

Patient Repository

↓

Prisma

↓

Database
```

Instead of remaining responsible for appointment persistence,

it now coordinates multiple data sources.

This is not the Repository's responsibility.

---

# Why Is This a Problem?

As the application grows,

Repositories become tightly coupled.

```
Doctor Repository

↓

Appointment Repository

↓

Patient Repository

↓

Diagnosis Repository
```

Eventually,

changing one Repository may unexpectedly affect several others.

The Repository layer becomes difficult to understand and maintain.

---

# Correct Implementation

Each Repository should work independently.

## Appointment Repository

```ts
async findById(
    id: string
): Promise<Appointment | null> {

    return prisma.appointment.findUnique({

        where: {

            id

        }

    });

}
```

---

## Patient Repository

```ts
async findById(
    id: string
): Promise<Patient | null> {

    return prisma.patient.findUnique({

        where: {

            id

        }

    });

}
```

---

## Service

```ts
const appointment = await appointmentRepository.findById(id)

if (!appointment) {
  throw new Error('Appointment not found')
}

const patient = await patientRepository.findById(appointment.patientId)

return {
  appointment,

  patient
}
```

---

# Explanation

Responsibilities are now clearly separated.

```
Service

├── Appointment Repository

└── Patient Repository
```

Repositories retrieve data.

The Service combines that data into a business response.

---

# Business Example

The doctor opens an appointment.

The application needs:

- Appointment details
- Patient information
- Doctor profile
- Diagnosis history

This is a business workflow.

The Service coordinates multiple Repository calls.

No Repository should orchestrate other Repositories.

---

# When Is It Acceptable?

Repositories may execute joins or use Prisma's `include` to retrieve related data from the database.

For example:

```ts
await prisma.appointment.findMany({
  include: {
    patient: true
  }
})
```

This is perfectly acceptable because:

- It is a single database query.
- No Repository depends on another Repository.
- The Repository still owns the data access.

The problem is **Repository-to-Repository dependency**, not loading related entities.

---

# Benefits

Keeping Repositories independent provides:

- Lower coupling.
- Better separation of concerns.
- Easier testing.
- Simpler dependency graphs.
- More predictable architecture.

---

# Common Mistakes

## Injecting Repositories into Other Repositories

Repositories should not depend on each other.

---

## Orchestrating Business Workflows

Repositories should retrieve data.

Services should coordinate workflows.

---

## Creating Circular Dependencies

Repository A

↓

Repository B

↓

Repository C

↓

Repository A

Circular dependencies make applications difficult to maintain and often introduce runtime issues.

---

# Production Notes

In production systems,

Repositories are treated as independent data access components.

Business operations frequently require data from multiple entities,

but coordinating those operations belongs in the Service layer.

Keeping Repositories independent makes the architecture easier to scale, test, and modify.

---

# Summary

Repositories should never orchestrate other Repositories.

Each Repository should manage access to its own data, while Services coordinate multiple Repository calls to implement business workflows.

This separation keeps the architecture clean, maintainable, and aligned with the Repository Pattern.

# Generic Repository Abuse

Many developers attempt to reduce duplicated code by creating a single Generic Repository.

The idea appears attractive.

Instead of creating:

- PatientRepository
- DoctorRepository
- AppointmentRepository

they create one Repository capable of handling every entity.

Although this reduces the amount of code,

it usually increases the complexity of the application.

Repositories should model business entities,

not generic database operations.

---

# Incorrect Implementation

A Generic Repository often looks like this.

## Code

```ts
export class GenericRepository<T> {
  async findById(id: string): Promise<T | null> {
    // Generic implementation
  }

  async findAll(): Promise<T[]> {
    // Generic implementation
  }

  async create(data: unknown): Promise<T> {
    // Generic implementation
  }

  async update(id: string, data: unknown): Promise<T> {
    // Generic implementation
  }

  async delete(id: string): Promise<void> {
    // Generic implementation
  }
}
```

Every entity now inherits these methods.

---

# What's Wrong?

The Repository no longer represents the business domain.

Instead,

it represents generic database operations.

```
Generic Repository

↓

Patient

Doctor

Appointment

Diagnosis

Prescription
```

Eventually,

every entity requires custom behavior.

The Generic Repository becomes increasingly difficult to maintain.

---

# Why Is This a Problem?

Business entities are rarely identical.

For example,

a Patient Repository may require:

- Search by phone number.
- Search by medical record number.
- Active patient filtering.

An Appointment Repository may require:

- Find today's appointments.
- Find appointments by doctor.
- Find appointments by clinic.
- Find upcoming appointments.

These methods do not belong in a Generic Repository.

---

# Correct Implementation

Create dedicated Repositories.

```
PatientRepository
```

↓

Patient-specific queries

---

```
AppointmentRepository
```

↓

Appointment-specific queries

---

```
DoctorRepository
```

↓

Doctor-specific queries

Each Repository models its own business entity.

---

# Business Example

The clinic introduces a new feature.

Doctors can search patients using:

- Phone Number
- Blood Group
- Insurance Number

Only the Patient Repository changes.

```
Patient Repository

↓

New Search Methods
```

No other Repository is affected.

This keeps the architecture modular.

---

# Repository Responsibility

Repositories should answer business questions.

Examples

```ts
findPatientByPhoneNumber()
```

```ts
findUpcomingAppointments()
```

```ts
findDoctorsByClinic()
```

These methods clearly describe the business operation being performed.

A Generic Repository cannot express these business concepts cleanly.

---

# Benefits

Using dedicated Repositories provides:

- Better readability.
- Better discoverability.
- Stronger business modeling.
- Easier maintenance.
- Easier testing.
- Cleaner architecture.

Each Repository evolves independently.

---

# When Can Generic Repositories Be Useful?

Generic Repositories can sometimes be useful for small internal utilities,

framework development,

or shared infrastructure libraries.

However,

for business applications,

dedicated Repositories are usually the better choice.

Business requirements almost always become more specific over time.

---

# Common Mistakes

## Optimizing Too Early

Developers often introduce Generic Repositories before duplicated code becomes a real problem.

Avoid unnecessary abstraction.

---

## Hiding Business Intent

Methods such as:

```ts
findAll()
```

provide little information about the business operation.

Prefer methods with meaningful names.

---

## Making Every Entity Fit One Design

Different entities have different requirements.

Repositories should reflect those differences rather than forcing every entity into the same abstraction.

---

# Production Notes

Large backend applications rarely rely on one Generic Repository.

Instead,

each business entity owns its own Repository with methods that describe business operations.

This approach improves maintainability,

keeps business intent clear,

and allows each Repository to evolve independently.

---

# Summary

Generic Repositories often reduce code duplication at the cost of readability and maintainability.

Repositories should represent business entities and expose business-focused operations rather than generic CRUD methods.

Dedicated Repositories produce cleaner architectures that scale more naturally as business requirements evolve.

# Large Repository Classes

Another common mistake is allowing a Repository to grow without clear boundaries.

Initially,

a Repository contains only a few methods.

```
PatientRepository

↓

findById()

create()

update()
```

As new features are added,

more methods are introduced.

```
PatientRepository

↓

findById()

findByEmail()

findByPhone()

findByInsurance()

findRecentPatients()

findInactivePatients()

findPatientsByClinic()

findPatientsWithAppointments()

findPatientsWithDiagnosis()

findPatientsWithPrescriptions()

...

50+ Methods
```

Eventually,

the Repository becomes extremely large and difficult to maintain.

---

# Why Does This Happen?

Every new feature needs patient data.

Developers continue adding methods to the same Repository because it already exists.

Over time,

the Repository becomes responsible for many unrelated business operations.

Its responsibilities continue to expand.

---

# What's Wrong?

A large Repository usually indicates that it is trying to solve too many problems.

```
Patient Repository

↓

Searching

↓

Reporting

↓

Statistics

↓

Dashboard Queries

↓

CRUD Operations

↓

Analytics
```

Instead of being a focused data access component,

it becomes a collection of unrelated queries.

---

# Why Is This a Problem?

Large Repository classes introduce several challenges.

- Difficult to navigate.
- Harder to review.
- Increased merge conflicts.
- Lower readability.
- Higher maintenance cost.
- Difficult onboarding for new developers.

Finding the correct method becomes increasingly difficult as the Repository grows.

---

# Business Example

Suppose the clinic introduces:

- Reception Dashboard
- Doctor Dashboard
- Admin Dashboard
- Reporting Module

Instead of creating focused query methods,

every new query is added to `PatientRepository`.

```
PatientRepository

↓

Reception Queries

↓

Doctor Queries

↓

Admin Queries

↓

Reports
```

The Repository no longer has a clear responsibility.

---

# Better Approach

Keep Repository methods focused on persistence.

Move feature-specific query composition into dedicated Services.

```
ReceptionService

↓

PatientRepository

--------------------

DoctorService

↓

PatientRepository

--------------------

ReportingService

↓

PatientRepository
```

The Repository provides reusable data access methods.

Services compose them according to business requirements.

---

# Split Responsibilities When Necessary

If an entity develops many specialized queries,

consider separating responsibilities.

Example

```
PatientRepository

↓

CRUD Operations

--------------------

PatientSearchRepository

↓

Search Queries

--------------------

PatientReportRepository

↓

Reporting Queries
```

Each Repository has a clear purpose.

This approach is especially useful in large applications with complex reporting requirements.

---

# Repository Responsibility

Ask the following question before adding a new method.

> Does this method represent normal persistence for this entity, or is it specific to one feature?

If it belongs to a specific feature,

consider whether it should be implemented elsewhere.

Repositories should remain cohesive.

---

# Common Mistakes

## One Repository for Every Feature

Avoid placing dashboard queries,

reporting queries,

analytics,

and CRUD operations inside the same Repository.

---

## Extremely Long Repository Files

If a Repository grows to hundreds or thousands of lines,

it is usually a sign that responsibilities should be reviewed.

---

## Ignoring Cohesion

Methods inside the same Repository should be closely related.

Unrelated operations often indicate that the Repository has become too broad.

---

# Production Notes

Production applications favor small,

focused Repositories.

When business requirements become more complex,

engineers often separate specialized query logic into dedicated Repositories or query services rather than continuously expanding a single Repository.

This improves readability,

reduces maintenance effort,

and keeps responsibilities well defined.

---

# Summary

Repositories should remain focused on a single business entity and its persistence responsibilities.

When a Repository grows excessively,

it becomes difficult to maintain and understand.

Keeping Repositories cohesive results in cleaner architecture, easier code reviews, and better long-term maintainability.

# Ignoring Transactions

Another common mistake is executing multiple related database operations without using a transaction.

A transaction ensures that a group of database operations either:

- All succeed.
- All fail.

There is no partial success.

When related operations are executed independently,

a failure in the middle of the process can leave the database in an inconsistent state.

---

# Incorrect Implementation

Suppose a patient books an appointment.

The application performs two operations.

- Create Appointment.
- Update Doctor Availability.

A developer writes:

## Code

```ts
const appointment = await prisma.appointment.create({
  data: appointmentData
})

await prisma.doctor.update({
  where: {
    id: doctorId
  },

  data: {
    availableSlots: {
      decrement: 1
    }
  }
})
```

---

# What's Wrong?

The operations execute independently.

```
Create Appointment

↓

Success

↓

Update Doctor

↓

Fails
```

The appointment now exists,

but the doctor's available slots were never updated.

The database contains inconsistent information.

---

# Business Example

A patient books an appointment.

The application performs:

- Create Appointment.
- Reserve Doctor Slot.
- Create Medical Record.
- Create Billing Record.

Suppose the billing service fails.

Without a transaction,

the first three operations succeed,

while the fourth does not.

```
Appointment

✓

Doctor Slot

✓

Medical Record

✓

Billing

✗
```

The clinic now has incomplete business data.

---

# Correct Implementation

Execute all related operations inside a transaction.

## Code

```ts
await prisma.$transaction(async tx => {
  await tx.appointment.create({
    data: appointmentData
  })

  await tx.doctor.update({
    where: {
      id: doctorId
    },

    data: {
      availableSlots: {
        decrement: 1
      }
    }
  })
})
```

---

# Explanation

The transaction behaves as one unit.

```
Transaction

↓

Appointment

↓

Doctor

↓

Commit
```

If any operation fails,

the database rolls back every previous operation.

```
Transaction

↓

Appointment

↓

Doctor

↓

Failure

↓

Rollback
```

The database returns to its previous state.

---

# Repository Responsibility

Repositories should support transactions whenever multiple related database operations must succeed or fail together.

However,

Repositories should not decide **when** a transaction is required.

That decision belongs to the Service layer because it understands the complete business workflow.

---

# Why the Service Starts the Transaction

The Service coordinates business operations.

Example

```
Appointment Service

↓

Begin Transaction

↓

Appointment Repository

↓

Doctor Repository

↓

Billing Repository

↓

Commit
```

The Service understands that all these operations belong to the same business process.

Each Repository simply performs its own database operation.

---

# Common Mistakes

## Multiple Independent Queries

Avoid executing related database operations separately when they must succeed together.

---

## Starting Transactions Everywhere

Not every Repository method requires a transaction.

Use transactions only when multiple dependent operations must be treated as one business operation.

---

## Catching Errors Without Rollback

If related operations execute outside a transaction,

catching an exception does not undo previously committed changes.

The database may already contain inconsistent data.

---

# Production Notes

Transactions are commonly used for operations such as:

- Appointment booking.
- User registration.
- Payment processing.
- Inventory updates.
- Order creation.
- Medical record creation.

Whenever business consistency is important,

transactions should be considered.

---

# Summary

Transactions protect data consistency by ensuring that related database operations succeed or fail together.

Repositories should support transactional operations,

while Services determine when a business workflow requires a transaction.

Using transactions appropriately prevents partial updates and keeps production data reliable.

# Ignoring Pagination

Another common Repository mistake is returning every matching record from the database.

During development,

the database usually contains only a small amount of data.

As a result,

queries appear fast,

and developers often overlook pagination.

In production,

however,

tables may contain millions of records.

Returning every row becomes inefficient and can significantly impact application performance.

---

# Incorrect Implementation

Consider a Repository that retrieves all patients.

## Code

```ts
async findAll(): Promise<Patient[]> {

    return prisma.patient.findMany();

}
```

---

# What's Wrong?

This query has no limit.

```
Database

↓

Patient Table

↓

Return Every Record
```

As the number of patients grows,

the Repository returns increasingly larger result sets.

---

# Why Is This a Problem?

Returning every record affects multiple parts of the application.

The database performs more work.

↓

More data is transferred across the network.

↓

Node.js allocates more memory.

↓

The API response becomes larger.

↓

The client takes longer to process the response.

A simple query can become a major performance bottleneck.

---

# Business Example

The clinic has been operating for eight years.

The Patient table now contains:

```
Patient

↓

2,500,000 Records
```

The receptionist opens the patient search page.

The Repository returns every patient.

```
2,500,000 Patients

↓

Network

↓

Application

↓

Browser
```

The user only needs the first page,

yet the application retrieves the entire dataset.

---

# Correct Implementation

Repositories returning collections should support pagination.

## Code

```ts
async findAll(
    page: number,
    limit: number
): Promise<Patient[]> {

    return prisma.patient.findMany({

        skip: (page - 1) * limit,

        take: limit,

        orderBy: {

            createdAt: "desc"

        }

    });

}
```

---

# Explanation

Instead of retrieving every patient,

the Repository retrieves only the requested page.

```
Database

↓

Patient Table

↓

20 Records

↓

Application
```

The amount of work performed by both the database and the application is significantly reduced.

---

# Repository Responsibility

Whenever a Repository returns a collection,

consider whether the collection can continue growing over time.

Examples include:

- Patients
- Doctors
- Appointments
- Diagnoses
- Prescriptions

If the answer is yes,

the Repository should support pagination.

---

# Common Mistakes

## Returning Entire Tables

Avoid Repository methods that retrieve every record without a clear reason.

---

## Leaving Pagination to the Client

The database should return only the required records.

Clients should not receive thousands of unnecessary objects.

---

## Forgetting Ordering

Pagination without a consistent ordering may produce inconsistent results between requests.

Always combine pagination with an appropriate `orderBy`.

---

# Production Notes

Production applications almost never expose unbounded collection queries.

Instead,

Repositories provide paginated methods that:

- Limit returned rows.
- Reduce database load.
- Lower memory consumption.
- Improve response times.
- Scale as datasets grow.

Pagination should be considered the default behavior for Repository methods returning collections.

---

# Summary

Ignoring pagination may not cause problems during development,

but it becomes a serious scalability issue as data grows.

Repositories should return only the records required for the current request, ensuring efficient database access and predictable application performance.

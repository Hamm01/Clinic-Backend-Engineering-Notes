# What Is the Repository Pattern?

The Repository Pattern is a software design pattern that introduces a dedicated layer responsible for communicating with the database.

Instead of allowing business logic to directly interact with the database or ORM, all persistence operations are delegated to a Repository.

```
Service

↓

Repository

↓

ORM

↓

Database
```

The Service focuses on business rules.

The Repository focuses on persistence.

---

# Why Was the Repository Pattern Introduced?

Imagine a Service that directly uses Prisma.

```ts
async create(dto: CreateDoctorDto) {

    await prisma.doctor.create({

        data: dto

    });

}
```

This code works.

However, the Service now performs two responsibilities.

```
Business Logic

+

Database Access
```

As applications grow, these responsibilities become tightly coupled.

The Repository Pattern separates them.

---

# Understanding the Separation

Without the Repository Pattern

```
Controller

↓

Service

↓

Prisma

↓

Database
```

The Service is responsible for business logic and persistence.

---

With the Repository Pattern

```
Controller

↓

Service

↓

Repository

↓

Prisma

↓

Database
```

Each layer owns a single responsibility.

---

# Responsibility Comparison

| Layer      | Responsibility      |
| ---------- | ------------------- |
| Controller | HTTP communication  |
| Service    | Business rules      |
| Repository | Database operations |
| Prisma     | ORM                 |
| PostgreSQL | Data storage        |

Notice that every layer answers a different question.

---

# Real Business Example

Suppose a receptionist registers a new patient.

The request flows like this.

```
Receptionist

↓

POST /patients

↓

PatientController

↓

PatientService

↓

PatientRepository

↓

Prisma

↓

PostgreSQL
```

The Controller handles the HTTP request.

The Service validates the business workflow.

The Repository stores the data.

---

# What Problems Does It Solve?

## Separation of Concerns

Business logic and persistence remain independent.

---

## Readable Services

Instead of writing database queries,

Services express business workflows.

Example

```ts
await this.patientRepository.create(dto)
```

This is easier to read than embedding Prisma queries throughout the Service.

---

## Reusable Persistence

Suppose multiple Services need to retrieve a patient.

Instead of repeating

```ts
prisma.patient.findUnique(...)
```

everywhere,

the Repository provides one reusable method.

```ts
patientRepository.findById(id)
```

---

## Easier Maintenance

If database queries change,

the update is usually limited to the Repository.

The rest of the application remains unchanged.

---

## Better Testing

During unit testing,

Repositories can be mocked.

Business logic can then be tested without connecting to a real database.

---

# Repository Pattern vs Prisma

A common misunderstanding is:

> "Prisma already abstracts the database, so why create another abstraction?"

The answer is that Prisma and the Repository Pattern solve different problems.

| Prisma                           | Repository Pattern                           |
| -------------------------------- | -------------------------------------------- |
| Maps objects to database queries | Organizes persistence inside the application |
| Generates SQL                    | Defines application boundaries               |
| ORM                              | Architectural Pattern                        |

Prisma is a tool.

The Repository Pattern is an architectural decision.

---

# When Should You Use the Repository Pattern?

The Repository Pattern becomes valuable when:

- The application has multiple business entities.
- Database queries are reused.
- Teams are growing.
- Business rules continue to evolve.
- Long-term maintainability is important.

Small prototypes may not require it.

Production applications usually benefit from it.

---

# Common Misconceptions

## "Repository Just Wraps Prisma"

A good Repository is more than a wrapper.

It provides meaningful persistence operations.

Example

```ts
findAvailableDoctors()

findUpcomingAppointments()

findPatientHistory()
```

These methods express business persistence needs.

---

## "Repository Contains Business Logic"

Repositories should not make business decisions.

They should only perform persistence operations.

Business rules remain inside Services.

---

## "One Repository Can Handle Everything"

Each Repository should manage one aggregate or domain entity.

Example

```
DoctorRepository

PatientRepository

AppointmentRepository

DiagnosisRepository
```

---

# Production Notes

Repositories help maintain a clean architecture by isolating persistence from business logic.

They become increasingly valuable as applications grow because they:

- Centralize database operations.
- Reduce duplication.
- Simplify testing.
- Improve maintainability.
- Keep Services focused on business workflows.

---

# Summary

The Repository Pattern is an architectural pattern that separates business logic from database access.

Instead of allowing Services to communicate directly with Prisma, production applications introduce a Repository layer responsible for persistence.

This separation leads to cleaner Services, reusable database operations, improved testing, and a more maintainable backend architecture.

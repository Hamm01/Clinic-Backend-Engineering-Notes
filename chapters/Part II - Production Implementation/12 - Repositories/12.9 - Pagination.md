# 12.9 - Pagination

> Modern backend applications rarely return every record from a database table. As applications grow, datasets become too large to load into memory or send over the network in a single response. Pagination allows repositories to return data in manageable chunks while improving performance, reducing bandwidth, and creating a better user experience.

# Why Does Pagination Exist?

Imagine the clinic has been running for several years.

The database now contains:

- 500 Doctors
- 50,000 Patients
- 1,200,000 Appointments
- 4,000,000 Diagnoses

Now consider the following API.

```http
GET /patients
```

Without pagination,

the Repository executes:

```ts
await prisma.patient.findMany()
```

This query attempts to retrieve every patient.

```
Database

↓

50,000 Patients

↓

Application

↓

Client
```

Although technically correct,

this approach creates several problems.

---

# Problems Without Pagination

Returning every record affects every layer of the application.

## Database

The database spends more time reading data.

---

## Backend

The application allocates more memory.

Large queries increase processing time.

---

## Network

Larger responses require more bandwidth.

Clients wait longer for responses.

---

## Frontend

The browser receives thousands of unnecessary records.

Rendering becomes slower.

---

```

↓

↓
```

The Repository should return only the records required for the current page.

# Offset Pagination

Offset pagination is the most common pagination strategy used in REST APIs.

Instead of retrieving every record,

the Repository retrieves only a specific portion of the dataset.

For example,

the first page may return:

```text
Records 1–20
```

The second page returns:

```text
Records 21–40
```

The third page returns:

```text
Records 41–60
```

Every page contains a fixed number of records.

---

# How Offset Pagination Works

Offset pagination is based on two values.

- Page Number
- Page Size (Limit)

Example

```text
Page = 3

Limit = 20
```

The Repository calculates:

```text
Skip = (Page - 1) × Limit

Skip = (3 - 1) × 20

Skip = 40
```

The database skips the first forty records

and returns the next twenty.

```
Database

↓

Skip 40

↓

Take 20

↓

Return Page 3
```

---

# Repository Implementation

Our Repository should expose a simple interface.

## Code

```ts
async findAll(

    page: number,

    limit: number

): Promise<Patient[]> {

    return prisma.patient.findMany({

        skip: (page - 1) * limit,

        take: limit,

        orderBy: {

            createdAt: "desc"

        }

    });

}
```

---

# Explanation

Prisma provides two important options.

```ts
skip
```

Number of records to ignore.

```ts
take
```

Number of records to return.

The Repository combines both values to retrieve only the requested page.

# Why Order Matters

Pagination should always be combined with a predictable ordering.

Example

```ts
orderBy: {
  createdAt: 'desc'
}
```

Without ordering,

the database may return records in a different sequence on every request.

Example

Page One

```
Patient A

Patient B

Patient C
```

Next request

```
Patient C

Patient A

Patient B
```

Users may see duplicate or missing records while navigating between pages.

Always define an ordering when paginating.

---

# Default Page Size

Every API should define a reasonable default page size.

Example

```text
20 Records
```

or

```text
25 Records
```

Avoid returning hundreds or thousands of records by default.

Large responses increase:

- Memory usage
- Network traffic
- Response time

---

# Limiting Maximum Page Size

Clients should not be allowed to request unlimited records.

Avoid

```http
GET /patients?limit=100000
```

Instead,

enforce a maximum.

Example

```text
Maximum Limit

100
```

If the client requests:

```text
500
```

the application should use:

```text
100
```

This protects the application from excessive resource consumption.

# Common Mistakes

## Forgetting orderBy

Pagination without ordering produces inconsistent results.

Always define a stable sorting column.

---

## Trusting Client Limits

Never allow unlimited page sizes.

Always enforce a maximum limit.

---

## Returning Entire Tables

Avoid

```ts
findMany()
```

without `skip` and `take` for list endpoints.

Large datasets should always be paginated.

---

## Hardcoding Values

Avoid writing

```ts
skip: 20
```

inside the Repository.

Pagination should be calculated dynamically based on the requested page and limit.

# Cursor Pagination

Offset pagination works well for many applications.

However, as the amount of data grows, it becomes less efficient.

Imagine the Appointment table contains:

```
25,000,000 Appointments
```

Now consider the following request.

```http
GET /appointments?page=50000&limit=20
```

The database must skip nearly one million records before returning the next twenty.

```
Database

↓

Skip 999,980 Records

↓

Return 20 Records
```

Although only twenty records are returned,

the database still performs significant work.

For extremely large datasets,

this approach becomes expensive.

---

# Why Cursor Pagination Exists

Instead of asking,

> "Give me page 50000"

Cursor pagination asks,

> "Give me the next twenty records after this record."

The database no longer counts pages.

Instead,

it starts from a known position.

```
Last Record

↓

Cursor

↓

Next Records
```

This allows the database to continue reading from where it previously stopped.

---

# Business Example

Suppose a doctor opens today's appointments.

The application loads:

```
Appointment 1

Appointment 2

Appointment 3

...

Appointment 20
```

The final appointment has the ID

```
APT-020
```

Instead of requesting

```
Page 2
```

the client sends

```
Cursor = APT-020
```

The Repository retrieves every appointment after that cursor.

---

# Repository Implementation

## Code

```ts
async findAfterCursor(

    cursor: string,

    limit: number

): Promise<Appointment[]> {

    return prisma.appointment.findMany({

        cursor: {

            id: cursor

        },

        skip: 1,

        take: limit,

        orderBy: {

            id: "asc"

        }

    });

}
```

---

# Explanation

Prisma starts reading from the specified cursor.

```ts
cursor
```

identifies the last record returned.

```ts
skip: 1
```

prevents Prisma from returning the cursor record again.

```ts
take
```

defines how many records should be returned.

---

# Visualizing Cursor Pagination

Suppose the database contains:

```
1

2

3

4

5

6

7

8
```

The client already received

```
1

2

3
```

The last record becomes the cursor.

```
Cursor = 3
```

The next request returns

```
4

5

6
```

The following cursor becomes

```
6
```

Next request

```
7

8
```

The Repository simply continues from the previous position.

---

# Why Is Cursor Pagination Faster?

Offset pagination

```
Skip

↓

Read

↓

Return
```

Cursor pagination

```
Known Position

↓

Continue Reading

↓

Return
```

The database does not repeatedly scan earlier records.

As tables grow,

cursor pagination remains efficient.

---

# Choosing a Cursor

The cursor must uniquely identify a record.

Good examples:

- Primary Key
- UUID
- Auto Increment ID
- Created Timestamp (when unique)

Avoid using columns that can change.

Examples:

- Patient Name
- Doctor Name
- Status

Changing cursor values may cause duplicate or missing records.

---

# Business Example

Patient Portal

↓

Medical History

↓

Load More

↓

Cursor

↓

Next Diagnoses

Instead of navigating between numbered pages,

the interface continuously loads more records.

This is commonly seen in modern applications.

---

# Where Cursor Pagination Is Used

Cursor pagination is ideal for:

- Infinite scrolling
- Activity feeds
- Chat applications
- Notifications
- Medical history
- Audit logs

Any dataset that continuously grows benefits from cursor pagination.

---

# Offset vs Cursor Pagination

| Offset Pagination            | Cursor Pagination              |
| ---------------------------- | ------------------------------ |
| Uses page numbers            | Uses cursor values             |
| Easy to implement            | Slightly more complex          |
| Good for admin tables        | Good for large datasets        |
| Slower on very large tables  | Consistent performance         |
| Supports jumping to any page | Best for sequential navigation |

Neither strategy is universally better.

Choose the approach that matches the business requirement.

---

# Common Mistakes

## Using Non-Unique Cursors

A cursor should uniquely identify a row.

Using non-unique values may return duplicate or missing records.

---

## Forgetting Stable Ordering

Cursor pagination still requires predictable ordering.

Always specify an `orderBy` clause that matches the cursor field.

---

## Using Cursor Pagination Everywhere

Admin dashboards often require direct navigation to specific pages.

Cursor pagination is not ideal for those scenarios.

Use offset pagination when numbered pages are more useful.

---

# Production Notes

Many large-scale applications rely on cursor pagination for high-volume data.

Examples include:

- Social media timelines
- Messaging systems
- Notification centers
- Banking transaction history
- Healthcare consultation history

Repositories should support the pagination strategy that best matches the application's access patterns.

---

# Summary

Cursor pagination retrieves records relative to a previously returned item rather than by counting pages.

This approach avoids scanning large numbers of rows, making it well suited for continuously growing datasets and applications that load information sequentially.

# Implementing Pagination Metadata

Earlier, we created a standard pagination response.

```ts
interface PaginationResult<T> {
  data: T[]

  meta: {
    page: number

    limit: number

    totalRecords: number

    totalPages: number

    hasNextPage: boolean

    hasPreviousPage: boolean
  }
}
```

Now we need to populate these values.

To do this, the Repository performs two database operations.

1. Retrieve the requested page of records.
2. Count the total number of matching records.

These values are then combined into a single response.

---

# Repository Implementation

## Code

```ts
async findAll(

    page: number,

    limit: number

): Promise<PaginationResult<Patient>> {

    const skip = (page - 1) * limit;

    const [patients, totalRecords] = await Promise.all([

        prisma.patient.findMany({

            skip,

            take: limit,

            orderBy: {

                createdAt: "desc"

            }

        }),

        prisma.patient.count()

    ]);

    const totalPages = Math.ceil(
        totalRecords / limit
    );

    return {

        data: patients,

        meta: {

            page,

            limit,

            totalRecords,

            totalPages,

            hasNextPage: page < totalPages,

            hasPreviousPage: page > 1

        }

    };

}
```

---

# Explanation

The Repository performs both database operations at the same time.

```
Repository

│

├── findMany()

│

└── count()

↓

PaginationResult
```

Running them in parallel reduces the overall response time.

---

# Why Use Promise.all()?

Consider the following implementation.

```ts
const patients = await prisma.patient.findMany()

const total = await prisma.patient.count()
```

The second query waits for the first query to finish.

Instead,

we execute both simultaneously.

```ts
const [patients, total] = await Promise.all([
    prisma.patient.findMany(...),
    prisma.patient.count()
]);
```

This improves performance because both queries are independent.

---

Suppose the application contains:


```text

=

```

Since a partial page still needs to be displayed,

we round upward.

```ts
Math.ceil(95 / 10)

↓

```

The application now knows there are ten pages.

# Common Mistakes

## Executing Queries Sequentially

Avoid running `findMany()` and `count()` one after another when they are independent.

Parallel execution improves response time.

---

## Incorrect Page Calculations

Do not truncate decimal values.

Always use `Math.ceil()` to calculate the total number of pages.

Otherwise, the final page may never be displayed.

---

## Returning Incorrect Metadata

Ensure that:

- `hasNextPage`
- `hasPreviousPage`
- `totalPages`

are calculated from the same dataset returned by the Repository.

Inconsistent metadata creates confusing user experiences.

---

# Production Notes

In large applications, pagination metadata becomes part of the API contract.

Frontend applications, mobile apps, and third-party consumers all rely on these values to build navigation components.

A consistent pagination response reduces duplication across the entire system.

# Combining Pagination with Filtering and Sorting

In production applications, users rarely request every record.

Instead, they search for specific information.

Consider the patient management screen.

The receptionist may want to:

- Search by patient name.
- View only active patients.
- Sort by registration date.
- Navigate through the results page by page.

All of these requirements should be supported by a single Repository method.

---

# Repository Method

Instead of exposing multiple methods such as:

```ts
findAll()

findByName()

findActivePatients()

findSortedPatients()
```

the Repository can expose one flexible interface.

## Code

```ts
interface PatientQuery {
  page: number

  limit: number

  search?: string

  status?: PatientStatus

  sortBy?: 'createdAt' | 'firstName'

  order?: 'asc' | 'desc'
}
```

---

# Explanation

The Repository receives one query object.

```
PatientQuery

↓

Repository

↓

Database
```

As the application grows,

additional filters can be added without changing the Repository's overall structure.

---

# Repository Implementation

## Code

```ts
async findAll(

    query: PatientQuery

): Promise<PaginationResult<Patient>> {

    const {

        page,

        limit,

        search,

        status,

        sortBy = "createdAt",

        order = "desc"

    } = query;

    const skip = (page - 1) * limit;

    const where: Prisma.PatientWhereInput = {

        ...(search && {

            firstName: {

                contains: search,

                mode: "insensitive"

            }

        }),

        ...(status && {

            status

        })

    };

    const [patients, totalRecords] = await Promise.all([

        prisma.patient.findMany({

            where,

            skip,

            take: limit,

            orderBy: {

                [sortBy]: order

            }

        }),

        prisma.patient.count({

            where

        })

    ]);

    return {

        data: patients,

        meta: {

            page,

            limit,

            totalRecords,

            totalPages: Math.ceil(
                totalRecords / limit
            ),

            hasNextPage:
                page < Math.ceil(totalRecords / limit),

            hasPreviousPage:
                page > 1

        }

    };

}
```

---

# Explanation

Notice how the Repository builds a single query.

```
Search

+

Filter

+

Sorting

+

Pagination

↓

Database Query
```

Instead of executing separate queries,

everything is combined into one database request.

This keeps the Repository efficient and easier to maintain.

```ts
return patients.filter(...);
```

Filtering should always be delegated to the database.

---

## Forgetting to Apply Filters to count()

The `count()` query should use the same filters as `findMany()`.

Otherwise,

the metadata becomes incorrect.

Example:

```
Returned Records

↓

20

Total Records

↓

1500
```

when only 42 records actually match the filter.

Always reuse the same `where` object.

# Combining Pagination with Filtering and Sorting

In production applications, users rarely request every record.

Instead, they search for specific information.

Consider the patient management screen.

The receptionist may want to:

- Search by patient name.
- View only active patients.
- Sort by registration date.
- Navigate through the results page by page.

All of these requirements should be supported by a single Repository method.

---

# Repository Method

Instead of exposing multiple methods such as:

```ts
findAll()

findByName()

findActivePatients()

findSortedPatients()
```

the Repository can expose one flexible interface.

## Code

```ts
interface PatientQuery {
  page: number

  limit: number

  search?: string

  status?: PatientStatus

  sortBy?: 'createdAt' | 'firstName'

  order?: 'asc' | 'desc'
}
```

---

# Explanation

The Repository receives one query object.

```
PatientQuery

↓

Repository

↓

Database
```

As the application grows,

additional filters can be added without changing the Repository's overall structure.

# Repository Implementation

## Code

```ts
async findAll(

    query: PatientQuery

): Promise<PaginationResult<Patient>> {

    const {

        page,

        limit,

        search,

        status,

        sortBy = "createdAt",

        order = "desc"

    } = query;

    const skip = (page - 1) * limit;

    const where: Prisma.PatientWhereInput = {

        ...(search && {

            firstName: {

                contains: search,

                mode: "insensitive"

            }

        }),

        ...(status && {

            status

        })

    };

    const [patients, totalRecords] = await Promise.all([

        prisma.patient.findMany({

            where,

            skip,

            take: limit,

            orderBy: {

                [sortBy]: order

            }

        }),

        prisma.patient.count({

            where

        })

    ]);

    return {

        data: patients,

        meta: {

            page,

            limit,

            totalRecords,

            totalPages: Math.ceil(
                totalRecords / limit
            ),

            hasNextPage:
                page < Math.ceil(totalRecords / limit),

            hasPreviousPage:
                page > 1

        }

    };

}
```

---

# Explanation

Notice how the Repository builds a single query.

```
Search

+

Filter

+

Sorting

+

Pagination

↓

Database Query
```

Instead of executing separate queries,

everything is combined into one database request.

This keeps the Repository efficient and easier to maintain.

Imagine new requirements are added.

# What is Prisma Client?

Prisma Client is a type-safe database client generated from our Prisma Schema.

Whenever we execute

```bash
npx prisma generate
```

Prisma reads

```
schema.prisma
```

and generates a fully typed database client.

Instead of writing SQL manually,

we write TypeScript.

Example

```ts
await prisma.patient.findUnique({
  where: {
    id
  }
})
```

Prisma converts this into SQL behind the scenes.

---

# Where Does Prisma Client Fit?

Our application architecture now looks like this.

```
Client

↓

Routes

↓

Middleware

↓

Controller

↓

Service

↓

Repository

↓

Prisma Client

↓

PostgreSQL
```

Repositories never communicate directly with PostgreSQL.

They communicate through Prisma Client.

---

# Why Not Create PrismaClient Everywhere?

A common beginner mistake is this.

```ts
const prisma = new PrismaClient()
```

inside every Repository.

Example

```ts
export class PatientRepository {
  private prisma = new PrismaClient()
}
```

It works.

But it creates multiple database connections.

---

# Why Is That a Problem?

Imagine

```
DoctorRepository

↓

new PrismaClient()
```

```
PatientRepository

↓

new PrismaClient()
```

```
AppointmentRepository

↓

new PrismaClient()
```

Every Repository opens its own connection pool.

As the application grows,

database resources are wasted.

---

# Production Solution

Production applications create one shared Prisma Client.

```
PrismaClient

↓

Shared Instance

↓

Imported Everywhere
```

Every Repository uses the same client.

---

# Project Structure

Create a dedicated folder.

```text
src/

├── config/

│   └── prisma.ts

├── repositories/

├── services/

├── controllers/
```

The Prisma Client belongs in the configuration layer.

# Explanation

The application creates a single Prisma Client instance.

Every Repository imports this object instead of creating a new one.

Example

```

↓

import prisma
```

```
PatientRepository

↓

import prisma
```

```
AppointmentRepository

↓

import prisma
```

One client.

Many repositories.

---

# Using Prisma Inside a Repository

Example

```ts
import { prisma } from '../config/prisma'

export class PatientRepository {
  async findById(id: string) {
    return prisma.patient.findUnique({
      where: {
        id
      }
    })
  }
}
```

Notice that the Repository never creates a new Prisma Client.

It only imports the shared instance.

---

# Why Export a Constant?

```ts
export const prisma = new PrismaClient()
```

This creates the client once.

Every import references the same object.

Node.js caches imported modules,

so the client is reused throughout the application.

# Best Practices

✔ Create one Prisma Client instance.

✔ Store it inside the configuration layer.

✔ Import it wherever persistence is required.

✔ Keep Prisma usage inside Repositories.

✔ Allow Services to remain independent of ORM details.

---

# Common Mistakes

## Creating Multiple Prisma Clients

Avoid

```ts
new PrismaClient()
```

inside every Repository.

---

## Using Prisma Inside Services

Avoid

```ts
await prisma.patient.create(...)
```

Services should communicate with Repositories,

not directly with Prisma.

---

## Using Prisma Inside Controllers

Controllers should never access the database.

Their responsibility ends after calling the Service.

---

## Mixing Business Logic with Persistence

Repositories retrieve and store data.

Business decisions belong to Services.

---

# Production Notes

Large applications may eventually extend the shared Prisma Client with:

- Logging
- Query middleware
- Soft delete helpers
- Performance monitoring
- Transaction helpers

However,

the core principle remains unchanged:

**Only one Prisma Client instance should exist.**

---

# Summary

Prisma Client is the application's gateway to the database.

Instead of creating multiple database clients throughout the project, production applications configure a single shared Prisma Client and reuse it across every Repository.

This approach improves performance, simplifies maintenance, and establishes a consistent persistence layer for the entire application.

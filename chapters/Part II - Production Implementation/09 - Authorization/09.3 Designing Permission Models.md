# 09.3 - Designing the Permission Database Schema

> Authorization should be driven by data, not hardcoded values. In this chapter, we redesign our database to support Roles, Permissions, and their relationships, creating a flexible authorization system that can evolve as business requirements change.

---

# Objective

After completing this chapter, you should understand:

- Why database-driven authorization is preferred.
- Why Roles should not be hardcoded.
- Why Permissions belong in their own table.
- How Many-to-Many relationships work.
- How Users, Roles, and Permissions are connected.
- How Prisma models represent these relationships.

---

# Current Project

Until now, our User model looked similar to this.

```prisma
model User {

  id        String   @id @default(uuid())

  email     String   @unique

  password  String

  role       String

  createdAt DateTime @default(now())

  updatedAt DateTime @updatedAt
}
```

Although this works,

the

```
role

↓

String
```

is extremely limited.

---

# Why This Design Becomes a Problem

Suppose the company introduces

```
Insurance Manager
```

With a String,

every place that checks

```ts
if (user.role === "ADMIN")
```

must now be reviewed.

As the application grows,

these checks become difficult to maintain.

---

# New Authorization Design

Instead of

```
User

↓

Role (String)
```

we move to

```
User

↓

Role

↓

Permissions
```

Roles become database records.

Permissions become database records.

The relationships define authorization.

---

# Final Database Design

```
User

↓

Role

↓

RolePermission

↓

Permission
```

This structure allows

- One Role → Many Permissions
- One Permission → Many Roles

This is a classic

```
Many-to-Many Relationship
```

---

# Database Relationship

```
                User
                  │
                  │
                  ▼
                Role
                  │
                  │
         RolePermission
          ▲          ▼
          │          │
      Permission
```

Notice

The User never directly owns permissions.

Permissions are inherited through the assigned Role.

---

# Step 1 — Create Role Model

Open

```
prisma/schema.prisma
```

Add

```prisma
model Role {

  id String @id @default(uuid())

  name String @unique

  description String?

  users User[]

  rolePermissions RolePermission[]

  createdAt DateTime @default(now())

  updatedAt DateTime @updatedAt
}
```

---

# Why These Fields?

## id

Every role needs a unique identifier.

Examples

```
Administrator

Doctor

Receptionist
```

Internally,

the database references

the

```
id
```

not the name.

---

## name

The business-readable name.

Example

```
ADMIN

DOCTOR

RECEPTIONIST
```

Marked

```
@unique
```

because duplicate role names are not allowed.

---

## description

Optional documentation.

Example

```
Can manage doctors and clinics.
```

Descriptions improve maintainability,

especially for large teams.

---

## users

```prisma
users User[]
```

One role

↓

Many users.

Example

```
Doctor

↓

John

↓

Alice

↓

Robert
```

---

## rolePermissions

One role

↓

Many permissions.

Example

```
Doctor

↓

patient:view

↓

diagnosis:create

↓

diagnosis:update
```

---

# Step 2 — Create Permission Model

Add

```prisma
model Permission {

  id String @id @default(uuid())

  name String @unique

  description String?

  rolePermissions RolePermission[]

  createdAt DateTime @default(now())

  updatedAt DateTime @updatedAt
}
```

---

# Why Separate Permissions?

Permissions represent

individual actions.

Examples

```
patient:create

patient:update

patient:view

patient:delete

doctor:create

doctor:update

doctor:delete

appointment:create
```

Every permission represents exactly one capability.

This keeps authorization simple and reusable.

---

# Step 3 — Create Junction Table

Because

```
Role

↓

Many Permissions
```

and

```
Permission

↓

Many Roles
```

Prisma requires

a junction table.

Add

```prisma
model RolePermission {

  roleId String

  permissionId String

  role Role
      @relation(fields: [roleId], references: [id])

  permission Permission
      @relation(fields: [permissionId], references: [id])

  @@id([roleId, permissionId])
}
```

---

# Why a Junction Table?

Imagine

```
Doctor
```

has

```
patient:view

patient:update
```

Now

```
Administrator
```

also needs

```
patient:view
```

Instead of duplicating permissions,

both roles reference

the same Permission record.

---

# Step 4 — Update User Model

Replace

```prisma
role String
```

with

```prisma
roleId String

role Role
    @relation(
        fields: [roleId],
        references: [id]
    )
```

---

# Why roleId?

Instead of storing

```
ADMIN
```

inside every user,

the database stores

```
Role ID
```

Benefits

- Less duplication
- Better integrity
- Easier updates
- Faster joins

---

# Final Prisma Models

```
User

↓

Role

↓

RolePermission

↓

Permission
```

This becomes the foundation for every authorization decision.

---

# Why Not Store Permissions Inside User?

Wrong

```
User

↓

Permissions
```

Imagine

10,000 doctors.

Every doctor would store

the same permissions repeatedly.

Instead,

permissions belong to the Role.

Users inherit them automatically.

# Production Notes

Always

- Store Roles in the database.
- Store Permissions separately.
- Use Many-to-Many relationships.
- Keep permissions granular.
- Avoid hardcoding role names throughout the application.

---

# Common Mistakes

## Using Strings Everywhere

Wrong

```ts
user.role === 'ADMIN'
```

for every permission check.

Prefer loading permissions through relationships.

---

## Creating Duplicate Permissions

Avoid

```
Doctor View Patient

Admin View Patient
```

Instead

```
patient:view
```

shared by multiple roles.

---

## Storing Permissions Per User

This creates unnecessary duplication and makes permission updates difficult.

Assign permissions to Roles whenever possible.

---

# Senior Engineer Notes

Most enterprise applications model authorization using database relationships rather than enums or hardcoded strings.

This design allows new roles and permissions to be introduced without changing application code, making the system easier to maintain, audit, and extend.

It also supports future features such as custom roles, tenant-specific permissions, and administrative permission management.

---

# Summary

In this chapter, we redesigned our authorization database to use Roles, Permissions, and a Many-to-Many relationship between them.

By replacing hardcoded role strings with relational data, we created a scalable authorization model that supports business growth while keeping application code clean and maintainable.

This schema becomes the foundation for the remaining Authorization implementation.

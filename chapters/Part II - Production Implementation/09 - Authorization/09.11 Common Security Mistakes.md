# 09.11 - Common Security Mistakes

> Authentication and Authorization are only effective when implemented correctly. Many production security incidents are not caused by weak encryption or broken JWTs, but by small authorization mistakes made during development. In this chapter, we will study common authorization vulnerabilities, understand how attackers exploit them, and learn how to prevent them in our Clinic Backend.

# Mistake 1 — Broken Access Control

Broken Access Control is one of the most common vulnerabilities reported by OWASP.

Example

```
PUT

/patients/123
```

Backend

```ts
await patientRepository.update(id, payload)
```

Nothing verifies

```
Ownership

Permission

Policy
```

Any authenticated user

can update

any patient.

---

# Correct Flow

```
Authentication

↓

Permission

↓

Ownership

↓

Policy

↓

Update
```

Never skip

authorization checks.

---

# Mistake 2 — IDOR

IDOR means

```
Insecure

Direct

Object

Reference
```

Example

Doctor

```
GET

/patients/101
```

returns

```
Patient A
```

Doctor changes URL

```
GET

/patients/102
```

Backend

```ts
findPatientById(req.params.id)
```

No ownership check.

Patient B's data

is leaked.

---

# Secure Implementation

Wrong

```ts
findPatientById(id)
```

Correct

```ts
findPatientById(
  id,

  req.user.id
)
```

The query itself

should verify ownership.

---

# Better Repository Example

```ts
async findOwnedPatient(

    patientId: string,

    doctorId: string

) {

    return prisma.patient.findFirst({

        where: {

            id: patientId,

            doctorId
        }
    });
}
```

Now

the database returns

```
Nothing
```

if ownership fails.

# Mistake 3 — Trusting Client Input

Suppose

client sends

```json
{
  "doctorId": "doctor_999"
}
```

Backend

```ts
repository.create({
  doctorId: req.body.doctorId
})
```

The attacker

assigns records

to another doctor.

---

# Correct Approach

Never trust

ownership fields.

Always use

```ts
doctorId: req.user.id
```

The JWT

identifies

the authenticated user.

---

# Mistake 4 — Privilege Escalation

Suppose

User

updates profile.

Request

```json
{
  "role": "ADMIN"
}
```

Backend

```ts
repository.update(req.body)
```

The attacker

promotes themselves

to Administrator.

---

# Correct Implementation

Ignore

protected fields.

```ts
repository.update({
  firstName,

  lastName,

  email
})
```

Never update

```
Role

Permissions

Status

Account Type
```

directly

from user input.

---

# Mistake 5 — Missing Ownership Check

Permission

```
diagnosis:update
```

exists.

Backend

updates

Diagnosis.

But

ownership

is never verified.

Doctor A

can modify

Doctor B's diagnosis.

Always verify

```
Owner

↓

Current User
```

before update.

---

# Mistake 6 — Returning Sensitive Errors

Wrong

```json
{
    "message":

    "Doctor ID 42

    does not own

    Diagnosis 100"
}
```

This leaks

internal information.

---

# Better Response

```json
{
  "message": "Access denied."
}
```

Keep detailed errors

inside logs,

not API responses.

---

# Mistake 7 — Authorization in Controllers

Wrong

```ts
if (user.role.name === 'ADMIN') {
}
```

Repeated across

20 controllers.

Eventually

one controller

forgets the check.

Security bug.

---

# Better Architecture

```
Middleware

↓

Policy

↓

Service

↓

Repository
```

Authorization

remains centralized.

# Mistake 8 — Missing Admin Override

Business rule

```
Admin

↓

Can Edit Everything
```

Service

only checks

ownership.

Administrator

cannot perform

their job.

Authorization

must match

business requirements.

---

# Mistake 9 — Exposing Internal IDs

Example

```
/patients/1

/patients/2

/patients/3
```

Attackers

can guess

every ID.

Prefer

UUIDs.

Example

```
/patients/

4f1be2a8...

```

Guessing

becomes impractical.

---

# Mistake 10 — Skipping Audit Logs

Suppose

Administrator

deletes

100 diagnoses.

Later

hospital asks

```
Who deleted them?
```

Without logs

there is

no answer.

Always log

```
User

↓

Action

↓

Resource

↓

Timestamp

↓

IP Address
```

for sensitive operations.

# Production Checklist

Before exposing an endpoint, verify

- Authentication
- Permission
- Ownership
- Business Policy
- Input Validation
- Audit Logging
- Secure Error Responses

If any item is missing,

the endpoint is incomplete.

---

# Common Mistakes Summary

| Mistake                  | Risk                      | Solution                |
| ------------------------ | ------------------------- | ----------------------- |
| Missing Permission Check | Unauthorized Access       | Permission Middleware   |
| Missing Ownership        | IDOR                      | Ownership Validation    |
| Trusting Client IDs      | Data Manipulation         | Use `req.user.id`       |
| Privilege Escalation     | Unauthorized Admin Access | Ignore Protected Fields |
| No Audit Logs            | No Accountability         | Log Sensitive Actions   |
| Controller Authorization | Duplicate Logic           | Middleware + Policies   |
| Sequential IDs           | Enumeration               | UUIDs                   |
| Detailed Error Messages  | Information Leakage       | Generic Responses       |

# Senior Engineer Notes

When reviewing code, experienced backend engineers rarely start by asking:

> "Does this endpoint return the correct response?"

Instead, they ask:

- Can someone access another user's data?
- Can someone modify data they don't own?
- Can someone become an administrator?
- Can IDs be guessed?
- Are sensitive operations logged?
- What happens if someone modifies the request body?

Thinking this way transforms security from an afterthought into part of the design process.

# Summary

A secure authorization system is more than checking JWTs or permissions.

It requires multiple layers of protection:

- Authentication
- Permissions
- Ownership
- Business Policies
- Input Validation
- Secure Error Handling
- Audit Logging

By understanding these common mistakes and designing our backend to prevent them, our Clinic Backend now follows security practices used in production healthcare, banking, and enterprise applications.

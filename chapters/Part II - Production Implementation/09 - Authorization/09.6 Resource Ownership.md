# 09.6 - Resource Ownership

> Having permission to perform an action does not always mean a user can perform that action on every resource. In many applications, users are allowed to modify only the resources they own. This chapter introduces Ownership-Based Authorization and implements it in our Clinic Backend.

# The Problem

Imagine the following request.

```
PUT

/diagnoses/123
```

Doctor

```
Dr. John
```

is authenticated.

The Permission Middleware checks

```
diagnosis:update
```

Result

```
Allowed
```

Everything looks fine.

But another question appears.

> Who created Diagnosis 123?

Suppose it was created by

```
Dr. Sarah
```

Should Dr. John be allowed to edit it?

No.

Even though

```
Permission

↓

Granted
```

Ownership

```
Denied
```

The request must fail.

---

# Permission vs Ownership

Permission answers

```
Can this user perform

diagnosis:update ?
```

Ownership answers

```
Does this diagnosis

belong to this doctor?
```

Both must be true.

# Why Not Middleware?

Some developers attempt

```
Permission Middleware

↓

Ownership Middleware
```

This becomes difficult because

ownership depends on

business data.

The middleware would need

database queries

for every resource type.

Instead,

ownership belongs

inside the Service.

async findById(id: string) {

    return prisma.diagnosis.findUnique({

        where: {

            id,
        },
    });
}

---

# Why Repository?

The repository is responsible only for

loading data.

It does not decide

who may edit it.

---

# Step 2 — Service

Modify

```
diagnosis.service.ts
```

```ts
async updateDiagnosis(

    diagnosisId: string,

    doctorId: string,

    payload: UpdateDiagnosisDto

) {

    const diagnosis =

        await repository.findById(
            diagnosisId
        );

    if (!diagnosis) {

        throw new NotFoundException(
            "Diagnosis not found."
        );
    }

    if (

        diagnosis.doctorId !== doctorId

    ) {

        throw new ForbiddenException(

            "You can only edit your own diagnoses."
        );
    }

    return repository.update(

        diagnosisId,

        payload
    );
}
```

---

# Understanding This Code

First

```
Load Diagnosis
```

Next

```
Compare

diagnosis.doctorId

↓

doctorId
```

If they differ

```
Forbidden
```

Otherwise

```
Update Diagnosis
```

---

# Step 3 — Controller

Modify

```
diagnosis.controller.ts
```

```ts
export async function update(
  req: Request,

  res: Response
) {
  const diagnosis = await service.updateDiagnosis(
    req.params.id,

    req.user.id,

    req.body
  )

  return res.json(diagnosis)
}
```

Notice

The client never sends

```
doctorId
```

We always use

```
req.user.id
```

obtained from the JWT.

# Why Use req.user.id?

Imagine this request.

```json
{
  "doctorId": "doctor_999"
}
```

If the backend trusted

the request body,

any user could impersonate

another doctor.

Instead,

the authenticated user

comes from

```
JWT

↓

req.user
```

This cannot be modified

by the client.

# Production Notes

Always

- Authenticate first.
- Check permissions.
- Load the resource.
- Verify ownership.
- Never trust resource ownership sent by the client.
- Return **403 Forbidden** when ownership validation fails.

# Common Mistakes

## Trusting Client IDs

Wrong

```json
{
  "doctorId": "doctor_2"
}
```

Correct

```
req.user.id
```

---

## Checking Ownership Before Authentication

Always identify

the user

before checking ownership.

---

## Putting Ownership Inside Repository

Repositories only access data.

Business rules belong

inside Services.

---

## Returning HTTP 401

Ownership failures are

```
403 Forbidden
```

Authentication failures are

```
401 Unauthorized
```

# Summary

Permission determines whether a user is allowed to perform an action.

Ownership determines whether the user is allowed to perform that action on a specific resource.

By combining both checks, we prevent users from modifying resources that belong to someone else, even when they possess the required permission.

Ownership is one of the most important authorization patterns in production backend systems and is widely used in healthcare, finance, collaboration platforms, and enterprise applications.

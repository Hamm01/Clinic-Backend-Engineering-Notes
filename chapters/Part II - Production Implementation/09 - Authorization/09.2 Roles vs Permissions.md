# 09.2 - Roles vs Permissions

> Roles describe who a user is within the organization, while permissions describe what that user is allowed to do. As applications grow, relying only on roles becomes difficult to maintain. This chapter introduces permission-based authorization and explains why it scales better than role-only systems.

---

- What a Role is.
- What a Permission is.
- Why Roles alone become difficult to manage.
- Why enterprise applications use Permissions.

# The Problem

Our application currently checks permissions like this.

```ts
authorize('ADMIN', 'DOCTOR')
```

At first, this looks simple.

If the authenticated user's role is

```
ADMIN
```

or

```
DOCTOR
```

the request is allowed.

Otherwise,

the request is rejected.

For a small application,

this works perfectly.

---

# The Problem With Roles

Suppose we protect an endpoint like this.

```ts
authorize('ADMIN', 'CLINIC_MANAGER', 'REGIONAL_MANAGER')
```

Another endpoint.

```ts
authorize('ADMIN', 'DOCTOR', 'SENIOR_DOCTOR')
```

Another endpoint.

```ts
authorize('ADMIN', 'LAB_TECHNICIAN')
```

As more roles appear,

every endpoint becomes harder to maintain.

---

# Changing Business Rules

Imagine the business decides

```
Receptionists

↓

Can Edit Patients
```

Yesterday

```
No
```

Today

```
Yes
```

Tomorrow

```
Only Senior Receptionists
```

Now every role check throughout the application must be updated.

Maintenance becomes difficult.

---

# Introducing Permissions

Instead of asking

```
Who is this user?
```

we ask

```
What can this user do?
```

Examples

```
patient:create
```

```
patient:update
```

```
patient:delete
```

```
doctor:create
```

```
doctor:update
```

```
doctor:delete
```

Notice

Permissions describe **actions**, not people.

---

# Role

A Role is simply a collection of responsibilities.

Examples

```
Administrator
```

```
Doctor
```

```
Receptionist
```

Roles help organize users.

They do not directly describe every capability.

---

# Permission

A Permission describes

one allowed action.

Examples

```
patient:create
```

```
patient:view
```

```
patient:update
```

```
patient:delete
```

Each permission represents

exactly one capability.

---

# Relationship Between Roles and Permissions

Instead of

```
User

↓

Role

↓

Route
```

we move to

```
User

↓

Role

↓

Permissions

↓

Route
```

The Role grants

many Permissions.

The Route checks

Permissions.

---

# Example

Role

```
Doctor
```

Permissions

```
patient:view

patient:update

diagnosis:create

diagnosis:update
```

Another role

```
Receptionist
```

Permissions

```
appointment:create

appointment:update

patient:view
```

Notice

Both roles

share

```
patient:view
```

No duplication is required.

---

# Business Scenario

Suppose the business asks

```
Receptionists

↓

Can View Patients
```

Simply assign

```
patient:view
```

No route changes.

No middleware changes.

No controller changes.

Only the role-to-permission mapping changes.

---

# Authorization Flow

```
User

↓

Role

↓

Permissions Loaded

↓

Permission Checked

↓

Business Logic
```

Notice

The application is no longer asking

```
Is Doctor?
```

Instead it asks

```
Has Permission?
```

---

# Why This Scales Better

Adding a new role

```
Insurance Officer
```

does not require changing

50 endpoints.

Simply assign

the appropriate permissions.

The authorization system continues working.

---

# Real Company Example

Imagine GitHub.

Roles

```
Owner

Administrator

Maintainer

Developer

Reporter

Guest
```

Internally,

GitHub does not hardcode

Instead,

---

# Comparing Both Approaches

| Role-Based Authorization    | Permission-Based Authorization |
| --------------------------- | ------------------------------ |
| Easy to start               | Slightly more setup            |
| Difficult to scale          | Highly scalable                |
| Duplicate role checks       | Reusable permissions           |
| Harder to maintain          | Easier to maintain             |
| Good for small applications | Ideal for production systems   |

Roles

---

# Common Mistakes

## Confusing Roles with Permissions

Wrong

```
Role

↓

Capability
```

Correct

```
Role

↓

Collection of Permissions
```

---

## Hardcoding Role Names

Avoid

```ts
if (role === "ADMIN")
```

Prefer

```ts
authorize('patient:create')
```

or

```ts
authorize('diagnosis:update')
```

depending on the implementation.

---

## Creating Too Many Roles

Adding new roles for every small requirement creates unnecessary complexity.

Instead,

reuse permissions.

---

# Senior Engineer Notes

Permissions provide long-term flexibility.

As organizations grow,

business rules change far more frequently than application code.

By assigning permissions to roles instead of embedding role checks throughout the application, backend systems remain easier to maintain, easier to audit, and easier to extend without major code changes.

This is why permission-based authorization has become the standard approach in modern enterprise applications.

---

# Summary

Roles describe **who a user is**.

Permissions describe **what a user can do**.

Small applications can rely on roles alone, but production systems combine roles with permissions to build flexible, scalable, and maintainable authorization.

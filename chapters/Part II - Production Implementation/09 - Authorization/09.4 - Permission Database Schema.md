# 09.4 - Permission Database Schema

> A scalable authorization system is built on a well-designed database schema. Instead of hardcoding roles and permissions inside the application, we store them in relational tables that can evolve as business requirements change. In this chapter, we will design, implement, migrate, and populate the authorization database using Prisma.

---

- How to seed default authorization data.
- How enterprise applications manage authorization.

# Current Authorization Architecture

Until now, our authorization looked like this.

```
User

↓

Role

↓

Route
```

Example

```ts
authorize('ADMIN', 'DOCTOR')
```

This works for small applications.

As applications grow,

it becomes difficult to maintain.

---

# Our New Authorization Architecture

Instead of checking roles directly,

we introduce permissions.

```
User

↓

Role

↓

RolePermission

↓

Permission

↓

Route
```

The application no longer asks

```
Is this user a Doctor?
```

Instead,

it asks

```
Does this user have

patient:update

permission?
```

This design is much more flexible.

---

# Why Database Driven Authorization?

Imagine the business creates a new role.

```
Insurance Manager
```

Should we modify

```
schema.prisma

↓

Migration

↓

Application Code

↓

Deployment
```

every time?

No.

Instead,

we insert a new Role record into the database and assign the appropriate permissions.

The application continues working without changing business logic.

# Step 1 — Update the User Model

Open

```
prisma/schema.prisma
```

Replace

```prisma
role String
```

with

```prisma
roleId String

role Role
    @relation(
        fields: [roleId],
        references: [id]
    )
```

### Why?

Previously,

every user stored

```
ADMIN

DOCTOR

RECEPTIONIST
```

inside the User table.

Now,

the user stores only

```
roleId
```

which references the Role table.

Benefits

- Better normalization.
- No duplicated role names.
- Easier updates.
- Strong foreign key integrity.

---

# Step 2 — Create the Role Model

```prisma
model Role {

  id String @id @default(uuid())

  name String @unique

  description String?

  users User[]

  rolePermissions RolePermission[]

  createdAt DateTime @default(now())

  updatedAt DateTime @updatedAt
}
```

---

# Why These Fields?

## id

Primary Key.

Every role must have a unique identifier.

---

## name

Business-readable name.

Examples

```
ADMIN

DOCTOR

RECEPTIONIST
```

Marked unique because duplicate roles are not allowed.

---

## description

Optional explanation.

Example

```
Responsible for managing clinics.
```

Useful for administrators and documentation.

---

## users

One Role

↓

Many Users

Example

```
Doctor

↓

John

↓

Alice

↓

Robert
```

---

## rolePermissions

One Role

↓

Many Permissions

through

```
RolePermission
```

---

# Step 3 — Create Permission Model

```prisma
model Permission {

  id String @id @default(uuid())

  name String @unique

  description String?

  rolePermissions RolePermission[]

  createdAt DateTime @default(now())

  updatedAt DateTime @updatedAt
}
```

---

# Why Separate Permissions?

Permissions represent

individual actions.

Examples

```
patient:create

patient:view

patient:update

patient:delete

doctor:create

appointment:create

diagnosis:update
```

One permission

↓

One capability.

---

# Permission Naming Convention

We will follow

```
resource:action
```

Examples

```
patient:view

patient:create

patient:update

patient:delete

doctor:view

appointment:create

clinic:update
```

Benefits

- Human readable
- Easy to search
- Consistent naming
- Easy to audit

---

# Step 4 — Create RolePermission Model

Roles and Permissions have a

```
Many-to-Many
```

relationship.

One Role

↓

Many Permissions

One Permission

↓

Many Roles

Create

```prisma
model RolePermission {

  roleId String

  permissionId String

  role Role
      @relation(
        fields: [roleId],
        references: [id]
      )

  permission Permission
      @relation(
        fields: [permissionId],
        references: [id]
      )

  @@id([roleId, permissionId])
}
```

---

# Why a Junction Table?

Without a junction table,

you would duplicate permission records.

Example

Wrong

```
Doctor View Patient

Admin View Patient

Manager View Patient
```

Correct

```
patient:view
```

shared by multiple roles.

---

# Final Database Design

```
                User
                  │
                  │
                  ▼
                Role
                  │
                  │
         RolePermission
          ▲          ▼
          │          │
      Permission
```

This design follows

Third Normal Form (3NF)

and avoids unnecessary duplication.

# Step 6 — Seed Roles

Instead of manually inserting data,

we use Prisma Seed.

Example Roles

```
ADMIN

DOCTOR

RECEPTIONIST
```

Always use

```ts
upsert()
```

instead of

```ts
create()
```

This allows the seed to run multiple times without duplicate record errors.

# Production Notes

Always

- Keep Roles in the database.
- Keep Permissions in the database.
- Use Many-to-Many relationships.
- Use consistent permission naming.
- Use seed files.
- Never hardcode IDs.
- Keep authorization data under version control.

---

# Common Mistakes

## Using Role Strings Everywhere

Wrong

```ts
if (user.role === "ADMIN")
```

Correct

Load permissions and check capabilities.

---

## Duplicate Permissions

Avoid

```
Doctor Update Patient

Admin Update Patient
```

Create one permission

```
patient:update
```

and assign it to multiple roles.

---

## Skipping Seed Files

Never rely on manually inserting authorization data.

Always use Prisma Seed.

---

# Senior Engineer Notes

Enterprise applications rarely hardcode authorization logic.

Instead,

they store roles and permissions as relational data, allowing business rules to evolve without modifying application code.

This design supports future requirements such as

- Custom Roles
- Organization-specific Permissions
- Multi-Tenant Authorization
- Administrative Permission Management

without changing the underlying architecture.

# 09.7 - Policy-Based Authorization

> Roles determine who a user is. Permissions determine what a user can do. Ownership determines whether a user owns a resource. However, many real-world business rules cannot be expressed using only these concepts. Policy-Based Authorization introduces dynamic authorization rules that are evaluated based on the current business context.

# What is a Policy?

A policy is a business rule that must be satisfied before an operation is allowed.

Unlike permissions,

a policy depends on

current business data.

Example

```
Doctor

↓

Has diagnosis:update

permission

↓

BUT

↓

Can only update diagnoses

inside their assigned clinic.
```

Permission alone

cannot answer this question.

---

# Why Permissions Are Not Enough

Imagine

Dr. John

has

```
diagnosis:update
```

permission.

Now imagine

there are two clinics.

```
Clinic A

Clinic B
```

Dr. John works in

```
Clinic A
```

Should he modify diagnoses

inside

```
Clinic B
```

No.

The permission exists,

but the business policy denies access.

# Business Rules in Our Clinic

Let's define a few policies.

## Policy 1

Doctors may only access patients

assigned to their clinic.

---

## Policy 2

Receptionists may create appointments

only during clinic working hours.

---

## Policy 3

A diagnosis cannot be modified

after it has been finalized.

---

## Policy 4

Only the assigned doctor

may prescribe medicines.

---

## Policy 5

Administrators

may bypass all clinic restrictions.

# Repository

Modify

```
diagnosis.repository.ts
```

```ts
async findWithClinic(

    diagnosisId: string

) {

    return prisma.diagnosis.findUnique({

        where: {

            id: diagnosisId,
        },

        include: {

            clinic: true,
        },
    });
}
```

---

# Why Include Clinic?

The authorization policy depends on

```
Clinic
```

Without loading it,

the Service cannot compare

clinic ownership.

---

# Service

Modify

```
diagnosis.service.ts
```

```ts
const diagnosis = await repository.findWithClinic(diagnosisId)

if (diagnosis.clinicId !== req.user.clinicId) {
  throw new ForbiddenException('Access denied.')
}
```

# Why Create a Policy File?

As applications grow,

policy logic becomes large.

Instead of placing

hundreds of lines

inside

```
diagnosis.service.ts
```

we can move

authorization rules into

```
diagnosis.policy.ts
```

Example

```ts
DiagnosisPolicy

↓

canUpdate()

↓

canDelete()

↓

canPrescribeMedicine()
```

This keeps the Service focused on orchestration while the Policy class contains business authorization rules.

---

# Production Notes

Always

- Authenticate first.
- Validate permissions.
- Validate ownership.
- Evaluate business policies.
- Keep policy logic reusable.
- Never trust client input for policy decisions.

---

# Common Mistakes

## Putting Policies in Middleware

Policies usually require business data.

They belong in the Service layer or dedicated Policy classes.

---

## Confusing Permissions with Policies

Permission

```
Can Update Diagnosis
```

Policy

```
Can update only before diagnosis is finalized.
```

They solve different problems.

---

## Hardcoding Business Rules Everywhere

Create reusable policy classes instead of duplicating authorization logic.

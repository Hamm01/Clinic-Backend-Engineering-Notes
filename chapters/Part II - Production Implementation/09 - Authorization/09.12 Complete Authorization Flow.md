```
                    Client
                       │
                       ▼

                 └────────────┘
                       │
```

# Step 1 — Client Sends Request

Example

```

```

Headers

```http
Authorization: Bearer eyJhbGciOi...
```

Body

```json
{
  "notes": "Blood pressure improving."
}
```

The client requests to update a diagnosis.

---

# Step 2 — Authentication Middleware

File

```
src/middlewares/authentication.middleware.ts
```

Responsibilities

- Read Authorization header.
- Verify JWT signature.
- Validate expiration.
- Extract user information.
- Attach authenticated user to the request.

Result

```ts
req.user = {
  id: 'doctor_001',

  roleId: 'role_doctor',

  clinicId: 'clinic_01'
}
```

If authentication fails

```
401 Unauthorized
```

The request stops here.

---

# Why Authentication Comes First

Without authentication

we do not know

who is making the request.

Authorization cannot happen

without identity.

---

# Step 3 — Permission Middleware

File

```
src/middlewares/permission.middleware.ts
```

Responsibilities

- Load user permissions.
- Check Redis cache.
- Load from PostgreSQL if cache misses.
- Compare required permissions.

Example

```ts
authorize('diagnosis:update')
```

Permission Required

```
diagnosis:update
```

User Permissions

```
patient:view

diagnosis:update

appointment:create
```

Permission exists.

Continue.

Otherwise

```
403 Forbidden
```

# Why Redis Is Used

Without Redis

Every request

↓

Database

↓

Permissions

↓

Continue

With Redis

Request

↓

Redis

↓

Continue

Database load

is significantly reduced.

# Step 5 — Service Layer

File

```
diagnosis.service.ts
```

Responsibilities

- Load diagnosis.
- Verify ownership.
- Evaluate business policies.
- Coordinate update operation.

Example

```ts
const diagnosis = await repository.findById(id)

if (diagnosis.doctorId !== req.user.id) {
  throw new ForbiddenException('Access denied.')
}
```

# Step 6 — Ownership Validation

Question

```
Does the diagnosis belong

to this doctor?
```

If

```
No
```

Return

```

```

Ownership protects

individual resources.

# Step 7 — Policy Evaluation

Ownership passed.

Now evaluate

business rules.

Examples

```
Clinic Match

↓

Diagnosis Locked?

↓

Appointment Completed?

↓

Doctor Active?
```

If any rule fails

```
403 Forbidden
```

# Step 8 — Repository

File

```
diagnosis.repository.ts
```

Responsibilities

- Execute Prisma queries.
- No business logic.
- No authorization decisions.

Example

```ts
return prisma.diagnosis.update({
  where: {
    id
  },

  data: payload
})
```

Repositories

only interact

with the database.

---

# Step 9 — PostgreSQL

Only after every validation succeeds

does the application

modify the database.

This minimizes

unnecessary writes

# Why This Architecture?

Every layer has

one responsibility.

| Layer                 | Responsibility        |
| --------------------- | --------------------- |
| Authentication        | Verify Identity       |
| Permission Middleware | Verify Capability     |
| Service               | Business Logic        |
| Ownership             | Verify Resource Owner |
| Policy                | Verify Business Rules |
| Repository            | Database Access       |
| PostgreSQL            | Data Storage          |
| Audit Logger          | Security Tracking     |

This separation makes the codebase easier to test, maintain, and extend.

# 401 vs 403

One of the most common interview questions.

## 401 Unauthorized

Meaning

```
Identity cannot be verified.
```

Examples

- Missing JWT
- Invalid JWT
- Expired JWT

---

## 403 Forbidden

Meaning

```
Identity verified

↓

Access denied.
```

Examples

- Missing permission
- Wrong owner
- Business policy failure

# Senior Engineer Notes

A common misconception is that authentication and authorization are the same thing.

They solve different problems.

Authentication answers:

> **Who are you?**

Authorization answers:

> **What are you allowed to do?**

Production systems then add two more questions:

> **Do you own this resource?**

> **Does the current business state allow this action?**

Thinking in these layers produces secure, scalable backend systems.

# 09.9 - Permission Caching

> Authorization is one of the most frequently executed operations in a backend application. Every protected request requires checking whether the authenticated user has the required permissions. Continuously querying the database for the same authorization data introduces unnecessary latency and increases database load. In this chapter, we will optimize authorization by introducing Permission Caching using Redis.

# Why Is This a Problem?

Permissions rarely change.

For example

```
Doctor

↓

patient:view

↓

diagnosis:update

↓

appointment:create
```

These permissions may remain unchanged for weeks.

Yet every request reloads them from PostgreSQL.

This wastes database resources.

---

PostgreSQL


At scale,

# Step 1 — Create Cache Service

Create

```
permission.cache.ts
```

```ts
import redisClient from '../config/redis'

export class PermissionCache {
  async get(userId: string) {
    return redisClient.get(`permissions:${userId}`)
  }

  async set(
    userId: string,

    permissions: string[]
  ) {
    await redisClient.set(
      `permissions:${userId}`,

      JSON.stringify(permissions),

      {
        EX: 3600
      }
    )
  }

  async remove(userId: string) {
    await redisClient.del(`permissions:${userId}`)
  }
}
```

---

# Understanding This Code

Redis stores

```
Key

↓

Value
```

Example

```

```

↓

```json

```

Every user has

their own permission cache.

```

```

```ts
const cachedPermissions = await permissionCache.get(userId)

if (cachedPermissions) {
  permissions = JSON.parse(cachedPermissions)
} else {
  permissions = await authService.getUserPermissions(userId)

  await permissionCache.set(
    userId,

    permissions
  )
}
```

---

# When Should Cache Be Cleared?

Whenever

- Role changes
- Permission changes
- User changes role
- User is suspended
- User is deleted

Always invalidate stale authorization data.

---

# Why Not Cache Forever?

Permissions can change.

Keeping stale data

creates

security problems.

Always define

a cache expiration time.

Example

```
1 Hour
```

or

```
15 Minutes
```

depending on business requirements.

Always

- Cache permissions.
- Use cache expiration.
- Invalidate cache after permission updates.
- Never trust stale authorization data.
- Keep Redis synchronized with PostgreSQL.

# Common Mistakes

## Forgetting Cache Invalidation

User permissions change,

but Redis still returns

old values.

This creates security vulnerabilities.

---

## Caching Entire Users

Only cache

authorization data.

Avoid caching unnecessary information.

---

## Infinite Cache Lifetime

Never cache permissions forever.

Authorization data changes.

---

## Skipping Redis

Loading permissions from PostgreSQL

for every request

does not scale.

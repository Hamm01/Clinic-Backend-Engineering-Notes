# Step 1 вЂ” Create Policy Class

Create

```
src/policies/diagnosis.policy.ts
```

```ts
export class DiagnosisPolicy {
  canUpdate(user: AuthenticatedUser, diagnosis: Diagnosis): boolean {
    if (user.role.name === 'ADMIN') {
      return true
    }

    return diagnosis.doctorId === user.id
  }
}
```

# Common Mistakes

## Putting Every Rule in Middleware

Middleware should remain generic.

Business-specific rules belong in policy classes.

---

## Mixing Repository and Authorization

Repositories fetch data.

Policies make authorization decisions.

Keep responsibilities separate.

---

## Hardcoding Admin Checks Everywhere

Avoid

```ts
if (user.role.name === "ADMIN")
```

inside multiple services.

Centralize the logic inside policy classes.

# Summary

Modern backend applications rarely rely on a single authorization mechanism.

Instead, they combine:

- Authentication
- Permissions
- Ownership
- Business Policies

Each layer answers a different question, resulting in a secure, scalable, and maintainable authorization system.

By separating responsibilities across Middleware, Policies, Services, and Repositories, our Clinic Backend now follows an authorization architecture similar to those used in enterprise healthcare, finance, and collaboration platforms.

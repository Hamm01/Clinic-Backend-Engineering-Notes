# 09.1 - Authorization Overview

> Authentication answers **"Who are you?"** Authorization answers **"What are you allowed to do?"** Although these concepts are closely related, they solve different problems. This chapter introduces authorization from first principles and explains why every production backend requires it.

# The Problem

Suppose three users log into our application.

```
Administrator

Doctor

Receptionist
```

Authentication verifies that all three users are legitimate.

The backend now knows:

```
Administrator

↓

Authenticated
```

```
Doctor

↓

Authenticated
```

```
Receptionist

↓

Authenticated
```

Authentication is complete.

But another question immediately appears.

> Should every authenticated user be allowed to perform every action?

The answer is **No**.

That is where Authorization begins.

---

# Real Business Scenario

Imagine our clinic has the following operations.

```
Create Patient

Delete Doctor

View Diagnosis

Manage Clinics

Create Appointment

Update Billing
```

Can everyone perform every operation?

No.

For example,

```
Receptionist
```

can schedule appointments.

But

```
Receptionist
```

should never

```
Delete Doctor
```

Likewise,

```
Doctor
```

can create diagnoses,

but usually cannot create a new clinic.

These decisions are business rules.

Authorization enforces those business rules.

---

# Authentication vs Authorization

Authentication asks

```
Who are you?
```

Authorization asks

```
What are you allowed to do?
```

Example

```
Login

↓

JWT Verified

↓

User Identified

↓

Role Identified

↓

Permission Checked

↓

Business Logic Executes
```

Notice

Authentication happens first.

Authorization happens second.

Without Authentication,

Authorization has nothing to evaluate.

---

# Why Authorization Exists

Imagine the following API.

```
DELETE /doctors/:id
```

Without Authorization,

any authenticated user could delete doctors.

Even

```
Receptionist

↓

Delete Doctor
```

This is clearly incorrect.

Authorization prevents these situations.

---

# Request Lifecycle

A production request now looks like this.

```
Client

↓

HTTP Request

↓

Authentication Middleware

↓

Identify User

↓

Authorization Middleware

↓

Check Permission

↓

Controller

↓

Service

↓

Repository

↓

Database

↓

Response
```

Notice

Authorization executes **before** business logic.

If permission is denied,

the Controller is never reached.

---

# Authorization Is a Business Rule

Consider this question.

Can a doctor prescribe medicine?

```
Yes
```

Can a receptionist prescribe medicine?

```
No
```

Can an administrator prescribe medicine?

```
Maybe
```

These answers do not come from Express.

They do not come from Prisma.

They do not come from PostgreSQL.

They come from the business domain.

Authorization is simply the backend enforcing business rules.

# Examples of Authorization Decisions

| Request                                 | Allowed? | Why?                              |
| --------------------------------------- | -------- | --------------------------------- |
| Doctor creates diagnosis                | ✅       | Doctor role has permission        |
| Receptionist creates diagnosis          | ❌       | Permission denied                 |
| Admin deletes clinic                    | ✅       | Administrative privilege          |
| Doctor deletes clinic                   | ❌       | Outside doctor's responsibilities |
| Doctor edits own diagnosis              | ✅       | Ownership rule                    |
| Doctor edits another doctor's diagnosis | ❌       | Ownership restriction             |

Notice

Authorization is about enforcing business rules consistently.

---

# Where Authorization Lives

Authorization should never be placed inside

- Prisma
- Database
- Repository

Instead,

it belongs in

```
Authentication

↓

Authorization Middleware

↓

Controller

↓

Service
```

Some advanced authorization decisions may also occur inside the Service layer when they depend on business data.

---

# Why Keep Controllers Thin?

Wrong

```ts
if (req.user.role !== "ADMIN") {
    return res.status(403).json(...);
}
```

Every controller repeats the same logic.

Correct

```
Request

↓

Authorization Middleware

↓

Controller
```

Controllers remain focused on handling requests and responses.

# Production Notes

---

# Common Mistakes

## Confusing Authentication with Authorization

Authentication verifies identity.

Authorization verifies permissions.

They solve different problems.

---

## Hardcoding Role Checks Everywhere

Avoid repeating

```ts
if (role === "ADMIN")
```

throughout the application.

Use reusable middleware and services.

---

## Returning the Wrong Status Code

Authentication failure

```
401 Unauthorized
```

Authorization failure

```
403 Forbidden
```

Understanding this distinction is important for designing REST APIs.

---

# Senior Engineer Notes

Authorization becomes more complex as applications grow.

A small application may only require roles.

A medium-sized application introduces permissions.

Large enterprise systems often implement policy engines, ownership rules, and fine-grained authorization.

By building our authorization system incrementally, we create an architecture that scales without major rewrites.

# Summary

Authorization determines what an authenticated user is allowed to do within the system.

While Authentication establishes identity, Authorization enforces business rules by evaluating roles, permissions, ownership, and policies before business logic executes.

Throughout this chapter, we will evolve our simple role-based middleware into a flexible authorization system capable of supporting real-world production applications.

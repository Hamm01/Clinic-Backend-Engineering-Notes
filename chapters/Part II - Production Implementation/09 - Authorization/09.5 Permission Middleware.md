# 09.5 - Permission Middleware

> Until now, our application authenticated users and associated them with roles. In this chapter, we will implement a reusable Permission Middleware that verifies whether an authenticated user has the required permission before allowing access to protected resources. This middleware becomes the central authorization mechanism used throughout the application.

# The Problem

Until now our middleware looked like this.

```ts
authorize('ADMIN', 'DOCTOR')
```

This middleware checks

```
Role

↓

Allowed?
```

This works,

but only while the application has a small number of roles.

As the project grows,

role checks become difficult to maintain.

---

# New Authorization Flow

Instead of checking

```
Role
```

we check

```
Permission
```

Example

```
patient:update

doctor:create

appointment:view
```

Our middleware becomes

```ts

```

This is much more flexible.

# Step 1 — Load User Permissions

Before checking permissions,

the middleware needs to know

which permissions belong to the authenticated user.

Modify


## Repository

```ts
async findUserWithPermissions(userId: string) {

    return prisma.user.findUnique({

        where: {

            id: userId,
        },

        include: {

            role: {

                include: {

                    rolePermissions: {

                        include: {

                            permission: true,
                        },
                    },
                },
            },
        },
    });
}
```

Everything required for authorization.

# Step 2 — Service Layer

Modify

```
auth.service.ts
```

```ts
async getUserPermissions(userId: string) {

    const user =
        await repository.findUserWithPermissions(
            userId
        );

    if (!user) {

        throw new Error(
            "User not found."
        );
    }

    return user.role.rolePermissions.map(

        (rolePermission) =>

            rolePermission.permission.name
    );
}
```

# Step 3 — Create Permission Middleware

Create

```
permission.middleware.ts
```

```ts
import { NextFunction, Request, Response } from 'express'
import { AuthService } from '../modules/auth/auth.service'

const authService = new AuthService()

export const authorize = (...requiredPermissions: string[]) => {
  return async (
    req: Request,

    res: Response,

    next: NextFunction
  ) => {
    const userId = req.user.id

    const permissions = await authService.getUserPermissions(userId)

    const allowed = requiredPermissions.some(permission =>
      permissions.includes(permission)
    )

    if (!allowed) {
      return res.status(403).json({
        message: 'Permission denied.'
      })
    }

    next()
  }
}
```

# Production Notes

Always

- Authenticate before Authorizing.
- Load permissions from the database.
- Keep Controllers free from permission logic.
- Return HTTP 403 when permission checks fail.
- Reuse middleware across all routes.

# Summary

In this chapter, we implemented a reusable Permission Middleware that protects API endpoints using permissions instead of hardcoded role names.

The middleware loads the authenticated user's permissions through the Service and Repository layers, compares them with the required permissions, and either allows the request to continue or returns **403 Forbidden**.

This approach keeps controllers clean, centralizes authorization logic, and provides a scalable foundation for future business rules.

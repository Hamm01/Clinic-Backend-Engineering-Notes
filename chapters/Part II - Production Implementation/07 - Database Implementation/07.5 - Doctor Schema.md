# 07.5 - Doctor Schema

> A doctor is the healthcare professional who consults patients, creates diagnoses, prescribes medicines, and manages appointments.

---

# Objective

In this chapter we will:

- Design the Doctor entity.
- Associate every doctor with a clinic.
- Understand every field.
- Implement the Prisma model.
- Prepare the Doctor entity for authentication and appointments.

---

# Business Requirement

A doctor works at a clinic.

Every doctor has:

- Personal information
- Contact information
- Professional information
- Employment information
- Audit information

Unlike a patient, a doctor is an employee of the clinic.

Every doctor belongs to exactly one clinic.

---

# Prisma Model

Open

```text
prisma/schema.prisma
```

Add the following model.

```prisma
model Doctor {

  id               String   @id @default(uuid())

  clinicId         String

  firstName        String

  lastName         String

  email            String   @unique

  phone            String?

  specialization   String

  licenseNumber    String   @unique

  yearsOfExperience Int?

  isActive         Boolean  @default(true)

  createdAt        DateTime @default(now())

  updatedAt        DateTime @updatedAt

  clinic           Clinic   @relation(fields: [clinicId], references: [id])
}
```

---

# Updating the Clinic Model

Add the following relationship inside the existing Clinic model.

```prisma
doctors Doctor[]
```

The updated Clinic model now knows that one clinic can have many doctors.

# Understanding Every Field

## id

```prisma
id String @id @default(uuid())
```

Primary key.

Every doctor receives a unique identifier.

This identifier is used throughout the application when creating appointments and medical records.

---

## clinicId

```prisma
clinicId String
```

Every doctor belongs to one clinic.

Instead of storing clinic information repeatedly,

we store only the clinic's identifier.

This field becomes the foreign key connecting Doctor to Clinic.

---

## firstName

```prisma
firstName String
```

Stores the doctor's first name.

---

## lastName

```prisma
lastName String
```

Stores the doctor's family name.

Keeping names separated improves searching and reporting.

---

## email

```prisma
email String @unique
```

Every doctor has a unique email address.

Later, this email will also be used during authentication.

The database guarantees uniqueness.

---

## phone

```prisma
phone String?
```

Optional contact number.

Some organizations may communicate using internal systems instead of personal phone numbers.

---

## specialization

```prisma
specialization String
```

Represents the doctor's medical specialty.

Examples

```
Cardiology

Orthopedics

Dermatology

Neurology
```

Appointments and patient searches often filter by specialization.

---

## licenseNumber

```prisma
licenseNumber String @unique
```

Every licensed doctor receives a unique registration number.

Examples

```
MED-458721

DOC-982134
```

The database prevents duplicate licenses.

---

## yearsOfExperience

```prisma
yearsOfExperience Int?
```

Optional field.

Useful for displaying doctor profiles and filtering experienced practitioners.

---

## isActive

```prisma
isActive Boolean @default(true)
```

Doctors sometimes leave an organization temporarily.

Instead of deleting the record,

we simply deactivate it.

Historical appointments remain valid.

---

## createdAt

```prisma
createdAt DateTime @default(now())
```

Stores when the doctor was registered.

Useful for auditing and reporting.

---

## updatedAt

```prisma
updatedAt DateTime @updatedAt
```

Automatically updated whenever the doctor's information changes.

---

## clinic

```prisma
clinic Clinic @relation(fields: [clinicId], references: [id])
```

Defines the relationship with the Clinic model.

Prisma uses this information to generate joins automatically.

---

# Relationship

The relationship between Clinic and Doctor is

```
Clinic (1)

      │

      │

      ▼

Doctor (Many)
```

One clinic can employ many doctors.

Every doctor belongs to one clinic.

---

# PostgreSQL Representation

Conceptually, PostgreSQL stores the following table.

```sql
CREATE TABLE doctors (

    id UUID PRIMARY KEY,

    clinic_id UUID NOT NULL,

    first_name TEXT NOT NULL,

    last_name TEXT NOT NULL,

    email TEXT UNIQUE NOT NULL,

    phone TEXT,

    specialization TEXT NOT NULL,

    license_number TEXT UNIQUE NOT NULL,

    years_of_experience INTEGER,

    is_active BOOLEAN DEFAULT TRUE,

    created_at TIMESTAMP,

    updated_at TIMESTAMP,

    FOREIGN KEY (clinic_id)

        REFERENCES clinics(id)
);
```

# Why This Design?

The Doctor table stores only information related to the doctor.

It intentionally does not store:

- Appointments
- Patients
- Diagnoses
- Prescriptions

Those belong to separate entities.

The Doctor model should remain focused on the doctor's identity and professional information.

---

# Common Mistakes

### Using Doctor Name as Identifier

Doctor names are not guaranteed to be unique.

Always use a primary key.

---

### Storing Clinic Details Inside Doctor

Do not duplicate clinic information.

The relationship already provides access to the clinic.

---

### Deleting Doctors

Deleting doctors may break historical appointment records.

Deactivate doctors instead.

---

### Forgetting Unique Constraints

Doctor emails and medical license numbers should always remain unique.

---

# Senior Engineer Notes

Before approving the Doctor schema, verify:

- Does every doctor belong to a clinic?
- Are license numbers unique?
- Can historical appointments still reference inactive doctors?
- Is business data separated from appointment data?

A well-designed Doctor entity becomes the foundation for appointments, authentication, diagnoses, and prescriptions.

# Summary

The Doctor entity represents healthcare professionals working within a clinic.

It stores identity, professional information, and the relationship to a clinic, while leaving appointments and medical records to their own dedicated entities.

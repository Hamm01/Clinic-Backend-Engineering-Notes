# 07.13 - Prisma Migrations

> A migration is a version-controlled change to your database schema. It converts the Prisma models you write into actual PostgreSQL tables.

---

# Objective

In this chapter we will:

- Understand what a migration is.
- Generate our first migration.
- Apply migrations to PostgreSQL.
- Understand migration history.
- Learn the migration workflow used in production.

# What is a Migration?

A migration converts

```
Prisma Schema

↓

SQL Statements

↓

PostgreSQL Tables
```

Instead of writing SQL manually,

Prisma generates the SQL for us.

Think of it as a translator.

```
schema.prisma

↓

Prisma Migration Engine

↓

CREATE TABLE ...

↓

PostgreSQL
```

---

```bash
npx prisma migrate dev --name init
```

# What Happens Internally?

When the command runs,

Prisma performs multiple steps.

```
Read schema.prisma

↓

Compare Current Database

↓

Generate SQL

↓

Execute SQL

↓

Create Migration Folder

↓

Generate Prisma Client
```

Everything happens automatically.

# Project Structure After Migration

```

│       └── migration.sql
│
├── schema.prisma

```

Notice the new folder.

```
migrations/
```

Every database change will be stored here.

# Generated SQL

Open

```
prisma/migrations/.../migration.sql
```

You will find SQL similar to

```sql
CREATE TABLE "Clinic" (

    "id" TEXT NOT NULL,

    "name" TEXT NOT NULL,

    PRIMARY KEY ("id")

);

CREATE TABLE "Doctor" (

    "id" TEXT NOT NULL,

    "clinicId" TEXT NOT NULL,

    PRIMARY KEY ("id")

);

CREATE TABLE "Patient" (

    "id" TEXT NOT NULL,

    PRIMARY KEY ("id")

);
```

Prisma generates this automatically.

You never write this SQL yourself.

# Migration History

Prisma also creates a table inside PostgreSQL.

```
_prisma_migrations
```

Example

```
Migration Name

Applied At

Checksum

Finished Time
```

This table tracks every migration applied to the database.

Never edit this table manually.

# Making Another Change

Suppose we update

```prisma
model Patient {

  bloodGroup String?

}
```

The schema changed.

Now create another migration.

```bash
npx prisma migrate dev --name add-blood-group
```

Prisma generates

```
migrations/

↓

20260713102000_init

↓

20260714131520_add_blood_group
```

Only the new change is applied.

The existing tables remain unchanged.

# Migration Workflow

The development workflow is always

```
Modify schema.prisma

↓

Create Migration

↓

Migration SQL Generated

↓

Database Updated

↓

Prisma Client Generated

↓

Commit to Git
```

The migration becomes part of the project.

---

# Migrations in Git

Always commit

```
prisma/schema.prisma

prisma/migrations/
```

Example

```bash
git add prisma/

git commit -m "Create appointment schema"
```

When another developer clones the repository,

they receive the complete database history.

---

# Development vs Production

Development

```bash
npx prisma migrate dev
```

Purpose

- Creates migrations
- Applies migrations
- Generates Prisma Client

---

Production

```bash
npx prisma migrate deploy
```

Purpose

- Applies existing migrations only
- Never creates new migrations

This ensures every production environment uses the same database schema.

# migrate dev vs db push

Both commands update the database,

but they serve different purposes.

### migrate dev

```bash
npx prisma migrate dev
```

- Creates migration files.
- Maintains migration history.
- Safe for team development.
- Used in almost every production project.

---

### db push

```bash
npx prisma db push
```

- Pushes schema directly.
- Does not create migration history.
- Mainly useful for prototypes and experiments.
- Not recommended for production.

---

# Common Mistakes

## Editing PostgreSQL Directly

Wrong

```
Open pgAdmin

↓

Create Table

↓

Modify Columns
```

Prisma no longer knows about those changes.

Always modify

```
schema.prisma
```

first.

---

## Deleting Migration Files

Migration files are part of the project.

Deleting them breaks migration history.

---

## Using db push in Production

Without migration history,

different environments can become inconsistent.

Use

```
migrate deploy
```

instead.

---

## Forgetting Prisma Client

Whenever the schema changes,

the Prisma Client must be regenerated.

Fortunately,

`migrate dev` does this automatically.

---

# Senior Engineer Notes

Before merging a database change, verify:

- Was the schema reviewed?
- Does a migration exist?
- Is the generated SQL correct?
- Has the migration been tested?
- Are migration files committed?

Database migrations are part of the source code and should be reviewed like any other code change.

# Summary

Prisma Migrations convert the models defined in `schema.prisma` into actual PostgreSQL tables while maintaining a complete history of database changes.

By storing migrations in Git, every developer and every deployment environment can reproduce the exact same database structure.

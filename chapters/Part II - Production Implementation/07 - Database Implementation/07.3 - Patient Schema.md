# 07.3 - Patient Schema

> The Patient is the first business entity in our clinic application.
> Everything else in the system eventually revolves around the patient.

---

# Objective

In this chapter we will:

- Design the Patient entity.
- Implement it using Prisma.
- Understand every field.
- See how Prisma maps to PostgreSQL.
- Understand why each field exists.
- Discuss alternative designs.

---

# Business Requirement

A patient visits the clinic for treatment.

The clinic must permanently store:

- Patient identity
- Contact information
- Date of Birth
- Gender
- Registration date
- Last update

Notice something.

A patient **does not** contain:

- Diagnosis
- Appointment
- Doctor
- Prescription

Those belong to other entities.

The Patient table should only store information that belongs to the patient.

---

# Prisma Model

Open

```text
prisma/schema.prisma
```

Add

```prisma
model Patient {

  id          String   @id @default(uuid())

  firstName   String

  lastName    String

  email       String?  @unique

  phone       String?

  gender      Gender

  dateOfBirth DateTime

  createdAt   DateTime @default(now())

  updatedAt   DateTime @updatedAt
}
```

We have also introduced our first enum.

```prisma
enum Gender {
  MALE
  FEMALE
  OTHER
}
```

---

# Understanding Every Field

## id

```prisma
id String @id @default(uuid())
```

This is the primary key.

Every patient must have one unique identifier.

We use UUID because:

- IDs can be generated by the application.
- They are difficult to guess.
- They work well in distributed systems.
- They remain unique across environments.

Without a primary key, we cannot uniquely identify a patient.

---

## firstName

```prisma
firstName String
```

Stores the patient's first name.

It is required because every patient record should have an identifiable name.

---

## lastName

```prisma
lastName String
```

Stores the patient's family name.

Keeping first and last names separate makes searching, sorting, and reporting easier.

---

## email

```prisma
email String? @unique
```

Notice two things.

### Optional

```
String?
```

Some patients may not provide an email.

The question mark tells Prisma that the value may be null.

---

### Unique

```
@unique
```

Two patients cannot register with the same email.

Even if the application forgets to validate this,

the database will enforce the rule.

The database is always the final source of truth.

---

## phone

```prisma
phone String?
```

Phone numbers are optional.

Some clinics register patients using only their name.

Keeping this optional makes registration more flexible.

---

## gender

```prisma
gender Gender
```

Instead of allowing free text,

we restrict the value to predefined options.

Benefits:

- Consistent data.
- Easier reporting.
- Prevents spelling mistakes.

Without an enum, users might enter:

```
Male

male

M

MALE

Boy
```

All of these represent the same meaning but create inconsistent data.

---

## dateOfBirth

```prisma
dateOfBirth DateTime
```

Why store the date instead of age?

Because age changes every year.

Date of birth never changes.

The application can always calculate the current age.

Never store data that can be derived from another value.

---

## createdAt

```prisma
createdAt DateTime @default(now())
```

Automatically stores when the patient was created.

Useful for:

- Audit logs
- Reports
- Monthly registrations
- Analytics

---

## updatedAt

```prisma
updatedAt DateTime @updatedAt
```

Prisma updates this field automatically whenever the record changes.

Examples:

- Phone updated.
- Email changed.
- Name corrected.

This helps identify the latest version of the record.

---

# How PostgreSQL Sees This

Although we wrote a Prisma model,

PostgreSQL stores a table.

Conceptually, it looks like this.

```sql
CREATE TABLE patients (

    id UUID PRIMARY KEY,

    first_name TEXT NOT NULL,

    last_name TEXT NOT NULL,

    email TEXT UNIQUE,

    phone TEXT,

    gender TEXT NOT NULL,

    date_of_birth TIMESTAMP NOT NULL,

    created_at TIMESTAMP,

    updated_at TIMESTAMP

);
```

Prisma generates SQL similar to this during migrations.

# Entity Structure

```
Patient

├── id

├── firstName

├── lastName

├── email

├── phone

├── gender

├── dateOfBirth

├── createdAt

└── updatedAt
```

At this stage, the Patient is an independent entity.

It has no relationships yet.

Those will be introduced when we build Appointments.

---

# Why This Design?

The Patient table stores only patient information.

It intentionally does **not** store:

- Doctor
- Appointment
- Diagnosis
- Prescription

Why?

Because a patient can have:

- Many appointments.
- Many diagnoses.
- Many prescriptions.
- Many doctors over time.

Those belong to separate entities.

Keeping responsibilities separate avoids duplicated data.

---

# Alternative Designs

## Option 1

Store the full name.

```text
fullName
```

Simple.

But searching and sorting become more difficult.

---

## Option 2

Store age.

```text
age
```

Not recommended.

Age changes every year.

The application would constantly need updates.

Instead, store the date of birth.

---

## Option 3

Store phone as required.

Useful if the clinic always contacts patients by phone.

Our application keeps it optional because not every clinic follows the same registration process.

---

# Common Mistakes

### Using Email as the Primary Key

Email addresses can change.

Primary keys should never change.

---

### Storing Age

Always store the date of birth.

Age is calculated.

---

### Forgetting Audit Fields

Without `createdAt` and `updatedAt`, it becomes difficult to answer:

- When was this patient registered?
- When was the record last modified?

---

### Mixing Business Data

The Patient table should not contain diagnosis information.

Diagnosis belongs to the Diagnosis entity.

---

# Senior Engineer Notes

Before approving this schema, ask:

- Does every field belong to the patient?
- Can any field be calculated instead of stored?
- Which fields are optional?
- Which fields must be unique?
- Which fields may change?
- Which fields should never change?

These questions help keep the model focused and maintainable.

---

# Follow the Data

A receptionist registers a patient.

↓

A new row is inserted into the Patient table.

↓

Later, the patient books an appointment.

↓

The Appointment entity will reference this patient's `id`.

↓

The doctor creates a diagnosis.

↓

The Diagnosis entity will be linked through the Appointment, not directly inside the Patient table.

This separation keeps the database normalized and avoids data duplication.

---

# Summary

The Patient entity is the foundation of the clinic system.

It stores only patient-specific information and leaves appointments, diagnoses, and prescriptions to their own entities.

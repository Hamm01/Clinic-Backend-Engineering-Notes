# 07.10 - Relationships

> A relationship connects two entities. Without relationships, every table becomes isolated and the application cannot represent how the business actually works.

---

# Objective

In this chapter we will:

- Understand what a relationship is.
- Learn the relationship types used in relational databases.
- Connect all entities in our clinic application.
- Understand foreign keys.
- Learn how Prisma models relationships.
- Visualize the complete database.

---

# What is a Relationship?

A database is more than individual tables.

Imagine we only have these tables.

```
Patient

Doctor

Appointment
```

If none of them are connected,

the database cannot answer questions like:

- Which appointments belong to a patient?
- Which doctor treated this patient?
- Which clinic does this doctor work at?
- Which prescriptions belong to this consultation?

Relationships solve this problem.

---

# What is a Foreign Key?

A foreign key is a column that stores the primary key of another table.

Example

```
Doctor

id = d1
```

Appointment

```
doctorId = d1
```

Instead of copying all doctor information,

Appointment stores only the doctor's ID.

---

# Why Use Foreign Keys?

Without relationships

```
Appointment

Doctor Name

Doctor Email

Doctor Phone
```

Every appointment duplicates doctor information.

If the doctor changes their phone number,

every appointment must be updated.

Instead

```
Doctor

↓

Appointment
```

Appointment stores only

```
doctorId
```

The rest of the information comes from the Doctor table.

This avoids duplicated data.

---

# Relationship Types

Relational databases primarily use three relationship types.

---

# One-to-One (1:1)

One record belongs to exactly one other record.

Example

```
Appointment

↓

Medical Record
```

Every appointment has one medical record.

Every medical record belongs to one appointment.

Prisma

```prisma
model Appointment {

  medicalRecord MedicalRecord?

}

model MedicalRecord {

  appointmentId String @unique

  appointment Appointment
      @relation(fields: [appointmentId], references: [id])

}
```

Notice

```
@unique
```

This guarantees only one medical record exists for an appointment.

---

# One-to-Many (1:N)

One record owns many other records.

Example

```
Clinic

↓

Doctors
```

One clinic

↓

Many doctors

Prisma

```prisma
model Clinic {

  doctors Doctor[]

}

model Doctor {

  clinicId String

  clinic Clinic
      @relation(fields: [clinicId], references: [id])

}
```

The parent contains an array.

The child contains the foreign key.

---

# Another One-to-Many

Patient

↓

Appointments

```
Patient (1)

↓

Appointment (Many)
```

Prisma

```prisma
model Patient {

  appointments Appointment[]

}

model Appointment {

  patientId String

  patient Patient
      @relation(fields: [patientId], references: [id])

}
```

One patient can visit multiple times.

---

# One Doctor

Many Appointments

```
Doctor

↓

Appointments
```

Prisma

```prisma
model Doctor {

  appointments Appointment[]

}

model Appointment {

  doctorId String

  doctor Doctor
      @relation(fields: [doctorId], references: [id])

}
```

A doctor's schedule is simply a collection of appointments.

---

# Medical Record

Many Diagnoses

```
Medical Record

↓

Diagnoses
```

Prisma

```prisma
model MedicalRecord {

  diagnoses Diagnosis[]

}

model Diagnosis {

  medicalRecordId String

  medicalRecord MedicalRecord
      @relation(fields: [medicalRecordId], references: [id])

}
```

A consultation may produce multiple diagnoses.

---

# Diagnosis

Many Prescriptions

```
Diagnosis

↓

Prescriptions
```

Prisma

```prisma
model Diagnosis {

  prescriptions Prescription[]

}

model Prescription {

  diagnosisId String

  diagnosis Diagnosis
      @relation(fields: [diagnosisId], references: [id])

}
```

One diagnosis may require multiple medicines.

# Why We Don't Duplicate Data

Imagine every appointment stores

```
Doctor Name

Doctor Email

Doctor Phone
```

Now suppose the doctor changes their phone number.

Every appointment becomes incorrect.

Instead,

Appointment stores

```
doctorId
```

Whenever we need doctor information,

we fetch it through the relationship.

This keeps the data consistent.

---

# Querying Relationships

Example

Find an appointment together with its patient and doctor.

```ts
const appointment = await prisma.appointment.findUnique({
  where: {
    id: appointmentId
  },
  include: {
    patient: true,
    doctor: true
  }
})
```

Prisma automatically joins the related tables.

---

# Nested Relationships

We can go even deeper.

```ts
const appointment = await prisma.appointment.findUnique({
  where: {
    id: appointmentId
  },
  include: {
    patient: true,
    doctor: {
      include: {
        clinic: true
      }
    },
    medicalRecord: {
      include: {
        diagnoses: {
          include: {
            prescriptions: true
          }
        }
      }
    }
  }
})
```

One query now returns:

- Patient
- Doctor
- Clinic
- Medical Record
- Diagnoses
- Prescriptions

This is one of Prisma's biggest strengths.

# Common Mistakes

## Duplicating Data

Do not store doctor information inside Appointment.

Store only the foreign key.

---

## Missing Foreign Keys

Without foreign keys,

tables become disconnected.

The application cannot navigate between entities.

---

## Wrong Relationship Direction

The child stores the foreign key.

Correct

```
Appointment

patientId
```

Wrong

```
Patient

appointmentId
```

One patient can have many appointments.

---

## Forgetting Referential Integrity

Relationships ensure that

```
Appointment

doctorId
```

must reference an existing doctor.

This prevents invalid data.

---

# Senior Engineer Notes

Before approving a schema verify:

- Does every relationship represent a real business rule?
- Is duplicated data avoided?
- Are foreign keys placed in the child table?
- Are relationship names clear?
- Can Prisma navigate the relationship naturally?
- Does every table have a single responsibility?

Relationships should model the business—not convenience.

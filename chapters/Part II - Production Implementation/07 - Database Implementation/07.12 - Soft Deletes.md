# 07.12 - Soft Deletes

> Soft Delete is a technique where data is marked as deleted instead of being permanently removed from the database.

---

# Objective

In this chapter we will:

- Understand Soft Delete.
- Learn why production systems rarely delete data.
- Implement Soft Delete using Prisma.
- Update queries to ignore deleted records.
- Understand the trade-offs.

---

# Why Not Delete Data?

Imagine this situation.

```
Patient

↓

Appointment

↓

Medical Record

↓

Diagnosis

↓

Prescription
```

Now suppose an administrator clicks

```
Delete Patient
```

If the patient is permanently removed,

what happens to

- Appointment history?
- Medical records?
- Prescriptions?
- Reports?
- Billing?

Everything becomes inconsistent.

Medical history should never disappear because someone clicked "Delete".

---

# Hard Delete

A hard delete permanently removes data.

```sql
DELETE FROM patients

WHERE id = 'patient-123';
```

Result

```
Database

❌ Patient Removed

❌ Appointment Links Broken

❌ Medical History Lost
```

The data cannot be recovered.

---

# Soft Delete

Instead of removing the row,

we mark it as deleted.

Example

```text
Patient

id

name

deletedAt

2026-07-13 14:30:00
```

The row still exists.

The application simply ignores it.

---

# Implementing Soft Delete

Add a new field to the Patient model.

```prisma
model Patient {

  id String @id @default(uuid())

  firstName String

  lastName String

  deletedAt DateTime?

}
```

Notice the `?`.

If

```
deletedAt = NULL
```

the patient is active.

If

```
deletedAt = 2026-07-13
```

the patient has been soft deleted.

---

# Applying the Same Pattern

Most business entities should support soft deletes.

Examples

```prisma
deletedAt DateTime?
```

Patient

Doctor

Clinic

Appointment

MedicalRecord

Diagnosis

Prescription

This provides a consistent deletion strategy across the application.

---

# Soft Delete Instead of Delete

Instead of

```ts
await prisma.patient.delete({
  where: {
    id: patientId
  }
})
```

Use

```ts
await prisma.patient.update({
  where: {
    id: patientId
  },
  data: {
    deletedAt: new Date()
  }
})
```

Nothing is removed.

Only the deletion timestamp is recorded.

---

# Reading Active Patients

If soft deletes are used,

queries should ignore deleted records.

Example

```ts
const patients = await prisma.patient.findMany({
  where: {
    deletedAt: null
  }
})
```

Only active patients are returned.

---

# Reading Deleted Patients

Administrators may need to review deleted data.

Example

```ts
const deletedPatients = await prisma.patient.findMany({
  where: {
    deletedAt: {
      not: null
    }
  }
})
```

---

# Restoring a Patient

Since the row still exists,

restoring is simple.

```ts
await prisma.patient.update({
  where: {
    id: patientId
  },
  data: {
    deletedAt: null
  }
})
```

The patient becomes active again.

# Why Clinics Prefer Soft Deletes

Medical systems must preserve historical records.

Examples include

- Previous diagnoses.
- Medicine history.
- Insurance audits.
- Legal investigations.
- Government compliance.
- Statistical reports.

Removing historical records can create legal and operational issues.

---

# Query Pattern

Most repository methods should automatically exclude deleted rows.

Example

```ts
const doctor = await prisma.doctor.findUnique({
  where: {
    id: doctorId,
    deletedAt: null
  }
})
```

Or

```ts
const doctors = await prisma.doctor.findMany({
  where: {
    deletedAt: null
  }
})
```

This pattern should be applied consistently.

---

# Trade-offs

Benefits

- Data recovery.
- Complete audit history.
- Legal compliance.
- Historical reporting.
- Prevents accidental data loss.

Costs

- Every query must account for `deletedAt`.
- Tables continue to grow.
- Requires discipline in repository implementation.

---

# Common Mistakes

## Mixing Hard and Soft Deletes

Choose one strategy per entity.

For business entities like patients, doctors, and appointments, prefer soft deletes.

---

## Forgetting to Filter Deleted Rows

If queries do not include

```ts
deletedAt: null
```

deleted records may appear in the application.

---

## Physically Deleting Medical Data

Medical history should generally remain available for auditing and legal purposes.

---

## Soft Deleting Lookup Tables Unnecessarily

Reference data such as

- Gender
- Blood Groups
- Appointment Status

is usually managed differently and often does not require soft deletes.

---

# Senior Engineer Notes

Before implementing soft deletes ask:

- Does this entity have historical value?
- Can other records reference it?
- Is recovery required?
- Are there legal or compliance requirements?

If the answer is yes,

soft deletes are usually the safer choice.

# Summary

Soft Deletes allow the application to hide records without removing them from the database.

Instead of deleting rows, the application records when they were deleted and excludes them from normal queries.

This approach preserves historical data, supports auditing, and reduces the risk of accidental data loss in production systems.

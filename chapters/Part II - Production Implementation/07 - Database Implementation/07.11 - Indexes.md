# 07.11 - Indexes

> An index is a data structure that allows the database to find rows much faster without scanning the entire table.

---

# Objective

In this chapter we will:

- Understand what an index is.
- Learn why indexes are necessary.
- Create indexes using Prisma.
- Decide which fields should be indexed.
- Understand the trade-offs of indexing.

---

# Why Do We Need Indexes?

Imagine our Patient table contains

```
10 Patients
```

Searching is fast.

Now imagine

```
10,000 Patients
```

Still acceptable.

Now imagine

```
10,000,000 Patients
```

Without an index, PostgreSQL must inspect every row until it finds the correct one.

```
Patient Table

↓

Row 1

↓

Row 2

↓

Row 3

↓

...

↓

Row 10,000,000
```

This is called a **Full Table Scan**.

As the table grows, queries become slower.

Indexes solve this problem.

---

# What is an Index?

Think of a textbook.

Without an index,

finding a topic requires reading every page.

```
Page 1

↓

Page 2

↓

Page 3

↓

...

↓

Page 900
```

With an index,

you immediately jump to the correct page.

```
Index

↓

"Diabetes"

↓

Page 542
```

A database index works in the same way.

---

# How an Index Works

Without an index

```
SELECT *

FROM Patient

WHERE email = 'john@example.com';
```

The database checks every patient.

With an index

```
Email Index

↓

john@example.com

↓

Patient Row
```

The database reaches the row almost immediately.

---

# Primary Key Index

Every primary key automatically creates an index.

Example

```prisma
id String @id @default(uuid())
```

No additional work is required.

Searching by

```
id
```

is already very fast.

---

# Unique Index

When we write

```prisma
email String @unique
```

Prisma creates

- Unique Constraint
- Index

Therefore

```
Find Patient by Email
```

is efficient.

The same applies to

```prisma
licenseNumber String @unique
```

---

# Creating an Index

Suppose appointments are frequently searched by date.

Prisma

```prisma
model Appointment {

  id String @id @default(uuid())

  appointmentDate DateTime

  @@index([appointmentDate])

}
```

This creates an index on

```
appointmentDate
```

Now calendar queries become much faster.

---

# Composite Index

Sometimes one field is not enough.

Suppose we frequently search

```
Appointments

WHERE

doctorId

AND

appointmentDate
```

Instead of two separate indexes,

we create one composite index.

```prisma
model Appointment {

  ...

  @@index([doctorId, appointmentDate])

}
```

This is especially useful for

```
Doctor's Daily Schedule
```

# Why Foreign Keys Need Indexes

Consider this query.

```
Find all appointments for Patient X.
```

The database searches using

```
patientId
```

If `patientId` is not indexed,

every appointment must be scanned.

With an index,

the matching appointments are located quickly.

Foreign keys are among the most important columns to index.

---

# Query Example

```ts
const appointments = await prisma.appointment.findMany({
  where: {
    doctorId: doctorId
  }
})
```

As the number of appointments grows,

an index on `doctorId` keeps this query efficient.

---

# Trade-offs

Indexes improve reads.

But they also have costs.

Every time data is inserted, updated, or deleted,

the indexes must also be updated.

Benefits

- Faster SELECT queries.
- Faster filtering.
- Faster joins.
- Better sorting.

Costs

- Additional disk space.
- Slightly slower INSERT.
- Slightly slower UPDATE.
- Slightly slower DELETE.

Adding indexes everywhere is not a good strategy.

Only index fields that are frequently searched or joined.

---

# Prisma Example

```prisma
model Doctor {

  id String @id @default(uuid())

  email String @unique

  specialization String

  clinicId String

  @@index([clinicId])

  @@index([specialization])

}
```

This allows efficient queries such as

```
Find all Cardiologists

Find all Doctors in Clinic A
```

---

# Common Mistakes

## Indexing Every Column

More indexes do not always improve performance.

They increase storage usage and slow down writes.

---

## Forgetting Foreign Keys

Foreign keys are frequently used in joins.

Index them.

---

## Ignoring Query Patterns

Indexes should support real application queries,

not hypothetical ones.

---

## Indexing Large Text Fields

Large notes or summaries are usually poor candidates for standard indexes.

---

# Senior Engineer Notes

Before creating an index ask:

- Is this field searched frequently?
- Is it used in joins?
- Is it used in sorting?
- Is it used in filtering?
- How often is the table written to?
- Does the performance gain justify the maintenance cost?

Indexes should be driven by application behavior.

# Summary

Indexes allow PostgreSQL to locate data efficiently as the database grows.

Primary keys, unique fields, foreign keys, and commonly searched columns are good candidates for indexing.

Well-designed indexes improve query performance without changing business logic.

# 07.6 - Appointment Schema

> An appointment represents a scheduled consultation between a patient and a doctor.

---

# Objective

In this chapter we will:

- Build the Appointment entity.
- Connect Patient and Doctor.
- Understand our first foreign keys.
- Store appointment details.
- Prepare the foundation for diagnoses and prescriptions.

---

# Business Requirement

Patients visit doctors.

Doctors consult patients.

Every consultation should be recorded.

The clinic wants to know:

- Which patient booked?
- Which doctor will see the patient?
- When is the appointment?
- What is its current status?
- Which clinic is handling it?

Instead of storing this information inside Patient or Doctor,

we create a separate Appointment entity.

The Appointment becomes the connection between both business objects.

---

# Prisma Model

Open

```text
prisma/schema.prisma
```

Add

```prisma
model Appointment {

  id              String   @id @default(uuid())

  patientId       String

  doctorId        String

  appointmentDate DateTime

  status          AppointmentStatus @default(SCHEDULED)

  reason          String?

  notes           String?

  createdAt       DateTime @default(now())

  updatedAt       DateTime @updatedAt

  patient         Patient @relation(fields: [patientId], references: [id])

  doctor          Doctor  @relation(fields: [doctorId], references: [id])
}
```

Create the enum.

```prisma
enum AppointmentStatus {

  SCHEDULED

  COMPLETED

  CANCELLED

  NO_SHOW
}
```

# Understanding Every Field

## id

```prisma
id String @id @default(uuid())
```

Every appointment receives its own identifier.

Notice that appointments are business events.

Multiple appointments may exist for the same patient.

Each one must have its own ID.

---

## patientId

```prisma
patientId String
```

Stores which patient booked this appointment.

Instead of storing the entire patient,

we only store the patient's identifier.

This is called a Foreign Key.

---

## doctorId

```prisma
doctorId String
```

Stores which doctor will conduct the consultation.

Again,

only the identifier is stored.

The doctor's complete information already exists in the Doctor table.

---

## appointmentDate

```prisma
appointmentDate DateTime
```

Stores the scheduled consultation date.

Example

```
2026-08-20 10:30 AM
```

Later this field is used for

- Daily schedules
- Calendar view
- Notifications
- Reports

---

## status

```prisma
status AppointmentStatus
```

Appointments move through different stages.

```
SCHEDULED

↓

COMPLETED
```

or

```
SCHEDULED

↓

CANCELLED
```

Using an enum prevents invalid values.

---

## reason

```prisma
reason String?
```

Why the patient booked.

Examples

```
Chest Pain

Fever

Headache

Routine Checkup
```

Optional because some clinics register appointments without symptoms.

---

## notes

```prisma
notes String?
```

Administrative notes.

Example

```
Patient requested evening appointment.
```

This is **not** the doctor's consultation notes.

Those belong in Medical Records.

---

## createdAt

Stores when the appointment was created.

---

## updatedAt

Automatically updates whenever the appointment changes.

---

## patient

```prisma
patient Patient
```

Creates the relationship to Patient.

Now Prisma understands

```
Appointment

↓

Patient
```

---

## doctor

```prisma
doctor Doctor
```

Creates the relationship to Doctor.

# PostgreSQL Representation

Conceptually

```sql
CREATE TABLE appointments (

    id UUID PRIMARY KEY,

    patient_id UUID NOT NULL,

    doctor_id UUID NOT NULL,

    appointment_date TIMESTAMP NOT NULL,

    status TEXT NOT NULL,

    reason TEXT,

    notes TEXT,

    created_at TIMESTAMP,

    updated_at TIMESTAMP,

    FOREIGN KEY (patient_id)

        REFERENCES patients(id),

    FOREIGN KEY (doctor_id)

        REFERENCES doctors(id)

);
```

---

# Why This Design?

Imagine storing appointments inside Patient.

```
Patient

↓

appointments[]
```

Soon the Patient record becomes enormous.

Likewise,

storing appointments inside Doctor creates the same problem.

Appointments deserve their own entity because:

- They have their own lifecycle.
- They have their own status.
- They change independently.
- They reference multiple business objects.

# Common Mistakes

### Saving Diagnosis Inside Appointment

Appointments schedule consultations.

Diagnoses describe medical findings.

Keep them separate.

---

### Saving Patient Details Again

The Patient table already stores

- name
- phone
- email

Do not duplicate them.

---

### Updating Patient Information Inside Appointment

Appointments should reference Patients,

not copy patient data.

---

### Using Free Text for Status

Avoid

```
Done

Finished

Completed

complete

Done ✅
```

Use an enum.

---

# Senior Engineer Notes

Before approving the Appointment schema ask:

- Can multiple appointments exist for one patient?

Yes.

- Can one doctor have many appointments?

Yes.

- Should appointments survive if patient details change?

Yes.

- Does Appointment own diagnosis information?

No.

Appointment represents the consultation itself.

# Summary

The Appointment entity is the first relationship-driven model in the clinic application.

It connects patients with doctors and represents the real-world consultation event.

The next chapter introduces **Medical Record**, where the doctor begins recording the outcome of that consultation.

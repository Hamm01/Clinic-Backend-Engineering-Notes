# 07.2 - Designing the Database

> Before creating tables, we need to understand the business.  
> A good database is a reflection of the business domain, not just a collection of tables.

---

# Objective

In this chapter we will learn:

- How to identify business entities.
- How to separate entities from relationships.
- How to decide what belongs in each table.
- How to think before writing a Prisma schema.
- Why good schema design reduces future complexity.

At the end of this chapter, we will know **what** needs to be stored, but we will intentionally avoid writing Prisma models until the next chapters.

---

# Start With the Business

Imagine a clinic.

Forget PostgreSQL.

Forget Prisma.

Forget TypeScript.

Ask one question.

> **What information must the business remember?**

Anything the business needs to remember usually becomes a database entity.

For our clinic, the business remembers:

- Patients
- Doctors
- Clinics
- Appointments
- Medical Records
- Diagnoses
- Prescriptions

These are our first business entities.

---

# Thinking Like a Backend Engineer

A common beginner question is:

> "Which tables should I create?"

A better question is:

> "What objects exist in the business?"

Backend engineers model **real-world objects**, not database tables.

Example:

```
Real World

Patient

↓

Database

Patient Table
```

Another example:

```
Real World

Doctor

↓

Database

Doctor Table
```

The database should represent the business, not the framework being used.

# Identifying the Main Entities

From the workflow above, we can identify the primary entities.

```
Clinic

Doctor

Patient

Appointment

Medical Record

Diagnosis

Prescription
```

Notice that these are all **nouns**.

A useful rule is:

> **Most nouns become entities.**

Examples:

| Business Object | Database Entity |
| --------------- | --------------- |
| Patient         | Patient         |
| Doctor          | Doctor          |
| Clinic          | Clinic          |
| Appointment     | Appointment     |
| Diagnosis       | Diagnosis       |

---

# Not Everything Becomes a Table

Some information belongs inside another entity.

Example:

Patient

```
First Name

Last Name

Email

Phone Number
```

These are attributes of a patient.

They are **not** separate tables.

Wrong design:

```
Patient

↓

FirstName Table

↓

Email Table

↓

Phone Table
```

Correct design:

```
Patient

├── firstName

├── lastName

├── email

└── phone
```

A separate table should only exist when the data has its own identity and lifecycle.

---

# Which Data Changes Independently?

This is one of the most important questions during schema design.

Example:

A patient's email.

Can it change?

Yes.

Does it still belong to the Patient?

Yes.

Now consider an appointment.

Can appointments change independently of the patient?

Yes.

Can a patient have many appointments?

Yes.

That means Appointment deserves its own entity.

---

# Identifying Relationships

Now let's connect the business objects.

```
Clinic

│

├── Doctors

│

└── Patients
```

Doctors treat patients.

Patients book appointments.

Appointments create medical records.

Medical records contain diagnoses and prescriptions.

Business flow:

```
Clinic

↓

Doctor

↓

Appointment

↓

Patient

↓

Medical Record

↓

Diagnosis

↓

Prescription
```

Notice that every entity has a clear responsibility.

---

# Why We Don't Store Everything in One Table

Imagine a single table.

```
Patient

Doctor

Clinic

Diagnosis

Medicine

Appointment Date

Prescription

Notes

Phone

Email
```

Problems:

- Duplicate data.
- Difficult updates.
- Poor scalability.
- Inconsistent records.
- Large table size.

Instead, we split the information based on responsibility.

Each table owns one type of data.

---

# Data Ownership

Every piece of information should have one owner.

Example:

Doctor's specialization.

Who owns it?

```
Doctor
```

Patient's phone number.

Who owns it?

```
Patient
```

Appointment date.

Who owns it?

```
Appointment
```

Diagnosis.

Who owns it?

```
Diagnosis
```

A good schema avoids storing the same information in multiple places.

# Designing for Future Requirements

Suppose next year the clinic expands.

One doctor now works at two clinics.

Would our current design support that?

These are the kinds of questions engineers ask before implementation.

Schema design should consider future business growth without over-engineering the current solution.

# Common Mistakes

### Creating tables before understanding the business

The database should reflect the business domain.

Never let the ORM decide the design.

---

### Treating every field as a separate table

Only create separate tables for data that has its own lifecycle.

---

### Mixing responsibilities

A Patient should not store diagnosis information.

A Diagnosis should not store doctor profile information.

Each entity owns its own data.

---

### Ignoring future growth

Ask:

- Can this entity exist independently?
- Can there be many of them?
- Will the business rules change later?

These questions influence the schema.

---

# Senior Engineer Notes

Before writing a single Prisma model, ask yourself:

- What business object am I modelling?
- Who owns this data?
- Can this object exist independently?
- Does this object change independently?
- Should this be an entity or just a field?
- Will this design still make sense if the application grows?

Good database design is not about writing tables.

It is about accurately modelling the business.

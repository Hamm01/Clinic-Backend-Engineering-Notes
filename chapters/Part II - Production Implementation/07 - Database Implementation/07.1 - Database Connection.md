---

# Business Requirement

Our clinic application needs a place to store data permanently.

The backend itself is **stateless**. If the server restarts, everything in memory is lost.

The database becomes the application's long-term memory.

Every feature we build later—patients, doctors, appointments, diagnoses—will ultimately be stored here.

---

# Choosing PostgreSQL

We will use PostgreSQL because it provides:

- ACID transactions
- Strong relational integrity
- Foreign key support
- Powerful indexing
- JSON support
- Excellent tooling
- Mature ecosystem

These characteristics make it suitable for healthcare applications where data consistency is more important than write speed.

# Why Create a Separate Database?

Instead of storing every application inside one database,

we isolate projects.

Example

```text
PostgreSQL Server

├── clinic_db

├── ecommerce_db

├── blog_db
```

Benefits:

- Easier backups.
- Easier migrations.
- Better security.
- Independent deployments.
- Cleaner development environment.

# Why Two Packages?

## prisma

Development tool.

Responsibilities:

- Generate migrations.
- Generate Prisma Client.
- Update schema.
- Inspect the database.

Used only during development.

## @prisma/client

Runtime library.

Responsibilities:

- Execute queries.
- Insert data.
- Update records.

Every repository will use this package.

# Understanding the Generated Files

## prisma/schema.prisma

This becomes the blueprint of our database.

Every model we create later will be defined here.

Think of it as the source of truth for the database structure.

---

## .env

Prisma does not hardcode database credentials.

Instead,

it reads them from environment variables.

Example

```env
DATABASE_URL="postgresql://postgres:password@localhost:5432/clinic_db"
```

---

# Why Use Environment Variables?

Never write credentials directly inside source code.

Wrong

```ts
const database = 'postgres://postgres:password@localhost'
```

Correct

```env
DATABASE_URL=...
```

Benefits:

- Different credentials for development, staging, and production.
- No passwords committed to Git.
- Easier deployment.

---

# Configuring Prisma

Open

```text
prisma/schema.prisma
```

Update

```prisma
generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}
```

---

# Understanding Each Section

## Generator

```prisma
generator client
```

Prisma automatically generates a TypeScript client.

Instead of writing SQL manually,

we write

```ts
await prisma.patient.findMany()
```

Prisma converts it into SQL.

---

## Datasource

```prisma
datasource db
```

Defines

- database type
- connection string

Changing from PostgreSQL to another supported database generally starts here, although schema and query differences may still require changes.

# Common Mistakes

### Hardcoding database credentials

Never commit database passwords.

Always use environment variables.

---

### Editing generated Prisma files

Only modify:

```text
schema.prisma
```

Never edit generated client files inside `node_modules`.

---

### Mixing development and production databases

Always use separate databases for:

- Development
- Testing
- Staging
- Production

---

# Senior Engineer Notes

Before creating any tables, ask these questions:

- What business problem are we solving?
- Which objects need to be stored permanently?
- Which relationships exist between them?
- Which fields are required today?
- Which fields might be required in six months?

Schema design should follow business requirements—not the other way around.

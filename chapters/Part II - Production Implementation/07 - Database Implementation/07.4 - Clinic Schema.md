# 07.4 - Clinic Schema

> A clinic is the organization where doctors work and patients receive treatment.

---

# Objective

In this chapter we will:

- Design the Clinic entity.
- Understand why a clinic deserves its own table.
- Implement the Prisma model.
- Explain every field.
- Prepare the foundation for Doctor relationships.

---

# Business Requirement

A healthcare company may operate multiple clinics.

For example

```
Apollo Healthcare

├── Delhi Clinic

├── Chandigarh Clinic

└── Mumbai Clinic
```

Every clinic has its own:

- Name
- Address
- Contact Number
- Email
- Operating Hours
- Status

Instead of duplicating this information for every doctor,

we store it once.

---

# Prisma Model

Open

```
prisma/schema.prisma
```

Add

```prisma
model Clinic {

  id          String   @id @default(uuid())

  name        String

  email       String?  @unique

  phone       String?

  address     String

  city        String

  state       String

  country     String

  postalCode  String?

  isActive    Boolean  @default(true)

  createdAt   DateTime @default(now())

  updatedAt   DateTime @updatedAt
}
```

---

# Understanding Every Field

## id

```prisma
id String @id @default(uuid())
```

Primary key.

Every clinic needs a unique identifier.

Doctors and appointments will later reference this ID.

---

## name

```prisma
name String
```

The public name of the clinic.

Example

```
Apollo Clinic Chandigarh
```

---

## email

```prisma
email String? @unique
```

Optional.

Not every clinic has a dedicated email.

If present,

it must be unique.

---

## phone

```prisma
phone String?
```

Reception phone number.

Patients may call this number for appointments.

---

## address

```prisma
address String
```

Physical location of the clinic.

Useful for:

- Maps
- Billing
- Patient visits

---

## city

Stores the city.

Example

```
Chandigarh
```

---

## state

Stores the state or province.

Example

```
Punjab
```

---

## country

Stores the country.

Example

```
India
```

Keeping location separated makes searching and reporting easier than storing one large address string.

---

## postalCode

Optional.

Some clinics may not require it initially.

---

## isActive

```prisma
isActive Boolean @default(true)
```

Instead of deleting clinics,

we simply mark them inactive.

Benefits

- Historical appointments remain valid.
- Doctors remain linked.
- Reports remain accurate.

---

## createdAt

Stores when the clinic was created.

---

## updatedAt

Automatically updated whenever the record changes.

# Why This Design?

Notice that we are **not** storing doctors inside the Clinic table.

Wrong

```
Clinic

↓

doctorNames
```

Wrong

```
Clinic

↓

doctorIds[]
```

Instead,

the Doctor entity will reference the Clinic.

This keeps the relationship flexible and avoids duplicating clinic data.

---

# Alternative Designs

## Single Address Field

```
address
```

Simple.

Good for small systems.

---

## Structured Address

```
address

city

state

country

postalCode
```

Better for filtering.

Useful when searching clinics by location.

---

## Separate Address Table

```
Clinic

↓

Address
```

Useful only when addresses become reusable across multiple entities.

For our clinic application,

this would be unnecessary complexity.

---

# Common Mistakes

### Embedding Doctor Information

A clinic should not store doctor details.

Doctors belong in their own table.

---

### Permanently Deleting Clinics

Historical appointments would lose their clinic reference.

Prefer marking clinics inactive.

---

### Using Clinic Name as Primary Key

Names change.

IDs should never change.

---

# Senior Engineer Notes

Before approving the schema ask:

- Can multiple clinics have similar names?
- Can a clinic move to another address?
- Can clinics become inactive?
- Will doctors always belong to one clinic?

These questions influence future relationships.

---

# Follow the Data

Administrator creates a clinic.

↓

Clinic row inserted.

↓

Doctors are assigned to the clinic.

↓

Patients book appointments.

↓

Appointments reference the clinic through the assigned doctor.

The clinic becomes the organizational root of the application.

# 07.14 - Seed Data

> Seed Data populates the database with realistic sample records so the application can be developed, tested, and demonstrated without manually entering data.

# Why Not Insert Records Manually?

Imagine creating

```
50 Patients

200 Appointments

300 Prescriptions
```

through the UI.

It would take hours.

Instead,

one command creates everything.

```
npx prisma db seed
```

---

# Seed File

Create

```
prisma/

└── seed.ts
```

---

# Configure package.json

Add

```json
{
  "prisma": {
    "seed": "tsx prisma/seed.ts"
  }
}
```

Now Prisma knows where the seed file is located.

---

# Seed Order

The order matters.

```
Clinic

↓

Doctor

↓

Patient

↓

Appointment

↓

Medical Record

↓

Diagnosis

↓

Prescription
```

Why?

Because every child depends on its parent.

Example

Appointment cannot exist until

- Doctor exists
- Patient exists

Diagnosis cannot exist until

Medical Record exists.

# Creating Prisma Client

```ts
import { PrismaClient } from '@prisma/client'

const prisma = new PrismaClient()
```

This allows the seed script to communicate with PostgreSQL.

---

# Creating Clinics

```ts
const clinic = await prisma.clinic.create({
  data: {
    name: 'City Care Clinic',
    email: 'contact@citycare.com',
    phone: '9999999999',
    address: 'Downtown'
  }
})
```

Clinic is created first because doctors belong to clinics.

---

# Creating Doctors

```ts
const doctor = await prisma.doctor.create({
  data: {
    firstName: 'Amit',

    lastName: 'Sharma',

    email: 'amit@clinic.com',

    specialization: 'Cardiology',

    licenseNumber: 'DOC-1001',

    clinicId: clinic.id
  }
})
```

Notice

```
clinicId
```

links the doctor to the clinic.

---

# Creating Patients

```ts
const patient = await prisma.patient.create({
  data: {
    firstName: 'Rahul',

    lastName: 'Verma',

    email: 'rahul@gmail.com',

    phone: '9876543210',

    dateOfBirth: new Date('1995-05-10')
  }
})
```

Patients are independent.

No relationship is required yet.

---

# Creating Appointments

```ts
const appointment = await prisma.appointment.create({
  data: {
    patientId: patient.id,

    doctorId: doctor.id,

    appointmentDate: new Date(),

    status: 'SCHEDULED'
  }
})
```

Appointment connects

Patient

↓

Doctor

---

# Creating Medical Record

```ts
const medicalRecord = await prisma.medicalRecord.create({
  data: {
    appointmentId: appointment.id,

    consultationSummary: 'Patient complained of chest pain.'
  }
})
```

Notice

Medical Record belongs to Appointment.

---

# Creating Diagnosis

```ts
const diagnosis = await prisma.diagnosis.create({
  data: {
    medicalRecordId: medicalRecord.id,

    diagnosisName: 'Hypertension',

    severity: 'MEDIUM'
  }
})
```

---

# Creating Prescription

```ts
await prisma.prescription.create({
  data: {
    diagnosisId: diagnosis.id,

    medicineName: 'Amlodipine',

    dosage: '5 mg',

    frequency: 'Once Daily',

    duration: '30 Days'

})
```

The complete business flow is now seeded.

# Running Seed Again

Suppose you execute

```bash
npx prisma db seed
```

again.

Without precautions,

duplicate records will be inserted.

Example

```
City Care Clinic

City Care Clinic

City Care Clinic
```

To avoid this,

development teams often

- Reset the database before seeding.
- Delete existing records.
- Use `upsert` for reference data.

---

# Reset and Seed

Development workflow

```bash
npx prisma migrate reset
```

Prisma

```
Drops Database

↓

Creates Tables

↓

Runs Migrations

↓

Runs Seed Script
```

This guarantees a clean database.

# Production

Seeding is usually used only for

- Development
- Testing
- Local environments

Production databases are normally populated through the application itself.

Example

```
Administrator creates Clinic

↓

Administrator registers Doctor

↓

Receptionist registers Patient
```

Real users create production data.

---

# Common Mistakes

## Wrong Seed Order

Creating appointments before patients or doctors will fail because foreign keys do not exist.

Always seed parent tables first.

---

## Hardcoding IDs

Wrong

```ts
doctorId: '123'
```

Use

```ts
doctor.id
```

from the previously created record.

---

## Forgetting Relationships

Every child record should reference an existing parent.

---

## Creating Unrealistic Data

Good seed data should resemble production.

Create

- Multiple doctors
- Multiple patients
- Different appointment dates
- Different diagnoses
- Different prescription combinations

This makes testing meaningful.

---

# Senior Engineer Notes

A good seed script should:

- Be repeatable.
- Follow relationship order.
- Avoid broken foreign keys.
- Create realistic data.
- Support local development.
- Help frontend and backend developers work independently.

Many teams invest time in high-quality seed data because it speeds up development and testing.

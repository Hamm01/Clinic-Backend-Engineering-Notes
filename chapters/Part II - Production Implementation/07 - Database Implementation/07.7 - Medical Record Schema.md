# 07.7 - Medical Record Schema

> A Medical Record stores the doctor's consultation for a specific appointment.

---

# Objective

In this chapter we will:

- Create the Medical Record entity.
- Associate every medical record with an appointment.
- Store consultation information.
- Prepare the foundation for diagnoses and prescriptions.

---

# Business Requirement

When a patient visits a doctor, the consultation must be recorded.

The doctor may write:

- Patient observations
- Symptoms
- Clinical findings
- Consultation summary
- Follow-up instructions

These notes should belong to the consultation itself.

Therefore,

every appointment has one medical record.

---

# Prisma Model

Open

```text
prisma/schema.prisma
```

Add

```prisma
model MedicalRecord {

  id                   String   @id @default(uuid())

  appointmentId        String   @unique

  chiefComplaint       String?

  examinationNotes     String?

  consultationSummary  String?

  followUpInstructions String?

  createdAt            DateTime @default(now())

  updatedAt            DateTime @updatedAt

  appointment Appointment @relation(
      fields: [appointmentId],
      references: [id]
  )
}
```

# Updating Appointment

Inside Appointment add

```prisma
medicalRecord MedicalRecord?
```

The question mark means

an appointment may exist before the consultation happens.

Medical records are created only after the doctor completes the consultation.

# Understanding Every Field

## id

```prisma
id String @id @default(uuid())
```

Unique identifier for the consultation record.

---

## appointmentId

```prisma
appointmentId String @unique
```

Every medical record belongs to exactly one appointment.

The `@unique` constraint guarantees that one appointment cannot have multiple medical records.

---

## chiefComplaint

```prisma
chiefComplaint String?
```

The patient's primary reason for visiting.

Examples

```
Chest Pain

Headache

Fever

Difficulty Breathing
```

---

## examinationNotes

```prisma
examinationNotes String?
```

Doctor's observations during examination.

Examples

```
Blood pressure elevated.

Pulse regular.

Temperature normal.
```

---

## consultationSummary

```prisma
consultationSummary String?
```

A concise summary of the consultation.

Example

```
Patient diagnosed with seasonal viral infection.

No hospitalization required.
```

---

## followUpInstructions

```prisma
followUpInstructions String?
```

Instructions given after consultation.

Examples

```
Return after 7 days.

Repeat blood test.

Continue medication.
```

---

## createdAt

Stores when the consultation record was created.

---

## updatedAt

Automatically updated whenever the consultation notes change.

---

## appointment

```prisma
appointment Appointment
```

Creates the relationship with the appointment.

This allows Prisma to retrieve the complete consultation together with appointment details.

# PostgreSQL Representation

Conceptually

```sql
CREATE TABLE medical_records (

    id UUID PRIMARY KEY,

    appointment_id UUID UNIQUE NOT NULL,

    chief_complaint TEXT,

    examination_notes TEXT,

    consultation_summary TEXT,

    follow_up_instructions TEXT,

    created_at TIMESTAMP,

    updated_at TIMESTAMP,

    FOREIGN KEY (appointment_id)

        REFERENCES appointments(id)

);
```

# Why This Design?

The Medical Record stores information that belongs to one consultation.

It does not store:

- Patient profile
- Doctor profile
- Prescriptions
- Diagnoses

Those belong to their own entities.

The Medical Record acts as the doctor's consultation document.

# Common Mistakes

### Saving Consultation Notes Inside Appointment

Appointments schedule visits.

Medical Records document what happened during the visit.

Keep these responsibilities separate.

---

### Creating Multiple Medical Records

One consultation should have one medical record.

Use the `@unique` constraint on `appointmentId`.

---

### Mixing Diagnoses With Notes

Consultation notes describe observations.

Diagnoses describe medical conclusions.

Store them separately.

---

# Senior Engineer Notes

Before approving this schema, verify:

- Does every medical record belong to one appointment?
- Can an appointment exist before a medical record?

Yes.

- Can a medical record exist without an appointment?

No.

The appointment is the source of the consultation.

# 11.11 - Service Layer Anti-Patterns

> Throughout this module, we have built production-ready Services for our Clinic Backend. However, writing a Service that works is only the beginning. As projects grow, poor architectural decisions gradually make Services difficult to understand, difficult to test, and difficult to maintain. In this chapter, we will review the most common Service Layer anti-patterns found in production codebases and learn how to identify and fix them.

# What Is an Anti-Pattern?

An anti-pattern is a solution that appears to work initially but creates long-term maintenance problems.

Example

```
Works Today

↓

Difficult Tomorrow

↓

Expensive Next Year
```

Many production problems are caused by anti-patterns rather than bugs.

---

# Anti-Pattern 1 — God Service

## Problem

One Service owns too many responsibilities.

Example

```ts
export class AppointmentService {
  async createAppointment() {}

  async createInvoice() {}

  async sendEmail() {}

  async sendSMS() {}

  async uploadPrescription() {}

  async verifyInsurance() {}

  async generateReport() {}

  async updateInventory() {}
}
```

---

## Why Is This Bad?

The Service is responsible for unrelated business domains.

```
Scheduling

↓

Billing

↓

Notifications

↓

Inventory

↓

Reports
```

Every new feature increases complexity.

---

## Better Design

Split responsibilities.

```text
AppointmentService

↓

BillingService

↓

NotificationService

↓

InsuranceService

↓

ReportService
```

Each Service owns one business capability.

---

# Anti-Pattern 2 — Fat Public Methods

## Problem

```ts
async create() {

    // 500 lines

}
```

Nobody can understand the workflow.

Reviewing the method becomes difficult.

Testing becomes difficult.

Refactoring becomes risky.

---

## Better Design

```ts
async create(dto) {

    await this.validate(dto);

    await this.checkAvailability(dto);

    const appointment =
        await this.persist(dto);

    await this.notify(appointment);

    return appointment;

}
```

The public method should explain the workflow,

not every implementation detail.

---

# Anti-Pattern 3 — Business Logic Inside Controllers

## Bad Example

```ts
if (!doctor.isActive) {

    return res.status(400).json(...);

}
```

---

## Why Is This Wrong?

Controllers manage HTTP.

They should not decide business rules.

Move the logic into the Service.

---

# Anti-Pattern 4 — Repository Becoming a Service

Sometimes developers write this.

```ts
appointmentRepository.createAppointment()

appointmentRepository.cancelAppointment()

appointmentRepository.completeAppointment()
```

The Repository now contains business rules.

Repositories should only communicate with the database.

Business decisions belong to Services.

---

# Anti-Pattern 5 — Duplicate Business Logic

Imagine this appears three times.

```ts
if (!doctor.isActive) {
  throw new BadRequestException()
}
```

Now every future change requires updating multiple places.

---

## Better Design

```ts
private async validateDoctor(
    doctorId: string
)
```

Reuse business rules.

Avoid duplication.

---

# Anti-Pattern 6 — Hidden Database Queries

Suppose this helper exists.

```ts
private async validatePatient() {

    await prisma.patient.findUnique(...);

}
```

Direct Prisma usage bypasses the Repository layer.

---

## Better Design

```ts
await this.patientRepository.findById(id)
```

Services coordinate.

Repositories access the database.

---

# Anti-Pattern 7 — Long Transactions

Imagine

```ts
await prisma.$transaction(async () => {

    await sendEmail();

    await uploadFile();

    await axios.post(...);

});
```

The transaction remains open while waiting for external systems.

This wastes database connections.

---

## Better Design

```
Transaction

↓

Commit

↓

Email

↓

Background Job

↓

External APIs
```

Keep transactions short.

---

# Anti-Pattern 8 — Circular Service Dependencies

Example

```
PatientService

↓

AppointmentService

↓

PatientService
```

Now neither Service can exist independently.

Circular dependencies make testing and maintenance difficult.

---

## Better Design

Extract shared business logic into another Service.

Example

```
SchedulingService
```

Both Services depend on it,

instead of depending on each other.

---

# Anti-Pattern 9 — Returning Different Shapes

Example

```ts
return patient
```

Later

```ts
return {
  patient,

  invoice,

  doctor
}
```

Later

```ts
return true
```

The API becomes inconsistent.

Define clear return types.

---

# Anti-Pattern 10 — Business Rules Hidden in Comments

Bad

```ts
// Don't allow appointments on Sundays
```

Good

```ts
this.validateClinicWorkingHours()
```

Business rules should be expressed through code,

not hidden inside comments.

# Service Review Checklist

Before approving a pull request, ask:

- Does the Service have one clear responsibility?
- Is the public workflow easy to read?
- Is business logic duplicated?
- Are Repositories responsible only for persistence?
- Are Controllers thin?
- Are transactions used correctly?
- Are helper methods reusable?
- Is the Service easy to unit test?
- Would a new developer understand this file quickly?

If most answers are "Yes",

the Service is probably well designed.

---

# Production Notes

Senior engineers rarely judge a Service by whether it works.

Instead, they ask:

- Can it be maintained?
- Can it be tested?
- Can new business rules be added safely?
- Will another engineer understand it six months from now?

Maintainability is often more valuable than clever code.

---

# Common Mistakes

- Creating God Services.
- Mixing HTTP with business logic.
- Duplicating validation.
- Performing database queries outside Repositories.
- Writing massive public methods.
- Keeping transactions open too long.
- Introducing circular dependencies.
- Returning inconsistent response types.

---

# Summary

A good Service layer is not measured by the number of features it contains, but by how clearly it expresses business workflows.

By avoiding common anti-patterns such as God Services, duplicated business logic, fat methods, misplaced responsibilities, and hidden database access, backend engineers can build applications that remain maintainable even as business requirements continue to grow.

Learning to recognize these code smells is an important step from writing functional code to designing production-quality software.

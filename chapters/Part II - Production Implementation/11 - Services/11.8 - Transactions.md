# The Problem We Created

In the previous chapter, our consultation workflow looked like this.

```
Create Diagnosis

↓

Create Prescription

↓

Create Lab Tests

↓

Update Medical Record

↓

Complete Appointment
```

Each step writes to the database.

Now imagine something goes wrong.

---

# Example Failure

```
Diagnosis Created

✓

↓

Prescription Created

✓

↓

Lab Test Created

✗ Database Error

↓

Medical Record

Not Executed

↓

Appointment

Not Updated
```

Now ask yourself.

What does the database contain?

```
Diagnosis

✓ Exists

Prescription

✓ Exists

Medical Record

✗ Missing

Appointment

✗ Still In Progress
```

The consultation is now partially completed.

This is called **inconsistent data**.

---

# Why Is This Dangerous?

Imagine a doctor opens the patient's history.

The diagnosis exists.

The prescription exists.

The appointment still appears active.

The medical record is incomplete.

Different parts of the application now disagree about reality.

This leads to:

- Incorrect reports.
- Duplicate consultations.
- Incorrect billing.
- Missing medical history.
- Broken business workflows.

---

# What Is a Transaction?

A transaction groups multiple database operations into **one logical unit of work**.

Instead of executing operations independently,

the database treats them as one operation.

```
Operation 1

↓

Operation 2

↓

Operation 3

↓

Operation 4

↓

Commit
```

If every operation succeeds,

the transaction is committed.

If any operation fails,

everything is rolled back.

---

# Mental Model

Imagine transferring money between bank accounts.

```
Withdraw ₹500

↓

Deposit ₹500
```

Would it be acceptable if

```
Withdraw

✓

Deposit

✗
```

Of course not.

Money disappears.

The same principle applies to databases.

---

# Without Transactions

```
Diagnosis

✓

Prescription

✓

Lab Tests

✗

Medical Record

✗

Appointment

✗
```

Database becomes inconsistent.

---

# With Transactions

```
Diagnosis

↓

Prescription

↓

Lab Tests

↓

Medical Record

↓

Appointment
```

If one fails

```
ROLLBACK

↓

Everything Removed
```

Database remains consistent.

---

# ACID Properties

Every modern relational database follows the ACID principles.

| Property    | Meaning                                        |
| ----------- | ---------------------------------------------- |
| Atomicity   | Everything succeeds or everything fails.       |
| Consistency | Database always remains valid.                 |
| Isolation   | Transactions do not interfere with each other. |
| Durability  | Once committed, data is permanently stored.    |

For backend engineers,

the most important concept is **Atomicity**.

# Introducing Prisma Transactions

Prisma provides the `$transaction()` method.

```ts
await prisma.$transaction(async tx => {
  // Database operations
})
```

Everything inside this callback becomes one database transaction.

---

# Rewriting Our Workflow

Instead of

```ts
await diagnosisRepository.create()

await prescriptionRepository.createMany()

await labTestRepository.createMany()

await medicalRecordRepository.addDiagnosis()

await appointmentRepository.updateStatus()
```

We wrap everything inside one transaction.

```ts
await prisma.$transaction(async tx => {
  // All database writes
})
```

---

# Example Implementation

```ts
await prisma.$transaction(async tx => {
  const diagnosis = await tx.diagnosis.create({
    data: diagnosisData
  })

  await tx.prescription.createMany({
    data: prescriptions
  })

  await tx.labTest.createMany({
    data: labTests
  })

  await tx.medicalRecord.update({
    where: {
      patientId
    },

    data: {
      latestDiagnosisId: diagnosis.id
    }
  })

  await tx.appointment.update({
    where: {
      id: appointmentId
    },

    data: {
      status: 'COMPLETED'
    }
  })
})
```

# Failure Scenario

Suppose

```
Diagnosis

✓

↓

Prescription

✓

↓

Lab Tests

✗
```

Prisma automatically performs

```
Rollback

↓

Diagnosis Removed

↓

Prescription Removed

↓

Database Restored
```

It becomes as though nothing ever happened.

---

# Transaction Boundaries

A transaction should include **only database operations**.

Good

```ts
Diagnosis

↓

Prescription

↓

Medical Record

↓

Appointment
```

All database writes.

---

# What Should NOT Be Inside a Transaction?

Avoid long-running operations.

❌

```ts
await emailService.send();

await redis.set();

await uploadFile();

await axios.post(...);
```

Why?

Because transactions keep database connections open.

External services are slow and unpredictable.

---

# Better Approach

```
Transaction

↓

Commit

↓

Send Email

↓

Publish Event

↓

Queue Background Job
```

Keep transactions short.

# Production Notes

Use transactions when:

- Multiple tables are updated.
- Financial operations.
- Booking systems.
- Inventory management.
- Medical systems.
- Order processing.

Avoid transactions for:

- Reading data only.
- Sending emails.
- Uploading files.
- Calling external APIs.
- Long-running operations.

---

# Common Mistakes

## Large Transactions

Do not keep transactions open for several seconds.

Keep them as short as possible.

---

## Including External APIs

Never include

```ts
Email

SMS

Redis

HTTP Requests
```

inside database transactions.

---

## Forgetting Rollback Scenarios

Always ask:

> "What happens if operation number 4 fails?"

If the answer is "the database becomes inconsistent",

you probably need a transaction.

---

## Using Transactions Everywhere

Transactions are powerful,

but they also consume database resources.

Use them only when business consistency requires them.

---

# Summary

In this chapter, we solved the consistency problem introduced in the previous chapter.

Instead of executing database operations independently, we grouped them into a single transaction using Prisma's `$transaction()` API. This guarantees that either every operation succeeds or every operation is rolled back, keeping the database consistent.

Transactions are one of the most important tools for backend engineers because real-world business workflows often modify multiple entities together. Understanding where transaction boundaries begin and end is essential for building reliable production systems.

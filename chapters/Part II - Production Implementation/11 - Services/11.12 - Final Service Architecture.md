# Responsibility Review

## Controller

Responsible for:

- Receiving requests.
- Reading parameters.
- Calling Services.
- Returning HTTP responses.

Should **not** contain business logic.

---

## Service

Responsible for:

- Business rules.
- Validation.
- Workflow orchestration.
- Transactions.
- Calling repositories.
- Calling other domain services.

Should **not** know about Express or HTTP.

---

## Repository

Responsible for:

- Database queries.
- Persistence.
- Mapping data.

Should **not** implement business rules.

---

# Characteristics of a Good Service

A production Service should:

- Represent one business capability.
- Be easy to read.
- Be easy to test.
- Hide implementation details.
- Coordinate workflows.
- Delegate persistence to repositories.

The public methods should read like business documentation.

Example:

```ts
async create(dto) {

    await this.validateDoctor(dto);

    await this.validatePatient(dto);

    await this.checkAvailability(dto);

    const appointment =
        await this.createAppointment(dto);

    await this.notifyParticipants(
        appointment
    );

    return appointment;

}
```

Even without comments, the workflow is understandable.

# Where Different Logic Belongs

| Concern          | Layer            |
| ---------------- | ---------------- |
| HTTP Request     | Controller       |
| Authentication   | Middleware       |
| Input Validation | Validation Layer |
| Business Rules   | Service          |
| Database Access  | Repository       |
| SQL Generation   | Prisma           |
| Data Storage     | PostgreSQL       |

Every concern has a dedicated home.

---

# Code Review Checklist

When reviewing a Service, ask:

- Does this Service have one responsibility?
- Is the public workflow readable?
- Are helper methods extracted?
- Is business logic duplicated?
- Are repositories used correctly?
- Are transactions limited to database work?
- Can this Service be unit tested?
- Is it independent of Express?

If the answer is "Yes" to all, the architecture is likely healthy.

---

# Common Design Principles

## Single Responsibility

Each Service owns one business capability.

---

## Separation of Concerns

HTTP, business logic, and persistence remain independent.

---

## Readability

Business workflows should be understandable without reading implementation details.

---

## Reusability

Validation and helper methods should be reused instead of duplicated.

---

## Testability

Business logic should be testable without HTTP or database dependencies.

# Key Takeaways

- Services implement business logic.
- Controllers handle HTTP communication.
- Repositories manage persistence.
- Transactions protect data consistency.
- Services should remain readable and focused.
- Architecture is maintained through continuous refactoring, not one-time design.

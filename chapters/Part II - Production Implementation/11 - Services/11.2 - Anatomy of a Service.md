# Business Requirement

Our Clinic Management System allows an administrator to register doctors.

Creating a doctor is much more than inserting a row into a database.

The application must:

- Verify the clinic exists.
- Check if the email is already registered.
- Hash the password.
- Save the doctor.
- Record an audit log.
- Send a welcome email.

These operations together form one business workflow.

The **DoctorService** is responsible for coordinating that workflow.

---

# Request Lifecycle

Let's revisit the complete request flow.

```text
Client

↓

POST /api/v1/doctors

↓

Routes

↓

Authentication Middleware

↓

Authorization Middleware

↓

DoctorController

↓

DoctorService

↓

DoctorRepository

↓

Prisma

↓

PostgreSQL
```

Notice something important.

The Service sits between the Controller and the Repository.

```
Controller

↓

Service

↓

Repository
```

Everything above the Service belongs to the HTTP layer.

Everything below the Service belongs to the persistence layer.

The Service is responsible for the business layer.

---

# Where Should a Service Live?

Each business module owns its own Service.

Our Doctor module contains:

```text
doctor/

├── doctor.controller.ts
├── doctor.service.ts
├── doctor.repository.ts
├── doctor.routes.ts
├── dto/
├── validators/
└── types/
```

Keeping related files together makes the project easier to navigate.

When someone needs to modify doctor-related business logic, they immediately know where to look.

# Creating the Service File

Create the following file.

```text
src/modules/doctor/doctor.service.ts
```

Initially, it will only contain imports and the Service class.

Do not worry about business logic yet.

Production software is usually built incrementally.

# Step 1 — Import Dependencies

Let's begin by importing everything the Service depends on.

```ts
import bcrypt from 'bcrypt'

import { DoctorRepository } from './doctor.repository'
import { ClinicRepository } from '../clinic/clinic.repository'

import { AuditService } from '../../shared/services/audit.service'
import { EmailService } from '../../shared/services/email.service'

import { CreateDoctorDto, UpdateDoctorDto } from './dto'
```

Before continuing, let's understand why every import exists.

---

# Import 1 — bcrypt

```ts
import bcrypt from 'bcrypt'
```

## Why do we need bcrypt?

Passwords should never be stored as plain text.

Instead,

```
User Password

↓

Hash Password

↓

Store Hash
```

Even if the database is compromised, attackers should not be able to read user passwords.

---

## Why is bcrypt used inside the Service?

Because password hashing is part of the business workflow.

Creating a doctor means creating secure credentials.

The Repository should never decide how passwords are generated.

That decision belongs to the Service.

---

# Import 2 — DoctorRepository

```ts
import { DoctorRepository } from './doctor.repository'
```

The Repository is responsible for communicating with Prisma.

Examples include:

```ts
create()

update()

delete()

findById()

findByEmail()

findAll()
```

Notice something important.

The Repository knows how to communicate with the database.

The Service knows **when** to call the Repository.

These are two completely different responsibilities.

---

# Why Doesn't the Service Use Prisma Directly?

Imagine writing this inside the Service.

```ts
await prisma.doctor.create({
  data: dto
})
```

This couples the Service directly to Prisma.

Now imagine switching to another ORM in the future.

Every Service would need modification.

Instead,

```
DoctorService

↓

DoctorRepository

↓

Prisma

↓

Database
```

Only the Repository knows about Prisma.

The Service remains independent of the persistence implementation.

---

# Import 3 — ClinicRepository

```ts
import { ClinicRepository } from '../clinic/clinic.repository'
```

Creating a Doctor also depends on another business entity.

Before saving the Doctor,

we must verify that the Clinic exists.

Without this validation,

our database could contain orphaned Doctors.

Example

```
Doctor

↓

Clinic ID = 123

↓

Clinic Does Not Exist
```

That is invalid business data.

The Service prevents this from happening.

---

# Why Use Another Repository?

Some developers are tempted to write

```ts
await prisma.clinic.findUnique(...)
```

inside DoctorService.

Avoid this.

Instead,

DoctorService collaborates with ClinicRepository.

Benefits include:

- Better separation of concerns.
- Easier testing.
- Reusable database logic.
- Consistent architecture.

Repositories own persistence.

Services own orchestration.

---

# Import 4 — AuditService

```ts
import { AuditService } from '../../shared/services/audit.service'
```

Production systems rarely perform business actions silently.

Imagine a Doctor being deleted.

Later someone asks

> "Who deleted this Doctor?"

Without an audit log,

there is no answer.

Audit logging provides accountability.

Typical audit information includes:

```text
Action

↓

Entity

↓

User

↓

Timestamp

↓

Metadata
```

DoctorService does not know how audit logs are stored.

It simply delegates the task to AuditService.

---

# Import 5 — EmailService

```ts
import { EmailService } from '../../shared/services/email.service'
```

After successfully creating a Doctor,

the application sends a welcome email.

Should DoctorService know how SMTP works?

No.

Should it know how SendGrid works?

No.

Should it know how AWS SES works?

No.

It only knows one thing.

```
Send Welcome Email
```

The implementation details remain hidden inside EmailService.

This separation allows us to change email providers without modifying DoctorService.

---

# Import 6 — DTOs

```ts
import { CreateDoctorDto, UpdateDoctorDto } from './dto'
```

DTO stands for **Data Transfer Object**.

A DTO defines the structure of data entering the business layer.

Instead of receiving

```ts
Request
```

the Service receives

```ts
CreateDoctorDto
```

This keeps the Service independent of Express.

Notice there is no

```ts
Request

Response

NextFunction
```

inside the Service.

That is intentional.

The Service should not know anything about HTTP.

---

At this point, our Service contains only imports.

Although this seems simple,

we have already made several important architectural decisions.

- External integrations are delegated to dedicated Services.
- HTTP objects never enter the business layer.

The foundation is now ready.

# 11.2 - Anatomy of a Service (Part 2)

> In the previous part, we prepared the Service by importing its dependencies and understanding why each dependency exists. In this part, we will build the actual structure of the `DoctorService`, inject its dependencies, define its public methods, and understand why production Services are organized this way.

---

# Step 2 — Creating the Service Class

Now that all dependencies have been imported, we can create the Service itself.

```ts
export class DoctorService {}
```

At this point, our Service contains no business logic.

This is intentional.

Every production Service starts with its structure before implementing its behavior.

---

# Why Use a Class?

Instead of writing functions like this

```ts
export async function createDoctor() {}

export async function updateDoctor() {}

export async function deleteDoctor() {}
```

we group related business operations inside a class.

```ts
export class DoctorService {}
```

## Benefits

- Related operations stay together.
- Dependencies are injected once.
- Easier to maintain.
- Easier to test.
- Clear ownership of business logic.

Think of a Service class as the home for one business domain.

---

# Step 3 — Constructor Injection

The Service needs access to several dependencies.

Let's inject them.

```ts
export class DoctorService {
  constructor(
    private readonly doctorRepository: DoctorRepository,

    private readonly clinicRepository: ClinicRepository,

    private readonly auditService: AuditService,

    private readonly emailService: EmailService
  ) {}
}
```

This is called **Constructor Injection**.

Instead of creating dependencies ourselves, they are provided from outside.

---

# Why Constructor Injection?

Imagine writing this.

```ts
const doctorRepository = new DoctorRepository()

const clinicRepository = new ClinicRepository()

const emailService = new EmailService()
```

This creates several problems.

- Tight coupling.
- Difficult unit testing.
- Difficult mocking.
- Harder to replace implementations.

Instead,

dependencies are passed into the constructor.

```
Application

↓

Creates Dependencies

↓

Injects DoctorService

↓

DoctorService Uses Them
```

The Service doesn't care who created them.

It simply uses them.

---

# Understanding `private readonly`

Let's examine one dependency.

```ts
private readonly doctorRepository: DoctorRepository
```

There are three important parts.

---

## private

```ts
private
```

The dependency can only be used inside this class.

Other classes cannot directly access it.

This protects internal implementation details.

---

## readonly

```ts
readonly
```

Once assigned,

the dependency cannot be changed.

Example

```ts
this.doctorRepository = anotherRepository
```

This is not allowed.

The dependency remains constant throughout the lifetime of the Service.

---

## DoctorRepository

```ts
DoctorRepository
```

This defines the type of the dependency.

TypeScript now knows exactly which methods are available.

For example

```ts
this.doctorRepository.findByEmail()

this.doctorRepository.create()

this.doctorRepository.findAll()
```

The IDE can provide autocomplete and type safety.

---

# Why Doesn't the Service Create Dependencies?

Imagine this.

```ts
export class DoctorService {
  private repository = new DoctorRepository()
}
```

Problems

- Cannot replace Repository during testing.
- Difficult to mock.
- Strong coupling.
- Violates Dependency Injection principles.

Production applications avoid this approach.

---

# Current Service Structure

At this point,

our file looks like this.

```ts
export class DoctorService {
  constructor(
    private readonly doctorRepository: DoctorRepository,

    private readonly clinicRepository: ClinicRepository,

    private readonly auditService: AuditService,

    private readonly emailService: EmailService
  ) {}
}
```

No methods yet.

Only the structure.

---

# Step 4 — Defining Public Methods

Now let's define the operations this Service will perform.

```ts
export class DoctorService {
  constructor(
    private readonly doctorRepository: DoctorRepository,

    private readonly clinicRepository: ClinicRepository,

    private readonly auditService: AuditService,

    private readonly emailService: EmailService
  ) {}

  async create(dto: CreateDoctorDto) {}

  async update(id: string, dto: UpdateDoctorDto) {}

  async findById(id: string) {}

  async findAll() {}

  async delete(id: string) {}
}
```

These methods represent business operations.

Notice they are empty.

We'll implement them in the following chapters.

---

# Why Async?

Every method is asynchronous.

```ts
async create(...)
```

Why?

Because almost every business operation eventually communicates with an external system.

Examples include

- Database
- Redis
- Email Provider
- File Storage
- Queue
- Notification Service

All of these operations are asynchronous.

Using `async/await` keeps the code readable.

---

# Why Doesn't the Service Receive Request?

Notice this.

```ts
async create(
    dto: CreateDoctorDto
)
```

Instead of

```ts
async create(
    req: Request,
    res: Response
)
```

The Service receives only business data.

It does not know

- Express
- HTTP
- Routes
- Status Codes

This keeps the business layer independent from the web framework.

If tomorrow we build

- REST API
- GraphQL API
- gRPC Service
- Background Worker

all of them can reuse the same DoctorService.

---

# Communication Between Layers

The interaction now becomes very clear.

```text
DoctorController

↓

doctorService.create(dto)

↓

DoctorRepository.create()

↓

Prisma

↓

PostgreSQL
```

Every layer performs one responsibility.

---

# What the Service Owns

DoctorService owns business operations such as

- Register Doctor
- Update Doctor
- Delete Doctor
- Activate Doctor
- Deactivate Doctor
- Search Doctor
- Assign Doctor to Clinic

Whenever a new Doctor-related requirement arrives,

this is the first file we inspect.

---

# What Does NOT Belong Here?

The Service should never contain

❌ Express routes

❌ HTTP responses

❌ Request objects

❌ Response objects

❌ Prisma queries

❌ SQL queries

Those responsibilities belong to other layers.

---

# Production Notes

A production Service should be

- Stateless.
- Focused.
- Reusable.
- Testable.
- Independent of Express.
- Independent of Prisma.

A Service coordinates business logic.

It should never become responsible for infrastructure concerns.

---

# Common Mistakes

## Creating Dependencies Manually

❌ Wrong

```ts
const repository = new DoctorRepository()
```

Always inject dependencies.

---

## Using Express Objects

❌ Wrong

```ts
create(req, res)
```

Use DTOs instead.

---

## Mixing Database Logic

❌ Wrong

```ts
await prisma.doctor.create(...)
```

Always delegate persistence to the Repository.

---

## Mixing Business Domains

❌ Wrong

```text
DoctorService

↓

Doctor

↓

Patient

↓

Billing

↓

Inventory
```

One Service should own one business domain.

```ts
  async update(id: string, dto: UpdateDoctorDto) {}

  async findAll() {}

```

Although the methods are empty,

the architecture is now complete.

# Business Requirement

The Clinic Management System allows an Administrator to register Doctors.

When a Doctor is created, the backend must ensure that:

- The clinic exists.
- The email address is unique.
- The password is securely hashed.
- The doctor is saved into the database.
- An audit record is created.
- A welcome email is sent.

This is not one database operation.

It is one **business workflow**.

The responsibility of coordinating this workflow belongs to the **DoctorService**.

---

# Why These Imports?

Let's understand every dependency before writing any business logic.

---

### bcrypt

```ts
import bcrypt from 'bcrypt'
```

Purpose

Used to securely hash passwords before storing them in the database.

Why inside the Service?

Because password hashing is a **business operation**, not a database operation.

The Repository should never know how passwords are generated.

---

### DoctorRepository

```ts
import { DoctorRepository } from './doctor.repository'
```

Purpose

Provides database operations related to Doctors.

Examples

- create()
- update()
- findById()
- findByEmail()
- delete()

The Service calls the Repository.

The Repository communicates with Prisma.

---

### ClinicRepository

```ts
import { ClinicRepository } from '../clinic/clinic.repository'
```

Purpose

Before creating a Doctor, we must verify that the Clinic actually exists.

This business rule requires data from another module.

Instead of querying Prisma directly,

the Service collaborates with another Repository.

---

### AuditService

```ts
import { AuditService } from '../../shared/services/audit.service'
```

Purpose

Every important business action should be recorded.

Example

```
Doctor Created

↓

Created By

↓

Timestamp

↓

IP Address

↓

Metadata
```

Keeping audit logic separate allows it to be reused throughout the application.

---

### EmailService

```ts
import { EmailService } from '../../shared/services/email.service'
```

Purpose

Responsible for sending emails.

The DoctorService should never know **how** emails are sent.

It simply requests

```
Send Welcome Email
```

The EmailService handles the implementation.

---

### DTOs

```ts
import { CreateDoctorDto, UpdateDoctorDto } from './dto'
```

Purpose

DTOs define the shape of incoming business data.

The Service receives structured objects instead of raw HTTP requests.

---

# Step 3 — Constructor Injection

Let's inject the dependencies.

```ts
export class DoctorService {
  constructor(
    private readonly doctorRepository: DoctorRepository,

    private readonly clinicRepository: ClinicRepository,

    private readonly auditService: AuditService,

    private readonly emailService: EmailService
  ) {}
}
```

---

# Why Constructor Injection?

Instead of creating objects manually

```ts
const repository = new DoctorRepository()
```

we receive them from outside.

Benefits

- Easier testing.
- Easier mocking.
- Loose coupling.
- Better maintainability.
- Supports Dependency Injection.

Production applications rarely instantiate dependencies inside Services.

---

# Why `private readonly`?

```ts
private readonly doctorRepository
```

### private

Prevents other classes from directly accessing the dependency.

Only DoctorService should communicate with it.

---

### readonly

Ensures the dependency cannot be reassigned after construction.

Once injected,

it remains constant throughout the lifetime of the Service.

---

# Production Notes

- Keep Services focused on one business domain.
- Inject dependencies instead of creating them.
- Never access Prisma directly from the Service.
- Never use Express Request or Response objects inside the Service.
- Think of the Service as the business orchestrator.

---

# Common Mistakes

### Creating Repositories inside the Service

❌ Wrong

```ts
const doctorRepository = new DoctorRepository()
```

Always inject dependencies.

---

### Accessing Prisma Directly

❌ Wrong

```ts
await prisma.doctor.create(...)
```

Always communicate through the Repository layer.

---

### Returning HTTP Responses

❌ Wrong

```ts
return res.status(201).json(...)
```

Services return business results.

Controllers return HTTP responses.

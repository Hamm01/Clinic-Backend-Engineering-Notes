# Revisiting Our Service

From the previous chapter, our Service currently looks like this.

```ts
export class DoctorService {
  constructor(
    private readonly doctorRepository: DoctorRepository,

    private readonly clinicRepository: ClinicRepository,

    private readonly auditService: AuditService,

    private readonly emailService: EmailService
  ) {}

  async create(dto: CreateDoctorDto) {}

  async update(id: string, dto: UpdateDoctorDto) {}

  async findById(id: string) {}

  async findAll() {}

  async delete(id: string) {}
}
```

The methods exist,

but they contain no business logic yet.

Before implementing them,

we need to understand **what belongs inside these methods.**

---

# Responsibility 1 — Execute Business Rules

The primary responsibility of a Service is enforcing business rules.

For example,

a Doctor can only be created if:

- The Clinic exists.
- The email is unique.
- The password meets security requirements.
- The user has permission.

These are not database rules.

These are business rules.

The Service owns them.

---

# Example

```ts
const existingDoctor = await this.doctorRepository.findByEmail(dto.email)

if (existingDoctor) {
  throw new ConflictException('Doctor already exists.')
}
```

### Explanation

Before creating a Doctor,

we verify that another Doctor with the same email does not already exist.

This decision belongs inside the Service because it represents a business requirement.

The Repository simply searches the database.

The Service decides what to do with the result.

---

# Responsibility 2 — Coordinate Multiple Repositories

A Service is not limited to one Repository.

Example

```ts
const clinic = await this.clinicRepository.findById(dto.clinicId)

if (!clinic) {
  throw new NotFoundException('Clinic not found.')
}
```

### Explanation

Creating a Doctor depends on another business entity.

Instead of accessing Prisma directly,

DoctorService collaborates with ClinicRepository.

This keeps each Repository responsible for its own data.

---

# Responsibility 3 — Prepare Data

Sometimes data must be transformed before saving.

Example

```ts
const hashedPassword = await bcrypt.hash(dto.password, 12)
```

### Explanation

Passwords should never be stored as plain text.

The Service prepares the data before passing it to the Repository.

The Repository should receive data that is already valid and ready to persist.

---

# Responsibility 4 — Delegate Persistence

Once the business rules have been validated,

the Service delegates persistence.

```ts
const doctor = await this.doctorRepository.create({
  ...dto,

  password: hashedPassword
})
```

### Explanation

Notice something important.

The Service does **not** call Prisma.

Instead,

it delegates persistence to the Repository.

This keeps database access centralized.

---

# Responsibility 5 — Coordinate External Services

Business workflows often involve more than the database.

Example

```ts
await this.auditService.log({
  action: 'DOCTOR_CREATED',

  entityId: doctor.id
})

await this.emailService.sendWelcomeEmail(doctor.email)
```

### Explanation

Creating a Doctor also creates an audit log and sends an email.

The Service coordinates these operations.

The Service does **not** know how emails are sent or where audit logs are stored.

Those responsibilities belong to dedicated Services.

---

# Responsibility 6 — Return Business Results

Finally,

the Service returns the business result.

```ts
return doctor
```

### Explanation

The Service returns domain data.

It does **not** return HTTP responses.

Controllers decide how to transform this result into an HTTP response.

---

# Complete Workflow

Putting everything together,

the workflow becomes

```text
Receive DTO

↓

Validate Business Rules

↓

Check Clinic

↓

Check Duplicate Email

↓

Prepare Data

↓

Hash Password

↓

Save Doctor

↓

Create Audit Log

↓

Send Welcome Email

↓

Return Doctor
```

This is the responsibility of the Service.

# What Does NOT Belong Inside a Service?

Avoid placing these responsibilities inside the Service.

---

## HTTP Requests

❌ Wrong

```ts
req.body

req.params

req.query
```

The Controller extracts HTTP data.

The Service receives DTOs.

---

## HTTP Responses

❌ Wrong

```ts
return res.status(201).json(...);
```

Controllers return HTTP responses.

Services return business results.

---

## Direct Prisma Queries

❌ Wrong

```ts
await prisma.doctor.create(...)
```

Always communicate through the Repository.

---

## Route Definitions

❌ Wrong

```ts
router.post(...)
```

Routes belong in the routing layer.

# Production Notes

A production Service should:

- Focus on one business domain.
- Coordinate workflows.
- Delegate persistence.
- Delegate infrastructure concerns.
- Remain independent of Express.
- Remain independent of Prisma.

The simpler the responsibility,

the easier the Service becomes to maintain.

---

# Common Mistakes

## Fat Controllers

Moving business logic into Controllers.

---

## Smart Repositories

Implementing business rules inside Repositories.

---

## Service Using Prisma

Skipping the Repository layer.

---

## Returning HTTP Responses

Mixing business logic with presentation logic.

---

# Summary

A Service is the business layer of the application.

Its responsibility is not to communicate with HTTP clients or databases directly.

Instead,

it coordinates business workflows, validates business rules, prepares data, collaborates with Repositories and other Services, and returns business results.

Keeping these responsibilities inside the Service creates applications that are easier to understand, easier to test, and easier to scale.

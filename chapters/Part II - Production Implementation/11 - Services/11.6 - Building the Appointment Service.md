# Business Requirement

A receptionist schedules an appointment.

Unlike registering a Doctor or Patient,

an appointment depends on several existing entities.

The application must verify:

- Clinic exists.
- Doctor exists.
- Patient exists.
- Doctor belongs to the selected clinic.
- Doctor is active.
- Patient is active.
- Appointment time is valid.
- Doctor is available.
- Time slot is not already booked.

Only after these validations should the appointment be created.

# Why Cancel Instead of Delete?

Hospitals rarely delete appointments.

Deleting removes historical information.

Instead,

appointments usually change status.

```
SCHEDULED

↓

CONFIRMED

↓

IN_PROGRESS

↓

COMPLETED
```

or

```
SCHEDULED

↓

CANCELLED
```

This preserves history for reporting and auditing.

Business rules often prefer **state transitions** over deletion.

# Step 1 — Implement create()

Unlike previous Services,

AppointmentService coordinates several independent workflows.

Our public method should still remain easy to read.

```ts
async create(
    dto: CreateAppointmentDto
) {

    const clinic =
        await this.validateClinic(dto.clinicId);

    const doctor =
        await this.validateDoctor(dto.doctorId);

    const patient =
        await this.validatePatient(dto.patientId);

    await this.validateDoctorClinic(
        doctor,
        clinic
    );

    this.validateAppointmentTime(
        dto.startTime,
        dto.endTime
    );

    await this.checkDoctorAvailability(
        doctor.id,
        dto.startTime,
        dto.endTime
    );

    const appointment =
        await this.appointmentRepository.create({

            ...dto,

            status: AppointmentStatus.SCHEDULED

        });

    await this.auditService.log({

        action: "APPOINTMENT_CREATED",

        entityId: appointment.id

    });

    return appointment;

}
```

# Step 2 — Validate Doctor

```ts
private async validateDoctor(
    doctorId: string
) {

    const doctor =
        await this.doctorRepository.findById(
            doctorId
        );

    if (!doctor) {

        throw new NotFoundException(
            "Doctor not found."
        );

    }

    if (!doctor.isActive) {

        throw new BadRequestException(
            "Doctor is inactive."
        );

    }

    return doctor;

}
```

---

## Why Validate Activity?

A doctor may still exist in the database.

However,

that does not mean appointments should be scheduled.

Example

```
Doctor

↓

Resigned

↓

Inactive
```

Business rules determine availability.

The database only stores data.

# Step 4 — Verify Doctor Belongs To Clinic

```ts
private async validateDoctorClinic(

    doctor: Doctor,

    clinic: Clinic

) {

    if (doctor.clinicId !== clinic.id) {

        throw new BadRequestException(

            "Doctor does not belong to this clinic."

        );

    }

}
```

---

## Why Is This Important?

Imagine this request.

```
Clinic A

↓

Doctor from Clinic B

↓

Appointment
```

The database allows the relationship.

The business should not.

The Service enforces this rule.

---

# Step 5 — Validate Appointment Time

```ts
private validateAppointmentTime(

    startTime: Date,

    endTime: Date

) {

    if (startTime >= endTime) {

        throw new BadRequestException(

            "Invalid appointment time."

        );

    }

}
```

---

## Why Does This Belong Here?

Repositories should never understand

```
Business Time

↓

Business Schedule

↓

Business Hours
```

Those decisions belong to the Service.

---

# Step 6 — Check Doctor Availability

```ts
private async checkDoctorAvailability(

    doctorId: string,

    startTime: Date,

    endTime: Date

) {

    const appointment =

        await this.appointmentRepository
            .findConflictingAppointment({

                doctorId,

                startTime,

                endTime

            });

    if (appointment) {

        throw new ConflictException(

            "Doctor already has an appointment during this time."

        );

    }

}
```

---

# Conflict Detection

Suppose

```
Doctor

↓

10:00 AM

↓

10:30 AM
```

A receptionist tries to create another appointment

```
10:15 AM

↓

10:45 AM
```

The Service prevents overlapping appointments.

Without this validation,

multiple patients could be scheduled at the same time.

# Production Notes

As appointment workflows evolve, additional business rules can be introduced without making `create()` difficult to understand.

Examples include:

- Clinic working hours.
- Doctor vacation schedules.
- Appointment duration limits.
- Maximum daily appointments.
- Insurance validation.
- Holiday calendars.

Each new rule should become another focused helper method rather than expanding the main workflow.

---

# Common Mistakes

## Skipping Conflict Detection

Never assume a doctor is available.

Always check existing appointments.

---

## Trusting Client Input

Do not trust that the frontend has already validated appointment times.

The backend must always enforce business rules.

---

## Putting Business Logic in the Repository

Avoid implementing scheduling rules inside database queries.

Repositories retrieve data.

Services decide business outcomes.

---

## Returning Success Before Validation

Always complete every validation before persisting data.

Fail fast when business rules are violated.

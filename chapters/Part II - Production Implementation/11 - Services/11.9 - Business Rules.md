# Principle 1 â€” One Method, One Responsibility

Instead of this.

```ts
async create() {

    // validation

    // billing

    // notifications

    // scheduling

    // persistence

}
```

Prefer

```ts
async create(dto: CreateAppointmentDto) {

    await this.validateAppointment(dto);

    await this.scheduleAppointment(dto);

    await this.createInvoice(dto);

    await this.notifyParticipants(dto);

}
```

The public method now reads like a business workflow.

# Principle 4 â€” Public vs Private Methods

Public methods represent business capabilities.

Example

```ts
create()

update()

cancel()

complete()
```

Private methods support those capabilities.

Example

```ts
validateDoctor()

validateClinic()

validateSchedule()

checkConflicts()

generateInvoiceNumber()
```

Ask yourself:

> "Should another class call this method?"

If the answer is **No**,

make it private.

---

# Principle 5 â€” Keep Services Readable

A good Service should almost read like English.

Example

```ts
async create(dto) {

    await this.validateClinic(dto);

    await this.validateDoctor(dto);

    await this.validatePatient(dto);

    await this.checkAvailability(dto);

    const appointment =
        await this.createAppointment(dto);

    await this.notifyParticipants(
        appointment
    );

    return appointment;

}
```

Even without comments,

the workflow is easy to understand.

# How Senior Engineers Think

Experienced engineers ask questions like:

- Can this workflow be understood in under one minute?
- Does each helper have one responsibility?
- Is this business rule duplicated?
- Can another Service own this responsibility?
- Does this method explain the business process clearly?

Notice they focus on readability,

not just making the code compile.

---

# Production Notes

As Services evolve:

- Keep public methods short.
- Extract reusable business rules.
- Group related logic.
- Split unrelated responsibilities.
- Prefer orchestration over duplication.

Good architecture is usually the result of continuous refactoring,

not perfect planning.

---

# Common Mistakes

## Huge Public Methods

Avoid hundreds of lines inside `create()`.

---

## Duplicating Validation

Extract reusable helpers.

---

## One Service Doing Everything

Split unrelated business concerns.

---

## Premature Abstraction

Do not create five Services for a project with only two business rules.

Refactor when complexity actually appears.

---

# Summary

As applications grow, Services naturally become larger because business requirements increase over time.

Rather than allowing Services to become difficult to maintain, experienced backend engineers continuously organize business rules by extracting helper methods, grouping related logic, separating business concerns, and introducing dedicated domain Services when responsibilities become too broad.

The goal is not to create more classes.

The goal is to make business workflows easy to understand, easy to modify, and easy to test.

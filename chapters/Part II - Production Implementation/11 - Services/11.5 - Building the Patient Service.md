# Business Requirement

A receptionist registers a new patient in the clinic.

Unlike creating a doctor, registering a patient has different business rules.

The application must:

- Verify the clinic exists.
- Normalize patient information.
- Detect possible duplicate patients.
- Generate a unique Patient Number.
- Save the patient.
- Record an audit log.
- Return the created patient.

Notice something important.

There is

- No password.
- No email verification.
- No authentication.

Instead, this workflow introduces new backend engineering concepts.

# Why Doesn't PatientService Need EmailService?

DoctorService required EmailService because new doctors receive login credentials.

Patients do not.

This demonstrates an important principle.

**Every Service should depend only on what it actually needs.**

Avoid injecting unnecessary dependencies.

# New Concept — Data Normalization

Users rarely enter data consistently.

Examples

```
RONAK SINGH

ronak singh

 Ronak Singh

RONAK   SINGH
```

Should all of these be stored differently?

No.

Before saving data,

we normalize it.

---

# Example

```ts
private normalizeName(name: string): string {

    return name

        .trim()

        .replace(/\s+/g, " ");

}
```

---

# Why Normalize Data?

Without normalization,

your database becomes inconsistent.

Example

```
Ronak Singh

↓

ronak singh

↓

RONAK SINGH

↓

Ronak    Singh
```

Searching,

sorting,

reporting,

and duplicate detection become unreliable.

Normalizing data before persistence improves consistency across the entire application.

---

# New Concept — Patient Number Generation

Doctors are identified by email.

Patients are usually identified by a hospital registration number.

Example

```
PT-2026-000001

PT-2026-000002

PT-2026-000003
```

This identifier is meaningful to doctors, receptionists, and hospital staff.

Unlike a database ID,

it is designed for humans.

---

# Why Not Use the Database ID?

Database IDs are implementation details.

```
id

↓

8a6c8d61...

↓

UUID
```

Hospital staff should never need to remember UUIDs.

Instead,

the application generates a business identifier.

```
Patient Number

↓

PT-2026-000015
```

This is easier to communicate and search.

---

# New Concept — Private Helper Methods

As Services grow,

the `create()` method can become very large.

Instead of writing everything inside one method,

we extract reusable logic.

Example

```ts
private generatePatientNumber() {}

private normalizeName() {}

private checkDuplicatePatient() {}
```

These helper methods improve readability.

---

# Why Private?

These methods are implementation details.

Controllers should never call them.

Other Services should never call them.

Only PatientService needs them.

Keeping them private protects the internal workflow.

---

# Production Notes

A production Service should expose only business operations.

Helper methods remain private.

Public methods describe the business.

Private methods support the business.

This separation keeps Services easy to read as they grow.

---

# Common Mistakes

## Huge create() Methods

Avoid writing hundreds of lines inside one method.

Extract reusable logic into private methods.

---

## Exposing Internal Helpers

❌ Wrong

```ts
public normalizeName() {}
```

This method is not part of the business API.

Keep it private.

---

## Using Database IDs as Business IDs

Avoid exposing UUIDs to hospital staff.

Generate meaningful identifiers for the business domain.

# Step 1 — Implementing create()

Our public business method remains small.

```ts
async create(
    dto: CreatePatientDto
) {

    const clinic =
        await this.validateClinic(dto.clinicId);

    const normalizedDto =
        this.normalizePatient(dto);

    await this.checkDuplicatePatient(normalizedDto);

    const patientNumber =
        await this.generatePatientNumber();

    const patient =
        await this.patientRepository.create({

            ...normalizedDto,

            clinicId: clinic.id,

            patientNumber

        });

    await this.auditService.log({

        action: "PATIENT_CREATED",

        entityId: patient.id

    });

    return patient;

}
```

---

# Why Keep create() Small?

Many developers write Services like this.

```ts
async create() {

    // 250 lines of code...

}
```

Large methods become difficult to understand.

Instead,

each step represents one business operation.

```
Validate

↓

Normalize

↓

Duplicate Check

↓

Generate Number

↓

Persist

↓

Audit

↓

Return
```

This makes the workflow obvious to anyone reading the code.

---

# Step 2 — Validate Clinic

```ts
private async validateClinic(
    clinicId: string
) {

    const clinic =
        await this.clinicRepository.findById(clinicId);

    if (!clinic) {

        throw new NotFoundException(
            "Clinic not found."
        );

    }

    return clinic;

}
```

---

## Why Extract This Method?

Clinic validation will be required in several workflows.

Examples

- Register Patient
- Transfer Patient
- Schedule Appointment
- Create Medical Record

Instead of copying the same logic,

we reuse one private helper.

---

# Step 3 — Normalize Patient Data

```ts
private normalizePatient(
    dto: CreatePatientDto
): CreatePatientDto {

    return {

        ...dto,

        firstName: dto.firstName
            .trim()
            .replace(/\s+/g, " "),

        lastName: dto.lastName
            .trim()
            .replace(/\s+/g, " "),

        phone: dto.phone
            .replace(/\D/g, "")

    };

}
```

---

## Why Normalize Before Saving?

Suppose three receptionists register the same patient.

```
Ronak

RONAK

 ronak
```

Without normalization,

duplicate detection becomes unreliable.

Normalizing data ensures consistency before persistence.

---

# Step 4 — Duplicate Detection

```ts
private async checkDuplicatePatient(
    dto: CreatePatientDto
) {

    const patient =
        await this.patientRepository.findByIdentity({

            phone: dto.phone,

            dateOfBirth: dto.dateOfBirth

        });

    if (patient) {

        throw new ConflictException(

            "Possible duplicate patient found."

        );

    }

}
```

---

## Why Phone + Date of Birth?

Unlike Doctors,

Patients may not always have an email address.

Business rules determine what makes a patient unique.

Depending on the hospital,

this could include

- Phone Number
- Date of Birth
- National ID
- Government Health Number

The Service enforces these rules.

---

# Step 5 — Generate Patient Number

```ts
private async generatePatientNumber() {

    const totalPatients =
        await this.patientRepository.count();

    const nextNumber =
        totalPatients + 1;

    const year =
        new Date().getFullYear();

    return `PT-${year}-${String(nextNumber).padStart(6,"0")}`;

}
```

---

## Example Output

```
PT-2026-000001

PT-2026-000002

PT-2026-000003
```

---

## Why Separate This Logic?

Patient numbering is a business rule.

Tomorrow,

the hospital may decide to generate numbers like

```
CLINIC01-PT-2026-000001
```

Only this helper changes.

The rest of the Service remains untouched.

---

# Step 6 — Persist the Patient

```ts
const patient = await this.patientRepository.create({
  ...normalizedDto,

  clinicId: clinic.id,

  patientNumber
})
```

---

## Why Doesn't the Repository Generate the Number?

Repositories should not decide business policies.

The Repository simply stores the data it receives.

The Service prepares the entity.

The Repository persists it.

---

# Step 7 — Record Audit Log

```ts
await this.auditService.log({
  action: 'PATIENT_CREATED',

  entityId: patient.id
})
```

---

## Why Audit After Saving?

Imagine logging before persistence.

```
Audit Success

↓

Database Insert Failed
```

Now the audit says a patient was created,

but the patient doesn't exist.

The audit should represent completed business actions.

# Production Notes

As Services grow,

continue extracting private helper methods.

Good Services read like business workflows.

A reader should understand the process without reading hundreds of lines of code.

Aim for methods that clearly express intent.

---

# Common Mistakes

## Huge Service Methods

❌

```ts
async create() {

    // 300 lines

}
```

Break the workflow into smaller methods.

---

## Repeating Validation Logic

Avoid validating the same entity in multiple places.

Extract reusable helpers.

---

## Mixing Database Logic

Never generate SQL or Prisma queries inside helper methods.

Repositories own persistence.

---

## Ignoring Data Normalization

Never trust user input exactly as it arrives.

Normalize before saving.

# 11.7 - Building the Diagnosis Service

> In the previous chapters, we built Services that managed a single business workflow. Creating a Diagnosis is different. A diagnosis is not just one database record—it represents an entire medical consultation. During one consultation, the system may create a diagnosis, prescribe medicines, order lab tests, update the patient's medical history, and complete the appointment. This chapter introduces business orchestration across multiple entities and prepares us for the Transactions chapter.

# Business Requirement

A doctor finishes examining a patient.

The application must:

- Validate the appointment.
- Verify the doctor.
- Verify the patient.
- Create the diagnosis.
- Save prescribed medicines.
- Save requested laboratory tests.
- Update the patient's medical record.
- Mark the appointment as completed.
- Record an audit log.

Unlike previous Services,

this workflow modifies several business entities.

# Implementing create()

The workflow should remain readable.

```ts
async create(
    dto: CreateDiagnosisDto
) {

    const appointment =
        await this.validateAppointment(dto.appointmentId);

    const doctor =
        await this.validateDoctor(dto.doctorId);

    const patient =
        await this.validatePatient(dto.patientId);

    const diagnosis =
        await this.createDiagnosis(dto);

    await this.createPrescription(
        diagnosis.id,
        dto.medicines
    );

    await this.createLabTests(
        diagnosis.id,
        dto.labTests
    );

    await this.updateMedicalRecord(
        patient.id,
        diagnosis.id
    );

    await this.completeAppointment(
        appointment.id
    );

    await this.auditService.log({

        action: "DIAGNOSIS_CREATED",

        entityId: diagnosis.id

    });

    return diagnosis;

}
```

Notice something important.

The `create()` method reads almost like business documentation.

Each helper represents one step of the consultation.

---

# Step 1 — Validate Appointment

```ts
private async validateAppointment(
    appointmentId: string
) {

    const appointment =
        await this.appointmentRepository.findById(
            appointmentId
        );

    if (!appointment) {

        throw new NotFoundException(
            "Appointment not found."
        );

    }

    if (
        appointment.status !==
        AppointmentStatus.IN_PROGRESS
    ) {

        throw new BadRequestException(
            "Diagnosis can only be created for an active consultation."
        );

    }

    return appointment;

}
```

---

# Why Validate Appointment?

A diagnosis should never be created for

- Cancelled appointments.
- Future appointments.
- Completed appointments.

Business rules belong inside the Service.

---

# Step 2 — Create Diagnosis

```ts
private async createDiagnosis(
    dto: CreateDiagnosisDto
) {

    return this.diagnosisRepository.create({

        symptoms: dto.symptoms,

        findings: dto.findings,

        diagnosis: dto.diagnosis,

        notes: dto.notes,

        patientId: dto.patientId,

        doctorId: dto.doctorId

    });

}
```

The Repository stores the diagnosis.

The Service decides **when** it should be stored.

---

# Step 3 — Create Prescription

```ts
private async createPrescription(

    diagnosisId: string,

    medicines: PrescriptionDto[]

) {

    if (!medicines.length) {

        return;

    }

    await this.prescriptionRepository.createMany({

        diagnosisId,

        medicines

    });

}
```

Not every consultation requires medication.

Business rules decide whether this step executes.

---

# Step 4 — Create Lab Tests

```ts
private async createLabTests(

    diagnosisId: string,

    tests: LabTestDto[]

) {

    if (!tests.length) {

        return;

    }

    await this.labTestRepository.createMany({

        diagnosisId,

        tests

    });

}
```

Some patients require laboratory investigations.

Others do not.

The workflow adapts accordingly.

---

# Step 5 — Update Medical Record

```ts
private async updateMedicalRecord(

    patientId: string,

    diagnosisId: string

) {

    await this.medicalRecordRepository.addDiagnosis({

        patientId,

        diagnosisId

    });

}
```

Every diagnosis becomes part of the patient's permanent medical history.

---

# Step 6 — Complete Appointment

```ts
private async completeAppointment(
    appointmentId: string
) {

    await this.appointmentRepository.updateStatus(

        appointmentId,

        AppointmentStatus.COMPLETED

    );

}
```

The consultation has ended.

The appointment lifecycle advances.

```
SCHEDULED

↓

CONFIRMED

↓

IN_PROGRESS

↓

COMPLETED
```

Notice that we change the status instead of deleting anything.

# Why Is This Chapter Important?

Imagine this sequence.

```
Diagnosis Created

✓

Prescription Saved

✓

Medical Record Failed

✗

Appointment Updated

✗
```

Now the database is inconsistent.

Some information was saved.

Some was not.

This introduces a serious problem.

---

# The Problem We Have Created

At this point,

our implementation is **not yet completely safe**.

Why?

Because several database operations execute independently.

If one operation fails,

previous operations are not automatically rolled back.

This can leave the system in an inconsistent state.

---

# Production Note

A production backend should execute this workflow inside a **database transaction**.

However,

we intentionally have **not** introduced transactions yet.

Why?

Because before learning transactions,

you first need to understand **why they are necessary**.

This chapter creates that problem naturally.

The next chapter will solve it.

# Common Mistakes

## Writing Everything Inside create()

Avoid writing hundreds of lines inside one method.

Extract helper methods.

---

## Mixing Repository Logic

Repositories save data.

Services coordinate business workflows.

---

## Ignoring Partial Failures

Whenever multiple tables are updated,

always think about consistency.

---

## Returning Before Completing Workflow

Never return success until the complete consultation has finished.

# 11.10 - Service vs Controller

> Throughout this handbook, we have built Controllers and Services separately. One of the most common questions asked by junior backend developers is: **"Should this code go inside the Controller or the Service?"** This chapter answers that question by comparing their responsibilities, identifying common mistakes, and demonstrating how production backends keep Controllers thin while allowing Services to own business logic.

# Request Lifecycle

Before comparing both layers, let's revisit the request lifecycle.

```

↓

Route

↓

Middleware

↓

Controller

↓

Service

↓

Repository

↓

Database
```

Every layer has a different responsibility.

When responsibilities overlap, code becomes difficult to maintain.

---

# Controller Responsibility

A Controller represents the application's HTTP layer.

It is responsible for:

- Receiving the HTTP request.
- Reading request parameters.
- Calling the appropriate Service.
- Returning the HTTP response.
- Converting business errors into HTTP responses.

A Controller **does not decide business rules.**

---

# Service Responsibility

A Service represents the application's business layer.

It is responsible for:

- Implementing business rules.
- Validating business workflows.
- Coordinating repositories.
- Calling other Services.
- Managing transactions.
- Returning business results.

A Service should not know anything about HTTP.

# Incorrect Controller

Many beginners write Controllers like this.

```ts
async create(
    req: Request,
    res: Response
) {

    const clinic =
        await prisma.clinic.findUnique(...);

    if (!clinic) {

        return res.status(404).json({
            message: "Clinic not found"
        });

    }

    const patient =
        await prisma.patient.create(...);

    await prisma.audit.create(...);

    return res.status(201).json(patient);

}
```

---

# Problems

The Controller now contains:

- Database queries.
- Business validation.
- Persistence logic.
- Audit logic.

The Controller has become responsible for everything.

---

# Correct Controller

Instead,

Controllers should delegate work.

```ts
async create(
    req: Request,
    res: Response
) {

    const patient =

        await this.patientService.create(
            req.body
        );

    return res.status(201).json({

        data: patient

    });

}
```

Notice how little code remains.

---

# Why Is This Better?

The Controller now focuses only on HTTP.

```
Receive Request

↓

Call Service

↓

Return Response
```

Business logic has moved elsewhere.

---

# Incorrect Service

The opposite mistake also happens.

```ts
async create(dto) {

    res.status(201).json(...);

}
```

or

```ts
req.body.name
```

or

```ts
req.user
```

---

# Why Is This Wrong?

Services should never know:

- Express
- Request
- Response
- HTTP Status Codes

Those belong to Controllers.

Otherwise,

the Service becomes tightly coupled to Express.

---

# Correct Service

```ts
async create(
    dto: CreatePatientDto
) {

    await this.validateClinic(
        dto.clinicId
    );

    const patient =

        await this.patientRepository.create(
            dto
        );

    return patient;

}
```

The Service receives plain data.

It returns plain business objects.

| Controller       | Service                 |
| ---------------- | ----------------------- |
| HTTP Request     | Business Rules          |
| HTTP Response    | Business Workflow       |
| Route Parameters | Validation              |
| Query Parameters | Repository Coordination |
| Status Codes     | Transactions            |
| Request Body     | Domain Logic            |

# Validation Example

HTTP validation

```
Email format

Password length

Required fields
```

Usually handled by middleware or validation libraries.

Business validation

```
Doctor exists

Clinic exists

Patient active

Doctor available
```

Handled by Services.

---

# Why Keep Controllers Thin?

Imagine changing from

```
Express

↓

Fastify
```

Would your business logic change?

No.

Only Controllers change.

Services remain exactly the same.

Keeping business logic inside Services makes framework migration much easier.

---

# Testing Benefits

Thin Controllers are easy to test.

Most unit testing focuses on Services.

Example

```
PatientService

↓

Validate Clinic

↓

Create Patient

↓

Audit Log
```

Controllers usually only verify:

- Correct Service called.
- Correct response returned.

# Common Mistakes

## Database Queries Inside Controllers

Avoid

```ts
await prisma.patient.create(...)
```

Move persistence into the Repository.

---

## Business Rules Inside Controllers

Avoid

```ts
if (doctor.isOnLeave)
```

Business decisions belong to Services.

---

## HTTP Objects Inside Services

Never pass

```ts
Request

Response

NextFunction
```

to Services.

Pass DTOs instead.

---

## Fat Controllers

If a Controller exceeds 100–150 lines,

review whether business logic should move into the Service.

---

# Production Notes

A useful guideline is:

**Controller asks.**

**Service decides.**

The Controller asks the Service to perform a business operation.

The Service decides whether that operation is allowed.

This simple rule keeps responsibilities clear throughout the application.

---

# Summary

Controllers and Services solve different problems.

Controllers manage communication between clients and the application by handling HTTP requests and responses.

Services manage business logic by validating workflows, enforcing business rules, coordinating repositories, and returning business results.

Separating these responsibilities makes applications easier to understand, easier to test, easier to maintain, and less dependent on any specific web framework.

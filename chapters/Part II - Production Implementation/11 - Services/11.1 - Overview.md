# 11.1 - Overview

> Services are the heart of every backend application. While Controllers are responsible for handling HTTP requests and Repositories are responsible for communicating with the database, Services contain the application's business logic. They decide **what should happen** when a request is received. In this module, we will design and implement production-ready Services for our Clinic Management System while understanding the architectural principles followed by experienced backend engineers.

---

# Learning Objectives

After completing this module, you will understand:

- What a Service layer is.
- Why Services exist.
- How Services fit into a layered architecture.
- What responsibilities belong inside a Service.
- How Services interact with Controllers and Repositories.
- How business rules are implemented.
- How to write clean, reusable, and maintainable Services.

# Why This Module Exists

So far we have built Controllers that receive HTTP requests.

Example

```
POST /api/v1/doctors
```

The Controller receives the request.

But then an important question arises.

```
What should happen next?
```

Should the doctor be created immediately?

Should we verify the email?

Should we hash the password?

Should we check whether another doctor already exists?

Should we create an audit log?

Should we send a welcome email?

Someone has to make these decisions.

That someone is the **Service layer**.

# What Is a Service?

A Service is responsible for executing the application's business rules.

Think of a Service as the decision maker.

The Controller asks

```
A request has arrived.

What should happen?
```

The Service answers

```
Here is the complete business workflow.
```

The Service coordinates everything required to complete the request.

---

# Real Clinic Example

A receptionist creates a new patient.

```
POST

/api/v1/patients
```

The Controller receives the request.

The Service now decides

```
Is the patient already registered?

↓

Generate Patient ID

↓

Normalize phone number

↓

Prepare patient data

↓

Call Repository

↓

Return result
```

Notice

The Controller never makes these decisions.

---

# Responsibilities of a Service

Services are responsible for

- Business rules.
- Business workflows.
- Coordinating multiple repositories.
- Data preparation.
- Business validation.
- Permission checks.
- Calling external services.
- Managing transactions.
- Publishing events.
- Returning business results.

---

# Responsibilities That Do NOT Belong in a Service

Services should not

❌ Read HTTP Requests

❌ Read Express Route Parameters

❌ Build HTTP Responses

❌ Return HTTP Status Codes

❌ Access Express Request or Response objects

Those responsibilities belong to Controllers.

---

# Relationship Between Layers

Every layer has exactly one responsibility.

```
Controller

↓

Reads HTTP Request

↓

Calls Service
```

```
Service

↓

Executes Business Logic

↓

Calls Repository
```

```
Repository

↓

Communicates with Database
```

Keeping responsibilities separate makes the application easier to understand and maintain.

---

# Why Do We Need Services?

Imagine we need to create a Doctor.

Business requirements say

- Email must be unique.
- Password must be hashed.
- The Doctor belongs to a Clinic.
- The Doctor should receive a welcome email.
- An audit log should be created.

Where should these rules live?

Not in the Controller.

Not in the Repository.

They belong inside the Service.

---

# Services Think in Business

Controllers think about

```
HTTP
```

Repositories think about

```
Database
```

Services think about

```
Business
```

This separation allows each layer to evolve independently.

---

# Benefits of the Service Layer

Using a dedicated Service layer provides

- Clear separation of responsibilities.
- Better code organization.
- Easier testing.
- Reusable business logic.
- Smaller Controllers.
- Simpler database access.
- Easier maintenance.
- Better scalability.

# Summary

The Service layer is where the application's business logic lives. It receives requests from Controllers, makes business decisions, coordinates workflows, communicates with Repositories, and returns results back to the Controller.

Unlike Controllers, which focus on HTTP communication, and Repositories, which focus on database access, Services focus entirely on implementing the rules and workflows that define how the business operates.

Understanding the Service layer is one of the most important milestones in backend engineering because it is where software stops reacting to requests and starts enforcing business behavior.

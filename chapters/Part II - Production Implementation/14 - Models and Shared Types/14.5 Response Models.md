# 14.5 - Response Models

## Overview

A Response Model represents the data returned from the application to the client.

It defines the public contract of an API by controlling which fields are exposed.

Instead of returning database records directly, production applications use dedicated Response Models to send only the required information.

---

## Why Response Models Exist

The database often stores more information than the client should receive.

Consider the following Patient record.

```text
Patient

id

firstName

lastName

email

passwordHash

createdAt

updatedAt
```

If this object is returned directly, sensitive or unnecessary information may be exposed.

A Response Model ensures that only the required fields are included in the API response.

---

---

## How It Is Used

Service

```ts
async getPatient(id: string): Promise<PatientResponse> {

    const patient = await this.patientRepository.findById(id);

    return {

        id: patient.id,

        firstName: patient.firstName,

        lastName: patient.lastName,

        email: patient.email

    };

}
```

The Service transforms the database object into a Response Model before returning it.

## One Response Model Per API

Different endpoints often require different responses.

Example

```text
PatientResponse

PatientProfileResponse

PatientSummaryResponse

AppointmentResponse

DoctorResponse
```

Each endpoint returns only the data required by that operation.

---

## What Should Be Included?

A Response Model should contain only the fields that the client needs.

Example

```ts
export interface AppointmentResponse {
  id: string

  appointmentDate: Date

  status: string

  doctorName: string
}
```

---

## What Should NOT Be Included?

Avoid exposing internal application fields.

Incorrect

```ts
export interface PatientResponse {
  passwordHash: string

  createdAt: Date

  updatedAt: Date
}
```

Internal implementation details should never be part of a public API.

---

## Best Practices

- Create dedicated Response Models for APIs.
- Return only required fields.
- Hide internal database fields.
- Keep responses consistent.
- Design responses based on client requirements.

---

## Common Mistakes

‚ùå Returning Prisma models directly.

‚ùå Exposing sensitive information.

‚ùå Returning unnecessary database columns.

‚ùå Using one Response Model for every endpoint.

‚ùå Returning raw database objects from Controllers.

---

## Summary

Response Models define the public structure of an API response.

They protect internal application data, improve API consistency, and provide a clear contract between the backend and its clients.

# 14.10 - Validation Models (Zod)

## Overview

Validation Models ensure that incoming data is correct before it enters the application's business logic.

Instead of trusting data received from clients, every request should be validated against predefined rules.

In our Clinic Backend, we use **Zod** to validate incoming HTTP requests and infer TypeScript types from the same schema.

---

## Why Validation Models Exist

Clients can send invalid or incomplete data.

Example

```json
{
  "firstName": "",
  "email": "invalid-email",
  "phone": 12345
}
```

Without validation:

- Invalid data reaches the Service.
- Database errors increase.
- Business logic becomes harder to maintain.

Validation stops invalid requests before they enter the application.

---

## Implementation

Validation schemas are stored separately.

```text
src/

models/

└── validation/

    ├── patient.validation.ts

    ├── doctor.validation.ts

    ├── appointment.validation.ts

    └── diagnosis.validation.ts
```

Each feature owns its own validation schema.

---

## Example

```ts
import { z } from 'zod'

export const createPatientSchema = z.object({
  firstName: z.string().min(2),

  lastName: z.string().min(2),

  email: z.email(),

  phone: z.string().optional()
})
```

This schema defines the rules for creating a patient.

---

## Inferring Types

One advantage of Zod is automatic type inference.

```ts
export type CreatePatientRequest = z.infer<typeof createPatientSchema>
```

Now the validation schema becomes the source of truth for both:

- Runtime validation
- TypeScript types

---

## How It Is Used

Controller

```ts
const request = createPatientSchema.parse(req.body)

return patientService.create(request)
```

If validation fails, Zod throws an error before the Service is called.

---

## Request Lifecycle

```text
Client

↓

HTTP Request

↓

Zod Validation

↓

Request Model

↓

Controller

↓

DTO

↓

Service
```

Validation is always the first step after receiving a request.

---

## Validation Rules

Typical validations include:

- Required fields
- Email format
- Password length
- String length
- Number ranges
- Optional fields
- Arrays
- Nested objects

Keep business rules inside Services and structural validation inside Zod schemas.

---

## Best Practices

- Create one schema for each API operation.
- Keep validation close to the feature.
- Reuse shared schemas where appropriate.
- Infer TypeScript types from Zod.
- Validate before entering the Service layer.

---

## Common Mistakes

❌ Skipping request validation.

❌ Duplicating validation rules.

❌ Mixing business logic with validation.

❌ Creating one large validation file.

❌ Trusting client input without validation.

---

## Summary

Validation Models protect the application from invalid input.

Using Zod provides both runtime validation and TypeScript type inference, ensuring that only valid data reaches the business layer.

# 14.4 - Request Models

## Overview

A Request Model represents the data received from the client through an HTTP request.

It defines the structure of incoming data before the application processes it.

Every API endpoint that accepts input should have its own Request Model.

This keeps the application's input predictable, organized, and easy to maintain.

---

## Why Request Models Exist

Clients send data in many different formats.

For example, when creating a new patient, the client sends:

```json
{
  "firstName": "John",
  "lastName": "Doe",
  "email": "john@example.com",
  "phone": "9876543210"
}
```

Instead of passing the raw request throughout the application, we represent it using a dedicated Request Model.

---

## Implementation

In our Clinic Backend, Request Models are stored separately from other models.

```text
src/

models/

└── requests/

    └── create-patient.request.ts
```

Example

```ts
export interface CreatePatientRequest {
  firstName: string

  lastName: string

  email: string

  phone?: string
}
```

This model represents only the data expected from the client.

---

## How It Is Used

Controller

```ts
@Post()
async createPatient(

    @Body() request: CreatePatientRequest

) {

    return this.patientService.create(request);

}
```

The Controller receives the Request Model and forwards it to the next layer.

---

## Request Lifecycle

```text
Client

↓

HTTP Request

↓

Request Model

↓

Controller

↓

Service
```

Request Models are only used at the application's entry point.

---

## What Should Be Included?

A Request Model should contain only the fields that the client is allowed to send.

Example

```ts
export interface CreateAppointmentRequest {
  patientId: string

  doctorId: string

  appointmentDate: Date

  reason: string
}
```

---

## What Should NOT Be Included?

Do not include fields that are generated by the system.

Incorrect

```ts
export interface CreatePatientRequest {
  id: string

  createdAt: Date

  updatedAt: Date

  passwordHash: string
}
```

These values are created by the application or database, not by the client.

---

## Best Practices

- Create one Request Model per endpoint.
- Include only client-provided fields.
- Keep Request Models focused on input.
- Name models clearly.
- Keep them separate from Database Models.

---

## Common Mistakes

❌ Using Database Models as Request Models.

❌ Accepting system-generated fields.

❌ Reusing one Request Model for every endpoint.

❌ Mixing response fields into requests.

❌ Passing the raw HTTP request through the application.

---

## Summary

Request Models define the structure of incoming HTTP data.

They provide a clear contract between the client and the application, ensuring that Controllers receive only the information required for a specific operation.

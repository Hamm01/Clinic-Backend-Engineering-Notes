# 14.7 - Shared Types

## Overview

As applications grow, the same data structures are often used in multiple modules.

Instead of redefining these types in every file, we create **Shared Types** that can be reused throughout the application.

Shared Types improve consistency, reduce duplication, and make the codebase easier to maintain.

---

## Why Shared Types Exist

Imagine every Repository defining its own ID type.

```ts
type PatientId = string
```

```ts
type DoctorId = string
```

```ts
type AppointmentId = string
```

All of these represent the same concept—a unique identifier.

Rather than duplicating definitions, we create one shared type that can be reused across the project.

---

## Implementation

In our Clinic Backend, Shared Types are stored in a dedicated folder.

```text
src/

models/

└── shared/

    ├── id.type.ts

    ├── pagination.type.ts

    └── api-response.type.ts
```

Each file contains reusable types that are shared between multiple modules.

---

## Example

### Identifier Type

```ts
export type Id = string
```

Usage

```ts
export interface PatientResponse {
  id: Id

  firstName: string
}
```

---

### Pagination Type

```ts
export interface Pagination {
  page: number

  limit: number
}
```

This can be reused by every paginated API.

---

### API Response Type

```ts
export interface ApiResponse<T> {
  success: boolean

  data: T

  message: string
}
```

Usage

```ts
const response: ApiResponse<PatientResponse> = {
  success: true,

  data: patient,

  message: 'Patient fetched successfully'
}
```

---

## Request Lifecycle

Shared Types are not tied to a single layer.

They can be used across the application wherever a common structure is required.

```text
Controller

↓

Service

↓

Repository

↓

Shared Types
```

Unlike Request Models or DTOs, Shared Types are reusable across multiple layers.

---

## When to Create a Shared Type

Create a Shared Type when:

- The same type is used in multiple files.
- Multiple modules depend on the same structure.
- The type represents a common concept across the application.

Avoid creating Shared Types for objects that belong to only one feature.

---

## Best Practices

- Keep Shared Types generic.
- Reuse them across modules.
- Keep names descriptive.
- Store them in a dedicated folder.
- Remove duplicate type definitions.

---

## Common Mistakes

❌ Creating Shared Types for feature-specific objects.

❌ Duplicating the same type in multiple files.

❌ Mixing Shared Types with DTOs.

❌ Placing Shared Types inside Controllers or Services.

❌ Creating unnecessary abstractions.

---

## Summary

Shared Types define reusable TypeScript structures that are used throughout the application.

They reduce duplication, improve consistency, and provide a single source of truth for commonly used data structures.

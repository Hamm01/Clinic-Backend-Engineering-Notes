# 14.12 - Best Practices

## Overview

Well-designed models improve maintainability, readability, and scalability.

By following a few consistent practices, every layer of the application remains independent and easier to modify without affecting other parts of the system.

This chapter summarizes the practices followed throughout our Clinic Backend.

---

## Give Every Model One Responsibility

Each model should represent a single purpose.

Example

```text
CreatePatientRequest

↓

CreatePatientDto

↓

PatientResponse
```

Avoid using one model for multiple responsibilities.

---

## Organize Models by Feature

Keep all models related to the same business entity together.

```text
models/

patient/

├── create-patient.request.ts

├── create-patient.dto.ts

├── patient.response.ts

└── patient.validation.ts
```

This improves navigation as the project grows.

---

## Keep Layers Independent

Each layer should use its own model.

```text
Controller

↓

Request Model

↓

DTO

↓

Service

↓

Repository

↓

Prisma Model

↓

Database

↓

Response Model

↓

Client
```

Avoid sharing the same object across multiple layers.

---

## Name Models by Their Purpose

Choose descriptive names.

Correct

```text
CreatePatientRequest

UpdatePatientRequest

PatientResponse

CreatePatientDto
```

Avoid

```text
PatientModel

PatientData

PatientObject
```

The name should clearly describe the model's responsibility.

---

## Keep Shared Resources Centralized

Store reusable resources in dedicated folders.

```text
models/

shared/

enums/

constants/
```

This prevents duplication across features.

---

## Keep Validation Separate

Validation should remain inside dedicated validation schemas.

Correct

```text
create-patient.validation.ts
```

Avoid placing validation rules inside:

- DTOs
- Response Models
- Database Models

---

## Do Not Expose Database Models

Repositories return database models internally.

Services transform them into Response Models before sending data to clients.

```text
Prisma Model

↓

Service

↓

PatientResponse
```

Never expose raw database objects through the API.

---

## Keep Models Lightweight

Models should only describe data.

Avoid placing:

- Business logic
- Database queries
- Validation logic
- Utility methods

inside model files.

---

## Best Practices Checklist

✓ One responsibility per model.

✓ Organize models by feature.

✓ Separate Request, DTO, and Response models.

✓ Keep validation independent.

✓ Centralize shared types.

✓ Use meaningful names.

✓ Hide database models from clients.

✓ Keep models focused on data representation.

---

## Summary

Well-designed models provide a clean contract between different layers of the application.

Following these practices results in a backend that is easier to understand, maintain, and scale as new features are added.

# 14.6 - DTOs (Data Transfer Objects)

## Overview

A Data Transfer Object (DTO) represents the data exchanged between layers inside the application.

Unlike Request Models, which represent incoming HTTP requests, DTOs represent business data that Services use to perform application logic.

DTOs help decouple the Service layer from HTTP-specific concerns and database models.

---

## Why DTOs Exist

The Service layer should not depend on HTTP requests.

Similarly, it should not depend on database models.

Instead, Controllers convert incoming Request Models into DTOs before calling the Service.

This allows the Service to work with business data rather than HTTP-specific objects.

---

## Request Model vs DTO

Request Model

```text
Represents HTTP Input

↓

Used by Controllers
```

DTO

```text
Represents Business Data

↓

Used by Services
```

Although they may contain similar fields, they serve different responsibilities.

---

## Implementation

In our Clinic Backend, DTOs are stored separately.

```text
src/

models/

└── dto/

    └── create-patient.dto.ts
```

Example

```ts
export interface CreatePatientDto {
  firstName: string

  lastName: string

  email: string

  phone?: string
}
```

This DTO contains only the business data required to create a patient.

---

## How It Is Used

Controller

```ts
@Post()
async createPatient(

    @Body() request: CreatePatientRequest

) {

    const dto: CreatePatientDto = {

        firstName: request.firstName,

        lastName: request.lastName,

        email: request.email,

        phone: request.phone

    };

    return this.patientService.create(dto);

}
```

The Controller converts the Request Model into a DTO before calling the Service.

---

## Request Lifecycle

```text
Client

↓

Request Model

↓

Controller

↓

DTO

↓

Service

↓

Repository
```

The Service communicates using DTOs rather than HTTP models.

---

## Why Not Pass Request Models?

Consider a registration endpoint.

Request Model

```ts
export interface RegisterPatientRequest {
  firstName: string

  lastName: string

  email: string

  password: string

  confirmPassword: string
}
```

The Service does not need `confirmPassword`.

Instead, the Controller validates the request and creates a DTO.

```ts
export interface CreatePatientDto {
  firstName: string

  lastName: string

  email: string

  password: string
}
```

The DTO contains only the data required for business logic.

---

## Best Practices

- Create DTOs for Service operations.
- Keep DTOs independent of HTTP.
- Include only business data.
- Keep DTOs immutable whenever possible.
- Name DTOs based on the operation they perform.

---

## Common Mistakes

❌ Using Request Models as DTOs.

❌ Passing HTTP-specific objects into Services.

❌ Including unnecessary fields inside DTOs.

❌ Using one DTO for every operation.

❌ Returning DTOs directly as API responses.

---

## Summary

DTOs provide a clean contract between Controllers and Services.

They allow the business layer to remain independent of HTTP requests and database models, resulting in a more maintainable and scalable architecture.

# 14.8 - Enums

## Overview

Enums define a fixed set of allowed values.

Instead of using plain strings throughout the application, enums provide a type-safe way to represent values that never change.

Examples include:

- Appointment Status
- Patient Status
- User Roles
- Gender
- Payment Status

Using enums improves readability, consistency, and reduces runtime errors.

---

## Why Enums Exist

Imagine storing appointment status as a string.

```ts
const status = 'Completed'
```

Another developer writes

```ts
const status = 'completed'
```

Someone else writes

```ts
const status = 'COMPLETE'
```

Although they represent the same concept, they are different string values.

Enums solve this problem by providing one predefined set of values.

---

## Implementation

In our Clinic Backend, enums are stored in a dedicated folder.

```text
src/

models/

└── enums/

    ├── appointment-status.enum.ts

    ├── patient-status.enum.ts

    ├── role.enum.ts

    └── gender.enum.ts
```

Each enum represents one business concept.

---

## Example

Appointment Status

```ts
export enum AppointmentStatus {
  SCHEDULED = 'SCHEDULED',

  COMPLETED = 'COMPLETED',

  CANCELLED = 'CANCELLED'
}
```

Usage

```ts
const status = AppointmentStatus.SCHEDULED
```

This prevents invalid values from being assigned.

---

## Another Example

Role Enum

```ts
export enum UserRole {
  ADMIN = 'ADMIN',

  DOCTOR = 'DOCTOR',

  RECEPTIONIST = 'RECEPTIONIST'
}
```

Usage

```ts
if (user.role === UserRole.ADMIN) {
  // Allow access
}
```

---

## Using Enums in Prisma

Enums can also be defined inside `schema.prisma`.

```prisma
enum AppointmentStatus {

    SCHEDULED

    COMPLETED

    CANCELLED

}
```

Model

```prisma
model Appointment {

    id String @id @default(cuid())

    status AppointmentStatus

}
```

Prisma generates the corresponding TypeScript enum automatically.

## Request Lifecycle

Enums can be used across multiple layers.

```text
Controller

↓

Service

↓

Repository

↓

Prisma

↓

Database
```

Unlike Request Models or DTOs, enums represent fixed business values and can be shared throughout the application.

---

## Best Practices

- Create one enum for each business concept.
- Use descriptive enum names.
- Keep enum values consistent.
- Prefer enums over hardcoded strings.
- Reuse enums across the application.

---

## Common Mistakes

❌ Hardcoding string values.

```ts
status = 'completed'
```

❌ Creating duplicate enums.

❌ Using enums for values that frequently change.

❌ Mixing unrelated values in one enum.

❌ Creating enums for data that belongs in the database.

---

## Summary

Enums define a fixed set of allowed values for the application.

They improve type safety, eliminate magic strings, and provide consistency across Controllers, Services, Repositories, and the database.

| Use Case              | Enum | Constant |

| Roles                 | ✅   | ❌       |
| Status                | ✅   | ❌       |
| API URLs              | ❌   | ✅       |
| Cache TTL             | ❌   | ✅       |
| File paths            | ❌   | ✅       |

A simple rule to remember:

Enums answer: "Which values are allowed?"
Constants answer: "What value should the application use?"

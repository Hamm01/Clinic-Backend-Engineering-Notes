# 14.13 - Common Mistakes

## Overview

Models are meant to create clear boundaries between different layers of the application.

However, many projects lose these boundaries over time by reusing models incorrectly or mixing responsibilities.

This chapter highlights the most common mistakes developers make when designing Models and Shared Types.

---

## Using One Model Everywhere

One of the biggest mistakes is using the same model throughout the entire application.

Incorrect

```text
Request

↓

Controller

↓

Service

↓

Repository

↓

Database
```

Every layer shares the same object.

This tightly couples the application and makes future changes difficult.

Correct

```text
Request Model

↓

DTO

↓

Prisma Model

↓

Response Model
```

Each layer has its own representation of the data.

---

## Returning Database Models Directly

Avoid exposing Prisma models to API clients.

Incorrect

```ts
return prisma.patient.findUnique({
  where: {
    id
  }
})
```

This may expose internal fields such as:

- passwordHash
- createdAt
- updatedAt

Instead, map the database model to a Response Model before returning it.

---

## Reusing Request Models as DTOs

Although Request Models and DTOs may contain similar fields, they represent different responsibilities.

Incorrect

```text
Controller

↓

Request Model

↓

Service
```

Correct

```text
Controller

↓

Request Model

↓

DTO

↓

Service
```

The Service should receive business data, not HTTP-specific objects.

---

## Duplicating Shared Types

Avoid defining the same type multiple times.

Incorrect

```ts
type Id = string
```

inside several files.

Instead, create one shared definition.

```ts
export type Id = string
```

and reuse it across the application.

---

## Mixing Business Logic with Models

Models should only describe data.

Avoid writing business logic inside model files.

Incorrect

```ts
export interface Patient {
  firstName: string

  lastName: string

  calculateAge(): number
}
```

Business logic belongs in the Service layer.

---

## Creating Large Generic Models

Avoid creating one model for every operation.

Incorrect

```ts
interface Patient {
  id: string

  firstName: string

  lastName: string

  email: string

  password: string

  createdAt: Date

  updatedAt: Date
}
```

This model tries to satisfy every layer.

Instead, create purpose-specific models.

---

## Mixing Validation with Models

Validation rules should remain inside validation schemas.

Incorrect

```ts
interface Patient {
  firstName: string // Minimum 2 characters
}
```

Correct

```text
create-patient.validation.ts
```

Validation belongs in Zod schemas, not model definitions.

---

## Poor Naming Conventions

Use names that describe the model's responsibility.

Correct

```text
CreatePatientRequest

PatientResponse

CreatePatientDto
```

Avoid

```text
PatientData

PatientModel

PatientObject

TempPatient
```

Meaningful names improve readability.

---

## Common Mistakes Checklist

Avoid these mistakes:

- Using one model throughout the application.
- Returning Prisma models directly.
- Reusing Request Models as DTOs.
- Duplicating shared types.
- Mixing business logic with models.
- Creating generic models.
- Mixing validation with models.
- Using inconsistent naming.

---

## Summary

Well-designed models have a single responsibility and belong to a specific layer of the application.

Separating Request Models, DTOs, Response Models, Shared Types, and Validation Models results in a backend that is easier to maintain, extend, and test.

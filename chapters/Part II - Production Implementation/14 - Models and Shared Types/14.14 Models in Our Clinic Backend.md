# 14.14 - Models in Our Clinic Backend

## Overview

Throughout this module, we learned about different types of models and their responsibilities.

Rather than using a single object throughout the application, our Clinic Backend uses different models at different stages of the request lifecycle.

This separation keeps the application modular, maintainable, and easy to extend.

---

## Complete Architecture

Models are used throughout the request lifecycle.

```text
                    HTTP Request
                          │
                          ▼
                 Request Model
                          │
                          ▼
                 Validation (Zod)
                          │
                          ▼
                    Controller
                          │
                          ▼
                         DTO
                          │
                          ▼
                      Service
                          │
                          ▼
                    Repository
                          │
                          ▼
                 Prisma Database Model
                          │
                          ▼
                     PostgreSQL
                          │
                          ▲
                 Prisma Database Model
                          │
                          ▲
                    Repository
                          │
                          ▲
                      Service
                          │
                          ▼
                  Response Model
                          │
                          ▼
                    Controller
                          │
                          ▼
                    HTTP Response
```

Every model has a specific responsibility within this flow.

## Folder Structure

Our Clinic Backend organizes models by feature.

```text
src/

├── patient/

│   ├── create-patient.request.ts

│   ├── update-patient.request.ts

│   ├── create-patient.dto.ts

│   ├── patient.response.ts

│

├── doctor/

├── appointment/

├── diagnosis/

├── prescription/

├── shared/
```

Each feature owns its own models, while reusable resources remain shared.

Creating a new patient.

Valid data continues.

### Step 3

Controller


### Step 4

## Model Responsibilities

| Model            | Responsibility                                         |
| ---------------- | ------------------------------------------------------ |
| Request Model    | Represents incoming HTTP requests                      |
| Validation Model | Validates client input                                 |
| DTO              | Transfers business data between Controller and Service |
| Prisma Model     | Represents database entities                           |
| Response Model   | Represents outgoing API responses                      |
| Shared Types     | Reusable application types                             |
| Enums            | Fixed business values                                  |
| Constants        | Reusable application values                            |

Each model exists for one purpose.

---

## Key Principles

Throughout our Clinic Backend, we follow these principles.

- One responsibility per model.
- Separate HTTP from business logic.
- Separate business logic from persistence.
- Never expose database models directly.
- Organize models by feature.
- Keep shared resources centralized.
- Validate data before business logic.
- Return Response Models to clients.

These principles remain consistent across every feature in the application.

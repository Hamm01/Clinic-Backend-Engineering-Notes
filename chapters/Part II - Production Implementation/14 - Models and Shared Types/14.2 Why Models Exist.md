# 14.2 - Why Models Exist

## Overview

As applications grow, the same data passes through multiple layers.

A client sends an HTTP request.

The Controller receives it.

The Service processes the business logic.

The Repository communicates with the database.

Finally, the response is returned to the client.

Although the business entity remains the same, each layer has different responsibilities and therefore requires different data.

This is why we create different models instead of using a single object throughout the application.

---

## The Problem

Imagine creating a new patient.

The client sends the following request.

```json
{
  "firstName": "John",
  "lastName": "Doe",
  "email": "john@example.com",
  "password": "secret123"
}
```

If this same object is passed through every layer of the application, several problems arise.

- Unnecessary fields travel through the system.
- Internal data may be exposed.
- Business rules become difficult to enforce.
- Layers become tightly coupled.

---

## Different Layers Need Different Data

Each layer works with only the information it requires.

```text
Client
        │
        ▼
Request Model
        │
        ▼
Service (DTO)
        │
        ▼
Repository
        │
        ▼
Prisma Model
        │
        ▼
Database
        │
        ▲
Response Model
        │
        ▲
Client
```

Instead of one large object, every layer receives an object designed for its own responsibility.

## Implementation

Consider creating a patient.

### Incoming Request

```json
{
  "firstName": "John",
  "lastName": "Doe",
  "email": "john@example.com",
  "password": "secret123"
}
```

---

### Database Record

```text
Patient

id

firstName

lastName

email

passwordHash

createdAt

updatedAt
```

Notice that the database stores:

- Generated ID
- Password Hash
- Timestamps

These values were never part of the HTTP request.

---

### API Response

```json
{
  "id": "pat_123",
  "firstName": "John",
  "lastName": "Doe",
  "email": "john@example.com"
}
```

The password hash is intentionally omitted.

Different layers expose different data.

---

## Why Not Use One Model?

Using one model everywhere seems simple initially, but it creates several problems.

Example

```ts
interface Patient {
  id: string

  firstName: string

  lastName: string

  email: string

  passwordHash: string

  createdAt: Date

  updatedAt: Date
}
```

Problems:

- Clients should not send IDs.
- Clients should not know password hashes.
- Responses should not expose internal fields.
- Services often require different data than Controllers.

One model cannot satisfy every layer.

---

## Our Clinic Backend

Throughout this project, every business entity follows the same approach.

```text
Patient

↓

Request Model

↓

DTO

↓

Prisma Model

↓

Response Model
```

The same applies to:

- Doctor
- Appointment
- Diagnosis
- Prescription
- Clinic

This keeps the architecture consistent across the entire application.

---

## Best Practices

- Design models for a single responsibility.
- Keep layers independent.
- Never expose internal database fields.
- Pass only the data required by each layer.
- Keep model names meaningful and consistent.

---

## Common Mistakes

❌ Using one model throughout the application.

❌ Returning database objects directly to clients.

❌ Exposing sensitive fields in API responses.

❌ Mixing HTTP concerns with database models.

❌ Making every layer depend on the same object.

---

## Summary

Models exist because different layers of an application have different responsibilities.

Instead of sharing one object everywhere, production applications define separate models for requests, business logic, database operations, and responses.

This separation improves security, maintainability, and scalability.

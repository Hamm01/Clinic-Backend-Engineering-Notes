# 14.11 - Folder Structure

## Overview

As the application grows, the number of models also increases.

Every feature contains:

- Request Models
- Response Models
- DTOs
- Validation Schemas

Without a proper folder structure, these files become difficult to locate and maintain.

A well-organized structure improves readability, scalability, and developer productivity.

---

## The Problem

Imagine a project with:

- 30 entities
- 10 API endpoints per entity

Organizing files only by type results in folders containing hundreds of files.

Example

```text
requests/

├── create-patient.request.ts

├── update-patient.request.ts

├── create-doctor.request.ts

├── update-doctor.request.ts

├── create-appointment.request.ts

...
```

Finding related files becomes difficult.

---

## Feature-Based Organization

In our Clinic Backend, models are organized by feature.

```text
src/

models/

├── patient/

│   ├── create-patient.request.ts

│   ├── update-patient.request.ts

│   ├── patient.response.ts

│   ├── create-patient.dto.ts

│   └── patient.validation.ts

│

├── doctor/

│   ├── create-doctor.request.ts

│   ├── doctor.response.ts

│   ├── create-doctor.dto.ts

│   └── doctor.validation.ts

│

├── appointment/

│

├── diagnosis/

│

├── prescription/

│

├── shared/

├── enums/

└── constants/
```

Everything related to one business feature stays together.

---

## Why Feature-Based?

When implementing a new feature, developers usually work within a single business domain.

For example, implementing Patients requires:

- Request Model
- DTO
- Validation
- Response Model

All related files are located in one folder.

This reduces navigation time and improves maintainability.

---

## Comparison

### Type-Based Structure

```text
requests/

responses/

dto/

validation/
```

Pros

- Easy for small projects.
- Simple to understand.

Cons

- Hundreds of files in one folder.
- Difficult to navigate as the project grows.

---

### Feature-Based Structure

```text
patient/

doctor/

appointment/

diagnosis/
```

Pros

- Related files stay together.
- Easier to maintain.
- Scales well.
- Faster navigation.

Cons

- Slightly more folders.

---

## Shared Resources

Some models are reused across the entire application.

These remain outside individual features.

```text
models/

shared/

enums/

constants/
```

Examples include:

- Shared Types
- Enums
- Constants

These are available to every module.

---

## How It Fits Into Our Project

```text
src/

controllers/

services/

repositories/

models/

database/

routes/

middlewares/

utils/
```

The `models` directory contains only data representations.

Business logic remains inside Services.

Database access remains inside Repositories.

---

## Best Practices

- Organize models by feature.
- Keep related files together.
- Separate shared resources.
- Use consistent naming conventions.
- Avoid deeply nested folders.

---

## Common Mistakes

❌ Mixing unrelated features.

❌ Creating one folder with hundreds of model files.

❌ Duplicating shared types.

❌ Storing business logic inside model files.

❌ Using inconsistent folder names.

---

## Summary

A feature-based folder structure keeps the project organized as it grows.

By grouping Request Models, DTOs, Response Models, and Validation Schemas by business feature, developers can quickly locate and maintain related files while keeping shared resources centralized.

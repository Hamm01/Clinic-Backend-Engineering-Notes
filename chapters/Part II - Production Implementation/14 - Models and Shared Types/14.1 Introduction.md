# 14.1 - Introduction

## Overview

In the previous modules, we built the foundation of our backend application.

We learned how requests travel through the application, how business logic is organized, how data is stored using PostgreSQL and Prisma, and how Repositories interact with the database.

Now, one important question remains:

> **How should data be represented inside the application?**

The same business entity travels through multiple layers, and each layer has different responsibilities. Instead of using one object everywhere, production applications represent data differently depending on where it is being used.

This module focuses on designing those models and organizing them in a clean, maintainable, and scalable way.

---

## Why This Module Exists

Consider the flow of creating a new patient.

A client sends an HTTP request.

The Controller receives the request.

The Service processes the business logic.

The Repository stores the data.

Finally, the database saves the record.

Although the business entity is the same, every layer requires a different representation of that data.

Using separate models keeps each layer independent and prevents unnecessary coupling between different parts of the application.

---

## What We Will Build

Throughout this module, we will design different types of models used in our Clinic Backend.

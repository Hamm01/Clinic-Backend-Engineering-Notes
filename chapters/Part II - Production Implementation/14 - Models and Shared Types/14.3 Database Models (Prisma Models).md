# 14.3 - Database Models

## Overview

A Database Model represents how business data is stored inside the database.

In our Clinic Backend, Database Models are defined using Prisma and act as the source of truth for our application's data.

Every business entity such as a Patient, Doctor, Appointment, or Diagnosis has its own database model.

These models define:

- What data is stored.
- The data types.
- Relationships between entities.
- Database constraints.

---

## Why Database Models Exist

The database needs a structured way to store information.

Without database models, there would be no definition for:

- Which fields belong to a Patient.
- How Appointments connect Doctors and Patients.
- Which fields are required.
- Which fields must be unique.

Database Models provide this structure.

---

## Implementation

In our Clinic Backend, all database models are located inside:

```text
prisma/

└── schema.prisma
```

Example

```prisma
model Patient {

    id          String   @id @default(cuid())

    firstName   String

    lastName    String

    email       String   @unique

    phone       String?

    createdAt   DateTime @default(now())

    updatedAt   DateTime @updatedAt

}
```

This model defines how a Patient is stored in PostgreSQL.

---

## One Business Entity, One Database Model

Every major business entity has its own model.

```text
Clinic

Doctor

Patient

Appointment

Diagnosis

Prescription
```

Each model represents one database table.

---

## Database Model Responsibilities

Database Models are responsible for:

- Defining fields.
- Defining relationships.
- Defining constraints.
- Defining default values.
- Mapping business entities to database tables.

They are **not** responsible for:

- HTTP requests.
- API responses.
- Business logic.
- Validation.

Those responsibilities belong to other layers.

---

## Request Lifecycle

Database Models participate only in the Repository layer.

```text
Client

↓

Controller

↓

Service

↓

Repository

↓

Database Model (Prisma)

↓

PostgreSQL
```

Controllers and Services never communicate with Database Models directly.

---

## Example

Creating a patient.

Repository

```ts
return prisma.patient.create({
  data: {
    firstName,

    lastName,

    email
  }
})
```

The Repository uses the Patient Database Model to insert the record into PostgreSQL.

---

## Best Practices

- Create one model for each business entity.
- Keep model names singular.
- Define relationships clearly.
- Keep database models focused on persistence.
- Let Prisma manage the schema.

---

## Common Mistakes

❌ Mixing API fields into database models.

❌ Storing unnecessary information.

❌ Using one model for multiple business entities.

❌ Putting business logic inside database models.

❌ Accessing database models outside the Repository layer.

---

## Summary

Database Models define how business entities are stored in the database.

They are the foundation of the persistence layer and are used exclusively by Repositories to communicate with PostgreSQL through Prisma.

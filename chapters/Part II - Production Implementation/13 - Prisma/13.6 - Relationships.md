# 13.6 - Relationships

## Overview

In real applications, data is rarely stored in isolation.

Patients book appointments.

Doctors treat patients.

Appointments contain diagnoses.

These connections are called **relationships**.

Prisma represents relationships using the `@relation` attribute.

---

## Why Relationships?

Without relationships, every model would exist independently.

```
Patient

Doctor

Appointment
```

The application would have no way to know:

- Which doctor treated a patient.
- Which patient booked an appointment.
- Which diagnosis belongs to an appointment.

Relationships connect these models together.

---

## One-to-One Relationship

One record is connected to exactly one other record.

Example

```
Doctor

↓

DoctorProfile
```

Each doctor has one profile.

Each profile belongs to one doctor.

Example

```prisma
model Doctor {

    id      String @id @default(cuid())

    profile DoctorProfile?

}

model DoctorProfile {

    id       String @id @default(cuid())

    doctorId String @unique

    doctor   Doctor @relation(fields: [doctorId], references: [id])

}
```

---

## One-to-Many Relationship

One record can own many other records.

Example

```
Doctor

↓

Appointments
```

One doctor

↓

Many appointments

---

Example

```prisma
model Doctor {

    id String @id @default(cuid())

    appointments Appointment[]

}

model Appointment {

    id String @id @default(cuid())

    doctorId String

    doctor Doctor @relation(fields: [doctorId], references: [id])

}
```

---

## Many-to-Many Relationship

Many records can be connected to many other records.

Example

```
Doctor

↓

Specializations

↑
```

One doctor

↓

Many specializations

One specialization

↓

Many doctors

Example

```prisma
model Doctor {

    id String @id @default(cuid())

    specializations Specialization[]

}

model Specialization {

    id String @id @default(cuid())

    doctors Doctor[]

}
```

Prisma automatically creates the junction table.

---

## The @relation Attribute

Relationships are defined using `@relation`.

Example

```prisma
doctor Doctor

@relation(

    fields: [doctorId],

    references: [id]

)
```

Here,

- `fields` → Foreign Key inside the current model.
- `references` → Primary Key in the referenced model.

## Querying Related Data

Relationships allow Prisma to load related records.

Example

```ts
const appointment = await prisma.appointment.findUnique({
  where: {
    id
  },

  include: {
    doctor: true,

    patient: true
  }
})
```

Result

```text
Appointment

↓

Doctor

↓

Patient
```

Everything is returned in a single query.

---

## Implementation In Our Project

Example Appointment model.

```prisma
model Appointment {

    id String @id @default(cuid())

    doctorId String

    patientId String

    doctor Doctor

        @relation(fields: [doctorId], references: [id])

    patient Patient

        @relation(fields: [patientId], references: [id])

}
```

This allows our Appointment Repository to retrieve:

- Appointment
- Doctor
- Patient

using one Prisma query.

---

## Best Practices

- Use relationships instead of duplicate data.
- Name relation fields clearly.
- Keep foreign key names consistent.
- Use `include` only when related data is required.
- Avoid unnecessary nested relationships.

---

## Common Mistakes

❌ Duplicating related information.

❌ Forgetting the foreign key field.

❌ Overusing nested `include`.

❌ Creating circular relationships unnecessarily.

---

## Summary

Relationships connect business entities together.

Prisma uses the `@relation` attribute to define these connections while automatically generating a type-safe API for querying related data.

Proper relationships reduce data duplication and simplify querying complex business data.

# 13.14 - Common Mistakes

## Overview

Prisma simplifies database development, but incorrect usage can still lead to performance problems, runtime errors, and maintainability issues.

This chapter covers the most common mistakes developers make while using Prisma and how to avoid them.

---

## Creating Multiple Prisma Clients

One of the most common mistakes is creating a new `PrismaClient` every time a query is executed.

Incorrect

```ts
import { PrismaClient } from '@prisma/client'

const prisma = new PrismaClient()
```

inside multiple files.

This creates unnecessary database connections.

Correct

Create one shared instance.

```ts
// src/database/prisma.ts

import { PrismaClient } from '@prisma/client'

export const prisma = new PrismaClient()
```

Import this instance throughout the application.

---

## Using Prisma Outside the Repository

Incorrect

```text
Controller

↓

Prisma
```

or

```text
Service

↓

Prisma
```

Correct

```text
Controller

↓

Service

↓

Repository

↓

Prisma
```

Keeping Prisma inside the Repository layer maintains a clean architecture.

---

## Forgetting to Generate the Prisma Client

After modifying `schema.prisma`, developers sometimes forget to regenerate the client.

Incorrect

```
Modify schema.prisma

↓

Run Application
```

Correct

```bash
npx prisma generate
```

Always regenerate the Prisma Client after schema changes.

---

## Forgetting to Run Migrations

Updating the schema does not update the database.

Incorrect

```
schema.prisma

↓

Application
```

Correct

```
schema.prisma

↓

Migration

↓

Database
```

Run

```bash
npx prisma migrate dev --name meaningful-name
```

---

## Returning Every Column

Avoid returning unnecessary data.

Incorrect

```ts
await prisma.patient.findMany()
```

Correct

```ts
await prisma.patient.findMany({
  select: {
    id: true,

    firstName: true,

    email: true
  }
})
```

Returning only required fields improves performance.

---

## Overusing include

Loading every relationship increases query complexity.

Incorrect

```ts
include: {

    doctor: true,

    patient: true,

    diagnosis: true,

    clinic: true,

    prescriptions: true

}
```

Correct

Only include relationships required for the current operation.

---

## Ignoring Transactions

Multiple related database operations should execute together.

Incorrect

```ts
await prisma.patient.create(...);

await prisma.appointment.create(...);
```

If the second operation fails, the database becomes inconsistent.

Correct

```ts
await prisma.$transaction(async tx => {
  // Database operations
})
```

---

## Using Raw SQL Without a Reason

Prisma supports raw SQL.

```ts
await prisma.$queryRaw`SELECT * FROM "Patient"`
```

Use raw SQL only when Prisma cannot express the required query efficiently.

Prefer Prisma Client whenever possible.

---

## Ignoring Database Errors

Database operations may fail because of:

- Unique constraint violations
- Missing records
- Foreign key constraints

Always handle database errors appropriately instead of assuming every query succeeds.

---

## Not Disconnecting in Scripts

Application servers usually keep one Prisma Client alive.

However,

scripts should disconnect after execution.

```ts
await prisma.$disconnect()
```

This is especially important for:

- Seed scripts
- Import scripts
- Export scripts

---

## Common Mistakes Checklist

Avoid these mistakes:

- Multiple Prisma Clients
- Prisma outside Repositories
- Forgetting `prisma generate`
- Forgetting migrations
- Returning unnecessary fields
- Overusing `include`
- Ignoring transactions
- Unnecessary raw SQL
- Ignoring database errors
- Forgetting `$disconnect()` in scripts

---

## Summary

Most Prisma problems are caused by incorrect usage rather than limitations of the tool.

Using a shared Prisma Client, following the application's architecture, keeping migrations up to date, and writing efficient queries results in a stable and maintainable backend.

# 13.7 - Migrations

## Overview

A migration is a set of database changes generated from the Prisma schema.

Whenever we add, remove, or modify models, Prisma generates SQL that updates the database without requiring us to write SQL manually.

---

## Why Migrations?

Suppose we create a new model.

```prisma
model Patient {

    id   String @id @default(cuid())

    name String

}
```

This only updates `schema.prisma`.

The actual PostgreSQL database is still unchanged.

A migration synchronizes the schema with the database.

---

## Creating a Migration

After modifying the schema, create a migration.

```bash
npx prisma migrate dev --name init
```

Example

```bash
npx prisma migrate dev --name create-patient-table
```

Prisma will:

- Generate SQL
- Create a migration
- Apply it to the database
- Regenerate Prisma Client

---

## Generated Folder

Prisma creates a migrations folder.

```text
prisma/

├── migrations/

│   ├── 20260718103000_init/

│   │   └── migration.sql

│   │

│   └── migration_lock.toml

└── schema.prisma
```

Every migration has its own SQL file.

---

## Migration Workflow

The typical workflow is:

```text
Update schema.prisma

↓

Run Migration

↓

Database Updated

↓

Prisma Client Generated

↓

Application Ready
```

---

## Example

Suppose we add a phone number.

Before

```prisma
model Patient {

    id   String @id @default(cuid())

    name String

}
```

After

```prisma
model Patient {

    id    String @id @default(cuid())

    name  String

    phone String

}
```

Run

```bash
npx prisma migrate dev --name add-phone
```

The Patient table is updated automatically.

---

## Reset Database

During development, we sometimes need to reset the database.

```bash
npx prisma migrate reset
```

This command:

- Drops the database
- Recreates it
- Runs all migrations
- Executes the seed script (if configured)

> Use this only during development.

---

## Deploying Migrations

In production, do not use `migrate dev`.

Instead, apply existing migrations.

```bash
npx prisma migrate deploy
```

This executes pending migrations without creating new ones.

---

## Implementation In Our Project

Development workflow:

```text
Modify schema.prisma

↓

npx prisma migrate dev --name meaningful-name

↓

Repository Development

↓

Testing
```

Every schema change follows this process.

---

## Best Practices

- Give migrations meaningful names.
- Commit migration files to Git.
- Review generated SQL when possible.
- Run migrations before testing.
- Use `migrate deploy` in production.

---

## Common Mistakes

❌ Editing the database manually.

❌ Forgetting to run migrations.

❌ Deleting migration history.

❌ Using `migrate reset` in production.

❌ Giving migrations meaningless names like `test`.

---

## Useful Commands

Generate a migration

```bash
npx prisma migrate dev --name migration-name
```

Deploy migrations

```bash
npx prisma migrate deploy
```

Reset database

```bash
npx prisma migrate reset
```

Generate Prisma Client

```bash
npx prisma generate
```

View database in Prisma Studio

```bash
npx prisma studio
```

---

## Summary

Migrations keep the database synchronized with `schema.prisma`.

Instead of manually writing SQL, Prisma generates and applies the required database changes while maintaining a history of every schema update.

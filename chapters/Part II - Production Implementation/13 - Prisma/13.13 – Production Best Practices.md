# 13.13 - Production Best Practices

## Overview

Prisma makes database access simple, but using it correctly in production requires following a few important practices.

These practices improve:

- Performance
- Stability
- Maintainability
- Resource utilization

---

## Create a Single Prisma Client

Create only one `PrismaClient` instance for the entire application.

Example

```ts
import { PrismaClient } from '@prisma/client'

export const prisma = new PrismaClient()
```

Reuse this instance across all Repositories.

---

## Share the Prisma Client

Our project structure:

```text
src/

database/

└── prisma.ts
```

Repository

```ts
import { prisma } from '../database/prisma'
```

Never create a new client inside a Repository.

Incorrect

```ts
const prisma = new PrismaClient()
```

inside every Repository.

---

## Keep Prisma Inside Repositories

Correct Architecture

```text
Controller

↓

Service

↓

Repository

↓

Prisma
```

Avoid

```text
Controller

↓

Prisma
```

or

```text
Service

↓

Prisma
```

The Repository remains the only layer responsible for database communication.

---

## Select Only Required Fields

Instead of returning every column,

retrieve only what is required.

Example

```ts
await prisma.patient.findMany({
  select: {
    id: true,

    firstName: true,

    email: true
  }
})
```

Smaller queries improve performance.

---

## Avoid Unnecessary Relationships

Only load related data when needed.

Correct

```ts
include: {
  doctor: true
}
```

Avoid deeply nested includes unless the business requirement demands them.

---

## Handle Errors Properly

Wrap database operations appropriately.

```ts
try {
  await prisma.patient.create({
    data
  })
} catch (error) {
  throw error
}
```

Allow the Service layer to decide how errors should be handled.

---

## Close Connections

Long-running servers typically maintain one Prisma Client instance.

For scripts such as:

- Seed
- Import
- Export

disconnect after execution.

```ts
await prisma.$disconnect()
```

---

## Review Generated Queries

Prisma simplifies SQL generation,

but developers should still understand the queries being executed.

Large joins,

missing indexes,

and unnecessary data retrieval can still affect performance.

---

## Keep Models Consistent

Maintain consistent naming.

Correct

```text
Patient

Doctor

Appointment
```

Avoid

```text
patient

PATIENT

tbl_patient
```

Consistency improves readability.

---

## Organize Repository Methods

Group related queries together.

Example

```text
PatientRepository

↓

create()

findById()

findByEmail()

update()

delete()
```

Avoid mixing unrelated business operations.

---

## Implementation In Our Project

Clinic Backend

```text
src/

database/

└── prisma.ts

↓

Repositories

↓

Services

↓

Controllers
```

Every database request follows the same architecture.

---

## Best Practices Checklist

✓ One Prisma Client

✓ Shared client instance

✓ Prisma only inside Repositories

✓ Select only required fields

✓ Load relations only when needed

✓ Handle errors correctly

✓ Disconnect in scripts

✓ Keep models consistent

✓ Review generated SQL

---

## Common Mistakes

❌ Creating multiple Prisma Clients.

❌ Accessing Prisma from Controllers.

❌ Returning unnecessary columns.

❌ Loading every relationship.

❌ Ignoring generated SQL.

❌ Forgetting to disconnect in scripts.

---

## Summary

Production applications use Prisma as a lightweight and efficient database client.

Following these practices helps maintain a clean architecture, reduces resource usage, and keeps database interactions predictable and maintainable.

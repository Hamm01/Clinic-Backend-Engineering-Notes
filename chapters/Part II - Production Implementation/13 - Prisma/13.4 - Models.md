# 13.4 - Defining Models

## Overview

Models define the structure of your application's data.

In Prisma, every model represents a table in the database.

Each field inside a model becomes a column in that table.

---

## Syntax

A model is declared using the `model` keyword.

```prisma
model Patient {

}
```

The model name should always be singular and written in PascalCase.

Examples:

```prisma
model Patient {}

model Doctor {}

model Appointment {}
```

---

## Defining Fields

Each model contains one or more fields.

```prisma
model Patient {

    id    String

    name  String

    email String

}
```

Each field has:

- Name
- Data Type
- Optional Attributes

---

## Basic Example

```prisma
model Patient {

    id        String

    firstName String

    lastName  String

    email     String

    phone     String

}
```

This defines five columns inside the Patient table.

## Model Naming Convention

Use:

- Singular names
- PascalCase
- Meaningful names

Correct

```prisma
model Patient {}

model Doctor {}

model Appointment {}
```

Avoid

```prisma
model Patients {}

model doctor {}

model tbl_patient {}
```

---

## Database Mapping

Prisma automatically maps models to database tables.

Example

```prisma
model Patient {

    id   String

    name String

}
```

Database

```text
Patient

----------------

id

name
```

## Folder Structure

Our project remains:

```text

└── schema.prisma

src/

├── database/

├── repositories/

└── services/
```

Only `schema.prisma` contains model definitions.

---

## Best Practices

- One model for one business entity.
- Use singular names.
- Keep names meaningful.
- Group related fields together.
- Keep formatting consistent.

---

## Common Mistakes

❌ Using plural model names.

❌ Using inconsistent naming.

❌ Creating duplicate models.

❌ Mixing unrelated data into one model.

---

## Summary

Models are the foundation of every Prisma project.

Each model represents a database table,

and each field represents a column.

Well-designed models lead to a well-structured database.

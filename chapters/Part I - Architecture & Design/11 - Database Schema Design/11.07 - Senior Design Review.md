# 11.07 - Senior Database Design Review

> Good database design is not measured by how many tables you create.
>
> It is measured by how many future problems you prevent.

---

At this stage, our database architecture is complete.

We have designed

- Business Entities

- ER Diagram
- Primary Keys

- Indexes

---

# How Senior Engineers Review a Database

Junior engineers often ask

> "Does this schema work?"

Senior engineers ask

> "What problems will this schema create in two years?"

That difference changes everything.

---

# Review 1 — Does Every Table Represent a Business Concept?

Ask yourself

```
Why does this table exist?
```

Good examples

- Doctor
- Patient
- Appointment
- Consultation
- Prescription

Poor examples

- Dashboard
- HomeScreen
- LoginPage

A database models the business, not the frontend.

---

# Review 2 — Does Every Table Have One Responsibility?

Example

Bad

```
Patient

↓

Appointments

↓

Consultations

↓

Payments

↓

Medicines

↓

Billing
```

One table is trying to solve many different problems.

Better

```
Patient

Appointment

Consultation

Prescription

Invoice
```

Each table has a single responsibility.

---

# Review 3 — Can Every Relationship Be Explained?

Never create relationships because "it might be useful."

Every relationship should answer a business question.

Example

```
Appointment

↓

Doctor
```

Question

Why?

Answer

Appointments belong to doctors.

Now ask

```
Medicine

↓

Clinic
```

Why?

There is no business reason.

Do not create the relationship.

---

# Review 4 — Is Any Information Duplicated?

Example

```
Prescription

patientId
```

and

```
Prescription

↓

Consultation

↓

Appointment

↓

Patient
```

Now patient information exists twice.

This creates inconsistency.

Keep one source of truth.

---

# Review 5 — Are Responsibilities in the Correct Layer?

Question

Should this rule belong to

- Database
- Repository
- Service

Example

```
Email must be unique.
```

Database.

Example

```
Doctor may only prescribe controlled medicine after approval.
```

Service Layer.

The database protects data.

The Service Layer protects business rules.

---

# Review 6 — Can the Schema Support Business Growth?

Today's clinic

```
One Clinic
```

Tomorrow

```
Five Clinics
```

Can the schema support this?

Today's doctors

```
One Clinic
```

Future

```
Multiple Clinics
```

Would that require redesign?

Think about realistic business growth.

Do not over-engineer for imaginary features.

---

# Review 7 — Does Every Query Have a Home?

Think about common application screens.

Doctor Dashboard

↓

Appointments

Patient Profile

↓

Medical History

Consultation Screen

↓

Prescriptions

Every important screen should map naturally to the schema.

If a simple screen requires ten complex joins,

the design deserves another review.

---

# Review 8 — Are Important Queries Indexed?

Ask

```
Which queries execute thousands of times every day?
```

Examples

- Doctor login
- Patient search
- Today's appointments
- Patient consultation history

Indexes should support these queries.

Not every column needs an index.

---

# Review 9 — Can We Explain Every Delete?

For every table ask

```
What happens if this record is deleted?
```

Patient

↓

Soft Delete

Doctor

↓

Soft Delete

OTP

↓

Hard Delete

Deletion strategy should match business value.

---

# Review 10 — Is History Preserved?

Healthcare systems rely on historical information.

Questions such as

- What medicine was prescribed six months ago?
- Who updated this diagnosis?
- Which doctor handled this consultation?

must always be answerable.

History is part of the product.

Not an optional feature.

---

# Review 11 — Can Another Engineer Understand This Schema?

Good architecture should communicate itself.

A new developer should understand

- what each table represents
- why relationships exist
- where business logic belongs

without reading thousands of lines of code.

If documentation is required to explain every relationship,

the schema is probably too complicated.

---

# Review 12 — Does the Database Reflect the Business?

This is the most important question.

Imagine the clinic closes tomorrow.

The database should still describe

- Doctors
- Patients
- Appointments
- Consultations
- Prescriptions

These concepts exist independently of software.

That is a sign of a healthy domain model.

---

# Common Architecture Smells

## Database Designed Around the UI

```
Dashboard Table

Settings Table

Profile Screen Table
```

Frontend changes frequently.

Business changes slowly.

Design for the business.

---

## Everything Stored in One Table

Large tables often indicate poor boundaries.

Split responsibilities.

---

## Missing Relationships

Without relationships

the database cannot guarantee integrity.

The application becomes responsible for correctness.

---

## Relationships Without Purpose

Every foreign key increases complexity.

Only create one when it answers a business question.

---

## Premature Optimization

Avoid adding extra tables,

indexes,

or relationships

for hypothetical future features.

Optimize when business requirements justify it.

---

# Database Review Checklist

Before implementation begins, verify

## Business

- [ ] Every table represents a business entity.
- [ ] Every entity has one responsibility.
- [ ] Every relationship has a business reason.

---

## Data Integrity

- [ ] Primary Keys defined.
- [ ] Foreign Keys defined.
- [ ] Constraints applied where appropriate.
- [ ] Duplicate data avoided.

---

## Performance

- [ ] Frequently queried columns indexed.
- [ ] Composite indexes justified.
- [ ] No unnecessary indexes.

---

## Maintainability

- [ ] Audit fields included where required.
- [ ] Soft delete strategy documented.
- [ ] Responsibilities separated cleanly.

---

## Scalability

- [ ] Schema supports expected business growth.
- [ ] No obvious bottlenecks.
- [ ] Business rules remain outside the database where appropriate.

# Summary

A production-ready backend begins long before the first controller or repository is created.

Strong architecture comes from understanding the business, defining clear responsibilities, protecting data integrity, and designing for maintainability.

The database should be viewed as the long-term representation of the business, while the application code becomes one possible way to interact with it.

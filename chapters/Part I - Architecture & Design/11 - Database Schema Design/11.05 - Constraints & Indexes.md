# 11.05 - Constraints & Indexes

> A good database does not trust the application.
>
> It protects itself.

---

# Objective

Our application already validates requests.

Our services already implement business rules.

So why does the database still need constraints?

Because applications can contain bugs.

Multiple services may write to the same database.

Background jobs may insert data.

Database migration scripts may bypass application logic.

The database is the final guardian of data integrity.

Constraints protect correctness.

Indexes protect performance.

Both are essential.

---

# Understanding the Layers

A backend has multiple layers of protection.

```
HTTP Request

↓

Validation

↓

Business Logic

↓

Repository

↓

Database Constraints
```

Notice something.

Even if every layer above the database fails,

the database should still reject invalid data.

---

# What is a Constraint?

A constraint is a rule enforced by the database.

Unlike validation,

constraints cannot be bypassed by application code.

Example

```
Patient Email

UNIQUE
```

Even if two backend servers accidentally attempt to insert the same email,

the database prevents duplicates.

---

# Why Constraints Exist

Imagine two receptionists register the same patient simultaneously.

Server A

```
Check Email

↓

Not Found
```

Server B

```
Check Email

↓

Not Found
```

Both now attempt

```
INSERT
```

Without a UNIQUE constraint,

both inserts succeed.

Now the clinic has duplicate patients.

Application validation was correct.

Concurrency created the problem.

The database must protect itself.

---

# Engineering Note

Validation improves user experience.

Constraints guarantee correctness.

Never assume validation alone is enough.

---

# Common Constraints

Most relational databases support several types of constraints.

---

## PRIMARY KEY

Guarantees

```
Identity
```

Example

```
doctorId
```

Every table should have exactly one primary key.

---

## FOREIGN KEY

Guarantees

```
Relationship Integrity
```

Example

```
appointment.doctorId
```

must reference an existing doctor.

Without this,

appointments could belong to doctors who do not exist.

---

## UNIQUE

Guarantees

```
No duplicates
```

Example

```
Doctor License Number
```

Every doctor must have a unique medical license.

---

## NOT NULL

Guarantees

```
Required Information
```

Example

```
Patient Name
```

A patient without a name is invalid.

The database should reject it.

---

## CHECK Constraint

Guarantees

```
Allowed Values
```

Example

```
Age >= 0
```

or

```
Appointment Duration > 0
```

The database refuses impossible values.

---

# Which Rules Belong in the Database?

Not every business rule should become a database constraint.

Consider

```
Doctor must belong to clinic.
```

Good candidate.

Relationship rule.

Now consider

```
Doctor can only prescribe antibiotics after lab confirmation.
```

Bad database constraint.

This is business logic.

It belongs in the Service Layer.

---

# Rule of Thumb

Database constraints protect

- identity
- relationships
- required fields
- data consistency

Services protect

- business workflows
- permissions
- policies
- calculations
- approvals

Do not confuse these responsibilities.

---

# Constraint Examples for Our Clinic

Patient

```
patientId

PRIMARY KEY
```

Doctor

```
licenseNumber

UNIQUE
```

Appointment

```
doctorId

FOREIGN KEY
```

Patient

```
name

NOT NULL
```

Consultation

```
appointmentId

UNIQUE
```

One appointment should only produce one consultation.

---

# What is an Index?

An index is a data structure that allows the database to locate records more efficiently.

Without an index,

the database may inspect every row.

With an index,

it can navigate directly to the required records.

Indexes improve read performance.

---

# Why Indexes Exist

Imagine

```
Appointment

5,000,000 rows
```

Doctor opens today's schedule.

Without an index,

the database scans

```
5,000,000 rows
```

to locate today's appointments.

With an index,

the database jumps directly to the relevant records.

---

# Engineering Note

Indexes are created because of queries,

not because of tables.

Always study how the application reads data.

---

# Which Columns Should Be Indexed?

Never ask

> Which columns exist?

Ask

> Which queries run every day?

---

## Example 1

Doctor Dashboard

Shows

```
Today's Appointments
```

Query

```
WHERE

doctorId

appointmentDate
```

This suggests a composite index.

---

## Example 2

Patient Search

```
Phone Number
```

Patients are frequently searched by phone.

Index

```
phone
```

---

## Example 3

Doctor Login

```
email
```

Every login searches by email.

Email should be indexed.

Usually,

it is also UNIQUE.

---

# Composite Indexes

Sometimes one column is not enough.

Example

```
doctorId

appointmentDate
```

Together they answer

```
Doctor's appointments for today.
```

A composite index is more efficient than two independent indexes.

Indexes should match real query patterns.

---

# Too Many Indexes

Indexes are not free.

Every insert,

update,

or delete

must also update every related index.

More indexes mean

- larger storage
- slower writes
- more maintenance

Only create indexes that support real business queries.

---

# Engineering Note

The fastest database is not the one with the most indexes.

It is the one with the right indexes.

---

# Constraints vs Validation

Example

Patient Registration

Validation

```
Valid Email Format
```

Constraint

```
Email UNIQUE
```

Validation checks the request.

The database guarantees the rule.

Both are required.

---

# Constraints vs Business Logic

Example

```
Appointment Date

Must be in future.
```

Should this be a CHECK constraint?

Usually,

No.

Business rules change.

The Service Layer should own this decision.

Database constraints should remain stable.

---

# Common Mistakes

## Every Column Indexed

Wrong.

Many unused indexes slow down writes.

---

## No Indexes

Queries eventually become slow as the database grows.

Performance problems appear only after production traffic increases.

---

## Business Rules as Constraints

Complex business decisions rarely belong inside the database.

Keep constraints simple and universal.

---

## Missing UNIQUE Constraints

Checking uniqueness only in application code creates race conditions.

Always enforce uniqueness where required.

---

# Suggested Indexes for Our Clinic

| Table            | Suggested Index           | Reason                |
| ---------------- | ------------------------- | --------------------- |
| Doctor           | email                     | Login                 |
| Doctor           | licenseNumber             | Unique identification |
| Patient          | phone                     | Patient search        |
| Appointment      | doctorId, appointmentDate | Doctor schedule       |
| Appointment      | patientId                 | Patient history       |
| Consultation     | appointmentId             | Consultation lookup   |
| PrescriptionItem | prescriptionId            | Prescription details  |
| Medicine         | name                      | Medicine search       |

Indexes should always be justified by a business query.

---

# Senior Engineer Checklist

Before approving indexes,

ask

- Which queries justify this index?
- Does this column participate in filtering?
- Does this column participate in joins?
- Does this column require uniqueness?
- Will this index slow down writes?
- Can two indexes be replaced with one composite index?

If every index has a business reason,

the design is likely correct.

---

# Summary

Constraints protect correctness.

Indexes protect performance.

Constraints ensure invalid data never enters the database.

Indexes ensure valid data can be retrieved efficiently.

Both should be designed from the application's business requirements and query patterns—not added arbitrarily.

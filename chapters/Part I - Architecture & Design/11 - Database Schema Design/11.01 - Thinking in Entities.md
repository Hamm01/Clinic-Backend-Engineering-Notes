# 11.01 - Thinking in Entities

> Before designing tables, think in **business entities**.
>
> A database should model **how the business works**, not how the code is written.

---

# Objective

Most developers open Prisma or PostgreSQL and immediately start creating tables.

A backend engineer starts much earlier.

The first question is never:

> "What tables do I need?"

Instead, it is:

> "What objects exist in this business?"

Only after identifying those objects do we think about tables, foreign keys, or indexes.

---

# The Wrong Approach

A common mistake is designing the database from the application's screens.

Example

```
Login Page

↓

Create User Table

↓

Dashboard

↓

Create Dashboard Table

↓

Appointment Page

↓

Create Appointment Table
```

This creates a database that follows the UI instead of the business.

The UI changes frequently.

The business changes much less often.

Design for the business.

---

# The Correct Approach

Instead of asking

```
Which API am I building?
```

Ask

```
What does this clinic actually manage?
```

Imagine a real clinic without software.

What physical things, people, or records exist?

Those become your entities.

---

# Understanding Our Clinic Business

Suppose we walk into a clinic.

We observe the daily operations.

```
Patient arrives

↓

Reception registers patient

↓

Appointment is booked

↓

Doctor sees patient

↓

Consultation happens

↓

Doctor writes diagnosis

↓

Medicines are prescribed

↓

Lab tests may be recommended

↓

Patient returns later

↓

Previous history is viewed
```

Now ask yourself:

**Which of these are real business objects?**

---

# Step 1 — Identify the Nouns

A useful technique is to underline every noun while discussing the business.

Example

```
Doctor

Patient

Clinic

Appointment

Consultation

Diagnosis

Prescription

Medicine

Lab Test
```

Most of these nouns become entities.

Not every noun becomes a table, but every table usually begins as a noun.

---

# Step 2 — Separate Business Entities from Technical Objects

Some things exist because of the business.

Others exist only because of technology.

Business Entities

```
Clinic

Doctor

Patient

Appointment

Consultation

Medicine

Prescription

Lab Test
```

Technical Objects

```
JWT

Express Request

HTTP Response

Redis Cache

Logger

Access Token

Middleware
```

Notice something.

Technical objects never become database entities.

They help the application run.

They are not part of the business domain.

---

# Our Core Business Entities

For this project, we will begin with the following entities.

```
Clinic

Doctor

Patient

Appointment

Consultation

Diagnosis

Prescription

Medicine

Lab Test
```

Each one represents something the business actually cares about.

Let's understand why.

---

# Clinic

A clinic represents a physical healthcare organization.

A clinic

- employs doctors
- registers patients
- schedules appointments
- stores consultation history

The clinic is the top-level business entity.

---

# Doctor

Doctors perform medical work.

Responsibilities

- Login to the portal
- View patients
- Write consultations
- Prescribe medicines
- Recommend lab tests

Notice something.

Doctors perform actions.

They generate business records.

---

# Patient

Patients receive healthcare.

Patients

- register once
- visit multiple times
- have multiple appointments
- have consultation history
- receive prescriptions

The patient is one of the most important long-lived entities.

Their medical history grows over time.

---

# Appointment

Appointments represent scheduled visits.

An appointment answers questions like

- Which patient?
- Which doctor?
- Which clinic?
- What date?
- What time?
- Current status?

Appointments are scheduling records.

They are not medical records.

---

# Consultation

A consultation represents the actual medical interaction.

It stores

- symptoms
- observations
- diagnosis
- doctor's notes

The consultation is the medical record created after an appointment.

---

# Diagnosis

Diagnosis represents the doctor's medical conclusion.

Example

```
Viral Fever

Type 2 Diabetes

Migraine

Hypertension
```

A diagnosis explains the patient's condition.

It belongs to a consultation.

---

# Prescription

A prescription stores what the doctor advises the patient to take.

It contains

- medicines
- dosage
- duration
- frequency
- instructions

The prescription belongs to a consultation.

Not directly to a patient.

---

# Medicine

Medicines are master data.

Examples

```
Paracetamol

Amoxicillin

Ibuprofen
```

A medicine exists independently.

Doctors simply reference it while creating prescriptions.

---

# Lab Test

Doctors sometimes recommend laboratory investigations.

Examples

```
CBC

Blood Sugar

Liver Function Test

X-Ray

MRI
```

Lab tests belong to a consultation because they are recommended during that visit.

---

# Engineering Note

Ask yourself:

> "Can this object exist even if my application is turned off?"

If the answer is **Yes**, it is likely a business entity.

Example

```
Doctor

✓ Exists
```

```
Patient

✓ Exists
```

```
JWT

✗ Exists only because of software
```

This simple question helps separate business concepts from technical implementation.

---

# What We Are NOT Creating Yet

Notice that we still have not discussed

- Tables
- Columns
- Primary Keys
- Foreign Keys
- Indexes
- Prisma Models

Those are implementation details.

At this stage, we only care about understanding the business.

---

# How a Backend Engineer Thinks

Most beginners think

```
API

↓

Database

↓

Business
```

A backend engineer thinks

```
Business

↓

Entities

↓

Relationships

↓

Database

↓

Code

↓

API
```

The order matters.

If you reverse it, the architecture usually becomes harder to maintain.

---

# Entity Ownership

Each entity has one clear responsibility.

| Entity       | Responsibility                        |
| ------------ | ------------------------------------- |
| Clinic       | Owns healthcare operations            |
| Doctor       | Provides medical care                 |
| Patient      | Receives medical care                 |
| Appointment  | Schedules a visit                     |
| Consultation | Records the medical visit             |
| Diagnosis    | Describes the medical condition       |
| Prescription | Stores prescribed treatment           |
| Medicine     | Represents available medicines        |
| Lab Test     | Represents recommended investigations |

Notice that none of these responsibilities overlap.

Each entity exists for one reason.

---

# Common Mistakes

## Creating Tables Too Early

Wrong

```
Open Prisma

↓

Start Writing Models
```

Without understanding the business, the schema often needs major redesign later.

---

## Designing from the UI

Screens change.

Business processes remain relatively stable.

Never let the frontend define the database.

---

## Mixing Technical Objects with Business Entities

Wrong

```
JWT Table

Session Table

Express Request Table
```

These are implementation details.

Not business concepts.

---

## One Entity Doing Everything

Example

```
Patient

↓

Appointments

↓

Billing

↓

Consultations

↓

Medicines

↓

Payments
```

This leads to oversized entities with multiple responsibilities.

Each business capability deserves its own entity.

---

# Senior Engineer Checklist

Before creating any table, ask

- Does this object exist in the real business?
- Is this a business concept or a technical concept?
- Does this entity have a single responsibility?
- Can I explain why this entity exists without mentioning the database?
- Would this entity still exist if the software did not?

If every answer is clear,

the entity is probably valid.

---

# Summary

Entities are the foundation of the database.

They represent the real objects that exist within the business, not the implementation details of the application.

By identifying entities first, the database naturally models the business instead of the user interface or the framework.

A well-designed backend always begins with understanding the business domain before creating tables or writing code.

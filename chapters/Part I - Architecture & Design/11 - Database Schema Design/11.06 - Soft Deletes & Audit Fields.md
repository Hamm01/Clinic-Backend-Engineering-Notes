# 11.06 - Soft Deletes & Audit Fields

> In production systems, deleting data is rarely the hardest problem.
>
> **Recovering deleted data is.**

---

# Objective

So far we have designed

- Business Entities
- Relationships
- Primary Keys
- Foreign Keys
- Constraints
- Indexes

Now we need to answer another architectural question.

> **What happens when someone deletes data?**

The answer is usually different between a learning project and a production system.

A beginner thinks

```
DELETE FROM patients;
```

A senior engineer asks

- Should this record ever disappear?
- Can we recover it?
- Is deleting it even legal?
- Do other records depend on it?
- Who performed the deletion?

---

# Why This Matters

Imagine a real clinic.

A receptionist accidentally deletes a patient.

Now imagine the patient returns after six months.

Questions immediately arise.

- Where is the patient's history?
- Where are previous prescriptions?
- Where are lab reports?
- Who deleted the record?
- Can it be restored?

If the answer is "No",

the system has failed the business.

---

# Hard Delete

A hard delete permanently removes a row.

```
Patient

↓

DELETE

↓

Gone Forever
```

SQL

```sql
DELETE FROM patients
WHERE patient_id = 'pat_001';
```

Once committed,

the record no longer exists.

---

# When Hard Deletes Are Acceptable

Hard deletes are appropriate when the data has no long-term business value.

Examples

```
Temporary OTP

Email Verification Token

Refresh Token

Cache Entries

Expired Sessions
```

These records are temporary.

Keeping them forever serves little purpose.

---

# When Hard Deletes Are Dangerous

Healthcare data

Financial transactions

Invoices

Medical consultations

Audit logs

Patient history

Prescription records

These should rarely disappear.

Deleting them may create legal, operational, or compliance problems.

---

# Soft Delete

Instead of removing the row,

mark it as deleted.

```
Patient

↓

deletedAt = current_timestamp

↓

Hidden From Normal Queries
```

The data still exists.

The application simply ignores it unless explicitly requested.

---

# Visual Example

Instead of

| patientId | Name  |
| --------- | ----- |
| P001      | Rahul |

After deletion

| patientId | Name  | deletedAt        |
| --------- | ----- | ---------------- |
| P001      | Rahul | 2026-07-12 15:45 |

Nothing was removed.

Only the status changed.

---

# Why Soft Deletes Exist

Imagine

```
Doctor

↓

Consultation

↓

Prescription
```

If the patient is deleted,

should the consultation disappear?

Should prescriptions disappear?

Should analytics become incorrect?

Usually,

No.

Historical records still matter.

---

# Engineering Principle

Delete means

> "The business no longer wants to use this record."

It does **not** necessarily mean

> "Destroy the record forever."

---

# Typical Soft Delete Columns

Most enterprise applications include

```text
deletedAt

deletedBy
```

Some applications also use

```text
isDeleted
```

Example

| Column    | Purpose              |
| --------- | -------------------- |
| deletedAt | When was it deleted? |
| deletedBy | Who deleted it?      |

Avoid using only

```
isDeleted
```

A timestamp provides much more information.

---

# How Queries Change

Without soft deletes

```sql
SELECT *
FROM patients;
```

With soft deletes

```sql
SELECT *
FROM patients
WHERE deleted_at IS NULL;
```

Every repository should automatically ignore deleted records unless explicitly requested.

The Service Layer should not need to remember this rule repeatedly.

---

# Engineering Note

Soft delete behavior belongs in the Repository Layer,

not scattered across controllers or services.

Repositories provide a consistent view of active data.

---

# Can Soft Deleted Records Be Restored?

Yes.

Instead of inserting a new row,

restore the existing one.

Example

```
deletedAt = NULL

deletedBy = NULL
```

The record becomes active again.

No business history is lost.

---

# What About Child Records?

Suppose

```
Patient

↓

Appointments

↓

Consultations

↓

Prescriptions
```

If the patient is soft deleted,

should every child record also be deleted?

Usually,

No.

Historical records remain valid.

The patient simply becomes inactive.

---

# Engineering Decision

In healthcare systems,

medical history should survive even if a patient account is deactivated.

The business values history more than deletion.

---

# Audit Fields

Deleting data is only part of accountability.

Production systems also answer

- Who created this?
- Who modified this?
- When?
- Who deleted it?

This information is called an **audit trail**.

---

# Common Audit Columns

Most enterprise tables contain

| Column    | Purpose                           |
| --------- | --------------------------------- |
| createdAt | Creation timestamp                |
| updatedAt | Last modification timestamp       |
| createdBy | User who created the record       |
| updatedBy | User who last modified the record |
| deletedAt | Soft delete timestamp             |
| deletedBy | User who deleted the record       |

Together,

these fields tell the complete story of a record.

---

# Example

Patient

```
Created

↓

Receptionist A
```

Later

```
Updated

↓

Receptionist B
```

Later

```
Deleted

↓

Administrator
```

The database now records

the entire lifecycle.

---

# Why Audit Fields Matter

Imagine a doctor asks

> "Who changed the patient's allergy information yesterday?"

Without audit fields,

there is no answer.

With audit fields,

the answer is immediate.

---

# Engineering Note

Audit fields are often more valuable than the business data itself.

They explain

**how** the data reached its current state.

---

# createdAt vs updatedAt

```
createdAt
```

Never changes.

```
updatedAt
```

Changes every time the record is modified.

These timestamps are useful for

- synchronization
- reporting
- debugging
- analytics

---

# createdBy vs updatedBy

These fields identify

the authenticated user responsible for the change.

Example

```
Doctor

Receptionist

Administrator
```

Knowing **who** performed an action is often as important as knowing **what** changed.

---

# Soft Delete vs Status

A common mistake is using

```
status = inactive
```

instead of soft delete.

These concepts are different.

Inactive

```
Business State
```

Deleted

```
Record Lifecycle
```

A patient may become inactive without being deleted.

Do not mix these concepts.

---

# Common Mistakes

## Physically Deleting Important Data

Wrong

```
DELETE Consultation
```

Medical records should usually be preserved.

---

## Forgetting Audit Fields

Without audit information,

debugging and accountability become difficult.

---

## Using Only isDeleted

A boolean answers

```
Was it deleted?
```

A timestamp answers

```
When was it deleted?
```

Choose the richer information.

---

## Repeating Soft Delete Logic Everywhere

Wrong

Controller

↓

Service

↓

Repository

↓

WHERE deletedAt IS NULL

This rule should live in one place.

Usually,

the Repository Layer.

---

# Should Every Table Support Soft Deletes?

No.

Examples

Suitable

- Patient
- Doctor
- Appointment
- Consultation
- Prescription

Usually unnecessary

- OTP
- Sessions
- Cache
- Verification Tokens
- Temporary Uploads

Architecture should match business value.

---

# Senior Engineer Checklist

Before approving a table,

ask

- Can this record be safely deleted?
- Should it be recoverable?
- Will another record depend on it later?
- Do we need to know who changed it?
- Is an audit trail required?
- Is this data legally important?

If the answer is "Yes, history matters,"

prefer soft deletes over hard deletes.

---

# Summary

Soft deletes protect business history.

Audit fields protect accountability.

Together,

they allow production systems to answer not only

> "What data exists?"

but also

> "Who created it, who changed it, who deleted it, and when?"

These capabilities are essential in domains such as healthcare, finance, education, and government, where historical accuracy is often more important than simply storing the latest state.
